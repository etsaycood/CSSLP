# 4.3 Performing Secure Interface Design

## Learning Objectives

- Describe the principles of secure API and interface design
- Explain input validation, output encoding, and canonicalization
- Apply contract-based design to interfaces
- Identify common interface vulnerabilities and their mitigations

---

## Secure Interface Principles

Interfaces are **attack surfaces** — every API endpoint, web form, file upload, and inter-service communication channel is a potential entry point for attackers.

### Interface Types

| Type | Description | Security Concern |
|------|-------------|-----------------|
| **User interfaces (UI)** | Web forms, mobile screens, desktop GUIs | Input validation, CSRF, clickjacking |
| **APIs (REST/SOAP/GraphQL)** | Programmatic interfaces between systems | Authentication, authorization, injection, rate limiting |
| **File uploads** | User-provided file ingestion | Malware, path traversal, file type validation |
| **Inter-service communication** | Service-to-service calls (microservices) | Mutual TLS, service mesh authentication |
| **Database interfaces** | Application-to-database connections | SQL injection, connection security, credential management |

---

## Input Validation

Input validation is the **first line of defense** against most injection attacks. All data entering the system across a trust boundary must be validated.

### Validation Strategies

| Strategy | Description | Preference |
|----------|-------------|:----------:|
| **Allowlist (Whitelist)** | Accept only known-good input that matches defined patterns | **Preferred** |
| **Denylist (Blacklist)** | Reject known-bad input | Less reliable — attackers find new patterns |
| **Sanitization** | Clean or transform input to remove dangerous content | Use when allowlisting is not feasible |

### Validation Rules

| Rule | Description |
|------|-------------|
| **Type** | Validate the data type (integer, string, date) |
| **Length** | Enforce minimum and maximum length |
| **Range** | Ensure numeric values are within acceptable bounds |
| **Format/Pattern** | Match against expected patterns (regex, format masks) |
| **Encoding** | Validate encoding and character set |
| **Canonicalization** | Normalize input to a standard form before validation |

> **Exam Tip**: Always validate on the **server side**. Client-side validation is a convenience for users but provides **no security** — it can be bypassed.

### Canonicalization

Canonicalization converts input to its **simplest, most standard representation** before validation:

| Attack | Example | Prevention |
|--------|---------|-----------|
| **URL encoding** | `%27` for `'` | Decode before validation |
| **Unicode encoding** | `\u0027` for `'` | Normalize to standard form |
| **Double encoding** | `%2527` (double-encoded `'`) | Decode iteratively until stable |
| **Path traversal** | `../../../etc/passwd` | Canonicalize path, validate against allowed directories |

---

## Output Encoding

Output encoding prevents **injection attacks** when displaying user-supplied data:

| Context | Encoding | Purpose |
|---------|----------|---------|
| **HTML body** | HTML entity encoding (`<` → `&lt;`) | Prevent XSS |
| **HTML attributes** | Attribute encoding | Prevent attribute injection |
| **JavaScript** | JavaScript encoding | Prevent JS injection |
| **URL** | URL encoding (percent encoding) | Prevent URL manipulation |
| **CSS** | CSS encoding | Prevent CSS injection |
| **SQL** | Parameterized queries (not encoding) | Prevent SQL injection |

## Contract-Based Design (Design by Contract)

Design by Contract (DbC) defines **formal specifications for interfaces** using three components:

| Component | Description | Example |
|-----------|-------------|---------|
| **Preconditions** | What must be true before calling the interface | Input must be valid, user must be authenticated |
| **Postconditions** | What is guaranteed after the interface completes | Data is encrypted, audit log is written |
| **Invariants** | What remains true throughout the interface's lifecycle | Account balance is never negative |

### Benefits for Security

- **Clear responsibility**: Caller responsible for preconditions; callee responsible for postconditions
- **Fail-fast**: Violations detected early — before security-critical operations
- **Documentation**: Formal specification of expected and guaranteed behavior
- **Testability**: Contracts define specific conditions that can be tested

## API Security Best Practices

| Practice | Description |
|----------|-------------|
| **Authentication** | Use strong authentication for all API calls (OAuth 2.0, API keys, mutual TLS) |
| **Authorization** | Enforce authorization on every endpoint — don't rely on client-side filtering |
| **Rate limiting** | Prevent abuse and DoS by limiting requests per client/time period |
| **Input validation** | Validate all input parameters, headers, and body content |
| **Output filtering** | Return only the data the client is authorized to see |
| **HTTPS only** | Encrypt all API traffic using TLS |
| **Versioning** | Version APIs to manage breaking changes |
| **Error handling** | Return generic error messages — never expose internal details |
| **CORS** | Properly configure Cross-Origin Resource Sharing policies |
| **Content type enforcement** | Validate and enforce expected Content-Type headers |

---

## Exam Focus Points

1. **Allowlisting > Denylisting**: Allowlisting is always preferred for input validation
2. **Server-side validation**: Client-side validation can be bypassed — always validate server-side
3. **Canonicalization**: Normalize before validating to prevent encoding-based attacks
4. **Output encoding**: Context-specific (HTML, JS, URL, CSS)
5. **Design by Contract**: Preconditions, postconditions, invariants
6. **Parameterized queries**: Correct defense against SQL injection (not output encoding)
7. **API rate limiting**: Defense against DoS and abuse

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Input Validation** | Verifying user input conforms to expected format, type, and range |
| **Allowlist** | Accepting only known-good input patterns |
| **Denylist** | Rejecting known-bad input patterns |
| **Canonicalization** | Converting input to a standard representation before validation |
| **Output Encoding** | Converting data to a safe representation for the target context |
| **Design by Contract** | Formal specification using preconditions, postconditions, and invariants |
| **Precondition** | Condition that must be true before a function is called |
| **Postcondition** | Condition guaranteed after a function completes |
| **Invariant** | Condition that remains true throughout an operation |
| **XSS** | Cross-Site Scripting |
| **CSRF** | Cross-Site Request Forgery |
| **CORS** | Cross-Origin Resource Sharing |
