# 4.4 Performing Architectural Risk Assessment

## Learning Objectives

- Explain the purpose and process of architectural risk assessment
- Describe attack surface evaluation and reduction
- Apply risk ranking techniques to prioritize architectural risks
- Identify countermeasures for high-risk architectural components

---

## Architectural Risk Assessment

An architectural risk assessment evaluates the **security posture of the system design** to identify weaknesses before implementation. It builds on the threat model by assessing the likelihood and impact of identified threats in the context of the system's architecture.

---

## Attack Surface Evaluation

The attack surface is the **sum of all entry and exit points** through which an attacker could interact with the system. Reducing the attack surface is one of the most effective ways to improve security.

### Attack Surface Components

| Component | Description | Examples |
|-----------|-------------|---------|
| **Network attack surface** | Open ports, protocols, services exposed to the network | HTTP (80), HTTPS (443), SSH (22), database ports |
| **Software attack surface** | Code accessible to unauthenticated or untrusted users | Login pages, public APIs, file upload endpoints |
| **Physical attack surface** | Physical access points to hardware and infrastructure | Server rooms, USB ports, console access |
| **Human attack surface** | People who can be targeted through social engineering | Employees, helpdesk, contractors |

### Measuring Attack Surface

| Metric | Description |
|--------|-------------|
| **Entry points** | Number of ways data can enter the system |
| **Exit points** | Number of ways data can leave the system |
| **Exposed services** | Number of services accessible externally |
| **Privilege levels** | Number of different privilege levels in the system |
| **Trust boundaries crossed** | Number of trust boundary transitions |

### Attack Surface Reduction Strategies

| Strategy | Description |
|----------|-------------|
| **Reduce entry points** | Disable unnecessary services, close unused ports |
| **Reduce privilege** | Run services with minimum required permissions |
| **Reduce code** | Remove unused code, features, and debug endpoints |
| **Reduce data** | Don't store data that isn't needed |
| **Reduce trust levels** | Limit who and what can access each component |
| **Network segmentation** | Isolate components into separate network segments |
| **API gateway** | Consolidate external access through a single controlled point |

## Risk Ranking

Risk ranking assigns **relative priority** to identified architectural risks based on likelihood and impact.

### Risk Matrix

|  | **Low Impact** | **Medium Impact** | **High Impact** |
|--|:-:|:-:|:-:|
| **High Likelihood** | Medium | High | Critical |
| **Medium Likelihood** | Low | Medium | High |
| **Low Likelihood** | Low | Low | Medium |

### Factors in Risk Ranking

| Factor | Description |
|--------|-------------|
| **Threat capability** | Skill level and resources of likely threat actors |
| **Vulnerability exploitability** | How easy the vulnerability is to exploit |
| **Business impact** | Financial, reputational, legal, and operational impact |
| **Exposure** | How accessible the vulnerable component is |
| **Existing controls** | Effectiveness of current countermeasures |

---

## Countermeasure Selection

For each high-risk architectural finding, select appropriate countermeasures:

| Architectural Risk | Countermeasure |
|-------------------|----------------|
| Large attack surface | Reduce exposed services, implement API gateway |
| Trust boundary violations | Input validation, authentication at every boundary |
| Single point of failure | Redundancy, failover, clustering |
| Insufficient isolation | Containerization, network segmentation, process isolation |
| Unencrypted data flows | TLS for data in transit, encryption for data at rest |
| Weak authentication | MFA, certificate-based authentication |
| Over-privileged components | Least privilege, service accounts with minimal permissions |

---

## Exam Focus Points

1. **Attack surface**: Sum of all entry/exit points — reduce to improve security
2. **Four surfaces**: Network, software, physical, human
3. **Attack surface reduction**: Reduce entry points, privilege, code, data, trust levels
4. **Risk ranking**: Likelihood × Impact matrix
5. **Countermeasure selection**: Match controls to specific architectural risks

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Attack Surface** | Sum of all points through which an attacker could interact with the system |
| **Attack Surface Reduction** | Minimizing the number of entry points and exposed components |
| **Risk Ranking** | Prioritizing risks based on likelihood and impact |
| **Risk Matrix** | Grid mapping likelihood against impact to determine risk level |
| **Countermeasure** | Security control implemented to address an identified risk |
| **Network Segmentation** | Dividing networks into segments with controlled access between them |
| **API Gateway** | Centralized entry point for managing and securing API access |
