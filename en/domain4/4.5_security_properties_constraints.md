# 4.5 Model (Non-Functional) Security Properties and Constraints

## Learning Objectives

- Define non-functional security properties at the architecture level
- Apply confidentiality, integrity, and availability constraints to architectural decisions
- Explain authentication, authorization, and accountability architecture patterns
- Describe how non-functional security requirements become architectural constraints

---

## Non-Functional Security Properties

Non-functional security properties define **quality attributes** that the architecture must satisfy. Unlike functional requirements (what the system does), these properties constrain **how well** the system performs its security functions.

### CIA at the Architecture Level

| Property | Architectural Implications |
|----------|---------------------------|
| **Confidentiality** | Encryption services, key management infrastructure, access control enforcement points, data masking services |
| **Integrity** | Hashing services, digital signature infrastructure, validation middleware, immutable audit logs |
| **Availability** | Load balancers, failover clusters, auto-scaling, health monitoring, disaster recovery sites |

### Authentication Architecture

| Pattern | Description | Use Case |
|---------|-------------|----------|
| **Centralized authentication** | Single authentication service shared across all applications | Enterprise SSO, IAM platform |
| **Token-based** | Stateless authentication using tokens (JWT, OAuth) | Microservices, APIs, mobile apps |
| **Certificate-based** | Mutual TLS using X.509 certificates | Service-to-service, high-security environments |
| **Federated** | Authentication delegated to external identity providers | Cross-organization, B2B integration |

### Authorization Architecture

| Pattern | Description | Use Case |
|---------|-------------|----------|
| **Centralized policy engine** | Single policy decision point (PDP) evaluating all access requests | Large-scale enterprise systems |
| **Embedded authorization** | Authorization logic embedded within each service | Simple applications, monolithic apps |
| **API gateway enforcement** | Authorization enforced at the API gateway level | Microservices, API-first architectures |
| **Attribute-based (ABAC)** | Dynamic authorization using subject, object, and environmental attributes | Fine-grained, context-aware access control |

#### XACML Architecture

| Component | Role |
|-----------|------|
| **PEP** (Policy Enforcement Point) | Intercepts access requests and enforces decisions |
| **PDP** (Policy Decision Point) | Evaluates access requests against policies |
| **PAP** (Policy Administration Point) | Creates and manages authorization policies |
| **PIP** (Policy Information Point) | Provides attribute data to the PDP for decision-making |

### Accountability Architecture

| Component | Purpose |
|-----------|---------|
| **Centralized logging** | Aggregate logs from all components in a central repository |
| **SIEM integration** | Correlate events across systems for threat detection |
| **Immutable audit trails** | Write-once logs that cannot be altered (tamper-evident) |
| **Log protection** | Encrypt and protect logs from unauthorized modification |
| **Retention policies** | Define how long logs are retained per regulatory requirements |

## Security Constraints

Security constraints are **restrictions imposed on the architecture** by security requirements, regulations, or organizational policies:

| Constraint Type | Examples |
|----------------|---------|
| **Technology constraints** | Must use FIPS 140-2 validated cryptographic modules |
| **Network constraints** | Database tier must not be directly accessible from the internet |
| **Data constraints** | PII must be encrypted at rest and in transit |
| **Performance constraints** | Authentication must complete within 2 seconds |
| **Regulatory constraints** | Audit logs must be retained for 7 years (SOX) |
| **Organizational constraints** | All services must use the corporate IAM platform |

1. **CIA at architecture level**: Know how each translates to architectural components
2. **XACML components**: PEP, PDP, PAP, PIP — know each role
3. **Token-based auth**: Stateless, suitable for microservices and APIs
4. **Centralized logging**: Essential for accountability and correlation
5. **Security constraints**: Technology, network, data, performance, regulatory, organizational

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Non-Functional Requirement** | Quality attribute defining how well the system performs |
| **Security Constraint** | Restriction imposed on architecture by security requirements |
| **PEP** | Policy Enforcement Point — intercepts and enforces access decisions |
| **PDP** | Policy Decision Point — evaluates access requests |
| **PAP** | Policy Administration Point — manages authorization policies |
| **PIP** | Policy Information Point — provides attribute data for decisions |
| **XACML** | eXtensible Access Control Markup Language |
| **JWT** | JSON Web Token — compact, URL-safe token for claims |
| **Mutual TLS** | Two-way TLS authentication using client and server certificates |
| **SIEM** | Security Information and Event Management |
