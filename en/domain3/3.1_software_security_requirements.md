# 3.1 Define Software Security Requirements

## Learning Objectives

- Differentiate between functional and non-functional security requirements
- Explain the Subject-Object-Activity Matrix
- Develop use cases that incorporate security considerations
- Identify timing and race condition requirements
- Describe session management, error handling, and configuration requirements

---

## Functional Security Requirements

Functional requirements describe **what the software must do** — they define specific functions, behaviors, and capabilities. From a security perspective, functional requirements define the security behaviors the system must exhibit.

### Business Requirements to Functional Requirements

Business requirements must be **translated into functional security requirements**. Every business goal has security implications that must be explicitly defined.

| Business Requirement | Functional Security Requirement |
|---------------------|-------------------------------|
| "Customers can view their account" | Authenticate user identity before displaying account data |
| "Employees can process orders" | Authorize users based on role; log all order modifications |
| "System must be available 99.9%" | Implement redundancy, failover, and health monitoring |

### Subject-Object-Activity Matrix

The Subject-Object-Activity Matrix establishes **relationships between users and resources**:

| Concept | Description |
|---------|-------------|
| **Subject** | Users or processes involved in the system |
| **Object** | Resources that subjects interact with |
| **Activity** | Permitted or denied actions for each subject-object combination |

**Example Matrix:**

| Subject | Object | Permitted Activities | Denied Activities |
|---------|--------|---------------------|-------------------|
| Customer | Own account data | Read, Update | Delete |
| Customer | Other accounts | — | Read, Update, Delete |
| Admin | All accounts | Read, Update, Delete | — |
| API Service | Transaction log | Write | Delete |

> **Exam Tip**: The Subject-Object-Activity Matrix permits **concise communication** about allowed system interactions. It is a tool for defining authorization requirements.

### Use Cases

Use cases describe **specific examples of intended behavior** of the system.

| Characteristic | Description |
|---------------|-------------|
| **Purpose** | Define intended behavior (functional requirement) of the system |
| **Scope** | NOT intended for ALL subject-object interactions — use for complex or ambiguous situations |
| **Format** | Typically presented in **graphical format** (use-case diagrams) |
| **Benefit** | Reduces ambiguous and incompletely articulated business requirements |

**Use-case modeling** shows intended system behavior (activity) for actors (users), explicitly specifying what the system does and does not do.

### Timing and Race Conditions

Software must address timing-related security vulnerabilities:

| Vulnerability | Description |
|--------------|-------------|
| **TOC/TOU** (Time of Check/Time of Use) | An attacker modifies a resource between the time it is checked and the time it is used |
| **Race Condition** | Concurrent threads compete to alter the same object during a race window |
| **Infinite Loop** | Processing that never terminates, consuming resources |
| **Mutual Exclusion** | Mechanism to prevent simultaneous access to shared resources |

## Non-Functional Security Requirements

Non-functional requirements define **how well the system performs** rather than what it does. They specify quality attributes including security, performance, usability, and reliability.

### Core Security Requirements (Non-Functional)

| Requirement | Description |
|-------------|-------------|
| **Confidentiality** | Data encryption, access controls, steganography, masking |
| **Integrity** | Input validation, hashing, checksums, CRC, parity checking, digital signatures |
| **Availability** | BIA, MTD, RTO, redundancy, failover, DRP |
| **Authentication** | MFA, digest authentication, credential validation |
| **Authorization** | DAC, MAC, RBAC, ABAC, delegation models |
| **Accountability** | Audit trails, logging, SIEM integration |

#### Integrity Verification Methods

| Method | Strength | Use Case |
|--------|----------|----------|
| **Parity bit check** | Weak | Basic error detection in data transmission |
| **CRC** (Cyclic Redundancy Check) | Moderate | Data integrity in network protocols |
| **Checksum** | Moderate | File integrity verification |
| **Cryptographic hash** | Strong | Strongest form of data integrity assurance; also provides confidentiality if combined with encryption |

### General Requirements

#### Session Management

- Maintains **state** in stateless protocols
- Implements **complete mediation** and **psychological acceptability**
- Uses **session identifiers (IDs)** that must be unpredictable and protected
- Must ensure established sessions are not susceptible to hijacking, fixation, or replay

#### Error and Exception Management

- Errors and exceptions are **potential sources of information disclosure**
- Error messages must not reveal internal system details, stack traces, or configuration
- Exceptions must be caught and handled gracefully — never expose raw error output
- Secure failure: when errors occur, default to a **secure state**

#### Configuration Parameters

- Identify and capture configuration settings that affect security
- Ensure appropriate level of protection for configuration data
- Baseline security configurations must be defined and enforced

### Operational Requirements

| Requirement | Description |
|-------------|-------------|
| **Anti-piracy** | Code obfuscation, code signing, anti-tampering, licensing, IP protection |
| **Archiving** | Organizational retention policies — location, duration, and format |
| **Deployment environment** | Secure by design, secure by default, secure in deployment |

> **Key Principle**: The default configuration must maintain security. If a user accepts all defaults, the system should be in a **secure state**.

### International Requirements

| Consideration | Description |
|--------------|-------------|
| **Legal and technological** | Ensure compliance with local regulations |
| **Character encoding** | Support different character sets and display directions |
| **Canonicalization** | Converting data to a standard form to prevent encoding-based attacks |

### Procurement Requirements

- **Contracts and SLAs** provide legal protection for software acquisitions
- **Software escrow** ensures access to source code if the vendor becomes unavailable
- Third-party components require security assessment before integration

### Protection Needs Elicitation (PNE)

The process of **eliciting security requirements** from stakeholders:

| Technique | Description |
|-----------|-------------|
| **Surveys/Questionnaires** | Collect functional and assurance requirements |
| **Interviews** | Detailed discussions with stakeholders about security needs |
| **Workshops** | Collaborative sessions to identify and prioritize security requirements |
| **Document analysis** | Review existing policies, standards, and compliance requirements |

PNE requires **strong communication and collaboration** with stakeholders and addresses:
- Business risk
- Project risk
- Technology risk

## Exam Focus Points

1. **Functional vs. non-functional**: Functional = what the system does; Non-functional = how well it performs
2. **Subject-Object-Activity Matrix**: Defines allowed/denied actions between users and resources
3. **TOC/TOU**: Time of Check / Time of Use race condition vulnerability
4. **Session management**: Session IDs must be unpredictable and protected
5. **Error messages**: Must not reveal internal system details
6. **Secure by default**: Default configuration must be secure
7. **Canonicalization**: Prevents encoding-based bypass attacks
8. **PNE**: Protection Needs Elicitation — structured approach to gathering security requirements

|------|-----------|
| **Functional Requirement** | Defines what the software must do (specific behaviors and capabilities) |
| **Non-Functional Requirement** | Defines quality attributes (security, performance, usability) |
| **Subject** | Entity requesting access (user, process, service) |
| **Object** | Resource being accessed (file, database, API) |
| **Use Case** | Specific example of intended system behavior |
| **TOC/TOU** | Time of Check / Time of Use — race condition vulnerability |
| **Race Condition** | Concurrent threads competing to alter the same resource |
| **Mutual Exclusion** | Mechanism preventing simultaneous access to shared resources |
| **PNE** | Protection Needs Elicitation |

| **BIA** | Business Impact Analysis |
| **CRC** | Cyclic Redundancy Check |
| **CONOPS** | Concept of Operations |
