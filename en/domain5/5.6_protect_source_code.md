# 5.6 Protect Source Code

## Learning Objectives

- Define version control system (VCS) security
- Describe secure branching and merging strategies
- Explain the importance of code signing and checksums
- Outline physical and logical protections for source code repositories

---

## Version Control System (VCS) Security

The Version Control System (e.g., Git, Subversion, Perforce) is the authoritative source of truth for an application's codebase. Compromise of the VCS can lead to intellectual property theft or the insertion of malicious backdoors into the product.

### Securing the Repository

| Control | Description |
|---------|-------------|
| **Authentication** | Require MFA for all developers accessing the repository. Use SSH keys or personal access tokens (PATs) instead of basic passwords. |
| **Access Control (RBAC)** | Enforce least privilege based on roles (Reader, Contributor, Maintainer, Admin). Restrict who can push to default branches or delete repositories. |
| **Audit Logging** | Log all clones, pulls, and pushes. Monitor for anomalous behavior (e.g., a developer cloning the entire corporate codebase on a weekend). |
| **Secret Scanning** | Use automated pre-receive hooks or native tools (e.g., GitHub Advanced Security) to scan commits and block pushes that contain API keys, passwords, or tokens. |
| **Network Isolation** | For highly sensitive IP, host the VCS on-premises or in a private cloud VPC restricted by VPN access, rather than on the public internet. |

## Secure Branching and Merging Strategies

Branching models dictate how developers collaborate and integrate changes. Security must be embedded into these workflows.

### Branch Protection Rules

The main/master branch must be protected to ensure code quality and security before integration.

1. **Require Pull Requests (PRs)**: Developers cannot push directly to `main`. All changes must be proposed via a branch and a Pull Request.
2. **Require Peer Reviews**: At least one (preferably two) authorized developers must review and approve the PR before it can be merged (enforcing Separation of Duties).
3. **Require Status Checks**: Automated CI checks must pass before merging. This includes:
    - Successful build (no compilation errors).
    - Unit tests pass.
    - SAST / SCA security scans pass with no critical findings.
    - Code coverage meets minimum thresholds.
4. **Require Linear History**: Prevent complex merge commits that can obscure malicious changes; enforce rebasing or squash merging.

### Commit Signatures

How do you prove that a commit supposedly made by "Alice" was actually made by Alice, and not an attacker spoofing her email address?
- **Signed Commits**: Developers use their personal GPG/PGP keys or S/MIME certificates to cryptographically sign individual Git commits.
- **Verification**: The VCS platform (e.g., GitHub, GitLab) verifies the signature against the public key on file for that user, badging the commit as "Verified."

## Code Signing and Validation

While commit signatures prove who wrote the code, **code signing** proves the integrity and origin of the final compiled artifact (the `.exe`, `.apk`, or Docker image) delivered to the customer.

### Purpose of Code Signing
1. **Authentication (Non-Repudiation)**: Assures users that the software legitimately comes from the stated publisher (e.g., Microsoft, Apple).
2. **Integrity**: Assures users that the software has not been tampered with or infected with malware since the publisher signed it.

### How Code Signing Works
1. The publisher calculates a cryptographic hash of the final compiled executable.
2. The publisher encrypts that hash using their **Private Key** (obtained from a trusted Certificate Authority).
3. The encrypted hash (the digital signature) and the publisher's Public Key/Certificate are appended to the executable.
4. When a user downloads the software, their operating system uses the Public Key to decrypt the hash, recalculates the hash of the downloaded file, and compares the two. If they match, the software is trusted.

### Securing the Signing Process
Code signing private keys are exceptionally valuable; if stolen, an attacker can sign malware, masquerading as your company.
- Code signing must be performed by the CI/CD pipeline, not manually by developers.
- The Code Signing Private Key must be stored in an **HSM (Hardware Security Module)** or a secure cloud vault, preventing export.
- Access to trigger the signing process must require strict separation of duties and approvals.

## Anti-Tampering and Obfuscation

For applications deployed into hostile environments (e.g., downloaded to user devices, mobile apps, IoT), source code and binaries must be protected against reverse engineering.

| Technique | Description |
|-----------|-------------|
| **Obfuscation** | Transforming the source or binary code to be intentionally difficult for humans to read/understand, without altering its function (e.g., renaming variables to meaningless characters, encrypting strings). |
| **Anti-Debugging** | Code that detects if it is being run inside a debugger or emulator and alters its behavior or crashes to thwart analysis. |
| **Anti-Tampering** | Implementing checksum validations within the code at runtime to verify its own integrity. If the binary has been modified, the application refuses to run. |

> **Exam Tip**: Obfuscation is *not* a primary security control (it is security through obscurity). It only delays a determined attacker; it does not replace strong cryptography or secure design.

---

## Exam Focus Points

1. **Branch Protection**: You must enforce PRs, peer reviews, and automated SAST/SCA checks before code hits the main branch.
2. **Commit Signing vs. Code Signing**: Commit signing (GPG) proves *developer identity* in the repo. Code signing (PKI) proves *artifact integrity/origin* to the end user.
3. **Secret Scanning**: Must be implemented at the repository level to prevent developers from committing plaintext API keys.
4. **Obfuscation Limits**: Obfuscation protects intellectual property and delays reverse engineering; it does not secure data or fix vulnerabilities.
5. **Separation of Duties**: Peer reviews enforce this—the person writing the code cannot be the same person approving it for production.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **VCS** | Version Control System (e.g., Git) |
| **Branch Protection** | Configuration rules that limit who can push code and what status checks must pass |
| **Pull Request (PR)** | A request to merge code changes, facilitating peer review |
| **Code Signing** | Cryptographically signing an executable to guarantee publisher identity and file integrity |
| **Signed Commit** | Cryptographically signing a Git commit using GPG/PGP to ensure author identity |
| **Obfuscation** | Making code intentionally difficult to read to thwart reverse engineering |
| **HSM** | Hardware Security Module (used to secure code signing keys) |
