# 5.2 Analyze Code for Security Vulnerabilities

## Learning Objectives

- Differentiate between manual and automated code review techniques
- Compare SAST, DAST, IAST, and SCA analysis methods
- Identify the appropriate lifecycle phase for different analysis types
- Describe the purpose and process of fuzz testing

---

## Manual Code Analysis

Manual analysis remains a critical component of identifying logic flaws that automated tools miss.

### Peer Code Review

Code review is the practice of having one or more developers examine source code written by someone else.

| Type | Description |
|------|-------------|
| **Over-the-shoulder** | Author explains code to a reviewer looking at the same screen. Informal and quick. |
| **Pair programming** | Two developers write code together at one workstation. Continuous review. |
| **Tool-assisted (Pull Requests)** | Reviewers use tools (e.g., GitHub, GitLab) to leave comments on diffs before merging. Standard industry practice. |
| **Formal Inspection (Fagan Inspection)** | Highly structured, rigorous review with predefined roles (Moderator, Author, Reader, Tester). Rare in mostly modern Agile, used in high-assurance systems. |

**Benefits**: Excellent for finding **business logic flaws**, design issues, and assessing code maintainability.
**Drawbacks**: Time-consuming, depends heavily on the reviewer's security expertise, scales poorly.

## Automated Application Security Testing (AST)

Automated tools analyze code or running applications to find known patterns of vulnerabilities.

### SAST (Static Application Security Testing)

SAST analyzes **source code, bytecode, or binaries without executing the program** (White-box testing).

| Characteristic | Detail |
|----------------|--------|
| **When to use** | Extremely early in the SDLC (development, IDE, commit hooks, CI/CD pipeline). |
| **What it finds** | Buffer overflows, injection flaws, hardcoded credentials, insecure crypto usage. |
| **Pros** | Identifies exact file and line number; finds issues before code is compiled/deployed; highly scalable. |
| **Cons** | **High false positive rate**; cannot find runtime environment issues or complex logic flaws; language-dependent. |

### DAST (Dynamic Application Security Testing)

DAST analyzes a **running application from the outside**, interacting with it like an attacker (Black-box testing).

| Characteristic | Detail |
|----------------|--------|
| **When to use** | Later in the SDLC (testing/QA, staging, pre-production). Application must be running. |
| **What it finds** | Authentication bypasses, server misconfigurations, XSS, injection (by observing responses). |
| **Pros** | **Low false positive rate** (if it finds it, it's usually real); language-independent; finds environment/runtime issues. |
| **Cons** | Cannot pinpoint the exact line of code; requires a stable, running environment; test coverage is only as good as the crawler/scripts driving it. |

### IAST (Interactive Application Security Testing)

IAST combines SAST and DAST elements by **instrumenting the application from within** while it runs (Glass-box testing).

| Characteristic | Detail |
|----------------|--------|
| **When to use** | During testing/QA (while automated functional tests are running). |
| **What it does** | An agent running inside the app monitors memory, traffic, and execution flow while tests (or DAST) interact with the app. |
| **Pros** | High accuracy (low false positives); identifies exact line of code; identifies runtime flaws. |
| **Cons** | Requires specific language/framework support; can impact application performance; depends on good functional test coverage to execute code paths. |

### RASP (Runtime Application Self-Protection)

Like IAST, RASP runs inside the application. However, RASP focuses on **protection rather than testing**, blocking attacks in real-time in production. (Covered more in Domain 7, but related to AST concepts).

### SCA (Software Composition Analysis)

SCA tools analyze the source code and build artifacts to **identify third-party and open-source components** and check them against vulnerability databases (e.g., NVD).

| Characteristic | Detail |
|----------------|--------|
| **When to use** | Every build/commit phase. |
| **What it does** | Generates an SBOM; flags known vulnerable components; identifies license compliance issues. |
| **Pros** | Solves the "Vulnerable Components" OWASP Top 10 issue; very fast. |
| **Cons** | Only finds *known* vulnerabilities (CVEs) in *third-party* code; does not analyze custom code. |

## AST Comparison Summary

| Tool | Analysis Type | State of App | Finds Line of Code? | False Positives |
|------|---------------|---------------|---------------------|-----------------|
| **SAST** | Static (White-box) | Not running | Yes | High |
| **DAST** | Dynamic (Black-box) | Running | No | Low |
| **IAST** | Interactive (Glass-box) | Running | Yes | Very Low |
| **SCA** | Composition | Not running | Yes (dependency list) | Low |

## Fuzzing (Fuzz Testing)

Fuzzing involves sending **invalid, unexpected, or random data** as input to a computer program to monitor for exceptions, crashes, or memory leaks.

### Types of Fuzzing

| Type | Description |
|------|-------------|
| **Generation-based (Smart Fuzzing)** | Generates inputs based on an understanding of the expected data format (e.g., generating malformed HTTP requests based on the RFC). |
| **Mutation-based (Dumb Fuzzing)** | Takes existing valid inputs and randomly mutates them (flipping bits, appending data) without understanding the protocol. |

**Objectives of Fuzzing**:
- Identify buffer overflows.
- Uncover unhandled exceptions and crash scenarios.
- Discover memory leaks.

Fuzzing is highly effective at finding zero-day vulnerabilities in parsing logic and network protocols, especially in C/C++ applications.

---

## Exam Focus Points

1. **SAST vs. DAST distinction**: SAST = early, source code, high false positives. DAST = later, running app, low false positives.
2. **IAST**: Instruments the app from within, combines benefits of SAST and DAST.
3. **SCA**: Used specifically to find vulnerabilities in open-source/third-party dependencies.
4. **Fuzzing targets**: Primarily looking for application crashes, unhandled exceptions, and memory leaks (buffer overflows).
5. **Logic flaws**: Automated tools (SAST/DAST) are generally poor at finding business logic flaws; **manual peer review** or threat modeling is necessary.
