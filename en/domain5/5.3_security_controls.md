# 5.3 Implement Security Controls

## Learning Objectives

- Select and implement appropriate cryptographic algorithms and protocols
- Implement secure authentication and identity management
- Manage cryptographic keys and secrets effectively
- Define secure error handling, logging, and monitoring practices

---

## Implementing Cryptography

Cryptography is notoriously difficult to implement correctly. The cardinal rule of cryptography is: **Never roll your own crypto (Do not write your own algorithms).**

### Algorithm Selection Guidelines

| Use Case | Recommended Algorithms/Standards | Legacy/Avoid |
|----------|----------------------------------|--------------|
| **Symmetric Encryption** (Data at Rest) | **AES-GCM** (AES-256), ChaCha20 | DES, 3DES, RC4 |
| **Asymmetric Encryption** (Key Exchange, Signatures) | **RSA** (2048+ bit), **ECC** (Elliptic Curve), Diffie-Hellman | RSA < 2048 bit |
| **Hashing** (Integrity Check) | **SHA-2** (SHA-256, SHA-512), SHA-3 | MD5, SHA-1 |
| **Password Hashing** (Storage) | **Argon2**, **Bcrypt**, PBKDF2 | Plain MD5, Plain SHA-2, cleartext |
| **Data in Transit** | **TLS 1.3**, TLS 1.2 | SSL (all versions), TLS 1.0, 1.1 |

### Key Concepts in Cryptographic Implementation

- **Salting (Passwords)**: A unique, random string added to the password *before* hashing. It prevents Rainbow Table attacks and ensures users with the same password have different hashes.
- **Key Stretching / Work Factor**: Algorithms like Bcrypt and PBKDF2 are designed to be intentionally slow/resource-intensive, making brute-force attacks computationally impractical.
- **Entropy (Randomness)**: High-quality randomness is required for generating keys, nonces, and session IDs. Always use a Cryptographically Secure Pseudo-Random Number Generator (**CSPRNG**; e.g., `/dev/urandom`), not standard language random functions (e.g., standard `Math.random()`).
- **Initialization Vectors (IVs) / Nonces**: Must be unique and unpredictable for every encryption operation to ensure identical plaintexts encrypt to different ciphertexts.

## Secrets and Key Management

Hardcoded secrets (API keys, database passwords, private keys) are a major source of breaches.

### Key Management Lifecycle

1. **Generation**: Use strong CSPRNGs.
2. **Distribution**: Securely transmit keys out of band.
3. **Storage**: Use dedicated **Secrets Management Vaults** (e.g., HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, HSMs). **Never store secrets in source code.**
4. **Rotation**: Change keys periodically, especially after personnel changes or suspected compromise.
5. **Revocation/Destruction**: Cryptographically erase keys when no longer needed.

## Implementing Identity and Access Management

### Authentication Implementation

- **Multi-Factor Authentication (MFA)**: Require two or more factors (Knowledge, Possession, Inherence). Implement for all external access and administrative functions.
- **Federation / SSO**: Use standard protocols rather than building custom authentication:
    - **SAML**: XML-based, heavily used in enterprise/B2B SSO.
    - **OAuth 2.0**: Authorization framework for delegating access (Token-based).
    - **OpenID Connect (OIDC)**: Authentication layer built on top of OAuth 2.0.

### Token Management (JWT)

JSON Web Tokens (JWTs) are commonly used for stateless API authentication.
- **Do**: Sign the token (JWS) to ensure integrity using strong algorithms (RS256).
- **Don't**: Rely on the "none" algorithm.
- **Don't**: Put sensitive data in the payload (unless encrypted via JWE).
- **Do**: Keep the token expiration (TTL) very short, and use refresh tokens to obtain new access tokens.

## Secure Exception Handling

When implementing error handling, balance the needs of debugging with the risk of information disclosure.

| Practice | Explanation |
|----------|-------------|
| **Catch Specific Exceptions** | Do not catch the top-level generic `Exception` class. Catch specific errors (e.g., `SQLException`) to handle them appropriately. |
| **Do Not Expose Internal Details** | Never return stack traces, database query strings, or server paths to the end user. Return generic, polite error messages (e.g., "An unexpected error occurred. Please try again later.") |
| **Log the Details Securely** | While the user sees a generic message, log the full technical details (stack trace) to a secure, centralized server for developer debugging. |

## Security Logging and Audit Trails

Logging is essential for non-repudiation, incident response, and compliance.

### What to Log

- Successful and failed authentication events.
- Authorization failures / access control denied events.
- Account lockouts and password changes.
- High-value transactions (e.g., fund transfers, PHI access).
- Application configuration changes.

### What NOT to Log (Sensitive Data)

- Cleartext passwords or session tokens.
- Complete credit card numbers (PAN) or CVV codes.
- Social Security Numbers or unmasked PII/PHI.
- **Log Forging Prevention**: Always sanitize and validate any user input before writing it to a log file to prevent CRLF injection or log poisoning attacks.

### Log Protection

- Logs must be protected against tampering (Integrity checks, Write-Once-Read-Many (WORM) storage).
- Logs should be centralized (sent off the local server to a SIEM) to prevent an attacker from deleting covering their tracks.

---

## Exam Focus Points

1. **Never roll your own crypto**: Always use proven, standard libraries and algorithms.
2. **Password storage**: Always Salt + Hash using a key-stretching algorithm (Argon2, Bcrypt, PBKDF2). MD5/SHA is insufficient.
3. **CSPRNG**: Ordinary `random()` functions are predictable; use cryptographically secure generators for security functions.
4. **Error handling**: Generic errors to the user, detailed verbose errors to the secure log.
5. **Secrets Vaults**: The correct answer for storing API keys or DB passwords is a vault/HSM, never source control or local config files.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **CSPRNG** | Cryptographically Secure Pseudo-Random Number Generator |
| **Salting** | Adding random data to a password before hashing |
| **Key Stretching** | Making a hashing function intentionally slow (e.g., Bcrypt) |
| **JWT** | JSON Web Token (commonly used in modern stateless APIs) |
| **SAML/OIDC** | Standard protocols for federation and Single Sign-On |
| **WORM** | Write-Once-Read-Many (a requirement for secure audit logs) |
