## The OWASP Top 10

The OWASP Top 10 is the universally recognized standard for the most critical web application security risks.

### 1. Broken Access Control
Failures in enforcing what authenticated users are allowed to do.
- **Examples**: Insecure Direct Object Reference (IDOR), privilege escalation, bypassing access control checks, metadata manipulation (e.g., JWT tampering).
- **Mitigation**: Default deny, implement access control centrally (middleware/filters), enforce record ownership, log access control failures.

### 2. Cryptographic Failures (formerly Sensitive Data Exposure)
Failures related to cryptography leading to sensitive data exposure.
- **Examples**: Transmitting data in cleartext, using weak or deprecated algorithms (MD5, SHA1), hardcoding cryptographic keys, insufficient entropy.
- **Mitigation**: Encrypt all sensitive data at rest and in transit, use strong standard algorithms (AES, RSA, TLS 1.3), manage keys securely (vaults).

### 3. Injection
Untrusted data is sent to an interpreter as part of a command or query.
- **Examples**: SQL injection, OS command injection, LDAP injection, Cross-Site Scripting (XSS).
- **Mitigation**: Use **parameterized queries** (prepared statements), separate data from commands, implement strict server-side **allowlist input validation**, use appropriate output encoding.

### 4. Insecure Design
Flaws related to missing or ineffective control design. (Focuses on pre-coding phase).
- **Examples**: Missing rate limiting resulting in credential stuffing, lack of business logic validation.
- **Mitigation**: Implement threat modeling during design, use secure design patterns, establish reference architectures.

### 5. Security Misconfiguration
Insecure default settings, open cloud storage, verbose error messages.
- **Examples**: Default passwords left unchanged, unpatched flaws, unused unneeded features enabled, error messages revealing stack traces.
- **Mitigation**: Automate configuration (Infrastructure as Code), remove unused features/services, segment architectures, implement a repeatable hardening process.

### 6. Vulnerable and Outdated Components
Using components with known vulnerabilities.
- **Examples**: Using an old version of Log4j (Log4Shell), unpatched OS, relying on abandoned libraries.
- **Mitigation**: Maintain an SBOM, use Software Composition Analysis (SCA) tools, implement a continuous patching process, remove unused dependencies.

### 7. Identification and Authentication Failures
Incorrectly implementing authentication and session management.
- **Examples**: Credential stuffing, brute force attacks permitted, weak password policies, exposing session IDs in URLs.
- **Mitigation**: Implement MFA, enforce strong password policies (NIST SP 800-63b), implement account lockout/rate limiting, use secure, built-in session managers.

### 8. Software and Data Integrity Failures
Code and infrastructure that does not protect against integrity violations.
- **Examples**: Deserialization of untrusted data, depending on plugins/libraries from untrusted sources, unverified CI/CD pipelines.
- **Mitigation**: Use digital signatures, verify artifact integrity, do not deserialize untrusted data (or use safe serialization formats like JSON), secure CI/CD pipelines.

### 9. Security Logging and Monitoring Failures
Inability to detect, escalate, and respond to active breaches.
- **Examples**: Not logging critical events (logins, failed access, valuable transactions), logs stored locally and lost during breach, no alerting configured.
- **Mitigation**: Ensure all login/access failures and high-value transactions are logged, use centralized log management (SIEM), establish alerts for suspicious activity.

### 10. Server-Side Request Forgery (SSRF)
A web application fetches a remote resource without validating the user-supplied URL.
- **Examples**: An attacker forces the server to connect to internal-only services or read internal files (e.g., AWS metadata endpoint).
- **Mitigation**: Use allowlists for domain/URL destinations, disable HTTP redirections, segment network access, do not send raw responses to clients.

## Common Weakness Enumeration (CWE)

While OWASP tracks the Top 10 risks, the **CWE list** is a community-developed list of common software security weaknesses (the underlying technical flaws).
- Examples: CWE-89 (Improper Neutralization of Special Elements used in an SQL Command), CWE-79 (Cross-site Scripting).
- **CWE/SANS Top 25**: A list of the most widespread and critical errors that can lead to serious vulnerabilities.

## Secure Coding Concepts

### Declarative vs. Imperative Security

| Type | Description | Example |
|------|-------------|---------|
| **Declarative Security** | Security rules defined outside the code (e.g., config files, annotations). Enforced by the framework/container. | `<security-constraint>` in web.xml, `@RolesAllowed("ADMIN")` annotations. |
| **Imperative Security** | Security logic written directly into the application code. | `if (user.getRole().equals("ADMIN")) { doSomething(); }` |

**Best Practice**: Use declarative security wherever possible. It is easier to audit, maintain, and less prone to developer error. Use imperative security only for fine-grained, business-logic-specific rules (e.g., "User can only edit their *own* documents").

### State Management

HTTP is intrinsically stateless. Applications must implement state management securely.

- **Cookies**: Small pieces of data sent by the server and stored on the client.
    - **Secure Flag**: Ensures the cookie is only transmitted over HTTPS.
    - **HttpOnly Flag**: Prevents client-side scripts (JavaScript) from accessing the cookie, mitigating XSS risks.
    - **SameSite Flag**: Controls whether cookies are sent with cross-site requests, mitigating CSRF risks.
- **Tokens (e.g., JWT)**: Must be cryptographically signed to prevent tampering. Sensitive data should not be stored in the payload unless encrypted, as the payload is base64 encoded, not encrypted.
- **Session IDs**: Must be randomly generated with high entropy, strictly validated, and regenerated or invalidated upon privilege changes (e.g., login/logout) to prevent **Session Fixation**.

### Memory Management

Improper memory management leads to vulnerabilities like buffer overflows (CWE-119).

| Issue | Description | Prevention |
|-------|-------------|------------|
| **Buffer Overflow** | Writing data beyond the bounds of an allocated buffer, potentially overwriting adjacent memory. | Use safe string functions (e.g., `strncpy` instead of `strcpy`), use memory-safe languages (Java, C#, Rust, Go). |
| **Memory Leak** | Failing to release allocated memory when no longer needed, leading to resource exhaustion (DoS). | Use garbage-collected languages; implement rigorous memory tracking in C/C++. |
| **Use After Free** | Accessing memory after it has been freed, potentially executing attacker-controlled data. | Set pointers to NULL after freeing them. |

### Concurrency and Threading

Concurrency issues occur when multiple threads access shared resources simultaneously without proper synchronization.

- **Race Conditions (TOC/TOU)**: Time of Check / Time of Use. The state of a resource changes between the time it is checked and the time it is used.
    - **Prevention**: Use thread-safe libraries, implement mutual exclusion (mutexes, locks), atomic operations.
- **Deadlocks**: Two or more threads are blocked forever, waiting for each other.
    - **Prevention**: Consistent lock acquisition ordering, timeouts.

## Exam Focus Points

1. **OWASP Top 10**: Memorize the latest Top 10 and their primary mitigation strategies (especially Injection -> Parameterized Queries).
2. **Cookie Flags**: Secure = HTTPS only, HttpOnly = No JS access (XSS mitigation), SameSite = CSRF mitigation.
3. **Declarative vs. Imperative**: Declarative (config/annotations) is preferred over imperative (code).
4. **Memory Management**: Buffer overflows are primarily an issue in non-memory-safe languages (C/C++).
5. **SSRF**: Server-Side Request Forgery targets internal resources behind firewalls.
