# 2.5 Define Security Metrics

## Learning Objectives

- Identify meaningful security metrics for software development
- Differentiate between Key Performance Indicators (KPIs) and Objectives and Key Results (OKRs)
- Explain criticality levels, remediation time, and complexity metrics
- Design a metrics program that drives security improvement

---

## Why Security Metrics Matter

Security metrics translate subjective assessments into **objective, quantifiable data**. Without metrics, organizations cannot answer fundamental questions: Are we getting more secure? Are our investments in security paying off? Where should we focus next?

Effective security metrics must be:

| Property | Description |
|----------|-------------|
| **Measurable** | Quantifiable through automated or manual data collection |
| **Actionable** | Drive decisions and behavior changes |
| **Relevant** | Aligned with business objectives and security goals |
| **Timely** | Available when decisions need to be made |
| **Repeatable** | Consistent methodology produces comparable results over time |

---

## Categories of Security Metrics

### Process Metrics

Measure the performance and maturity of security processes:

| Metric | Description | Example |
|--------|-------------|---------|
| **Percentage of projects with threat models** | How consistently threat modeling is applied | 85% of projects have completed threat models |
| **Security training completion rate** | Percentage of developers who completed security training | 92% completion rate this quarter |
| **Security gate pass rate** | How often projects pass security gates on the first attempt | 70% first-pass rate at design gate |
| **Mean time to remediate (MTTR)** | Average time from vulnerability discovery to fix | MTTR for critical: 3 days; high: 14 days |

### Product Metrics

Measure the security posture of the software itself:

| Metric | Description | Example |
|--------|-------------|---------|
| **Vulnerability density** | Number of vulnerabilities per thousand lines of code (KLOC) | 2.3 vulnerabilities per KLOC |
| **Open vulnerability count by severity** | Current backlog of unresolved vulnerabilities | 0 critical, 3 high, 12 medium |
| **Defect removal efficiency (DRE)** | Percentage of defects found before production | 95% DRE (found in dev/test, not prod) |
| **Code coverage** | Percentage of code exercised by security tests | 78% security test coverage |
| **Third-party component risk** | Number of dependencies with known CVEs | 4 dependencies with known high CVEs |

### Operational Metrics

Measure security effectiveness in production:

| Metric | Description | Example |
|--------|-------------|---------|
| **Security incidents per release** | Number of security incidents attributed to a release | 0.5 incidents per release |
| **Mean time to detect (MTTD)** | Average time to detect a security incident | 4 hours MTTD |
| **Patch deployment time** | Time from patch availability to deployment | 95% of critical patches deployed within 48 hours |

## Key Metric Definitions

### Criticality Level

Criticality level classifies the **severity and business impact** of a vulnerability or defect. Common classification systems include:

| Level | CVSS Range | Description | Response Expectation |
|-------|-----------|-------------|---------------------|
| **Critical** | 9.0–10.0 | Exploitable remotely, no authentication required, complete system compromise | Immediate remediation (hours) |
| **High** | 7.0–8.9 | Significant impact, likely exploitable | Remediate within days |
| **Medium** | 4.0–6.9 | Moderate impact, exploitable under certain conditions | Remediate within sprint/release cycle |
| **Low** | 0.1–3.9 | Minor impact, difficult to exploit | Remediate when practical |

### Complexity Metrics

Complexity correlates with security risk — more complex code tends to harbor more vulnerabilities:

| Metric | Description |
|--------|-------------|
| **Cyclomatic complexity** | Number of linearly independent paths through code; higher values = more paths to test and more potential for bugs |
| **Coupling** | Degree of interdependence between modules; high coupling increases the blast radius of vulnerabilities |
| **Code churn** | Frequency of code changes; high churn in security-critical areas warrants additional review |
| **Attack surface size** | Number of entry points, interfaces, and exposed services |

## KPIs and OKRs

### Key Performance Indicators (KPIs)

KPIs are **quantitative measurements** that track progress toward specific security objectives:

| KPI | Target | Frequency |
|-----|--------|-----------|
| Vulnerability density | < 1.0 per KLOC | Per release |
| Critical vulnerability remediation time | < 24 hours | Continuous |
| Security gate first-pass rate | > 80% | Per gate review |
| Security training completion | 100% | Quarterly |
| Third-party components with known CVEs | 0 critical | Per release |

### Objectives and Key Results (OKRs)

OKRs provide a **goal-setting framework** that connects high-level objectives to measurable results:

**Example OKR:**

| Element | Content |
|---------|---------|
| **Objective** | Reduce the security risk of our application portfolio |
| **Key Result 1** | Reduce average MTTR for critical vulnerabilities from 5 days to 1 day |
| **Key Result 2** | Achieve 100% SAST coverage in CI/CD pipeline across all repositories |
| **Key Result 3** | Reduce vulnerability density from 3.0 to 1.0 per KLOC |

### KPIs vs. OKRs

| Aspect | KPIs | OKRs |
|--------|------|------|
| **Nature** | Ongoing performance measurement | Time-bound goal setting |
| **Focus** | Maintaining standards | Driving improvement |
| **Duration** | Continuous / recurring | Quarterly or annually |
| **Scope** | Specific operational metrics | Strategic objectives with measurable results |

## Building a Metrics Program

### Maturity Progression

| Level | Description | Example Metrics |
|-------|-------------|-----------------|
| **Level 1: Ad hoc** | No consistent metrics collection | None — decisions based on intuition |
| **Level 2: Basic** | Collecting basic vulnerability counts | Total open vulnerabilities by severity |
| **Level 3: Managed** | Metrics tracked over time with trends | MTTR trends, vulnerability density trends |
| **Level 4: Optimized** | Metrics drive decisions and resource allocation | ROI of security investments, risk-adjusted metrics |

### Common Pitfalls

| Pitfall | Impact | Mitigation |
|---------|--------|-----------|
| **Vanity metrics** | Track numbers that look good but drive no action | Focus on metrics that trigger decisions |
| **Gaming** | Teams optimize for the metric rather than security | Use balanced scorecards with multiple metrics |
| **Too many metrics** | Information overload, analysis paralysis | Start with 5–7 core metrics |
| **No baseline** | Cannot measure improvement without a starting point | Establish baselines before setting targets |

---

## Exam Focus Points

1. **MTTR by severity**: Know typical remediation targets (critical < 24h, high < 7d)
2. **Vulnerability density**: Vulnerabilities per KLOC — key product quality metric
3. **KPIs vs. OKRs**: KPIs = ongoing monitoring; OKRs = time-bound improvement goals
4. **Criticality levels**: CVSS-based classification (Critical 9.0+, High 7.0+, Medium 4.0+, Low 0.1+)
5. **Cyclomatic complexity**: Higher complexity = higher security risk
6. **Defect Removal Efficiency**: Percentage of defects found before production
7. **Metrics must be actionable**: Drive decisions, not just reports

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **KPI** | Key Performance Indicator — quantitative measure of security performance |
| **OKR** | Objectives and Key Results — goal-setting framework linking objectives to measurable results |
| **MTTR** | Mean Time to Remediate — average time from vulnerability discovery to verified fix |
| **MTTD** | Mean Time to Detect — average time to identify a security incident |
| **Vulnerability Density** | Number of vulnerabilities per thousand lines of code |
| **DRE** | Defect Removal Efficiency — percentage of defects found before production |
| **Cyclomatic Complexity** | Measure of the number of independent paths through code |
| **CVSS** | Common Vulnerability Scoring System |
| **Code Churn** | Frequency and volume of code changes over time |
| **Attack Surface** | Total number of entry points and interfaces exposed to potential attackers |
| **Bug Bar** | Minimum acceptable quality threshold for security defects |
