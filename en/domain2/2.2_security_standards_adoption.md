# 2.2 Identify and Adopt Security Standards

## Learning Objectives

- Identify the major security frameworks and standards relevant to secure software development
- Differentiate between prescriptive frameworks and maturity models
- Explain how to promote security awareness across the organization
- Describe the role of standards in establishing measurable, repeatable security practices

---

## The Role of Security Standards

Security standards provide **defined levels of activity** that can be measured and monitored for compliance. They establish a common language and set of expectations across the organization and industry. Adopting recognized security standards offers several benefits:

- **Consistency**: Repeatable processes across teams and projects
- **Measurability**: Defined criteria against which compliance can be assessed
- **Credibility**: Third-party recognition and customer confidence
- **Legal protection**: Demonstration of due diligence and due care
- **Efficiency**: Leverage proven practices rather than inventing from scratch

## Security Frameworks and Standards

### ISO Standards

| Standard | Full Name | Focus |
|----------|-----------|-------|
| **ISO 27001** | Information Security Management Systems | Requirements for establishing, implementing, maintaining, and improving an ISMS |
| **ISO 27002** | Code of Practice for Information Security Controls | Guidance on selecting and implementing security controls |
| **ISO/IEC 15408** | Common Criteria (CC) | Evaluation framework for IT product security |
| **ISO/IEC 9126** | Software Product Quality | Six quality characteristics: Functionality, Reliability, Usability, Efficiency, Maintainability, Portability |
| **ISO/IEC 12207** | Software Life Cycle Processes | Defines processes covering the entire software lifecycle |
| **ISO/IEC 33001:2015** | Process Assessment | Process capability determination (Levels 0–5) |

#### Common Criteria (ISO/IEC 15408)

The Common Criteria framework allows vendors to make security claims about products and have them independently evaluated:

| Concept | Description |
|---------|-------------|
| **TOE** (Target of Evaluation) | The product or system being evaluated |
| **ST** (Security Target) | Security properties associated with a specific TOE |
| **PP** (Protection Profile) | Security requirements for a class of products |
| **EAL 1–7** | Evaluation Assurance Levels (1 = lowest, 7 = highest) |

> **Exam Tip**: EAL levels indicate the **depth of evaluation**, not the strength of security. EAL 4 is the highest level at which it is economically feasible to retrofit into existing products.

### NIST Frameworks

| Framework | Focus |
|-----------|-------|
| **NIST SP 800 Series** | Comprehensive security guidelines (cryptographic protocols, risk management, system security) |
| **NIST RMF** (Risk Management Framework) | Six-step process: Categorize → Select → Implement → Assess → Authorize → Monitor |
| **NIST SSDF** (SP 800-218) | Secure Software Development Framework (PO, PS, PW, RV) |
| **FIPS** (Federal Info Processing Standards) | Mandatory requirements for federal agencies and specific contractors |
| **FedRAMP** | Standardized security assessment for cloud products used by federal agencies |
| **SCAP** | Security Content Automation Protocol — standardized approach for security measurement |

### OWASP (Open Web Application Security Project)

OWASP is an open community producing freely available security resources:

| Resource | Description |
|----------|-------------|
| **OWASP Top 10** | Most critical web application security risks |
| **OWASP ASVS** | Application Security Verification Standard — three verification levels |
| **OWASP SAMM** | Software Assurance Maturity Model — measures and improves secure development practices |
| **OWASP Testing Guide** | Comprehensive web application security testing methodology |
| **OWASP Secure Coding Practices** | Quick reference guide for secure coding |

### Software Security Maturity Models

Maturity models help organizations **assess and improve** their software security posture over time. Unlike prescriptive frameworks that tell you *what* to do, maturity models help you understand *where you are* and *where you should go*.

#### BSIMM (Building Security In Maturity Model)

BSIMM is a **descriptive model** based on observing actual software security initiatives across many organizations. It measures what you are doing rather than prescribing what you should do.

| Characteristic | Description |
|---------------|-------------|
| **Approach** | Data-driven, observational — based on real practices |
| **Domains** | Governance, Intelligence, SSDL Touchpoints, Deployment |
| **Use** | Benchmark your SSI against industry peers |
| **Activities** | ~120 activities organized across 12 practices |

#### SAMM (Software Assurance Maturity Model)

SAMM is a **prescriptive model** maintained by OWASP that provides a structured approach to improving software security.

| Characteristic | Description |
|---------------|-------------|
| **Approach** | Prescriptive — tells you what practices to adopt |
| **Business Functions** | Governance, Design, Implementation, Verification, Operations |
| **Maturity Levels** | Four levels (0–3) per practice within each business function |
| **Use** | Create a roadmap for improving your security posture |

#### BSIMM vs. SAMM

| Aspect | BSIMM | SAMM |
|--------|-------|------|
| **Type** | Descriptive (what others do) | Prescriptive (what you should do) |
| **Basis** | Empirical data from organizations | Expert-defined best practices |
| **Primary use** | Benchmarking | Roadmap creation |
| **Maintained by** | Synopsys (commercial) | OWASP (community) |

> **Exam Tip**: BSIMM **describes** current industry practices; SAMM **prescribes** what you should do. Both are valuable — BSIMM for benchmarking, SAMM for planning improvement.

### SAFECode (Software Assurance Forum for Excellence in Code)

SAFECode is an **industry-backed organization** committed to increasing communication on software assurance:
- Publishes free best-practice guidance
- Applicable across organizations of **all sizes** (not just large enterprises)
- Focuses on practical, implementable secure development practices
- Includes guidance on development and distribution use cases

## Promoting Security Awareness

Security standards adoption is only effective if the people involved understand and embrace them. Security awareness programs ensure that all stakeholders — from developers to executives — understand their responsibilities.

### Components of Security Awareness

| Component | Target Audience | Content |
|-----------|----------------|---------|
| **Executive briefings** | C-suite, board | Business risk, regulatory requirements, investment justification |
| **Developer training** | Engineering teams | Secure coding practices, common vulnerabilities, tool usage |
| **Security champion programs** | Selected team members | Advanced security training, mentoring, internal advocacy |
| **Onboarding training** | New hires | Security policies, acceptable use, incident reporting |
| **Ongoing awareness** | All staff | Phishing awareness, social engineering, password hygiene |
| **Role-based training** | Specialized roles | DBA security, admin security, architect security review |

### Security Culture

Building a security culture requires:
- **Tone from the top**: Executive sponsorship and visible commitment
- **Integration into performance**: Security goals in job descriptions and performance reviews
- **Positive reinforcement**: Recognize and reward security-conscious behavior
- **Blameless post-mortems**: Encourage incident reporting without fear of punishment
- **Continuous learning**: Regular training updates to address evolving threats

## Exam Focus Points

1. **ISO 15408 Common Criteria**: Know TOE, ST, PP, and EAL levels (1–7)
2. **BSIMM vs. SAMM**: Descriptive (benchmarking) vs. Prescriptive (roadmap)
3. **NIST RMF six steps**: Categorize → Select → Implement → Assess → Authorize → Monitor
4. **FedRAMP**: Standardized cloud security assessment for federal government use
5. **OWASP resources**: Top 10 (risks), ASVS (verification), SAMM (maturity)
6. **SAFECode**: Industry-backed, applicable to all organization sizes
7. **Security awareness**: Must include executive buy-in, developer training, and cultural integration
8. **ISO 9126**: Six software quality characteristics (Functionality, Reliability, Usability, Efficiency, Maintainability, Portability)
