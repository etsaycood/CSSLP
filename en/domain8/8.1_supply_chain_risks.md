# 8.1 Software Supply Chain Risks

## Learning Objectives

- Define the software supply chain
- Identify common supply chain attack vectors
- Understand the impact of relying on third-party code
- Explain the risks of outsourced development environments

---

## What is the Software Supply Chain?

Decades ago, organizations wrote 100% of their software in-house. Today, a typical enterprise application consists of 80% to 90% **third-party open-source components, commercial SDKs, and external APIs**. The custom code written by the organization often merely glues these external pieces together.

The **Software Supply Chain** encompasses everything that touches your software before it reaches the end user:
- The open-source libraries (e.g., jQuery, React, Log4j).
- The developers (in-house, offshore contractors).
- The tools used to build it (IDEs, compilers, CI/CD pipelines).
- The infrastructure it runs on (AWS, Azure, Docker Hub).

## The Attack Vector

Why do attackers target the supply chain? **Scale and Efficiency.**

Instead of trying to hack 10,000 individual companies, an attacker hacks **one supplier** (e.g., a popular open-source library maintainer, or a company providing network monitoring software like SolarWinds). By inserting a subtle backdoor into that one supplier's update file, the attacker instantly gains access to all 10,000 companies who download that update.

---

## Common Supply Chain Risks

### 1. Compromised Third-Party Libraries
Attackers gain access to the credentials of an open-source maintainer (e.g., on npm, PyPI, or RubyGems) and publish a malicious update to a widely used package. When developers automatically pull the latest version, they import the malware directly into their application.

### 2. Dependency Confusion (Namesquatting)
Many organizations use a mix of private internal packages and public open-source packages. If an attacker discovers the name of an internal package (e.g., `company-auth-lib`), they can publish a malicious package with the exact same name to the public repository (npm) but give it a significantly higher version number (e.g., `v99.0`). The developer's package manager gets "confused" and downloads the higher-versioned public malware instead of the safe internal code.

### 3. Typo-Squatting
An attacker publishes a malicious package with a name almost identical to a highly popular package, hoping developers will make a typo when running the install command.
- **Example**: Creating a package named `requsts` (malicious) instead of `requests` (legitimate Python HTTP library).

### 4. Compromised Build Environments (CI/CD)
The pipeline itself is a massive risk. If an attacker gains access to Jenkins or GitHub Actions, they can inject malicious code into the final binary *after* the developer peer reviews are complete, right before it is digitally signed. The resulting file looks perfectly legitimate and is signed by the company's valid certificate, but contains a backdoor.

### 5. Outsourced Offshore Development
Hiring external contractors introduces risk regarding the security of their development environments (laptops, networks). If a contractor's laptop is infected with malware, that malware could easily migrate into your corporate source code repository when they push a commit.

---

## Mitigating the Risk (High Level)

- **Zero Trust**: Treat third-party code with the same suspicion as user input. Do not blindly trust an update just because it comes from a known vendor.
- **Pinning Dependencies**: Never configure package managers to automatically pull the `latest` or `*` version of a library. Pin them to specific, verified version numbers (e.g., `1.4.2`).
- **Internal Repositories**: Do not allow developers to download code directly from the public internet (npm/Maven). Force them to download packages through an internal proxy repository (like JFrog Artifactory or Sonatype Nexus) where the security team can scan them first.

---

## Exam Focus Points

1. **Efficiency of Attack**: Supply chain attacks are devastating because compromising one target yields access to thousands of downstream victims (e.g., the SolarWinds breach).
2. **Dependency Confusion**: An attack exploiting the package manager's preference for higher version numbers, tricking it into downloading public malware instead of private code.
3. **Pinning**: The practice of locking down library versions to prevent unexpected, malicious auto-updates.
4. **Third-Party Risk**: Even if your custom code is flawless, an insecure imported library compromises your entire application.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Software Supply Chain** | The interconnected network of vendors, tools, and code used to build an application |
| **Dependency Confusion** | Forcing a package manager to download public malware over a private internal library |
| **Typo-Squatting** | Registering maliciously named packages that rely on common developer misspellings |
| **Version Pinning** | Specifying the exact version of a dependency to prevent unauthorized updates |
