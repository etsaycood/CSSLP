# 8.3 Verify Provenance and Integrity

## Learning Objectives

- Define Provenance and Integrity in the context of the supply chain
- Explain the role of Code Signing and Digital Certificates
- Describe how to use Cryptographic Hashes to verify integrity
- Understand the SLSA (Supply-chain Levels for Software Artifacts) framework

---

## Provenance vs. Integrity

When a developer clicks "Download" on a new open-source library, two critical questions must be answered before that code is allowed into the corporate network.

| Concept | The Question | The Tool | Security Value |
|---------|--------------|----------|----------------|
| **Provenance** | "Where did this code actually come from? Who built it?" | Code Signing (Digital Signatures/Certificates) | Prevents Impersonation (e.g., stopping an attacker from pretending to be Microsoft). |
| **Integrity** | "Has this code been tampered with since the author published it?" | Cryptographic Hashing | Prevents Modification (e.g., stopping an attacker from inserting a backdoor into the download file while it's in transit). |

Both provide **Non-Repudiation**—the author cannot deny having published the code if it is later found to be malicious.

---

## Verifying Integrity: Hashing

A hash function (like SHA-256) takes an input file of any size and produces a fixed-length string of characters (the hash value or digest).

- **The Integrity Check**: 
    1. The legitimate developer writes the code, compiles it into `library.zip`, and calculates the SHA-256 hash.
    2. The developer publishes `library.zip` to a repository and publishes the hash value securely on their official website.
    3. You download `library.zip`, calculate the SHA-256 hash locally on your machine, and compare your result to the developer's published hash.
    4. **If they match exactly**, the file's integrity is verified. It has not been tampered with. If even a single bit was changed by a hacker in transit, the hash would be completely different (the Avalanche Effect).

> **Weak Hashes**: Do not rely on MD5 or SHA-1 for integrity checks, as they are vulnerable to collision attacks (where an attacker can create a malicious file that produces the same hash as the safe file).

---

## Verifying Provenance: Code Signing

Hashing proves the file didn't change in transit, but it doesn't prove *who* wrote it. An attacker could simply replace the file on the download server and replace the published hash next to it. 

**Code Signing** solves this by combining hashing with Asymmetric Cryptography (Public Key Infrastructure - PKI).

- **The Process**:
    1. The developer calculates the hash of `library.zip`.
    2. The developer encrypts that hash using their **Private Key** (which only they possess). This encrypted hash is the **Digital Signature**.
    3. The developer attaches the signature and their Public Key (certified by a trusted Certificate Authority) to the file.
    4. You download the file, use the developer's Public Key to decrypt the hash, and compare it to a hash you calculate yourself.
- **The Result**: Because the hash could only be decrypted by the Public Key, you are mathematically guaranteed that the file was signed by the holder of the Private Key (Provenance). Because the hashes match, you are guaranteed the file wasn't altered (Integrity).

## The SLSA Framework

**Supply-chain Levels for Software Artifacts (SLSA)** is an industry framework (sponsored by Google and the Linux Foundation) designed to prevent tampering, improve integrity, and secure packages and infrastructure.

It aims to answer: *How much confidence can we have that the software we just downloaded wasn't tampered with during its build process?*

SLSA defines incremental levels of maturity:
- **Level 1**: The build process is fully scripted, and provenance (who built it) is available. (Basic defense).
- **Level 2**: The build service generates authenticated provenance. (Prevents tampering during the build).
- **Level 3**: The source and build platforms meet specific high-security standards (Ephemeral environments, isolated builds).
- **Level 4**: Highest level. Requires two-person review of all changes and hermetic (completely sealed, offline) reproducible builds.

> **Exam Tip**: SLSA is an emerging standard heavily focused on securing the automated **CI/CD build pipeline** itself against supply chain attacks.

---

## Exam Focus Points

1. **Provenance equals Origin**: Who wrote it. Established via digital signatures (PKI, Private/Public Keys).
2. **Integrity equals Unchanged**: Has it been tampered with. Established via cryptographic hashing (SHA-2).
3. **Non-Repudiation**: The combination of provenance and integrity ensures the sender cannot deny sending the specific message/code.
4. **SLSA**: A framework specifically designed to secure the software build process and guarantee artifact integrity.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Provenance** | The chronological history of ownership or origin of a software artifact |
| **Integrity** | Assurance that data/code has not been altered in an unauthorized manner |
| **Code Signing** | The process of digitally signing executables to confirm the software author and guarantee that the code has not been altered |
| **Digital Signature** | A cryptographic mechanism providing origin authentication, data integrity, and non-repudiation |
| **SLSA** | Supply-chain Levels for Software Artifacts; a security framework for CI/CD pipelines |
| **Hermetic Build** | A build process that has absolutely no network access to the outside world, guaranteeing reproducibility |
