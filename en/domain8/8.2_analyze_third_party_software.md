# 8.2 Analyze Third-Party Software

## Learning Objectives

- Explain the role of Software Composition Analysis (SCA)
- Differentiate between restrictive and permissive Open Source software licensing
- Identify methods for discovering vulnerabilities in third-party code

---

## Software Composition Analysis (SCA)

Because a modern application is a patchwork quilt of imported libraries, manual code review is impossible. The primary tool used to analyze third-party software is the **Software Composition Analysis (SCA)** tool.

**How SCA Works:**
1. The developer triggers a build or commits code.
2. The SCA tool scans the package manifests (e.g., `package.json`, `pom.xml`, `requirements.txt`) and occasionally the raw binary files.
3. It identifies every exact version of every library the application uses, *including transitive dependencies* (the libraries that your libraries rely on).
4. It compares this inventory against a database of known vulnerabilities (e.g., the National Vulnerability Database - NVD) to see if any components have active CVEs.
5. It checks the licenses of the imported libraries to ensure legal compliance.

> **Security Value**: SCA solves the OWASP Top 10 issue "Vulnerable and Outdated Components."

## Analyzing Open Source Licensing Risks

Security isn't just about preventing hackers; it is also about preventing massive legal liability. Open Source Software (OSS) is free to use, but it comes with strict legal conditions.

If a developer imports a library with the wrong type of license into your proprietary commercial software, your organization could be legally forced to publish your proprietary source code for free on the internet.

### License Types

| License Category | Examples | Description | Business Risk |
|------------------|----------|-------------|---------------|
| **Permissive** | MIT, Apache 2.0, BSD | Highly flexible. Users can do almost anything, including modifying the code and using it in closed-source commercial software without releasing their own source code. | **Very Low** |
| **Copyleft (Restrictive / Viral)** | GPL (GNU General Public License) v2 / v3 | "Viral" nature. If you include GPL code in your application, your entire application legally becomes GPL-licensed, and you must publish your source code if you distribute the software. | **Extremely High** (for commercial software companies) |

### The Developer Blind Spot
Developers generally do not understand licensing law. They simply search GitHub for a library that solves their technical problem and import it. 
- **The Control**: SCA tools automatically scan imported libraries, flag any GPL or restrictive licenses, and break the build before the code can be committed.

## Discovering Vulnerabilities in Third-Party Code

When a major vulnerability drops (e.g., Log4Shell or Heartbleed), the CISO will immediately ask: *"Are we vulnerable?"*

Historically, answering that question took weeks of manual labor, searching across hundreds of development servers to see if that specific library was installed. This is unacceptable.

### The SBOM (Software Bill of Materials)
The modern solution to this problem is generating an **SBOM** for every piece of software the organization builds or buys.

- **Definition**: An SBOM is a formal, machine-readable inventory detailing the supply chain relationships of various components used in building software. It is essentially an "ingredients label" for code.
- **Formats**: CycloneDX (OWASP) and SPDX (Linux Foundation).
- **Incident Response Value**: When the CISO asks, "Are we vulnerable to Log4Shell?", the security team queries their centralized SBOM repository. Within seconds, the system returns a list of exactly which 5 internal applications contain the vulnerable Log4j library, allowing for immediate, targeted patching.

## Code Escrow

What happens if you purchase critical proprietary software from a small, 5-person startup, and two years later they go bankrupt and disappear? You lose all support and the ability to fix bugs.

- **Source Code Escrow**: A legal arrangement where the vendor provides their proprietary source code to a trusted, neutral third party (the Escrow Agent). 
- If the vendor goes bankrupt or breaches the support contract (the "release condition"), the Escrow Agent releases the source code directly to the customer, allowing the customer to maintain it themselves.

---

## Exam Focus Points

1. **SCA vs. SAST**: Both are white-box tools. SAST finds bugs in *your custom code*. SCA finds known CVEs and license issues in *third-party imported code*.
2. **Copyleft (GPL) Risk**: Including a GPL library in commercial software forces you to open-source your proprietary code. Permissive licenses (MIT/Apache) are safe for commercial use.
3. **SBOM Importance**: An SBOM is the authoritative inventory of software components, critical for rapid incident response when a new third-party vulnerability is announced.
4. **Code Escrow**: A mitigation for the business risk of a software supplier going out of business.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **SCA** | Software Composition Analysis |
| **Copyleft / Viral License** | A software license (e.g., GPL) requiring derived works to carry the same open-source terms |
| **Permissive License** | A software license (e.g., MIT, Apache) with minimal restrictions on how the code can be used or modified |
| **SBOM** | Software Bill of Materials; an "ingredients list" for software components |
| **Escrow** | A third-party arrangement holding source code to protect a buyer against a vendor's bankruptcy |
