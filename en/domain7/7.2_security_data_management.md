# 7.2 Security Data Management and Continuous Monitoring

## Learning Objectives

- Define Information Security Continuous Monitoring (ISCM)
- Identify different sources of application telemetry and security data
- Explain the role of metrics and KPIs in operation security
- Describe the integration of application data with organizational SIEMs

---

## Information Security Continuous Monitoring (ISCM)

ISCM is the process of maintaining ongoing awareness of information security, vulnerabilities, and threats to support organizational risk management decisions. 

Because threats evolve daily, point-in-time assessments (like an annual penetration test) are insufficient. **Continuous monitoring** ensures that the security posture remains acceptable long after deployment.

### Core Components of ISCM
1. **Define**: Determine what to monitor (metrics) and how often based on risk tolerance.
2. **Establish**: Implement monitoring architecture (agents, log forwarders).
3. **Implement**: Collect the data continuously.
4. **Analyze**: Review the data for anomalies and actionable intelligence.
5. **Respond**: Take action on findings (incident response, patching).
6. **Review**: Update the monitoring strategy as threats and business environments change.

## Telemetry and Security Data Sources

To effectively monitor an application, it must generate high-quality telemetry. "Telemetry" refers to the automated communications process by which measurements and other data are collected at remote or inaccessible points and transmitted to receiving equipment for monitoring.

### Key Data Sources

| Source | Description | Security Value |
|--------|-------------|----------------|
| **Application Logs** | Custom logs written by the application code. | Records who logged in, who accessed what data, and business logic exceptions. |
| **Web Server Logs** | Logs from IIS, Apache, Nginx, or Load Balancers. | Identifies attack volume, scanning behavior (404 spikes), and source IPs. |
| **WAF/IPS Alerts** | Web Application Firewall or Intrusion Prevention System logs. | Identifies specific blocked attack payloads (e.g., blocked SQLi attempts). |
| **Host/OS Metrics** | CPU, Memory, Disk I/O. | Sudden spikes may indicate a DoS attack or crypto-mining malware. |
| **Database Logs** | Query logs, transaction logs. | Identifies unsanctioned data exports or schema manipulation. |
| **APM Tools** | Application Performance Monitoring (e.g., New Relic, AppDynamics). | Identifies slow API endpoints which might be utilized in an application-layer DoS (Slowloris). |

## Log Aggregation and SIEM

A single enterprise application may consist of dozens of microservices spanning hundreds of containers. Looking at individual server logs is impossible. Data must be aggregated.

### SIEM (Security Information and Event Management)
A SIEM (e.g., Splunk, QRadar, ELK Stack) collects logs from across the enterprise, normalizes them, and correlates them to identify attacks that span multiple systems.

**The Correlation Example:**
- *Event A*: Web server logs show 50 failed login attempts for `admin`.
- *Event B*: Active Directory logs show a successful login for `admin`.
- *Event C*: Database logs show `admin` exporting the `users` table.
- **SIEM Correlation**: The SIEM connects A, B, and C in real-time and fires an alert: **"Brute force success followed by data exfiltration."**

### Application Logging Requirements for SIEM Integration
For an application to play nicely with a SIEM, developers must follow logging standards:
- **Standard Format**: Use structured logging (e.g., JSON) so the SIEM can easily parse fields (UserIP, Action, Status).
- **Consistent Timestamps**: All logs must use **UTC** (Coordinated Universal Time) and be synchronized via NTP. Without accurate time, correlating events across global servers is impossible.
- **Context**: Include transaction IDs or request IDs that follow a user's request across multiple microservices.

## Metrics and Key Performance Indicators (KPIs)

Management relies on metrics to determine if the ISCM program is effective.

| Metric | Description | What It Indicates |
|--------|-------------|-------------------|
| **Mean Time to Detect (MTTD)** | Average time it takes to discover a security breach. | Efficiency of the monitoring and SIEM alerts. (Lower is better). |
| **Mean Time to Respond (MTTR)** | Average time it takes to contain and eradicate the threat after detection. | Efficiency of the Incident Response team. (Lower is better). |
| **Log Volume / EPS** | Events Per Second arriving at the SIEM. | A sudden drop may indicate remote agents have crashed or attackers disabled logging. |
| **Alert Fatigue Rate** | Percentage of SIEM alerts that are False Positives. | If this is high, analysts will start ignoring alerts. Tuning is required. |

---

## Exam Focus Points

1. **Continuous Monitoring**: Replaces point-in-time scanning to account for dynamic infrastructure and evolving threats.
2. **NTP & UTC**: Network Time Protocol and UTC are absolute requirements for log correlation. Without them, reconstructing an attack timeline is impossible.
3. **SIEM Correlation**: The true value of a SIEM is *correlating* events from different sources to detect complex attacks.
4. **Structured Logging**: Developers should log in JSON to facilitate automated parsing.
