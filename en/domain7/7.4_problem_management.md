# 7.4 Problem Management

## Learning Objectives

- Distinguish clearly between Incident Management and Problem Management
- Describe the goal of Problem Management
- Apply Root Cause Analysis (RCA) techniques
- Explain the "5 Whys" and Ishikawa (Fishbone) diagrams

---

## Incident Management vs. Problem Management

These are two highly integrated ITIL (Information Technology Infrastructure Library) concepts, but their goals are distinct.

| Feature | Incident Management | Problem Management |
|---------|---------------------|--------------------|
| **Goal** | Restore normal service operation **as quickly as possible**. | Find the **root cause** and prevent the incident from ever happening again. |
| **Speed** | Urgent / Reactive. Put out the fire. | Deliberate / Proactive. Investigation and analysis. |
| **Analogy** | Taking painkillers for a headache. | Going to the doctor to find out *why* you have chronic headaches, and curing the illness. |
| **Output** | Workarounds, temporary patches. | Permanent structural fixes, architectural changes. |

### The Relationship
Multiple identical *incidents* indicate an underlying *problem*. 
- Ex: Web Server 1 crashes (Incident 1). It is rebooted. 
- Web Server 2 crashes (Incident 2). It is rebooted. 
- Problem Management is initiated to figure out a memory leak in the software code is causing the crashes.

## Root Cause Analysis (RCA)

Root Cause Analysis is the systematic process of identifying the fundamental reason a failure occurred. 

Fixing a symptom (Incident Management) provides temporary relief. Fixing the root cause (Problem Management) provides a permanent solution.

### Technique 1: The "5 Whys"
A simple, iterative interrogative technique used to explore cause-and-effect relationships. You ask "Why?" roughly five times until you hit the core issue.

**Example Scenario**: The main e-commerce database was compromised via SQL Injection.
1. **Why was the database compromised?** 
   - Because an attacker executed a SQL Injection attack on the login page.
2. **Why did the SQL Injection succeed?**
   - Because the login input fields were not validated or parameterized.
3. **Why were the fields not parameterized?**
   - Because the developer used old, legacy database connectivity code.
4. **Why did the developer use legacy code?**
   - Because the developer had never received secure coding training.
5. **Why was there no secure coding training?**
   - Because there is no budget or mandate for developer security training. *(Root Cause!)*

> **Security Value**: Instead of just patching the one SQL flaw (Incident Management), the organization implements mandatory secure coding training, preventing hundreds of future flaws (Problem Management).

### Technique 2: Ishikawa (Fishbone) Diagram
A visual tool used to categorize potential causes of a problem to identify its root causes. Also known as a Cause-and-Effect diagram.

The "head" of the fish is the problem. The "bones" are major categories of causes, typically:
- **People** (Training, fatigue, malicious intent)
- **Processes** (Lack of QA, poor change management)
- **Technology/Equipment** (Outdated servers, missing WAF)
- **Materials** (Third-party vulnerable libraries)
- **Environment** (Network issues)

By brainstorming potential failures in each category, the team can holistically identify contributing factors rather than focusing on a single point of failure.

## Known Errors and Workarounds

In Problem Management, when a root cause is identified but cannot be immediately fixed (due to budget, time, or complexity), it is classified as a **Known Error**.

- **Known Error Database (KEDB)**: An IT service management database containing all known errors and their approved **Workarounds**.
- **Workaround**: A temporary way to restore service or mitigate the risk until a permanent fix is deployed. (e.g., "The software has a memory leak. *Workaround*: Run a script to reboot the server every 24 hours at 3 AM until the patch is delivered next quarter.")

---

## Exam Focus Points

1. **Incident vs. Problem**: Incident = restore service fast; Problem = find root cause permanently.
2. **5 Whys**: Simple, questioning technique to drill down past symptoms to the fundamental process failure.
3. **Fishbone Diagram**: Categorizes potential causes (People, Process, Technology).
4. **Known Error**: A problem that has an identified root cause and a workaround, but no permanent fix yet.

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Problem Management** | The process responsible for managing the lifecycle of all problems; finding the root cause |
| **Root Cause Analysis (RCA)** | Systematic process of identifying the origin of a problem |
| **5 Whys** | Iterative interrogative technique to determine root cause |
| **Ishikawa / Fishbone Diagram** | A cause-and-effect diagram categorizing potential causes of a problem |
| **Known Error** | A problem with a documented root cause and a workaround |
| **Workaround** | A temporary solution to restore service or mitigate risk while a permanent fix is developed |
