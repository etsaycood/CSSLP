# 7.1 Perform Secure Software Deployment

## Learning Objectives

- Define the requirements for secure software release and deployment
- Explain different deployment strategies (Blue/Green, Canary)
- Describe the importance of the boot sequence (Secure Boot)
- Explain the role of Infrastructure as Code (IaC) in secure deployments

---

## Secure Release Preparation

Deployment is the transition of software from the controlled testing environment into the hostile production environment. It requires strict governance.

### Authorization to Operate (ATO)
Before deployment, the system must undergo **Assessment and Authorization (A&A)**. 
- The **Data Owner / Information Owner** reviews the residual risk report.
- If the risk is acceptable, they grant an **Authorization to Operate (ATO)**.
- Deploying without an ATO is a severe compliance violation.

### Pre-Deployment Checklist
- [ ] Code signing completed (ensures artifact integrity).
- [ ] SAST/DAST/SCA scans passed.
- [ ] Penetration test remediations validated.
- [ ] Configuration baseline established and hardened.
- [ ] **Rollback Plan** documented and tested.

> **Exam Tip**: Every deployment plan **must** include a tested rollback plan. If the deployment fails or causes a critical security issue, the team must be able to revert to the previous known-good state immediately.

## Deployment Strategies

Modern agile environments use deployment strategies that minimize downtime and reduce the blast radius of a failed deployment.

| Strategy | Description | Security Benefit |
|----------|-------------|------------------|
| **Blue/Green Deployment** | Two identical production environments (Blue is live, Green is idle). Deploy new code to Green, test it, then simply flip the router/load balancer to point to Green. | Instant rollback capability (just flip the router back to Blue). Zero downtime. |
| **Canary Release** | Deploy the new version to a small subset of users (e.g., 5%). Monitor for errors or security alerts. If stable, roll out to 100%. | Limits the blast radius. If a critical bug exists, only 5% of users are impacted. |
| **Rolling Deployment** | Slowly replace instances of the old version with the new version one-by-one across the cluster. | High availability during deployment, but rollback is slower than Blue/Green. |
| **Feature Toggles (Flags)** | Code is deployed to production but hidden behind a configuration flag. Pushing a config change turns the feature "on." | Extremely fast rollback (just turn the flag off) without requiring a new code deployment. |

## Environment Boot Security

Securing the software isn't enough; the underlying host must boot securely to ensure the OS hasn't been compromised by rootkits.

### Secure Boot and Measured Boot

| Technology | Description |
|------------|-------------|
| **Secure Boot** | A UEFI feature that ensures the system only boots using software (bootloaders, OS kernels) that are trusted and cryptographically signed by the hardware manufacturer. Prevents rootkits from loading before the OS. |
| **Measured Boot** | Records cryptographic hashes of boot components into the TPM (Trusted Platform Module) as the system starts. A remote server can verify these hashes (Remote Attestation) to confirm the system hasn't been tampered with. |
| **TPM** | Dedicated hardware chip on the motherboard that securely generates and stores cryptographic keys, passwords, and digital certificates. |

## Infrastructure as Code (IaC)

Modern deployments rarely involve humans logging into servers to install software. Instead, the infrastructure itself is defined in code (e.g., Terraform, Ansible, AWS CloudFormation).

### Security Benefits of IaC
- **Consistency**: Eliminates "configuration drift" (where servers slowly become misconfigured over time due to manual tweaks).
- **Immutability**: Servers are replaced, not patched. If a server is compromised, it is destroyed and rebuilt from the clean IaC template.
- **Shift-Left Security**: IaC files can be scanned for security misconfigurations (e.g., open S3 buckets, permissive firewall rules) using SAST tools *before* the infrastructure is even built.
- **Auditability**: Every change to the infrastructure is tracked in Version Control (Git), providing a perfect audit trail of who changed the firewall and when.

---

## Exam Focus Points

1. **Rollback Plan**: Essential for every deployment. Never deploy without a tested way to undo it.
2. **ATO**: Authorization to Operate must be granted by the risk owner before deploying.
3. **Blue/Green vs. Canary**: Blue/Green is for zero-downtime and instant rollback. Canary is for limiting the blast radius of new bugs.
4. **Secure Boot limits Rootkits**: It ensures only signed, trusted OS components load at startup.
5. **IaC prevents Config Drift**: Managing infrastructure as code ensures all servers are identical and hardened exactly to specification.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **ATO** | Authorization to Operate; formal management approval to deploy |
| **Blue/Green Deployment** | Using two identical environments to enable zero-downtime releases and instant rollback |
| **Canary Release** | Releasing software to a small subset of users to test stability |
| **Secure Boot** | UEFI feature process that checks the digital signature of the bootloader |
| **TPM** | Trusted Platform Module; hardware chip storing cryptographic material |
| **IaC** | Infrastructure as Code; managing and provisioning compute centers through machine-readable definition files |
