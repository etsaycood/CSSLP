# 7.3 Incident Management

## Learning Objectives

- Explain the difference between an Event and an Incident
- Describe the standard Incident Response Lifecycle (PICERL)
- Identify the goals of containment, eradication, and recovery
- Explain the importance of the internal/post-incident review

---

## Event vs. Incident

- **Event**: Any observable occurrence in a system or network. (e.g., a user logs in, a firewall blocks a packet, a server reboots). Events are normal.
- **Adverse Event**: An event with a negative consequence (e.g., a server crashes).
- **Security Incident**: A violation or imminent threat of violation of computer security policies, acceptable use policies, or standard security practices. (e.g., an attacker steals a database, malware infects a system, a DoS attack takes down a website).

All incidents are events; not all events are incidents.

---

## The Incident Response Lifecycle (PICERL)

The industry standard for incident response follows the six-phase PICERL methodology (often aligned with NIST SP 800-61).

### 1. Preparation
The most critical phase occurs *before* an incident happens.
- **Establish the CSIRT** (Computer Security Incident Response Team).
- **Create Runbooks / Playbooks**: Standard operating procedures for handling specific types of incidents (e.g., "Ransomware Playbook", "DDoS Playbook").
- **Tools and Access**: Ensure the team has "jump bags," access to network diagrams, out-of-band communication tools (Slack/Teams might be compromised!), and forensics tools.
- **Training and Tabletop Exercises**: Simulating incidents to practice the response.

### 2. Identification (Detection & Analysis)
Determining if an incident has actually occurred, its scope, and its severity.
- Analyzing alerts from the SIEM, WAF, or user reports.
- **Triage**: Is this a false positive or a true positive?
- **Scope**: Did they just hit the web server, or did they pivot into the database?
- **Declaration**: Formally declaring an incident, which triggers the escalating response procedures.

### 3. Containment
Stopping the bleeding. The goal is to quickly limit the scope and magnitude of the incident.
- **Short-term containment**: Taking immediate action (e.g., unplugging the server from the network, disabling the compromised user account) while preserving evidence.
- **Long-term containment**: Rebuilding systems cleanly, applying temporary patches, or routing traffic through a "sinkhole" to monitor the attacker while preparing the final fix.

> **Warning**: Do not immediately reboot a compromised server unless necessary. Rebooting destroys volatile memory (RAM) which holds critical forensic evidence (running malware, active network connections).

### 4. Eradication
Removing the root cause and the threat actor from the environment.
- Deleting malware, disabling breached accounts, closing the firewall hole they used to get in.
- This phase often overlaps with the problem management phase, where developers issue a hotfix to patch the underlying software vulnerability (e.g., fixing the SQL injection flaw).

### 5. Recovery
Returning the affected systems to normal business operations.
- Restoring from clean backups.
- Rebuilding OS environments from clean IaC templates.
- **Validation**: Heavily monitoring the restored systems for several days to ensure the attacker doesn't immediately return through a backdoor that was missed during eradication.

### 6. Lessons Learned (Post-Incident Review)
The most frequently skipped, yet most valuable phase. Conducted within a few weeks of the incident.
- **Goal**: Improve the response process for next time.
- **Questions asked**: What happened? Why didn't we catch it sooner? Did the playbook work? What tools do we need to buy?
- This phase directly loops back into **Phase 1 (Preparation)**.

## Forensics & Evidence Handling

If the incident may lead to legal action or prosecution, evidence must be handled meticulously.

- **Order of Volatility**: Collect evidence starting from the most volatile (easily lost) to the least volatile.
    1. CPU Registers / Cache
    2. RAM (Memory)
    3. Network State / Routing tables
    4. Hard Drives
    5. Backup tapes
- **Chain of Custody**: A rigorously maintained chronological log showing who collected, handled, transferred, or accessed the evidence. A broken chain of custody makes evidence inadmissible in court.
- **Write Blockers**: Never do forensic analysis on the original hard drive. Make a bit-for-bit copy using a hardware write blocker to ensure the original is mathematically unaltered, and perform analysis on the copy.

---

## Exam Focus Points

1. **PICERL Order**: Memorize the phases: Preparation, Identification, Containment, Eradication, Recovery, Lessons Learned.
2. **Preparation is paramount**: Without runbooks and practice, real incidents turn into chaos.
3. **Do not reboot**: Rebooting destroys volatile memory necessary for forensics. Contains by isolating from the network, not by pulling the power plug.
4. **Order of Volatility**: Collect RAM before touching the hard drive.
5. **Chain of Custody**: A legal tracking document for evidence.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Incident** | A violation of security policies or practices |
| **CSIRT** | Computer Security Incident Response Team |
| **Runbook / Playbook** | A documented set of procedures for responding to a specific incident type |
| **Containment** | Isolating the threat to prevent further damage |
| **Order of Volatility** | The sequence in which digital evidence should be collected, based on how easily it is lost |
| **Chain of Custody** | A chronological paper trail documenting the handling of evidence |
