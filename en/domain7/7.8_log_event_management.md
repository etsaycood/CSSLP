# 7.8 Log and Event Management

## Learning Objectives

- State the importance of immutable audit logging
- Describe log aggregation and normalization concepts
- Identity legal and compliance requirements for log retention
- Explain log analysis techniques

---

## The Purpose of Logging

Logging is the foundation of accountability, non-repudiation, incident response, and forensics. If an application does not generate accurate, protected logs, it is impossible to definitively prove who did what, when they did it, and whether a breach occurred.

### What Must Be Logged?
At a minimum, application security logs must capture:
- All authentication events (Successes and Failures).
- All authorization decisions (Especially Access Denied events).
- All administrative actions (Adding/removing users, changing permissions).
- High-value transactions (Financial transfers, viewing PHI).
- Configuration changes to the application or underlying server.

### What Must *Not* Be Logged?
(As noted in Domain 5, but reiterated here for operations context):
- Plaintext passwords or session tokens.
- Unmasked PII/PHI or full credit card numbers (PANs).
- Detailed database query strings or application stack traces, *unless* they are routed to a highly secured developer-only logging tier.

## Log Protection

Attackers know that logs are their enemy. Upon breaching a system, their immediate priority is deleting or altering the logs to cover their tracks.

### WORM (Write-Once-Read-Many) Storage
Logs must be forwarded immediately away from the local application server to a centralized log server or SIEM. Once received, they must be stored on **WORM media**.
- **WORM** ensures that once a log entry is written, it cannot be altered or deleted by anyone, not even a System Administrator with root privileges, until the retention period expires.
- **Example**: AWS S3 Object Lock (Compliance mode).

### Time Synchronization (NTP)
If Server A says a user logged in at 12:05:00 based on its local clock, and the database says data was accessed at 12:06:00, but they are in different time zones, correlating the events is highly problematic.
- **Rule**: All servers writing logs must sync their clocks using **NTP (Network Time Protocol)**.
- **Rule**: All logs must be written using **Coordinated Universal Time (UTC)**, regardless of the server's physical location.

## Log Aggregation and Normalization

Enterprise logs come in hundreds of different formats: IIS logs, Windows Event IDs, Linux syslog, Custom application JSON.

1. **Aggregation**: Centralizing all disparate logs into a single repository (the SIEM).
2. **Normalization**: The process where the SIEM parses the different formats and translates them into a single, common schema.
    - *Windows Event ID 4624* (Logon Success) and *Linux syslog `sshd: accepted password`* are both normalized to a common event type, e.g., `Authentication:Success`.
    - This allows analysts to write a single query to find all login successes across the entire enterprise, regardless of OS.

## Log Retention and Compliance

Logs consume massive amounts of storage space, which is expensive. However, compliance regulations dictate strict retention periods.

| Framework | Typical Retention Requirement |
|-----------|-------------------------------|
| **PCI DSS** | 1 year (with at least 3 months immediately available for online analysis). |
| **HIPAA** | Up to 6 years for audit logs related to PHI access. |
| **SOX** | Generally 7 years for financial audit trails. |
| **GDPR** | No explicit timeframe, but implies logs containing PII must not be kept longer than necessary for their purpose (Data Minimization). |

### Log Lifecycle
1. **Hot Storage**: Stored in the SIEM for 30-90 days for rapid querying and incident response. (Very expensive).
2. **Cold Storage / Archiving**: Moved to cheap, slow storage (e.g., AWS Glacier or tape backups) for 1–7 years purely to satisfy legal and compliance requirements.

## Log Analysis Techniques

- **Pattern Matching (Signatures)**: Searching for known malicious strings (e.g., an automated alert if the string `UNION SELECT` appears in a web log).
- **Heuristics / Anomaly Detection**: Establishing a baseline of normal behavior and alerting when deviations occur. (e.g., User typically logs in from New York at 9 AM. Today, they logged in from Moscow at 3 AM. No specific signature was matched, but the behavior is highly anomalous).
- **Correlation**: Connecting multiple events across different systems (as discussed in 7.2) to identify a cohesive attack pattern.

---

## Exam Focus Points

1. **WORM Storage**: The only way to guarantee log integrity against a compromised administrator.
2. **NTP and UTC**: Absolute prerequisites for log correlation. Without accurate, standardized time, logs are useless for forensics.
3. **Normalization**: Translating disparate log formats into a common schema for centralized analysis.
4. **Retention**: Driven by legal, regulatory, and business requirements—not by technical limitations.

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **WORM** | Write-Once-Read-Many; storage that prevents modification or deletion of data |
| **Normalization** | Parsing disparate log formats into a standard, common schema |
| **Aggregation** | Collecting logs from multiple sources into a centralized location (SIEM) |
| **Anomaly Detection** | Identifying behavior that deviates from an established baseline |
| **Cold Storage** | Inexpensive, slow-retrieval storage used for long-term archiving |
