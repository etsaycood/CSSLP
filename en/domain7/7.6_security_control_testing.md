# 7.6 Security Control Testing (Post-Deployment)

## Learning Objectives

- Explain the necessity of testing security controls after deployment
- Distinguish between pre-deployment AST and post-deployment control testing
- Describe the role of internal and external continuous auditing
- Understand SLA monitoring and cloud service provider (CSP) auditing

---

## Why Test Controls After Deployment?

During Domain 6 (Testing), the application was validated in a staging environment. However, once the software is deployed to production, the environment changes, user behavior introduces new variables, and the threat landscape evolves.

**Security control testing post-deployment validates that:**
1. The controls are actually turned on in production.
2. The controls are configured correctly for the live environment.
3. The controls have not degraded or been secretly turned off to improve performance.

| Pre-Deployment (Domain 6) | Post-Deployment (Domain 7) |
|-------------|------------|
| Focus: "Does the code contain flaws?" | Focus: "Are the runtime defenses working?" |
| SAST, Code Review, functional DAST | WAF tuning, SIEM alert verification, configuration audits |

---

## Continuous Auditing and Vulnerability Scanning

Instead of waiting for an annual penetration test, mature operations employ continuous scanning against production environments.

### 1. Production Vulnerability Scanning
Running automated authenticated and unauthenticated scans against the live infrastructure (web servers, databases, load balancers) on a weekly or daily basis.
- **Goal**: Identify missed patches, newly discovered CVEs for existing software, or configuration drift (e.g., someone accidentally opening port 3389 to the internet).
- **Rule**: Must be scheduled during low-traffic windows to avoid impacting production performance or causing an accidental Denial of Service.

### 2. WAF and IDS/IPS Tuning
A Web Application Firewall is not a "set-and-forget" tool.
- At deployment, it is often placed in "Learning" or "Log Only" mode to avoid breaking legitimate traffic (False Positives).
- **Security Control Testing**: Operations teams must continually test the WAF against known malicious payloads (e.g., sending an EICAR test file or a benign SQL injection string) to ensure the WAF correctly blocks the attack and fires an alert to the SIEM.

## Third-Party and Cloud Auditing

If your application relies on a Cloud Service Provider (CSP) like AWS, Azure, or a SaaS vendor, you still own the risk of a breach, but you cannot physically audit their datacenters.

### Service Level Agreements (SLAs)
You must monitor the vendor's compliance with agreed-upon SLAs (e.g., 99.99% uptime, 4-hour remediation for critical security patches).

### SOC 2 Reports (System and Organization Controls)
Because you cannot pentest AWS's underlying infrastructure, you rely on third-party attestation reports. A **SOC 2 Type II** report is the industry standard. An independent auditor verifies that the CSP has implemented, and consistently enforces, adequate security controls over a period of time (usually 6-12 months).

- **Type I**: A snapshot in time (Controls are designed correctly).
- **Type II**: A period of time (Controls actually work day-to-day). Type II is far superior.

---

## Synthetic Transactions

To continuously verify that application security controls (like authentication) are functional, operations teams use synthetic transactions.

- A script runs every 5 minutes simulating a user logging in, placing an item in a cart, and checking out. 
- If the script fails, or if it succeeds without triggering the expected MFA prompt, the operations team is alerted immediately that a critical security control has failed in production.

---

## Exam Focus Points

1. **Continuous Auditing**: Replaces annual assessments. Scans must be scheduled during maintenance windows to avoid DoS.
2. **WAF Tuning**: Essential post-deployment step. A newly deployed WAF will either block legitimate traffic or let attacks through until tuned.
3. **SOC 2 Type II**: The primary method for verifying the security controls of a Cloud Service Provider.
4. **Configuration Drift**: The primary reason post-deployment testing is necessary—administrators make undocumented changes that weaken security over time.

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Control Testing** | Validating that a security mechanism (e.g., WAF) functions correctly in the live environment |
| **SOC 2 Type II** | An auditor's report proving a service provider's security controls operate effectively over time |
| **Synthetic Transaction** | An automated script simulating user activity to verify application health and security |
| **Configuration Drift** | Unplanned and undocumented changes to system configurations over time |
| **WAF Tuning** | Adjusting Web Application Firewall rules to minimize false positives and false negatives |
