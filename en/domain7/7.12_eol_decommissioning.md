# 7.12 End of Life (EOL) and Decommissioning

## Learning Objectives

- Define End of Life (EOL) and End of Support (EOS)
- Explain the security risks associated with operating EOL software
- Describe the data migration and system decommissioning process
- Identify compliance requirements when terminating vendor services

---

## The Software Twilight: EOL and EOS

Software does not last forever. Eventually, maintaining old code becomes too expensive, or the underlying technology becomes obsolete.

| State | Description | Security Implication |
|-------|-------------|----------------------|
| **End of Life (EOL)** | The vendor (or your internal dev team) stops selling the product and stops adding new features. | Low risk. The vendor will typically still provide critical security patches for a defined period. |
| **End of Support (EOS)** | The vendor completely severs support. **No more security patches will ever be released.** | **Extreme Risk**. Running EOS software (e.g., Windows 7, Python 2.7) violates PCI DSS and heavily exposes the organization to unpatchable vulnerabilities. |

### The "Legacy System" Dilemma
Organizations frequently get trapped running EOS software because the business process relying on it is too critical, and migrating to a new system is too expensive.

- **Mitigation**: If you absolutely must run EOS software (such as an old XP machine running a million-dollar MRI scanner), you must **isolate it heavily**. Place it on a dedicated, air-gapped VLAN with no internet access, and strictly control lateral movement using firewalls and zero-trust policies.

---

## The Decommissioning Process

Decommissioning an application (Sunsetting) must follow a structured, secure procedure, not just unplugging the server.

### 1. Sunset Planning
- Communicate the shutdown to stakeholders and customers well in advance (e.g., 6-12 months).
- Stop accepting new user sign-ups or new data inputs.

### 2. Data Preservation / Migration
The data within an old application is often valuable and legally mandated for retention.
- **Migration**: Moving the data securely to its modern replacement system via encrypted ETL (Extract, Transform, Load) processes. Ensure data integrity checks (hashing) are used to verify the migration.
- **Archival**: Exporting the data to cold storage (e.g., AWS S3 Glacier, Long-term Tape Vaults) to satisfy 7-year legal retention requirements before securely destroying the database.

### 3. Service Termination
- **Revoke Service Accounts**: The sunset application usually has highly privileged access to the core database or external APIs. You must delete these service accounts *before* turning off the server, preventing orphaned credentials from being exploited later.
- **DNS / Routing**: Update DNS records and load balancers. If you abandon `api.old-app.com` but do not unregister the domain, an attacker might buy it via a subdomain takeover attack and use it to harvest credentials from users who have not updated their bookmarks.

---

## Terminating Third-Party Providers

When decommissioning a SaaS product or terminating a contract with a Cloud Service Provider (CSP), special contractual steps must be taken.

- **Data Return/Destruction clause**: You must invoke the Right to be Forgotten contract clauses. The vendor must securely return your massive raw data sets, and then provide a **Certificate of Destruction** proving they securely wiped your data from their multi-tenant environment.
- **Revoking External Access**: Immediately sever VPN tunnels, SAML SSO federation trusts, and API keys previously granted to the vendor.

---

## Exam Focus Points

1. **End of Support (EOS)**: Operating software past EOS violates compliance rules because zero-day vulnerabilities will never be patched by the vendor.
2. **Orphaned Accounts**: The highest security risk of decommissioning. Failing to delete the old application's administrative service accounts leaves backdoors in the network.
3. **Data Archival**: You cannot simply hit "Delete" on the database. Financial and healthcare regulations mandate keeping archival records for years.
4. **Isolation of Legacy Systems**: If forced to run EOS systems, they must be air-gapped or heavily segmented via VLANs/firewalls.

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **EOL (End of Life)** | The software is no longer sold or actively developed with new features |
| **EOS (End of Support)** | The vendor no longer supplies security patches or technical support |
| **Decommissioning (Sunsetting)** | The formal, secure process of permanently retiring an IT system |
| **Orphaned Account** | A valid user or service account left active after an employee leaves or a system is decommissioned |
| **Subdomain Takeover** | An attacker claiming an abandoned subdomain that still points to a deactivated cloud resource |
