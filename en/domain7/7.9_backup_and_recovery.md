# 7.9 Backup and Recovery

## Learning Objectives

- Define standard backup types (Full, Differential, Incremental)
- Explain the importance of testing backups
- Identify backup integrity and security requirements
- Calculate RTO and RPO relative to backup strategies

---

## Backup Types

A robust backup strategy balances storage costs, acceptable data loss, and the speed required to restore service after a failure (e.g., ransomware attack or hardware failure).

| Backup Type | What is Backed Up? | Restoration Speed | Storage Required | Explanation |
|-------------|--------------------|-------------------|------------------|-------------|
| **Full Backup** | Every single file and block of data on the system. | **Fastest**. You only need to restore one file (the most recent Full Backup). | **Highest**. Takes massive amounts of storage and time to complete. | Usually performed weekly (e.g., Sunday night) due to the heavy resource load. |
| **Differential** | All files that have changed *since the last Full Backup*. | Moderate. Requires restoring two files: the Full Backup, plus the latest Differential. | Moderate, but grows larger every day until the next Full Backup. | If a crash happens on Thursday, restore Sunday's Full, then Thursday's Differential. (Files changed Mon/Tue/Wed are included in Thu's Differential). |
| **Incremental** | All files that have changed *since any previous backup* (Full or Incremental). | **Slowest**. Requires restoring the Full Backup, plus *every* Incremental in chronological order. | **Lowest**. Only saves the tiny daily changes. Extremely fast to perform. | If a crash happens on Thursday, restore Sunday's Full, plus Mon, Tue, Wed, and Thu Incrementals. If Tuesday's file is corrupt, Wed and Thu cannot be restored. |

## The 3-2-1 Backup Rule

The industry standard securing against data loss, especially ransomware.

- **3**: Maintain at least **three copies** of your data (The production data, plus two backups).
- **2**: Store backups on at least **two different media types** (e.g., disk arrays and cloud object storage, or disk and magnetic tape). Prevents media-specific failures.
- **1**: Keep at least **one backup offsite** (or in an isolated "air-gapped" cloud vault). If a fire destroys the datacenter, or ransomware encrypts the entire local network, the offsite backup survives.

## Metrics: RPO and RTO

These two metrics, defined by business leadership through a Business Impact Analysis (BIA), dictate exactly what kind of backup technology the IT team must purchase.

### RPO (Recovery Point Objective)
- **Question**: "How much data can the business afford to lose permanently?"
- **Determines**: Backup Frequency.
- **Example**: If a bank's database has an RPO of 15 minutes, doing daily tape backups is unacceptable. Because if a crash happens at 4:00 PM, they lose all data since the 12:00 AM backup (16 hours of data loss). They require real-time database replication or snapshots every 15 minutes.

### RTO (Recovery Time Objective)
- **Question**: "How long can the application be offline before the business goes bankrupt?"
- **Determines**: Restoration Methodology and Speed.
- **Example**: If Amazon.com goes down, the RTO is minutes/seconds. Tape backups are unacceptable because pulling tapes from a vault and restoring terabytes of data takes days. They require hot site failovers for instant recovery.

## Securing the Backups

Attackers specifically target backups. If they encrypt the backups first, you cannot recover and are forced to pay the ransom.

- **Encryption**: Backups must be strongly encrypted at rest and in transit.
- **Integrity Checking**: Cryptographic hashes (SHA-256) of the backup files ensure they haven't been tampered with or silently corrupted.
- **Immutability (WORM)**: Modern backups must be set to "immutable." Even a domain administrator cannot delete or modify the backup files until the retention period expires.
- **Separation of Duties**: The service account used to write the backup should not have permission to delete the backup or modify the source database.

### The Most Important Rule
**An untested backup is not a backup.** 
It is merely a "hope." Organizations must perform regular, scheduled **Restore Operations** to verify the backups are actually readable and the RTO can realistically be met.

---

## Exam Focus Points

1. **Incremental vs. Differential**: Incremental requires all preceding tapes to restore (slow restore). Differential requires only the Full and the latest tape (medium restore).
2. **RTO vs. RPO**: RTO = Maximum tolerable downtime (drives speed). RPO = Maximum tolerable data loss (drives frequency).
3. **Immutability**: Essential defense against ransomware destroying backups.
4. **Testing**: Backups must be tested by performing actual restorations.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Full Backup** | Complete copy of all data |
| **Incremental Backup** | Backs up data changed since the last backup of any kind |
| **Differential Backup** | Backs up data changed since the last *Full* backup |
| **RPO** | Recovery Point Objective; the acceptable amount of data loss |
| **RTO** | Recovery Time Objective; the acceptable amount of downtime |
| **Air-Gap** | Physical or logical isolation of a backup from the primary network |
