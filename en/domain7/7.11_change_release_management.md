# 7.11 Change and Release Management

## Learning Objectives

- Describe the Change Management process
- Explain the role of the Change Advisory Board (CAB)
- Define secure release strategies and version control
- Identify the separation of duties required in release management

---

## The Change Management Process

In a production environment, unauthorized or poorly planned changes are the primary cause of system outages and security breaches. Change Management prevents rogue developers or admins from "just fixing a quick typo" on a live server.

**The Golden Rule**: No modifications may be made to production systems without formal request, review, approval, testing, and a rollback plan.

### Core Steps of Change Management
1. **Request for Change (RFC)**: A formal request detailing the proposed change (patch, new feature, configuration file edit).
2. **Review & Risk Assessment**: Evaluating the impact on security, performance, and users.
3. **Approval (The CAB)**: The Change Advisory Board reviews the RFC.
4. **Testing**: The change must be successfully deployed in a Non-Production environment (Staging/QA) identical to production.
5. **Implementation (Release)**: Deploying the change during an approved maintenance window.
6. **Post-Implementation Review**: Did the change succeed? Did it cause incidents? If so, trigger the rollback plan.

### Emergency Changes
In the event of a critical zero-day vulnerability or production outage, standard Change Management is too slow (e.g., waiting two weeks for a CAB meeting).
- **Emergency Change Advisory Board (ECAB)**: A smaller, high-level group authorized to instantly approve emergency patches over the phone or via chat.
- **Rule**: The paperwork/formal documentation still occurs, but it happens *retroactively*, after the emergency is resolved.

## Release Management

While Change Management governs the *approval* process, Release Management governs the tactical *deployment* of the software.

### Key Controls in Release Management
- **Separation of Duties (SoD)**: The developer who wrote the code **must not** have the credentials to deploy it to production. A dedicated Release Engineer (or the automated CI/CD pipeline) performs the deployment. This prevents a rogue developer from bypassing code reviews or writing a backdoor and deploying it silently.
- **Configuration baselines**: Knowing exactly which versions of which files and libraries comprise "Version 2.0" of the software so it can be reliably rebuilt.
- **Rollback / Back-out Plans**: You cannot deploy if you cannot undo it. If a deployment fails catastrophically, Release Management dictates how to revert to Version 1.9 instantly.

## Versioning and Semantic Versioning

Tracking which release contains which security fixes is critical for vulnerability management. Most software engineering uses Semantic Versioning (`MAJOR.MINOR.PATCH`).

Example: `1.4.2`
- **1 (MAJOR)**: Incompatible API changes or a massive rewrite.
- **4 (MINOR)**: Adding new functionality in a backwards-compatible manner.
- **2 (PATCH)**: Backwards-compatible bug fixes and **security patches**.

> **Security Implication**: If the security team tells the business they must upgrade a library from `1.4.2` to `1.4.3` to fix a CVE, the business knows it is safe to do so automatically, because MINOR/MAJOR versions didn't change (the API shouldn't break).

## Exam Focus Points

1. **Separation of Duties**: The cornerstone of release management. Developers write code; Release Managers (or CI/CD pipelines) deploy it. Never the same person.
2. **Rollback Plan Requirement**: An RFC that lacks a tested rollback plan must be rejected by the CAB automatically.
3. **Emergency Changes**: Security must balance agility with process. Emergency CABs bypass standard workflows to patch zero-days, but document retroactively.
4. **Change equals Risk**: The most statistically likely time for a system to crash or open a vulnerability is exactly after a change is implemented.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **RFC** | Request for Change; the formal document proposing a modification |
| **CAB** | Change Advisory Board; the committee that approves or rejects changes |
| **ECAB** | Emergency CAB; authorized to rapidly approve critical out-of-band patches |
| **Separation of Duties (SoD)** | Ensuring no single individual has the authority to both create and execute a sensitive action |
| **Semantic Versioning** | A standardized `MAJOR.MINOR.PATCH` numbering system tracking software releases |
| **Rollback Plan** | A documented, tested procedure for reversing a failed deployment |
