# 6.4 Perform Verification and Validation (V&V)

## Learning Objectives

- Distinguish clearly between Verification and Validation
- Identify which SDLC phases utilize V&V
- Explain Independent Verification and Validation (IV&V)
- Describe User Acceptance Testing (UAT)

---

## The Difference Between Verification and Validation

V&V are distinct processes used to ensure software quality and security. Confusing the two is a common error on the exam.

| Concept | The Core Question | Focus | Output |
|---------|-------------------|-------|--------|
| **Verification** | **"Are we building the product right?"** | Checks the product against the **specifications and requirements**. | Ensures architecture/code matches the design document and SRTM. |
| **Validation** | **"Are we building the right product?"** | Checks the product against the **user's actual needs and expectations** in the real world. | Ensures the software solves the customer's business/security problem. |

### Verification Details
- **Timing**: Occurs *throughout* the SDLC (planning, design, coding).
- **Activities**: Peer code reviews, design walkthroughs, static analysis (SAST), unit testing, architecture reviews.
- **Goal**: Find defects early. Ensure compliance with internal standards.

### Validation Details
- **Timing**: Occurs at the *end* of the SDLC (testing phase, deployment).
- **Activities**: Penetration testing, Dynamic testing (DAST), Beta testing, User Acceptance Testing (UAT).
- **Goal**: Ensure the finished product actually functions securely in its intended environment and meets the owner's true intent.

> **Analogy**: 
> - **Verification**: Checking the blueprints against the building code, and making sure the builders followed the blueprint.
> - **Validation**: Having the family walk through the finished house to see if it's actually comfortable and safe to live in.

## IV&V: Independent Verification and Validation

IV&V means that the Verification and Validation processes are performed by an **entity entirely separate from the development team**.

### Why use IV&V?
- **Avoids bias**: Developers have blind spots regarding their own code ("They know how it's *supposed* to work").
- **Regulatory Requirement**: High-assurance systems (aerospace, defense, healthcare) often legally require IV&V.
- **True Security Posture**: An independent third-party penetration test provides a more realistic validation of the system's defenses against external attackers.

### Levels of Independence (from lowest to highest)
1. Different developer on the same team (Peer Review).
2. Separate QA/Security team within the same company.
3. Fully independent third-party organization (External Auditor/Pentester).

## User Acceptance Testing (UAT)

UAT is the final phase of the software testing process, representing the ultimate **Validation** step.

- **Conducted by**: The client, end-users, or business owner (not the developers or QA engineers).
- **Goal**: To confirm the system meets business requirements and is ready for production deployment.
- **Security in UAT**: While UAT is primarily functional, it includes validating security scenarios:
    - *Can the manager approve the payroll? (Functional)*
    - *Is the standard employee blocked from seeing the payroll screen? (Security Validation)*
- **Outcome**: Formal sign-off and approval to release the software to Production.

---

## Traceability in V&V

V&V relies heavily on the **Security Requirements Traceability Matrix (SRTM)**.

- **Verification** uses the SRTM to ensure that every security requirement written in Phase 1 has a corresponding piece of code written in Phase 3.
- **Validation** uses the SRTM to execute test cases showing that the code successfully fulfills the security requirement in the live environment.

---

## Exam Focus Points

1. **Verification**: "Are we building the product right?" (Matches requirements). Occurs throughout the SDLC.
2. **Validation**: "Are we building the right product?" (Meets user needs in the real world). Occurs at the end of the SDLC.
3. **IV&V**: Independence is key for high-assurance systems to remove bias. Third-party pentesting is a form of IV&V.
4. **UAT**: The final Validation step performed by the end-user/client to provide formal sign-off for release.

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Verification** | Evaluating software to determine whether it satisfies the specified requirements |
| **Validation** | Evaluating software at the end of development to ensure compliance with intended real-world use |
| **IV&V** | Independent Verification and Validation; performed by an entity separate from the development team |
| **UAT** | User Acceptance Testing; final validation step performed by stakeholders/users |
