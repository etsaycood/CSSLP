# 6.5 Perform Security Testing (Tools and Execution)

## Learning Objectives

- Select the appropriate Application Security Testing (AST) tool for the right SDLC phase
- Differentiate between structural (white-box) and dynamic (black-box) testing principles
- Explain the role and limitations of Penetration Testing
- Understand Fuzzing and Code Coverage metrics

---

## Application Security Testing (AST) Landscape

Security testing requires a combination of tools; no single tool finds every type of vulnerability.

| Tool Type | Analysis Method | Primary Use Case | Strengths | Weaknesses |
|-----------|-----------------|------------------|-----------|------------|
| **SAST** (Static) | White-box (Source Code) | Early in SDLC; developer IDE, CI/CD commit hooks. | Finds exact line of code; scales well. | High false positives; cannot find runtime configuration flaws. |
| **SCA** (Composition) | White-box (Dependencies) | Identify vulnerable third-party open-source libraries. | Fast; generates SBOMs necessary for supply chain security. | Only finds *known* CVEs in *external* code; doesn't analyze custom logic. |
| **DAST** (Dynamic) | Black-box (Running App) | Later in SDLC; QA/Staging environments. | Low false positives; finds runtime issues (auth bypass, XSS). | Cannot pinpoint exact line of code; requires a stable, running environment. |
| **IAST** (Interactive) | Glass-box (Inside App) | During QA functional testing; instruments the code at runtime. | High accuracy; identifies exact line of code *and* runtime issues. | Can impact performance; depends on thorough functional test coverage. |

> **Note**: These concepts were introduced in Domain 5, but Domain 6 focuses on *executing* them as a cohesive testing strategy to validate the software.

## Penetration Testing (Pentesting)

Penetration testing is an authorized, simulated cyberattack on a computer system, performed to evaluate the security of the system.

- **Role in SDLC**: Pentesting is a **Validation** activity occurring at the end of the SDLC (Pre-production or Production). It should *not* be the primary method for finding bugs. It is meant to validate the effectiveness of the entire security program.
- **Goal**: Emulate a real-world attacker to identify business logic flaws, chained vulnerabilities (combining two low-risk flaws to create a high-risk exploit), and misconfigurations that automated tools (SAST/DAST) cannot find.

### The Problem with Pentesting
- **Late in the Cycle**: Finding an architectural flaw during a pentest right before launch is extremely expensive (Shift-Left principle).
- **Point-in-Time**: The report is only valid for the specific version of the code tested on that specific day. Agile development requires continuous testing, not just annual pentests.

## Fuzzing

Fuzzing involves sending massive amounts of invalid, unexpected, or random data (the "fuzz") to an application to induce a crash, exception, or memory leak.

- **Objective**: Identify buffer overflows (in C/C++), unhandled exceptions, and logic errors in parsers or network protocols.
- **Generation-based (Smart)**: The fuzzer understands the expected file format or protocol (e.g., HTTP) and creates structurally valid but malicious inputs.
- **Mutation-based (Dumb)**: The fuzzer takes an existing valid input and randomly flips bits or appends data without understanding the underlying format.

## Code Coverage Metrics

How much of the application has actually been tested? Code coverage metrics quantify the thoroughness of the test suite.

| Metric | Definition | Security Context |
|--------|------------|------------------|
| **Statement Coverage** | Percentage of executable statements that have been executed by tests. | Easiest to achieve, but weakest metric. (e.g., A basic positive test hits most statements). |
| **Branch/Decision Coverage** | Percentage of boolean expressions (if/else) where both True and False outcomes have been tested. | Better. Ensures the application's logic paths have been validated. |
| **Condition Coverage** | Percentage of individual sub-conditions within a boolean expression that have been evaluated to both True and False. | Strongest. Critical for complex access control logic (`if (User == Admin AND File == Public)`). |
| **Function/Method Coverage** | Percentage of functions called during testing. | A starting point to identify completely untested modules. |

> **Exam Tip**: You cannot realistically achieve 100% path coverage in complex software due to the sheer number of combinational possibilities. Testers must prioritize risk-based coverage.

---

## Exam Focus Points

1. **SAST vs. DAST limitations**: SAST has high false positives; DAST cannot point to the line of code.
2. **Pentesting is a Validation step**: It should find complex logic flaws and chained exploits, not basic SQL injection (which SAST/DAST should have caught earlier).
3. **Fuzzing**: Inducing crashes to find buffer overflows and unhandled exceptions.
4. **Code Coverage**: Branch/Decision coverage provides significantly more assurance than simple Statement coverage.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **SAST** | Static Application Security Testing (White-box, early SDLC) |
| **DAST** | Dynamic Application Security Testing (Black-box, late SDLC) |
| **Penetration Testing** | Simulating real-world attacks to validate overall security posture |
| **Fuzzing** | Testing with invalid or random data to induce system failure |
| **Statement Coverage** | Metric measuring the percentage of code lines executed during testing |
| **Branch Coverage** | Metric measuring the percentage of decision paths executed during testing |
