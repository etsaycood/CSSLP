# 6.3 Develop Security Test Cases

## Learning Objectives

- Differentiate between positive and negative testing
- Design positive and negative test cases
- Apply Boundary Value Analysis (BVA) and Equivalence Partitioning
- Formulate misuse and abuse test cases

---

## The Anatomy of a Test Case

A test case is a specific set of conditions, data, and expected results used to verify a security requirement.

A standard test case includes:
- **Test ID / Name**: Unique identifier mapping back to the SRTM.
- **Preconditions**: State the system must be in before the test (e.g., "User is logged in").
- **Test Steps**: Exact actions the tester (or automated script) will take.
- **Test Data**: The specific inputs used (e.g., username, password, malicious payload).
- **Expected Result**: What the system *should* do if it is secure.
- **Actual Result**: What the system *did* (Pass/Fail).

---

## Positive vs. Negative Testing

Security testing heavily relies on understanding the difference between how the system handles proper vs. improper input.

### Positive Testing (Happy Path)
Determines if the application **works as expected** when provided with valid input and authorized actions.
- **Focus**: Functionality and Accessibility. "Does it do what it is supposed to do?"
- **Example**: Logging in with the correct username and password, expecting to reach the dashboard.
- **Security Context**: Ensuring legitimate users are not falsely denied access (Availability).

### Negative Testing (Sad Path)
Determines if the application **handles invalid input gracefully** and rejects unauthorized actions.
- **Focus**: Resilience and Security. "Does it prevent what it is *not* supposed to do?"
- **Example**: Entering `' OR 1=1 --` into the login field, expecting the system to reject the input and throw a generic error, rather than granting access or crashing.
- **Security Context**: Validating error handling, fail-secure postures, and injection defenses.

> **Exam Tip**: In functional testing, the focus is mostly on Positive testing. In **security testing**, the vast majority of effort revolves around **Negative testing**.

## Input Testing Techniques

When designing test data for negative testing, you cannot test every infinite possible combination of inputs. Testers use specific techniques to maximize coverage with minimal test cases.

### 1. Boundary Value Analysis (BVA)
Bugs frequently occur at the edges (boundaries) of input domains. BVA targets exactly on, just below, and just above the allowed boundaries.
- **Scenario**: A password must be between 8 and 12 characters.
- **BVA Test Cases**:
    - 7 characters (Invalid - just below)
    - 8 characters (Valid - exact minimum)
    - 12 characters (Valid - exact maximum)
    - 13 characters (Invalid - just above)

### 2. Equivalence Class Partitioning
Dividing all possible inputs into groups (partitions) where the system is expected to behave the exact same way for every value in that group. You only need to test *one* value from each partition.
- **Scenario**: Application accepts age from 18 to 65.
- **Partitions**:
    - Partition 1: `Age < 18` (Invalid) -> Test with `10`
    - Partition 2: `18 <= Age <= 65` (Valid) -> Test with `30`
    - Partition 3: `Age > 65` (Invalid) -> Test with `80`

### 3. Misuse and Abuse Case Testing
Deriving test cases directly from the Threat Models and Misuse/Abuse cases developed during Domain 3 and 4.
- **Use Case**: User updates profile.
- **Misuse Case**: User accidentally uploads a 500MB PDF instead of a profile picture. (Test: Verify file size limit and type validation).
- **Abuse Case**: Attacker uploads a `.php` web shell disguised as a `.jpg` to gain remote code execution. (Test: Verify content-type inspection and secure storage execution restrictions).

## State and Workflow Testing

Security tests must look beyond single input fields and test the state machine of the application.

- **Forced Browsing / URL Manipulation**: Attempting to access step 3 of a checkout process without completing steps 1 and 2.
- **Parameter Tampering**: Intercepting a request (using a proxy like Burp Suite) and changing hidden fields, prices, or user IDs before it reaches the server.
- **Race Conditions**: Firing concurrent requests simultaneously to see if the system issues the same promotion code twice or transfers funds faster than balances update (TOC/TOU).

---

## Exam Focus Points

1. **Positive Testing**: Validating normal, expected behavior with valid inputs.
2. **Negative Testing**: Validating the system fails securely when given invalid or malicious inputs. Crucial for security.
3. **Equivalence Partitioning**: Testing one representative value from a group of values that should all yield the same result.
4. **Boundary Value Analysis (BVA)**: Testing values immediately at the edges of acceptable limits (e.g., n-1, n, n+1).

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Positive Testing** | Testing with valid data to ensure expected functionality |
| **Negative Testing** | Testing with invalid data to ensure the system handles errors securely |
| **BVA** | Boundary Value Analysis; testing values immediately on, above, and below boundaries |
| **Equivalence Partitioning** | Dividing input domains into classes and testing one value per class |
| **Test Case** | A specific set of conditions and data used to verify a security requirement |
