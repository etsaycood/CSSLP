# 6.1 Develop Security Testing Strategy

## Learning Objectives

- Define the purpose and components of a security testing strategy
- Explain how risk drives testing scope and depth
- Differentiate between a test strategy and a test plan
- Identify standards and guidelines used to formulate a testing strategy

---

## What is a Security Testing Strategy?

A security testing strategy is a **high-level document** that defines the overall approach, objectives, and parameters for testing the software. It bridges the gap between organizational security policies and the actual execution of tests.

| Characteristic | Description |
|----------------|-------------|
| **Focus** | High-level (The "What" and "Why") |
| **Audience** | Management, Stakeholders, Project Managers |
| **Lifespan** | Long-term; often applies across multiple projects or the entire organization |
| **Origin** | Derived from organizational policies, risk appetite, and compliance requirements |

### Strategy vs. Plan

> **Exam Tip**: STRATEGY = High-level approach ("We will use automated SAST and manual pentesting"). PLAN = Specific execution details ("Jane will run Checkmarx on Tuesday at 10 AM").

---

## Components of a Testing Strategy

A comprehensive testing strategy encompasses several key elements:

### 1. Objectives
What is the testing trying to achieve? (e.g., "Ensure compliance with PCI DSS," or "Identify OWASP Top 10 vulnerabilities before release.")

### 2. Risk-Based Testing Approach
Testing coverage is determined by the **risk profile** of the application.
- **High Risk App** (e.g., Financial transactions): Requires SAST, DAST, IAST, manual code review, and third-party penetration testing.
- **Low Risk App** (e.g., Internal cafeteria menu): May only require automated SAST and DAST.

### 3. Standards and Guidelines
The strategy identifies which industry standards the testing will align with:
- **OWASP ASVS** (Application Security Verification Standard)
- **NIST SP 800-115** (Technical Guide to Information Security Testing and Assessment)
- **PTES** (Penetration Testing Execution Standard)
- **OSSTMM** (Open Source Security Testing Methodology Manual)

### 4. Environments
Defines the environments where testing is permitted. (e.g., "Destructive testing is only permitted in the Staging environment. No automated scanning against Production without explicitly scheduled maintenance windows.")

### 5. Automation Strategy
Defines the balance between manual and automated testing, and how testing tools integrate into the CI/CD pipeline (DevSecOps integration).

## Types of Security Testing

The strategy dictates *which combinations* of these tests to use:

| Test Type | Description |
|-----------|-------------|
| **White-Box Testing** | Testing with full knowledge of the internal architecture and source code (e.g., SAST, manual code review). |
| **Black-Box Testing** | Testing with no knowledge of internal workings, acting as an external attacker (e.g., DAST, Penetration Testing). |
| **Gray-Box Testing** | Testing with partial knowledge (e.g., possessing user credentials but not source code) (e.g., IAST, authenticated DAST). |
| **Fuzz Testing** | Providing invalid, unexpected, or random data to induce a crash or exception. |
| **Regression Testing** | Retesting previously fixed vulnerabilities to ensure new code changes haven't reintroduced them. |

---

## Exam Focus Points

1. **Strategy vs. Plan**: Strategy is high-level and long-term; Plan is specific, scheduled, and tactical.
2. **Risk-based testing**: Testing effort must be proportional to the application's risk profile.
3. **Black vs. White vs. Gray box**: Know what level of knowledge the tester possesses in each.
4. **Standards**: NIST 800-115 and OWASP ASVS are primary testing standards.

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Test Strategy** | High-level document defining the overall testing approach and objectives |
| **Risk-Based Testing** | Allocating testing resources based on the risk profile of the application |
| **White-Box Testing** | Testing with full knowledge of source code and architecture |
| **Black-Box Testing** | Testing with zero knowledge of internal system workings |
| **Regression Testing** | Retesting to ensure recent code changes haven't reintroduced old bugs |
| **ASVS** | OWASP Application Security Verification Standard |
