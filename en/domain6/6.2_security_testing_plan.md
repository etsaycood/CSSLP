# 6.2 Develop Security Testing Plan

## Learning Objectives

- Differentiate a test plan from a test strategy
- Identify the core components of a security test plan
- Define test scope, schedule, and resource allocation
- Explain the role of the Security Requirements Traceability Matrix (SRTM) in planning

---

## What is a Security Testing Plan?

A security testing plan is a **tactical, project-specific document** derived from the overall testing strategy. It details the **logic, methods, scheduling, and resources** required to execute the tests for a specific release or application.

| Characteristic | Description |
|----------------|-------------|
| **Focus** | Low-level, tactical (The "How", "Who", "When") |
| **Audience** | QA Teams, Security Testers, Developers |
| **Lifespan** | Short-term; specific to a project, sprint, or software release |
| **Origin** | Derived from the Test Strategy and the specific project requirements |

---

## Components of a Security Testing Plan

A robust test plan answers logistical questions. Key sections include:

### 1. Scope (In-Scope and Out-of-Scope)
Clearly defines exactly what will and will not be tested.
- **In-Scope**: "The main web application API, login portal, and database backend."
- **Out-of-Scope**: "Third-party payment gateway, staging environment infrastructure."
> **Security Note**: Clearly defining out-of-scope items is critical in penetration testing to prevent accidental disruption of interconnected or third-party systems.

### 2. Schedule and Logistics
- **Timelines**: When does testing begin and end?
- **Test Windows**: What time of day will destructive or load testing occur? (e.g., "Only between 2:00 AM and 4:00 AM EST").
- **Milestones**: Deadlines for initial scanning, manual testing, and final reporting.

### 3. Resource Allocation
- **Personnel**: Who is conducting the tests? (Internal QA, internal security team, or external third-party pentester).
- **Tools**: Exactly which tools will be used (e.g., "Burp Suite Pro for proxying, Nessus for infrastructure scanning").
- **Environments**: Which specific environment URL/IP will be tested.

### 4. Test Data Requirements
What data is needed to perform the tests, and how will it be secured?
- E.g., "QA will require 5 active test accounts with varying privilege levels (Admin, User, Read-Only)."

### 5. Entrance and Exit Criteria
- **Entrance Criteria**: What must be true before testing can begin? (e.g., "Code is feature complete, deployed to staging, and smoke tests pass.")
- **Exit Criteria**: What must be true for testing to conclude? (e.g., "All planned test cases executed, NO Critical or High vulnerabilities remain unresolved.")

### 6. Defect Management (Incident Response)
How are findings communicated during the test?
- Where are bugs logged? (e.g., Jira)
- What is the emergency contact procedure if a test accidentally brings down the staging server?

## The Role of the SRTM

The **Security Requirements Traceability Matrix (SRTM)** (developed in Domain 3) is a vital input to the Test Plan.
- The Test Plan uses the SRTM to ensure **Full Coverage**.
- For every Security Requirement listed in the SRTM, there must be at least one scheduled Test Case in the Test Plan to verify it.
- **Bi-directional Traceability**: Ensure tests map to requirements, and requirements map to tests.

---

## Approvals and Sign-off

Because security testing (especially DAST and Pentesting) can be disruptive, the Test Plan requires **formal sign-off** from stakeholders:
- **Project Manager / Asset Owner**: Acknowledges the scope and schedule.
- **IT / Operations**: Acknowledges potential performance impacts during testing windows.
- **Security Leadership**: Validates that the plan appropriately addresses the application's risk.

---

## Exam Focus Points

1. **Plan vs. Strategy**: Plan is specific to a project (Who, What, When, Where); Strategy is overall organizational approach.
2. **Scope boundary**: Defining what is explicitly *out-of-scope* is just as important as defining what is in-scope to prevent unauthorized testing.
3. **Entrance/Exit Criteria**: Strict conditions dictating when testing can start and when it is considered finished.
4. **SRTM**: Used by the test plan to ensure every requirement has a corresponding test (Coverage).

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Test Plan** | Tactical document detailing the schedule, scope, and resources for a specific testing engagement |
| **In-Scope** | Systems, features, or networks specifically authorized for testing |
| **Out-of-Scope** | Systems explicitly off-limits for testing |
| **Entrance Criteria** | Preconditions that must be met before testing can begin |
| **Exit Criteria** | Conditions that signify the completion of the testing phase |
| **SRTM** | Security Requirements Traceability Matrix |
