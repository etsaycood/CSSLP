# 6.8 Analyze and Report Test Results

## Learning Objectives

- Explain defect management and tracking
- Describe the process of analyzing test results for false positives
- Evaluate residual risk post-testing
- Define the components of a comprehensive security test report

---

## Defect Management and Analysis

Running security tests (SAST, DAST, Pentests) generates raw data. That data must be analyzed, prioritized, and converted into actionable work items for developers.

### 1. Analysis and Qualification
Automated tools produce noise. The first step for a security analyst is to qualify the findings:
- **False Positives**: The tool flagged a vulnerability, but upon human inspection, the issue does not exist or is mitigated by an environmental control (e.g., a WAF). These must be weeded out so developers are not overwhelmed.
- **True Positives**: The vulnerability is real and requires action.
- *False Negatives*: The most dangerous category; a vulnerability exists, but the testing tools/processes failed to find it.

### 2. Prioritization
Not all True Positives are equal. Findings must be prioritized based on risk context (likelihood x impact).
- **CVSS (Common Vulnerability Scoring System)**: Assigns a baseline severity (0.0 - 10.0).
- **Contextual Risk**: A CVSS 9.0 Cross-Site Scripting (XSS) flaw in an internal, HR-only admin portal may be prioritized *lower* than a CVSS 7.0 Business Logic bypass in the main public customer portal.

### 3. Defect Tracking
Vulnerability findings must be tracked in the **developer's native workflow system** (e.g., Jira, Azure Boards, GitHub Issues).
- Do not maintain a separate "security spreadsheet." If it is not in the developer's backlog, it will not get fixed.
- Track metrics: **MTTR** (Mean Time to Remediate) and defect density.

## Residual Risk Evaluation

When the testing phase concludes, Management must evaluate the **Residual Risk** before authorizing the software for deployment to Production.

$$ \text{Residual Risk} = \text{Total Risk} - \text{Mitigating Controls (Fixes Applied)} $$

### The Sign-off Decision
At the end of testing, one of the following scenarios exists:
1. **All flaws fixed**: Residual risk is acceptable. Proceed to deployment.
2. **Minor flaws remain**: Residual risk is acceptable. Proceed to deployment with the minor flaws added to the technical debt backlog for future sprints.
3. **Critical/High flaws remain**: Residual risk is **unacceptable**. Deployment is halted (the build breaks), *unless* the Business Owner/Executive formally accepts the risk in writing (Risk Acceptance).

> **Exam Tip**: The development team or the security tester *cannot* accept risk. Only Management (the Data/System Owner) holds the authority to accept residual risk and release flawed software.

## Security Test Reporting

A formal test report documents the validation efforts and provides Management with the information needed to make the deployment decision.

### Components of a Comprehensive Report

| Section | Target Audience | Description |
|---------|-----------------|-------------|
| **Executive Summary** | Management (CISO, Business Owners) | High-level overview of the security posture. Was it a pass/fail? What is the overall risk? Written in **non-technical business language**. |
| **Scope & Methodology** | Security Leadership, Auditors | What was tested, when it was tested, and with what tools/standards (e.g., OWASP ASVS). |
| **Findings Summary** | Project Managers, QA Leads | Aggregate metrics: How many Critical, High, Medium, Low bugs were found? How many were remediated during the test window? |
| **Detailed Findings** | Developers, Engineers | Technical details for each vulnerability: Description, URL/Line of Code, Proof of Concept (steps to reproduce), and Remediation instructions. |

### Feedback Loops
Reporting is not just about the current project; it provides a feedback loop to improve earlier SDLC phases:
- If SAST constantly finds SQL Injection, the feedback loop triggers additional secure coding training for developers (Domain 5).
- If architecture reviews constantly miss flawed authentication designs, the feedback loop improves Threat Modeling processes (Domain 4).

---

## Exam Focus Points

1. **False Positives**: Must be filtered out before sending reports to developers to maintain trust and prevent alert fatigue.
2. **Defect Tracking**: Security bugs must be tracked in the same system as functional bugs (e.g., Jira).
3. **Executive Summary**: Must be non-technical, focusing on business risk, tailored for management decision-making.
4. **Risk Acceptance**: Only authorized Management/Business Owners can formally sign off on accepting critical residual risk. Security analysts only advise.

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **False Positive** | A test result incorrectly indicating that a vulnerability is present |
| **False Negative** | A test failing to identify a vulnerability that actually exists |
| **Residual Risk** | The risk remaining after security controls and bug fixes have been applied |
| **MTTR** | Mean Time to Remediate (metric tracking how fast teams fix bugs) |
| **Risk Acceptance** | Formal management decision to accept the consequences of an unremediated vulnerability |
