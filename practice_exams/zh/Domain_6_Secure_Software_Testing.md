# 領域 6：安全的軟體測試 (Secure Software Testing) (18題)

#### **1. 一個資安測試小隊接獲命令去攻堅一個網頁應用程式。上級什麼都沒給，只發給他們一組跟普通平民一樣的普通使用者帳號。沒有原始碼、沒有建築架構設計圖。這群特種部隊必須完全從系統「外面」，試圖鑽進去。這群人是在搞什麼名堂的測試手法？**
A. 已知環境測試 (白箱測試 White-box testing)
B. 未知環境測試 (黑箱測試 Black-box testing)
C. 靜態應用程式安全測試 (SAST)
D. 程式碼涵蓋率分析 (Code coverage analysis)

*   **答案：B**
*   **解析：** 這就是 **未知環境測試 (黑箱測試 Black-box testing / 動態測試)**。這種類型的猛攻，極致模擬了一個來自網際網路、對系統骨架與原始瑪一無所知的真實路人駭客的視角。反之，**已知環境測試 (白箱測試 White-box testing)** 則像是給了測試者一整份的藏寶地圖。

#### **2. 一套全自動獵犬被裝到了持續整合 (CI/CD) 產線上。每當應用程式被部署並「跑起來」後，這隻獵犬就會開始對這台活生生、會呼吸的伺服器狂噴幾千個 HTTP 炸彈 (如 SQLi 或是 XSS 負載)，並分析伺服器回傳的哀嚎聲來判定系統漏洞。這套凶狠的工具被歸類為：**
A. 靜態應用程式安全測試 (SAST)
B. 動態應用程式安全測試 (Dynamic Application Security Testing, DAST)
C. 軟體組成分析 (SCA)
D. 單元測試 (Unit Testing)

*   **答案：B**
*   **解析：** **DAST (動態應用程式安全測試)** 就是黑箱測試的自動化加農砲。這項科技擅長在應用程式「已經在環境中跑起來 (Executing)」的狀態下，到處爬網頁，不斷扔出惡意負載炸彈去痛毆這套活著的系統。

#### **3. 品管 (QA) 團隊需要一個塞滿「看起來跟真的一樣的個資資料庫」來進行壓力測試。隱私法律絕對不允許正式環境真實客戶的個資直接拉到測試環境來玩。於是 QA 團隊架設了一套演算法，憑空生出了一個極度龐大、裡面的人物栩栩如生、關聯性完美的資料集；但這個資料集裡連一滴都不屬於真實人類。這偉大的行為稱為：**
A. 資料匿名化 (Data Anonymization)
B. 資料混淆處理 (Data Obfuscation)
C. 綜合/合成資料產生 (Synthetic Data Generation)
D. 代碼化 (Tokenization)

*   **答案：C**
*   **解析：** 創造奇蹟叫做 **生成測試資料 (Synthetic Data / 綜合資料)**。它的魔法在於從虛空中憑空計算重構出資料集，跟正式營運區裡的資料庫一模一樣完美；但重點是它裡面「連一滴滴真實世界的個資也沒有」，完美閃躲過了一切隱私法規。

#### **4. 一家公司定期把正式上線的資料庫 (Production Database) 複製下來丟到測試環境區 (Staging) 測試。這位有良心的 DBA 寫了一支腳本：在資料落地到測試區前，把裡面所有的真實客戶姓名、地址、身分證字號，全部用星號 $(*)$ 或是亂碼塗掉或替換。請問 DBA 正在做什麼？**
A. 資料混淆 / 淨化 (Data Obfuscation / Sanitization)
B. 綜合資料產生 (Synthetic Data Generation)
C. 錯誤灌注 (Fault Injection)
D. 資料庫正規化 (Database Normalization)

*   **答案：A**
*   **解析：** 綜合資料是從無到有憑空捏造的；而 **混淆 / 淨化 / 遮罩 (Obfuscation / Sanitization / Masking)** 則是拿了一塊真實正式生產資料，動刀去挖除或遮蓋那些致命的「敏感識別碼」，讓它勉強符合法律規範。

#### **5. 為了測試一款渲染伺服器到底能承受多大極限，工程師故意開始朝這台伺服器的核心運算引擎輸入一堆「不合法、徹底隨機產生、格式崩壞」的巨量資料。唯一目的是祈禱伺服器當機，或是榨出記憶體洩漏 (Memory Leak)。這套測試方法是什麼？**
A. 互動式應用程式安全測試 (IAST)
B. 模糊測試 (Fuzzing / Fuzz Testing)
C. 滲透測試 (Penetration Testing)
D. 整合測試 (Integration Testing)

*   **答案：B**
*   **解析：** **模糊測試 (Fuzzing)** 就是狂塞「無效的、或是完全隨機產生」的垃圾資料當作輸入值。其終極目標，就是盯著程式什麼時候會噴出例外錯誤、發生記憶體洩漏或崩潰，通常代表挖到了「緩衝區溢位 (Buffer Overflow)」。

#### **6. 硬體工程師必須測試：當這台植入人體的物聯網心律調節器的 Wi-Fi 突然斷線，或電池電壓突然暴走時，它到底會怎麼反應。為了精準重現這種災難，這位工程師最好請出：**
A. 壓力測試 (Stress testing)
B. 模糊測試 (Fuzzing)
C. 錯誤/故障注入 (Fault injection)
D. 滲透測試 (Penetration testing)

*   **答案：C**
*   **解析：** **故障注入 (Fault injection / 錯誤注入)** 指在系統裡「刻意搞破壞」：比如斷網路、抽乾記憶體、惡意干擾電壓。這是為了考驗系統對災難處理的容忍度，驗證是否會進入「失效安全 (Fail Secure)」保護狀態。

#### **7. 開發工程師寫了小腳本。鎖定身分驗證類別裡極度邊緣的一個小函式 (例如：算 `hashPassword()` 回傳的長度對不對)。只要在 IDE 裡一按下存檔，腳本就會觸發運算。這種流派叫做：**
A. 系統整合測試 (System integration testing)
B. 單元測試 (Unit testing)
C. 動態應用程式安全測試 (DAST)
D. 滲透測試 (Penetration testing)

*   **答案：B**
*   **解析：** **單元測試 (Unit testing)** 是開發者操刀的白箱測試。它將最微觀的細胞組件 (一個函式或者是一個類別)，從應用程式本體獨立抽離出來進行拷打，確保它完成微小任務。

#### **8. 品管 (QA) 小隊測完了薪資系統偉大的「2.0 版」！上線前夕，隊長下令：「把當年針對舊版 1.0 寫的那套自動化腳本搬出來！用它去把這個 2.0 重新狂轟一遍！」這個舉動目的是為了執行：**
A. 威脅建構建模 (Threat Modeling)
B. 模糊測試 (Fuzzing)
C. 整合測試 (Integration testing)
D. 迴歸測試 (Regression testing)

*   **答案：D**
*   **解析：** **迴歸測試 (Regression testing)** 就是怕「改了這個，壞了那個」。每當軟體經歷改變，都必須重新測試一次，保證這次變更沒把以前活得好好的舊程式碼搞出新漏洞來。

#### **9. 一場滲透測試後，紅軍甩了一本記載了 45 個漏洞的戰報。安全主管在其中 5 個漏洞旁邊寫下：「此漏洞就算存在，只要駭客沒拿到內部管理員金鑰，也是絕對無法被觸發的，結案。」這番神操作被稱為：**
A. 產生安全測試案例 (Generating security test cases)
B. 分析測試結果之安全影響 (Analyzing security implications of test results)
C. 驗證與核實官方文件 (Verifying and validating documentation)
D. 互動式應用程式安全測試 (IAST)

*   **答案：B**
*   **解析：** 不是每個紅色警報都需要立刻被修補。**分析測試結果之安全影響 (Analyzing security implications of test results)** 是從厚重報告中釐清輕重緩急，精算真實世界發生的機率風險，再來分配修補預算跟時間。

#### **10. 一名網路上不知名的資安研究員，參加了你們的「Bug Bounty (漏洞懸賞大賽)」，附帶程式碼證明他有辦法「繞過你們的多因素身分驗證 (MFA)」。公司趕緊發懸賞金封口，並在 Jira 追蹤系統上開緊急單命令開發大軍修復。這個劇本，是哪兩項實務工作的典範？**
A. 使用 CVSS 對漏洞風險進行評分
B. 揪出未經官方記錄的功能
C. 資安研究員的對外拓展 與 臭蟲追蹤 (Security researcher outreach and bug tracking)
D. 密碼學亂數的熵度驗證

*   **答案：C**
*   **解析：** 營運 **漏洞狩獵懸賞 (Bug Bounty program)**，就是你在幹 **資安研究員對外拓展 (Security researcher outreach)** 外交活。吸引白帽駭客免費當打工仔幫你找漏洞。這群人找出的缺陷必須扔進正式的 **臭蟲追蹤 (Bug tracking)** 系統如 Jira 中被管理。

#### **11. 資安分析師想在不見盡頭的漏洞清單中排出修補優先順序。他拿著業界秘笈，根據「攻擊難度門檻」、「攻擊者特權」及「對機密、完整與可用性的傷害」來替每個漏洞算出一個 0.0 到 10.0 的精確小數。這位老兄使用的是：**
A. SLSA (軟體產物供應鏈級別)
B. OWASP Top 10
C. CVSS (通用漏洞評分系統 Common Vulnerability Scoring System)
D. STRIDE

*   **答案：C**
*   **解析：** **CVSS (通用漏洞評分系統)** 是稱霸全世界、用來幫軟體漏洞進行量化分類與嚴峻程度評分的絕對標準。

#### **12. 建置產線主管在管線上動了手腳：只要 SAST 大砲打中了一個 CVSS 分數大於或等於 7.0 的漏洞，管線就會立刻中斷，程式碼別想跨進預備環境 (Staging)。這套規則是實踐何種精神的典範？**
A. 滲透測試 (Penetration testing)
B. 實施 中斷/建置 門檻關卡 (Implementing Break/Build criteria)
C. 官方文件的核實驗證 (Verifying documentation)
D. 創造誤用與濫用的測試案例

*   **答案：B**
*   **解析：** 建立 **中斷與建置標準 (Break/Build criteria)** 是在軟體交付設下的安檢關卡。只要違背鐵律 (不准攜帶高等級漏洞)，閘門會立刻從天而降把管線截斷 ("breaks the build")。

#### **13. 在逆向工程後，怪胎工程師挖出這支商用軟體居然在一個隱密遠端連線埠上，開了一個專供「管理員進行深層神經網路除錯」的祕密後門。官方架構圖、使用手冊連個標點都沒提到。踢爆這個天大秘密完成了什麼目標？**
A. 驗證密碼學亂數產生器
B. 對邊界值進行模糊測試
C. 揪出未被官方記載的神祕暗黑功能 (Identifying undocumented functionality)
D. 針對展示層進行迴歸測試

*   **答案：C**
*   **解析：** 漏洞滲透最重要的任務之一，就是挖掘那些 **未具備文件紀錄的隱藏功能 (Identifying undocumented functionality)**。這些寄生蟲通常是下班忘記關的除錯隧道 (Debugging Portals)、或測試用後門。

#### **14. 菜鳥測試員寫自動化腳本：試圖用極為無腦的密碼「123」註冊新帳號。腳本期望得到：「系統必須冷酷地把請求踢飛並噴錯。」這測試神功是：**
A. 誤用 / 濫用 的測試案例逆襲
B. 功能性安全測試 (針對安全邏輯) (Functional security testing - Logic)
C. 非功能性安全測試
D. 綜合體的虛擬沙盤推演

*   **答案：B**
*   **解析：** 當你的測試目的，純粹只是想驗證「某一個針對資安所蓋出來的防護設施是否有照著原本設計去運作攔阻」，這就是專注於測試你所宣稱安全能力的 **功能性安全測試 (Functional security testing)**。

#### **15. 資安總管要求血滴子證據，證明產出加密初始向量 (IVs) 的隨機亂數真的是出自無法預測的深淵數學模型。測試團隊必須開啟哪一扇大門來應付？**
A. 瘋狂模糊測試 (Fuzzing)
B. 對 API 進行模糊測試
C. 密碼學數學大驗證 (Cryptographic validation - Entropy testing)
D. 朝防火牆發射滲透測試飛彈

*   **答案：C**
*   **解析：** **密碼學驗證 (Cryptographic validation)** 是一場極度考驗數學信仰的試煉。它會瘋狂解剖隱藏的偽隨機數產生器 (PRNGs)，確保吐出來的那些數字具備連神都無法參透的絕對數學 **熵數 (Entropy)**。

#### **16. UAT 上線前，工程師把一套代理人 (Agent) 神祕測試套件，死死綁定並潛伏在應用程式的 JVM 裡面。無聲無息地監視主記憶體與執行的程式碼，自己在地圖上精準告訴你哪一行的 Code 正在漏血。這黑科技叫做：**
A. 靜態應用程式安全測試 (SAST)
B. 動態應用程式安全測試 (DAST)
C. 互動式應用程式安全測試 (Interactive Application Security Testing, IAST)
D. 硬體輔助威脅模擬

*   **答案：C**
*   **解析：** **IAST (互動式應用程式安全測試)** 是「灰箱測試 (grey-box)」巨獸。它會把探針種進受測軟體執行的皮層裡面。當軟體被測試敲打時，IAST 就冷冷地看穿任何不正常的資料內流，高傲地指出漏洞在確切的「第幾行」。

#### **17. 應用程式準備就緒。主管派人把「五彩繽紛安裝手冊」、「快速入門指南」拿放大鏡一個字一個字去盯！確保公開廢報紙裡絕對沒有洩漏網路拓樸大秘密或忘記把「大明碼：admin/password」初始密碼塗掉。這項校稿工作隸屬於：**
A. 原始碼大卸八塊分析
B. 官方公文與文件的徹底核驗與驗證大法 (Verifying and validating documentation)
C. 捏造誤用案例
D. 威脅地圖紙上談兵

*   **答案：B**
*   **解析：** 對手駭客在攻擊前最愛從公開的教學手冊肉搜。**核驗及驗證文件資料 (Verifying and validating documentation)** 的存在就是在傻瓜說明書出門害人之前，確保它們早已被徹底淨化。

#### **18. 為了證明這神仙應用程式能在那場幾百萬人瘋狂的購物血祭中屹立不搖。QA 決定向雲端借兵，十萬個殭屍用戶軍隊去同時衝撞伺服器。這個宏大測試歷史上會歸檔在：**
A. 探討商業邏輯的生死鬥
B. 單元測試
C. 非功能性安全測試 (Nonfunctional security testing - Performance / Scalability / Reliability)
D. 在已知白箱裡演練滲透戰

*   **答案：C**
*   **解析：** 考驗這軟體在極限流量湧入的地獄絕境中，它的可靠度彈性極限。這種考驗被冠上學名：**非功能性安全測試 (Nonfunctional security testing)**。它只深切拷問這個軟體在 **效能、可靠度與不可崩壞的極限吞吐規模 (scalability)**，也就是考驗(A)可用性。
