# 領域 5：安全的軟體實作 (Secure Software Implementation) (17題)

#### **1. 一名開發人員依賴外部框架來強制執行授權規則，他只是在一個敏感的 API 函式上方加上了 `@RolesAllowed("ADMIN")` 這樣的標註 (Annotation)，而不是在函式內部親自撰寫各種 `if-else` 的權限檢查程式碼。這是哪一種安全方法的實踐？**
A. 指令式安全 (Imperative security)
B. 宣告式安全 (Declarative security)
C. 程式化安全 (Programmatic security)
D. 並行管理 (Concurrency management)

*   **答案：B**
*   **解析：** **宣告式安全 (Declarative security)** 依賴底層框架或容器來強制執行安全政策 (通常透過標註 / Annotations 或 XML 設定檔)，而無須將資安邏輯與業務邏輯混雜在一起。相反地，**指令式 (程式化) 安全 (Imperative / Programmatic security)** 則是開發人員必須親自在函式內一行行撰寫自訂的控制流程 (例如：`if (user.role == "ADMIN")`)。

#### **2. 下列哪一種漏洞是「未能正確執行輸出淨化 (Output Sanitization / Encoding)」所導致的最直接結果？**
A. SQL 注入 (SQL Injection)
B. 跨網站指令碼 (Cross-Site Scripting, XSS)
C. 緩衝區溢位 (Buffer Overflow)
D. 不安全的直接物件參考 (IDOR)

*   **答案：B**
*   **解析：** 當應用程式將不受信任的資料直接包含在網頁中，卻沒有進行適當的驗證或跳脫處理 (即輸出淨化或前後文編碼) 時，就會發生 **跨網站指令碼 (XSS)**。如果不對輸出進行編碼，受害者的瀏覽器就會把那些被駭客注入的惡意 JavaScript 當成正常指令來執行。SQL 注入是輸入驗證的問題，而 XSS 從根本上來說是輸出編碼的失敗。

#### **3. 一個開發團隊正在修復一個嚴重的 Bug：當兩個不同的用戶在同一毫秒搶購最後一張門票時，會導致超賣的慘劇。開發人員在資料庫實作了鎖定 (Locking) 機制，以確保這些交易會被乖乖排隊並循序處理。這項緩解措施解決了與下列何者相關的資安風險？**
A. 並行處理 (Concurrency / 執行緒安全 / 競態條件)
B. 輸出混淆 (Output Obfuscation)
C. 微架構推測執行 (Microarchitecture speculative execution)
D. 沙箱隔離 (Sandboxing)

*   **答案：A**
*   **解析：** **並行 (Concurrency)** 相關的問題 (像是競態條件 Race Conditions 或檢查時與使用時 TOCTOU 漏洞)，發生在多個非同步執行緒或行程「同時」存取共享資源，卻缺乏適當的同步機制 (例如：資料庫鎖定或執行緒安全機制) 的時候。這種混亂狀態會導致未經授權的修改或嚴重的邏輯錯誤 (例如把 100 塊錢重複花了兩次)。

#### **4. 一個應用程式被設計用來捕捉致命的資料庫查詢錯誤。但它並沒有把赤裸裸的 ODBC連線字串與SQL語法錯誤直接呈現在消費者的網頁螢幕上，而是把詳細的錯誤日誌默默寫進內部硬碟裡，然後在畫面上只顯示一個優雅、籠統的「發生未預期的錯誤，請稍後再試」訊息給用戶看。這項安全編碼實務阻止了哪一種特定的威脅？**
A. 跨網域請求偽造 (CSRF)
B. 權限提升 (Elevation of Privilege)
C. 資訊洩露 (Information Disclosure)
D. 否認 (Repudiation)

*   **答案：C**
*   **解析：** 妥善的 **錯誤與例外處理 (Error and Exception Handling)** 能夠防止 **資訊洩露 (Information Disclosure)**。將赤裸裸的堆疊追蹤 (Stack traces)、資料庫綱要細節或連線字串暴露給終端用戶，等同於免費發送極具價值的偵察情報給駭客，讓他們能輕鬆描繪系統的架構並打造精準致命的漏洞利用程式。

#### **5. 為了保護資料庫中極度敏感的個人身分資訊 PII，一個組織從數學層面上，將這些敏感資料替換成了一組隨機產生、能對應回主鍵的「代理值」。如果這個資料庫整包被偷走，這些代理值跟原始真實資料之間，存在著絕對「零」的數學關聯。這項處理技術稱為什麼？**
A. 加密 (Encryption)
B. 雜湊 (Hashing)
C. 代碼化 (Tokenization)
D. 混淆 (Obfuscation)

*   **答案：C**
*   **解析：** **代碼化 (Tokenization)** 使用不具敏感性的代替品 (代碼/Tokens) 來替換敏感資料。與依賴數學公式和金鑰來轉換資料的「加密」截然不同，代碼化是把真實的敏感資料鎖在實體隔離的「保險庫」裡，然後只把毫無意義的代碼交給下游系統去傳遞使用。它徹底從數學上斬斷了這些代碼與真實資料之間的聯結。

#### **6. 一名資安工程師在評估一個應用程式的 Session (工作階段/連線階段) 管理實作。當一名用戶輸入正確的帳號和密碼、成功通過身分驗證的「那一瞬間」，關於這名用戶手上的 Session ID，應用程式最必須做出的關鍵致命動作是什麼？**
A. 在傳輸過程中使用 AES-256 將現有的 Session ID 加密起來。
B. 強制註銷作廢舊的，並立刻產生並核發一組全新的、具備密碼學隨機性的 Session ID 給他。
C. 把他現有的 Session ID 存入一個安全的資料庫資料表裡面。
D. 將這名用戶的密碼雜湊值偷偷塞進這個 Session ID 後面來增加安全性。

*   **答案：B**
*   **解析：** 一旦使用者成功通過驗證登入，應用程式必須強制執行 **防止連線階段固定 (Session Fixation)** 的保護措施：將登入前核發的那張預設 Session ID 直接作廢，並立刻換發一張 **全新的、極度隨機的 Session ID**。

#### **7. 一個軟體開發團隊決定直接導入一套世界知名的密碼學函式庫 (例如：OpenSSL)，而不是自己白手起家去寫一套客製化的 AES 加密程式碼。當他們在享受「重複使用第三方開源程式碼」所帶來的好處時，下列哪一項是他們終其一生都「必須」積極去評估並緩解的連帶夢魘風險？**
A. 使用者介面到底有沒有符合心理可接受度。
B. 遞移依賴的連鎖漏洞 (Transitive vulnerabilities) 以及必須導入持續不斷的軟體組成分析 (SCA)。
C. 確保這個第三方圖書館採用了「程式化」而不是「宣告式」的安全機制。
D. 確保這個密碼學圖書館還順便防堵了 SQL 注入攻擊。

*   **答案：B**
*   **解析：** 業界建議 **安全地重複使用現成程式碼**。但引進第三方函式庫就等於繼承了它肚子裡可能藏有的所有漏洞。要馴服這頭怪獸，組織必須建立 SBOM 軟體物料清單，並動用 **軟體組成分析 (Software Composition Analysis, SCA)** 工具來追蹤依賴項的漏洞。

#### **8. 駭客從外面往你的軟體系統上傳了一個偽裝成正常大小的 ZIP 壓縮檔。結果裡面裝的竟然是惡名昭彰的「拉鍊炸彈 (Zip Bomb)」。為了防堵它，開發人員在系統執行解壓縮程序的當下，實施了極為嚴苛且無情的「檔案大小膨脹上限限制」。這項防禦措施屬於下列哪一種分類？**
A. 沙箱隔離 (Sandboxing)
B. 啟動編譯器參數 (Compiler switches)
C. 資源管理 (Resource management)
D. 靜態應用程式安全測試 (SAST)

*   **答案：C**
*   **解析：** 「Zip Bomb」的目的就是耗盡硬碟或撐爆記憶體來達成阻斷服務攻擊 (DoS)。嚴格限制 CPU、記憶體、與儲存空間的分配上限，是落實安全 **資源管理 (Resource management)** 的核心面向。

#### **9. 開發人員的 IDE 開發環境裡被強制綁定了一套全自動掃描工具。這套工具不用花時間把程式給跑起來，只是冷冷地盯著那些靜態原始碼猛看，就能精準地在畫面上標出漏洞。這套神兵利器被歸類為：**
A. 動態應用程式安全測試 (DAST)
B. 靜態應用程式安全測試 (SAST)
C. 互動式應用程式安全測試 (IAST)
D. 軟體組成分析 (SCA)

*   **答案：B**
*   **解析：** **SAST (靜態應用程式安全測試)** 屬於白箱測試方法。它直接去分析最赤裸的原始碼，在程式「完全沒有被執行」的狀態下精準提供漏洞所在的確切「行號」。

#### **10. 攻擊者發現只要在輸入框裡瘋狂塞滿亂碼，就能逼得伺服器消耗巨大資源當機。為了防禦這種爛招，開發人員決定祭出「允許清單 (Allow-list)」驗證。請問這份清單是怎麼殺敵的？**
A. 它會封鎖那些已知很危險的黑名單字元，然後放行其他剩下的所有東西。
B. 它極度獨裁地定義了「只有符合條件才能過關」，然後把不符合的「預設全部擋下」。
C. 它是把使用者的輸入變成代碼，藉此確保處理器無法執行它。
D. 它靠門口的 WAF 防火牆來檢查封包。

*   **答案：B**
*   **解析：** **允許清單 (Accept / Allow list)** 死板且嚴格地定義出系統所預期的、絕對安全的輸入條件。只要有不符的東西出現，預設就是全部拒絕 (Default Deny)。這種獨裁的作法勝過試圖過濾「已知壞東西」的封鎖清單 (Block / Deny list)。

#### **11. 一支寫 C/C++ 的開發團隊發現應用程式存在緩衝區溢位漏洞。為了爭取時間，資安工程師下令：「先用開啟了 ASLR 與 DEP 旗標的模式，重新編譯一次送上線！」這些旗標屬於：**
A. 編譯器參數 (Compiler switches)
B. 靜態程式碼分析器的 Linting 規則
C. 微架構處理器安全擴展指令
D. 沙箱隔離環境

*   **答案：A**
*   **解析：** 現代編譯器內建了防護機制 (如 ASLR、DEP/NX 位元等)。開啟這些 **編譯器參數 (Compiler switches)**，能在不改 code 的前提下，強制把編譯出來的二進位執行檔加上防護罩。

#### **12. 應用程式從記憶體不小心讀取到另一個處理程序 (Process) 先前的機密殘留資料。在硬體殿堂上，到底都是靠什麼防止這種記憶體走漏事件？**
A. 執行緒安全鎖定 (Thread safety locking)
B. 隔離邊界與微架構處理器安全擴展指令 (Isolation boundaries and Processor microarchitecture security extensions)
C. 強制存取控制清單 (MAC lists)
D. 看門狗計時器 (Watchdog timers)

*   **答案：B**
*   **解析：** 為了防止 Processes 在記憶體中偷窺 (如 Spectre/Meltdown 漏洞)，軟體世界依賴作業系統級別的 **隔離** 法寶，並強烈依賴實體矽晶片上的 **微架構處理器安全擴展指令** 來堵死快取記憶體旁道洩漏。

#### **13. 在對核心演算法進行審查時，公司規定這段 Code 要被合併前，必須由另一位資深老鳥工程師親自看過並點頭才能過關。因為人工審查：**
A. 執行起來比機器還要快、還要便宜。
B. 能更新 OWASP Top 10 資安弱點庫。
C. 擁有能揪出「機器看不懂」的複雜商業邏輯缺陷、甚至是惡意暗門的超群能力。
D. 能提供高達 100% 的程式碼執行涵蓋率。

*   **答案：C**
*   **解析：** SAST 掃描器抓語法錯誤一把罩，但缺乏商業邏輯概念。**人肉程式碼審查 (Manual Code Review / Peer Review)** 才能從錯綜複雜的流程中，揪出邏輯上不合理的地方或開發者蓄意埋伏的「後門」。

#### **14. 祖傳應用程式裡，這個檔案的「擁有者(創建人)」可以隨自己的心情，愛把讀寫權限拔掉或派給任何人都可以。這種管理權力下放的模型是：**
A. 角色基礎存取控制 (RBAC)
B. 強制存取控制 (MAC)
C. 自由選擇存取控制 (Discretionary Access Control, DAC)
D. 屬性基礎存取控制 (ABAC)

*   **答案：C**
*   **解析：** 這是 **自由選擇/自主存取控制 (Discretionary Access Control, DAC)**。它最迷人的特色是：決定誰能看這份檔案的偉大權力，完全交給了該物件的「擁有者 (Owner)」。管理員也管不到你怎麼分享。

#### **15. 為了貫徹機密性決定加密最敏感的資料庫。但要求：萬一這套演算法被破解，系統要有能力在「不改寫底層應用程式原始碼」的前提下，以迅雷不及掩耳的速度抽換全新加密演算法。這種原則被稱作：**
A. 密碼學敏捷性 (Cryptographic Agility)
B. 加鹽與金鑰延展 (Salting and Key Stretching)
C. 徹底中介 (Complete Mediation)
D. 密碼學驗證 (Cryptographic Validation)

*   **答案：A**
*   **解析：** **密碼學敏捷性 (Cryptographic Agility)** 要求系統在設計初期就具備「換腦能力」，能在現役演算法死亡時極快切換，不需要經歷架構重構的災難。

#### **16. 在一場PCI查帳中，稽核員發現應用程式居然把客人輸入的完整、未經遮掩的「純文字信用卡號」老實地寫進了稽核日誌裡！這是一宗什麼級別的災難？**
A. 輸出淨化 (Output sanitization) 失敗
B. 效能與資源管理崩壞
C. 安全日誌與稽核紀錄的「機密性與隱私」被嚴重踐踏
D. 並行處理錯誤

*   **答案：C**
*   **解析：** 寫下 **安全的日誌 (Secure logging)** 是門藝術。必須像殺手一樣，將敏感設定、登入憑證與個資信用卡號 (PII/payment data) 徹底清洗與遮罩。原封不動塞進 Log 會把日誌變成摧毀機密性的核彈。

#### **17. 為了向測試團隊證明「這個二進位執行檔確確實實是從授權編譯伺服器生出來的，且沒有被竄改過」，這個自動化產線必須在這個檔案上烙下什麼印記？**
A. 程式碼混淆器 (Obfuscation)
B. 程式碼簽章 - 數位簽章與雜湊 (Code Signing / Digital Signatures and Hashes)
C. 位址空間配置隨機化 (ASLR)
D. 看門狗計時器 (Watchdog Timers)

*   **答案：B**
*   **解析：** 套用 **程式碼簽章 (Code Signing)** 就是給予這個產物完整性與血統 (Provenance) 保證。只要拿著「公開金鑰」檢驗檔案簽章與雜湊值，就能拆穿竄改詭計。
