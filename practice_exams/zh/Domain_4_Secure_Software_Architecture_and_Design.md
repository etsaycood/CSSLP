# 領域 4：安全的軟體架構與設計 (Secure Software Architecture and Design) (19題)

#### **1. 一位資安架構師決定將一個龐大的單體式 (monolithic) 應用程式切塊，分割成數個極小、可獨立部署、每個都擁有自己專屬資料庫的輕量級服務，並讓它們透過 HTTP APIs 互相溝通。這位架構師採用了哪一種架構模式？**
A. 主從式架構 (Client-Server)
B. 點對點 (Peer-to-Peer, P2P)
C. 微服務架構 (Microservices / Service-Oriented Architecture SOA)
D. N-tier (多層次架構)

*   **答案：C**
*   **解析：** 將一個大應用程式按照業務領域 (business domains) 切割成許多微小且獨立的服務，並透過網路 APIs 進行通訊，就是定義了 **微服務 (Microservices)** 架構 (現代 SOA 的一種實踐)。這種模式雖提供了驚人的開發敏捷性，但也引入了極其龐大的攻擊面 (成百上千個 API 端點)，每一個端點現在都必須被獨立認證與授權。

#### **2. 一支工程團隊正在開發一款移動端 APP，該 APP 會將暫存的極機密用戶資料，小心翼翼地儲存在手機裝置內部一顆「物理上完全隔離的特殊矽晶片」中；此舉是為了防禦即使作業系統整個被勒索軟體給黑掉，資料也無法被提取。這支團隊正在運用哪一種硬體平台的緩解措施？**
A. 安全元件 / 可信賴平台模組 (Secure Element / TPM)
B. 旁道攻擊緩解 (Side-channel mitigation)
C. 韌體驅動虛擬記憶體 (Firmware driver virtual memory)
D. 推測執行緩解 (Speculative execution mitigation)

*   **答案：A**
*   **解析：** **安全元件 (Secure Element)** (例如手機裡的蘋果 Secure Enclave) 或是主機板上的 **可信賴平台模組 (TPM)**，是一顆實體上被徹底隔離出來的、獨立運作的密碼學協同處理器；它存在的目的就是為了極其安全地產生、儲存與運算密碼學金鑰與最高敏感度資料，徹底免疫來自 OS 作業系統層級的全面淪陷。

#### **3. 一名架構師正使用 STRIDE 威脅建構模型來找出應用程式設計上的缺陷。他發現了一個駭人的漏洞：用戶在把商品放進購物車並結帳時，竟然可以在網路傳輸中介途中「攔截並竄改 HTTP 請求」，藉此把極度昂貴的商品價格全部改成 1 元。這個漏洞完美對應到了 STRIDE 中的哪一個恐怖字母？**
A. 欺騙 (Spoofing)
B. 竄改 (Tampering)
C. 否認 (Repudiation)
D. 資訊洩露 (Information Disclosure)

*   **答案：B**
*   **解析：** **竄改 (Tampering)** 指的是一種惡意的行為：不管是在「傳輸中 (In transit)」(正如這題的 HTTP 請求) 或是在「靜止狀態 (At rest)」，未經授權地去修改、操縱或破壞資料的完整性。

#### **4. 一個原生的雲端應用程式被設計得極度「具有彈性 (Elastic)」：當流量尖峰湧入時，後端伺服器會自動且瞬間擴容到 100 台；而在深夜無人時則會自動縮容到只剩 2 台。為了在極度動態的環境中確保安全的架構，架構師下達死命令：伺服器絕對不允許人工登入配置，必須全自動透過極度嚴謹且經審核的腳本產生。這正在利用哪一項神經網路級別的現代技術？**
A. 無所不在的運算 (Pervasive computing)
B. 基礎設施即程式碼 (Infrastructure as Code, IaC)
C. 認知運算 (Cognitive computing)
D. 內容傳遞網路 (Content Delivery Networks, CDN)

*   **答案：B**
*   **解析：** **基礎設施即程式碼 (IaC)** 拋棄了傳統的人工門戶設定與點擊，改用配置腳本 (設定檔) 來實現「自動化、可重複且絕對安全」的基礎設施 (如虛擬機器、容器、防火牆網路規則) 佈建。唯有依靠 IaC，雲端架構才有可能安全地擴張與收縮。

#### **5. 在一個沒有中央伺服器 (Serverless) 的分散式點對點 (P2P) 網路架構中，最讓人頭痛的架構資安難題，就是如何保證一個節點傳給另一個節點的資料「沒有被惡意變造過」。下列哪一種密碼學機制，是建立這份信任「最決定性的關鍵」？**
A. 透過 AES-256 進行對稱式加密 (Symmetric encryption)
B. 雜湊與數位簽章 (Hashing and digital signatures / 不可否認性)
C. 資料庫觸發程序與視圖 (Database triggers and views)
D. 螢幕佈局與 UI 遮罩 (Screen layouts and UI masking)

*   **答案：B**
*   **解析：** 在一個缺乏中心化信任機構、彼此互不信任的分散式 P2P 網路中，要保證資料的完整性 (Integrity) 並強硬核實資料的原始發送者身分，唯有依賴強大的 **雜湊 (Hashing) 與非對稱式的數位簽章 (Digital Signatures)**。對稱式加密雖然能對外保密，卻無法原生解決分散式節點間互生疑鬼的「不可否認性 (Non-repudiation)」問題。

#### **6. 一群資安工程師對他們的新架構進行了一場血腥的兵棋推演，藉以模擬「進階持續性威脅 (APT) 國家級駭客」的攻擊。這個團隊所採用的方法論，不只死盯著軟體元件，更無所不用其極地去分析、揣摩「對手 (駭客) 真正的動機、手上的武器庫，以及他們渴望實現的最終目標」。他們有極大可能是在使用哪一套威脅建構方法論？**
A. 通用漏洞評分系統 (CVSS)
B. STRIDE
C. 攻擊模擬與威脅分析流程 (Process for Attack Simulation and Threat Analysis, PASTA)
D. 基礎設施即程式碼 (IaC)

*   **答案：C**
*   **解析：** **PASTA** 是一套以「攻擊者 (駭客)」為極端中心的、基於風險的威脅建構方法論。它總共七個步驟的主旨，就是要將組織的業務目標與技術細節對齊，並深深陷入攻擊者的視角，瘋狂進行「攻擊模擬 (Attack Simulation)」來挖掘連設計師自己都想不到的恐怖威脅。

#### **7. 一位工程師設計了一套系統，該應用程式目前有一個棘手的需求：它必須在「絕對不能讓使用者交出他的超級密碼」的前提下，暫時授權一個外部的第三方服務，讓其擁有有限度的權限去動用該使用者的帳號資源。針對這種極度敏感的介面，下列哪一種協定是「唯一絕佳的架構選擇」？**
A. 安全性聲明標記語言 (SAML)
B. OAuth 2.0
C. 輕量級目錄存取協定 (LDAP)
D. Kerberos

*   **答案：B**
*   **解析：** **OAuth 2.0** 是一個授權 (Authorization) 框架，它打從骨子裡被設計出來的目的，就是為了允許第三方應用程式 (Client) 能夠代表資源擁有者 (User)，向擁有資源的伺服器索取有限的存取權 (透過發放 Token)，而在這整個過程中，資源擁有者的「帳號密碼」被保護得死死的，絕不會被暴露給第三方。

#### **8. 企業架構委員會下達了一項軍令：新的架構模式必須能讓所有的 B2B 外部合作夥伴，用他們「自己公司內部的」帳號密碼系統進行登入，最後卻能無縫接軌並暢行無阻地登入我們組織內的 Web 應用程式。這種充滿魔力的設計，強烈依賴必須實作：**
A. 自由選擇存取控制 (DAC)
B. 強制存取控制 (MAC)
C. 聯合身分管理系統 (Federated Identity)
D. 推測執行 (Speculative Execution)

*   **答案：C**
*   **解析：** **聯合身分 (Federated Identity / Federation)** 就是在多個不同組織的「身分孤島」之間，搭起了一座雄偉的「信任橋樑」。這讓身處甲網域的用戶，能拿著甲網域核發的憑證通行證，直接光明正大地去敲開乙網域資源的大門 (業界最常使用 SAML 或 OIDC 協定來實現這項奇蹟)。

#### **9. 架構師正盯著一個全新 Web 應用程式的「攻擊面 (Attack Surface)」猛看、直冒冷汗。該應用程式竟然大剌剌地把 15 個根本沒有認證機制的 REST API 端點暴露在公網上，還使用了 4 套來路不明、尚未審查的開源函式庫，並且允許用戶用兩種不同的途徑上傳檔案。請問執行「攻擊面評估 (Attack Surface Evaluation)」這項苦差事的終極目標究竟為何？**
A. 為了刻意增加 API 端點的數量，用來打造迷宮癱瘓駭客。
B. 列出、看透駭客可能利用來鑽進與鑽出的每一個出入口，並動用一切手段無情地「最小化」這些破口。
C. 為了精準算出這支 Web APP 那高到破表的 CVSS 底分。
D. 強制作為在威脅偵測中導入認知運算 (AI) 的科學根據。

*   **答案：B**
*   **解析：** **攻擊面評估 (Attack Surface Evaluation)** 不留情面地記錄下駭客能與系統進行互動的「所有可能管道」(包括裸奔的 APIs、開著的連接埠、輸入表單介面、甚至是第三方套件)。其最崇高的架構目標就是「削減 (Reducing) 或最小化」這個表面積；門開得越少，駭客得逞的機率就越低，整體的架構風險也會隨之崩跌。

#### **10. 一套物聯網 (IoT) 架構極度依賴部署在荒郊野外的數百顆無線感測器。為了堅決防止有人偷拔機器進行物理破壞並竄改作業系統軟體，架構師使出殺手鐧：硬體在通電喚醒後，在將控制權交給作業系統核心 (Kernel) 之前，擁有絕對執行順位，並且必須強硬驗證「數位簽章」是否合法。這種死守硬體底線的防護罩被稱為：**
A. 旁道攻擊緩解 (Side-channel mitigation)
B. 安全啟動 (Secure Boot)
C. 頻外管理 (Out-of-band management)
D. 持續整合 (Continuous Integration)

*   **答案：B**
*   **解析：** 開啟電源的瞬間最危險！**安全啟動 (Secure Boot)** 這個神聖的機制確保了：設備在加電啟動的過程中，只願意吞下並執行那些來自「被信任方」、擁有合法簽名的韌體層與作業系統載入程式；這從物理上斷絕了各種變態的 Rootkit 惡意程式或被竄改的髒韌體，試圖在無人看管的 IoT 設備上生根發芽的可能。

#### **11. 一個古早味的單體式巨型架構，歷經千辛萬苦被重構為優雅的「多層次 (N-tier) 架構」，包含了：展示層 (網頁前端)、業務邏輯層 (API 後端) 以及資料存取層 (資料庫)。在這張架構設計圖中，最血腥、最嚴苛的「輸入驗證大屠殺」應該被部署在哪裡？**
A. 單憑展示層 (網頁客戶端) 那華麗的驗證就非常足夠了。
B. 所有的驗證工作，都只讓資料存取層的守衛來扛。
C. 展示層可以有驗證來提升使用者體驗，但最殘酷的驗證必須重兵集結在業務邏輯層 (伺服器端)，並且在此把所有上游傳來的資料視為「致命的毒物」不可信任。
D. 多層次架構根本不需要輸入驗證這種過時的東西。

*   **答案：C**
*   **解析：** 在一個 N-tier 的完美設計中，雖然前端客戶端 (展示層) 上的驗證能讓使用者操作起來很流暢並節省不必要的來回連線，但前端驗證防君子不防小人，駭客敲幾個指令就能直接叫用 API 把它給狠狠繞開！因此，**伺服器端的驗證 (Server-side validation，駐守在業務邏輯層的紅線)** 才是攸關生死的護城河；它絕對、永遠不可把上游丟過來的任何一個位元組視為理所當然的「清潔安全」。

#### **12. 一名資安架構師在審視一張存放極機密個資 (PII) 的資料庫唯讀架構圖時，立刻暴跳如雷。他下達鐵腕命令：必須強制替這個資料庫實作隔離的「視圖 (Views)」，絕對禁止任何應用程式直接去裸查最底層的「基礎資料表 (Base Tables)」。這項針對資料庫的獨裁安全控制政策，本質上就是在擁護哪一項最高安全原則？**
A. 需知原則 / 最小權限 (Need-to-Know / Least Privilege)
B. 不可否認性 (Non-repudiation)
C. 開放式設計 (Open Design)
D. 預設失效安全 (Fail Secure)

*   **答案：A**
*   **解析：** 資料庫的 **視圖 (Views)** 就如同一道隱形的濾網或虛擬的桌布，它只暴露出發出查詢的應用程式當下所「必須」看見的那幾欄、那幾列資料；對於它工作範疇外那些不能看的敏感病歷或信用卡號，會被完全屏蔽與隱藏。這徹徹底底地實踐了 **需知 (Need-to-Know) 原則** 以及 **最小權限 (Least Privilege)**。

#### **13. 在一場架構設計聯合審查會上，一位自信滿滿的工程師提出了絕妙點子：「為了提升伺服器路由速度，我們應該把包含至高通行權的用戶 Session Token，全部塞進 URL Query 參數裡明碼傳送！」坐在對面的資安架構師立刻毫不留情地重重打臉並否決這項草案。因為他深知，URL 在傳輸過程與其上下游的所有代理商、以及每個用戶的瀏覽器歷史紀錄中，都會被系統性地被無差別留存紀錄 (Log)。這個恐怖的設計情境深刻凸顯了在軟體開發中，除了考慮程式碼本身，也必須極度敬畏以下哪一層面設計的重要性？**
A. 挑選可重複使用的技術的能力
B. 安全的介面設計，以及對上下游依賴關係的深層忌憚 (Secure interface design and upstream/downstream dependencies)
C. 認知運算與人工智慧
D. 可信賴運算基礎 (TCB)

*   **答案：B**
*   **解析：** **安全的介面設計 (Secure Interface Design)** 要求架構師必須用放大鏡嚴格審視「各個組件彼此之間到底是怎麼交換口水 (資料) 的」。把能定生死的機密 (像是 Token 或 API Key) 直接塞進 URL 參數裡傳來傳去，這是一項極其低級且致命的架構設計缺陷。因為所有在請求路徑上的攔路虎 (像透明代理伺服器、負載平衡器、甚至是瀏覽器的歷史記錄簿) 預設都會毫不留情地把 URL 完整的模樣給 Log 記錄下來，導致機密大規模崩塌外流，成為駭客現成的自助餐。

#### **14. 在進行架構藍圖規劃的階段，組織決心導入「威脅情資 (Threat Intelligence)」。為此大灑幣買下了針對金融產業的「資訊分享與分析中心 (ISAC)」提供的高貴情資串流服務。藉此提前「看透」目前活躍的敵對駭客組織都在金融圈裡使用哪些骯髒手段。這項高成本戰略活動的最主要目標，是為了什麼？**
A. 算出公司自己系統裡面那些現存古董漏洞的 CVSS 分數。
B. 從厚厚的過期程式碼裡，去肉搜那些被遺忘的零時差漏洞。
C. 揪出那些「極度可信、攸關生死且真真切切存在」的威脅，將它們化為設計圖紙上那些最被優先關注並加強防禦工事的防彈裝甲重點。
D. 無奈地只為了應付乙方廠商合約裡那條「稽核權利」條款。

*   **答案：C**
*   **解析：** 把 **威脅情資 (Threat Intelligence)** 帶進枯燥的架構設計階段，能讓成天關在冷氣房畫圖的架構師們看見真實戰場的樣貌。透過了解現實世界中針對他們產業的惡意駭客到底在用什麼戰術、技術和程序 (TTPs)，架構師們就能針對性地 **優先分配預算與資源 (Prioritize)**，把防禦裝備點在真正有用的設計決策上。

#### **15. 全球組織展開史詩大遷徙！將一台躲在公司溫室裡的地端網頁伺服器，一行 code 都不改，連著作業系統皮帶肉，一毛一樣原封不動地搬上 Amazon Web Services (AWS) 去跑它的 EC2 虛擬主機。請注意！在這個極為針對的「雲端架構服務模型」裡，除了硬體是雲端廠商的之外，整顆作業系統 (OS) 它的修補、它的伺服器防火牆、甚至是它上面跑的應用程式，100% 全部是由這家組織繼續自己含淚管理！這個苦情的雲端模型，其正式名稱為：**
A. 軟體即服務 (Software as a Service, SaaS)
B. 平台即服務 (Platform as a Service, PaaS)
C. 基礎設施即服務 (Infrastructure as a Service, IaaS)
D. 無伺服器運算架構 (Serverless Architecture)

*   **答案：C**
*   **解析：** 你花錢純粹只是向雲端龍頭爸爸買「沒有靈魂的暴力運算資源」(比如說純粹買一顆虛擬機器 VM)。雲端業者佛心地幫你處理並管理了底下的冷氣房、實體伺服器網路線跟主機板等硬體雜事；但是，只要是跑在那顆虛擬機器上面的靈魂 (作業系統要灌什麼、打什麼安全補丁修補系統漏洞、以及你的應用程式生殺大權)，通通都是你要「自己負責與承擔」的共同責任模型，它的名字就叫做 **基礎設施即服務 (IaaS)**。

#### **16. 接到一張處理最高級別國防機密情報的應用程式建築藍圖。資安總設計師毫不猶豫地下手：徹底閹割了所有可疑的外部網路連線、粗暴地套用「強制存取控制 (MAC)」的極權體系，並佈下一張網，讓這個系統內的任何一隻螞蟻想咬一口蘋果，都必須乖乖立正站好，接受並強迫通過「參考監視器 (Reference Monitors)」的無情審查。請問，這整個系統中 (把硬體、軟體、軔體全部打包丟進攪拌機裡算進去)，那些全部聯手站在一起，咬死並貫徹最終安全防線機制的「防護長城大聯盟」，在資安界裡被賦予了一個極其崇高的尊稱。這個尊稱叫做：**
A. 通用漏洞評分系統 (CVSS)
B. 可信賴運算基礎 (Trusted Computing Base, TCB)
C. 推測執行防護罩 (Speculative Execution)
D. 軟體物料清單 (SBOM)

*   **答案：B**
*   **解析：** 各位觀眾，這就是 **可信賴運算基礎 (Trusted Computing Base, TCB)** 的究極全貌。TCB 是你這台電腦系統裡，從生到死、從硬體電路板、韌體到你的每一條作業系統軟體防護線裡面，那些「負責扛起所有防禦重任」、「全體總動員」去貫徹執行你們安全政策機制的超級守門員。你系統的安全防線如果崩了，那就是你的 TCB 破了洞。

#### **17. STRIDE 威脅模型教室：一名變態攻擊者，發現目標應用程式處理網路連線的消化速度極度緩慢。於是他寫了一支無腦外掛，向那可憐的伺服器狂吐數千個「還沒握完手就切斷的半殘破爛 TCP 連線請求」。結果，這台伺服器的體力 (記憶體) 就這樣被活生生榨乾，導致門口真正的衣食父母 (合法客戶) 連個首頁都進不去。這套毀滅性的爛招，它在 STRIDE 的罪狀分類圖鑑中，被釘在死刑台哪一個位置？**
A. 欺騙 (Spoofing)
B. 資訊洩露 (Information Disclosure)
C. 阻斷服務攻擊 (Denial of Service, DoS)
D. 權限提升 (Elevation of Privilege)

*   **答案：C**
*   **解析：** **阻斷服務攻擊 (Denial of Service, DoS)** (不管它是靠運算資源枯竭、記憶體塞滿，還是網路頻寬擠爆) 的唯一歹毒信念，就是藉由海量的癱瘓招式，導致目標系統瞬間失能，破壞系統最脆弱的「可用性 (Availability)」，讓那些花錢買你服務的無辜顧客們「就是永遠吃不到你的服務」。

#### **18. 資安稽核大隊來到，在檢視應用程式藍圖時他們冷冷下達了一道不可違逆的聖旨：「貴系統內，任何一位大老闆或管理員對伺服器發出的每一道手動控制指令，它都必須在被執行的當下，被『順手同時無差別地發送並強行抄寫在一台放置在極遠的海外孤島上，並且只准寫、不准改的遠端 Syslog 稽核日誌伺服器中』」。好啦，請問這道粗暴強硬的架構護城河，正是 STRIDE 理論教科書中，為了專剋哪一頭魔獸而量身訂做的殺手鐗？**
A. 權限提升 (Elevation of Privilege)
B. 否認 (Repudiation)
C. 欺騙 (Spoofing)
D. 資訊洩露 (Information Disclosure)

*   **答案：B**
*   **解析：** 這招太絕了。當系統日誌的運作方式被強硬隔離到位於遠端、一個「不可變更 (Immutable)」且沒有人有權力刪文的神聖唯讀紀錄儀表板上時。這就如同買了一套防死角攝影機！即使最高管理員偷下了一道邪惡指令 (像是刪除整個生產庫)，未來就算他們再瞎掰說「不是我幹的」，那套唯讀不漏記的神聖日誌也必定會成為呈堂證供。這就是最經典的用以粉碎「**否認 (Repudiation)**」的終極緩解措施保障。

#### **19. 建築大師畫出最後一張新世代手機 APP 的架構藍圖！並且狂傲地宣告：「我們這個新 APP 體內，絕對不要留存或建立擁有自己資料庫的密碼！我們自己也不負責雜湊它！我們要高傲且專一地、只去依靠那些巨頭大佬的『身分提供者 (Identity Providers / IdP)』(例如：Google 的 G 鍵登入、蘋果的 Apple ID 或微軟的 Azure AD) 來替我們接客做身分查核！」請問這位有個性的大師，他在這張架構決策上，是依偎並沾光了哪一個強大核心理念與設計思維，從而在源頭上瘋狂收斂、徹底消滅「帳號被偷的攻擊面」？**
A. 仔細評估並聰明挑選「可重複使用且已被千錘百鍊證明過」的架構技術 (在身分與憑證管理領域) (Evaluating and selecting reusable technologies)
B. 擁抱工業物聯網 (Industrial IoT) 機器人大軍
C. 實作豐富網際網路應用程式模式來抵抗「客戶端的連環爆炸 (Client-side exploits)」
D. 高大上的機器學習與認知運算系統

*   **答案：A**
*   **解析：** 傻瓜才自己在那邊造車輪子！將這吃力不討好且動不動就會搞砸的「登入驗證、密碼加密、忘記密碼邏輯與儲存庫」，統統發包並外包依賴給那些擁有世界頂級防護外裝機動步兵團守護著的超大型 IdP (身分提供巨頭)，透過 SAML 或 OIDC 等標準來驗身分。這是極為教科書且優雅的神仙走位。完美展示了如何靠著 **「評估與挑選那些經歷過血與火考驗的可重複使用技術 (Evaluating and selecting reusable technologies specifically for Credential Management)」**，來規避那自建帳戶資料系統所引來無比巨大且危險的架構風險。
