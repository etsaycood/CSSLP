# 領域 7：安全的軟體部署、營運與維護 (Secure Software Deployment, Operations, and Maintenance) (14題)

#### **1. 一個軟體開發團隊正在使用 Docker 容器來部署他們的新 Web 應用程式。為了防止萬一容器內部被入侵時，攻擊者能直接「逃逸 (Escape)」並挾持底層的主機作業系統 (Host OS)，系統管理員最強制、最關鍵的安全部署基準 (Security Baseline) 應該是什麼？**
A. 確保容器只透過 HTTP (Port 80) 通訊。
B. 確保容器內部執行的應用程式預設「絕對不」以 Root (最高權限) 執行。
C. 在容器內部安裝防毒軟體。
D. 使用相同密碼登入所有的容器。

*   **答案：B**
*   **解析：** 容器 (如 Docker) 與虛擬機器 (VM) 不同，容器實質上是與主機作業系統(Host OS) 共享同一個系統核心 (Kernel) 的。如果一個被攻陷的容器是以 **Root** 權限 (特權模式) 執行，攻擊者可以輕易發動「容器逃逸 (Container Escape)」大屠殺，直接奪走底層實體伺服器的控制權。確保容器以**非 Root (非特權) 使用者** 執行，是保護容器化環境最重要且最致命的一條防線。

#### **2. 公司要上線一個全新的財務應用程式。管理層對於「一次就取代掉舊系統」感到深深的恐懼與不安。因此，資安營運主管建議：先將新系統與舊系統同時平行運作 (Parallel run) 兩個禮拜；這段期間內舊系統依舊是主力，新系統只吃相同的資料串流當做影子測試，萬一新系統爆炸了也不會影響公司。請問這種部署策略稱為什麼？**
A. 藍綠部署 (Blue-Green Deployment)
B. 階段式 / 灰度發布 (Phased / Canary Release)
C. 影子 / 平行營運部署 (Shadow / Parallel Deployment)
D. 全面替換 (Big Bang Replacement)

*   **答案：C**
*   **解析：** 教科書上標準的 **平行營運 / 影子部署 (Parallel / Shadow Operations)** 策略。這招讓新舊兩套獨立系統在同一段時間內共同吃下一模一樣的輸入資料，但只讓舊系統負責產出真正的商業結果。它的唯一目的就是為了「無痛驗證」新系統的效能與準確度，萬一新系統有雷，也能保證對客戶達到「零衝擊 (Zero Impact)」的安全著陸。

#### **3. 為了防禦在軟體供應鏈中駭客偷偷把建置好的安裝檔偷梁換柱，開發產線 (CI/CD) 在拋出最終的二進位執行檔時，工程師用一台上鎖伺服器裡面的私鑰，對這個執行檔蓋了一個印章。到了客戶的機器要安裝前，客戶的作業系統會先去檢查這個印章對不對。請問這是應用了哪一種密碼學控制？**
A. 對稱式加密 (Symmetric Encryption)
B. 程式碼數位簽章 (Code Signing)
C. 位址空間配置隨機化 (ASLR)
D. 資料混淆 (Obfuscation)

*   **答案：B**
*   **解析：** **程式碼簽章 (Code Signing)** 是開發者使用非對稱式密碼學 (私鑰) 來為軟體打上數位封條。它保證了兩件神聖的事：(1) **真實性/身分 (Authenticity)**：這軟體真的是這家公司寫的；(2) **完整性 (Integrity)**：這軟體在網路上傳輸投遞的過程中，絕對沒有被任何駭客偷偷打開塞後門進去。

#### **4. 一家雲端代管公司正在經歷一場痛苦的資安稽核。稽核員發現：只要系統管理員離職，他的權限通常要拖上兩個禮拜才會被 IT 部門手動刪除；而在這段空窗期，這位前員工憑著他熟記的密碼，依舊可以在家裡大搖大擺地登入公司的資料庫。這代表這家公司徹底搞砸了哪一項維運管理流程？**
A. 工作階段管理 (Session Management)
B. 事件應變 (Incident Response)
C. 身分與存取管理配置/解除配置 (IAM Provisioning and Deprovisioning)
D. 修補程式管理 (Patch Management)

*   **答案：C**
*   **解析：** **解除配置 (Deprovisioning / 撤銷權限)** 是使用者生命週期中最容易致命的一環。當員工離職或約聘結束時，如果未能「立即且徹底」地拔除他們在所有系統與應用程式上的帳號存取權，這會留下一個巨大無比的安全黑洞，極易引發內部威脅 (Insider Threat) 或前員工的惡意報復。

#### **5. 深夜三點，資安維運中心 (SOC) 的警報大作！監控儀表板顯示：公司最核心的電子商務網頁伺服器，正持續對外向一個位於國外的未知可疑 IP 位址，發送海量且未經加密的信用卡資料封包。請問 SOC 團隊第一步「最立刻必須採取」的標準止血行動應該是什麼？**
A. 立刻把那台伺服器的硬碟格式化重灌。
B. 將該台受感染的伺服器從網路上進行隔離 (Isolation / Containment)。
C. 花一個月時間找出是哪個駭客寫的惡意程式。
D. 發布修補程式更新 (Patching)。

*   **答案：B**
*   **解析：** 在事件應變計畫 (Incident Response, IR) 的黃金準則中，一旦確認(識別) 了災難正在發生，下一個絕對優先動作就是 **控制 / 包含 (Containment / Isolation)**。也就是在不破壞數位證據的前提下，果斷將那台正在噴血的伺服器從網路架構中「拔線隔離」或移入沙箱，以阻止災難的大火繼續蔓延到其他健康的伺服器身上。

#### **6. 作為 DevOps 文化的一部分，開發團隊決定將他們建置伺服器 (CI/CD) 的所有設定、雲端 AWS 的防火牆規則、甚至是作業系統的安裝腳本，全部用純文字檔案 (YAML/JSON) 寫下來，並且直接放進 Git 程式碼版本控制系統裡面，像寫程式一樣地去管理它們。這項極致強大的自動化實務做法被稱作：**
A. 靜態分析 (SAST)
B. 基礎設施即程式碼 (Infrastructure as Code, IaC)
C. 持續監控 (Continuous Monitoring)
D. 威脅建模 (Threat Modeling)

*   **答案：B**
*   **解析：** **基礎設施即程式碼 (IaC)** 是雲端原生時代的魔法。傳統上需要人工點擊、手動拉線設定的建置與維運工作，全部被轉換成了可以被機器閱讀、可以被 Git 追蹤版本、每一次執行結果都保證「絕對一模一樣 (Idempotent)」的腳本設定檔。這徹底消滅了因為工程師「手滑」所導致的伺服器設定漂移 (Configuration Drift) 與安全地雷風險。

#### **7. 一個應用程式被發現存在一個「嚴重的零時差 (Zero-day) 漏洞」，駭客可以利用這招直接吃掉資料庫。但原廠的開發團隊表示：「這個底層架構太古老了，如果要寫更新修補程式 (Patch) 來補這個洞，至少要花三個月的時間連根拔起重寫。」在這絕望的三個月空窗期裡，為了不讓系統被駭客攻破，資安工程師只能無奈地在應用程式大門口，緊急架設一台強大的「網路應用程式防火牆 (WAF)」，用人工寫死規則來硬擋那些駭客的惡意攻擊封包。請問工程師架 WAF 的這個動作，在資安教條裡稱為什麼？**
A. 正規的漏洞修補 (Vulnerability Patching)
B. 虛擬修補 (Virtual Patching / Mitigating Control)
C. 回復 / 降級 (Rollback / Downgrade)
D. 災難復原 (Disaster Recovery)

*   **答案：B**
*   **解析：** 當你因為某些悲慘的原因 (例如廠商倒閉、系統太老、修補需要三個月) 而「無法親手去修改原始碼根除漏洞」時，你在應用程式外圍套上一層防護罩 (像是架設 WAF、IPS 或是修改網路關卡規則) 來死死堵住駭客打進來的攻擊路徑，這種續命手段就稱為 **虛擬修補 (Virtual Patching)** 或是實施補償性控制措施。它不治本，但在這三個月裡能保你一命。

#### **8. 公司規定：任何工程師都絕對不允許自己一個人就把程式碼直接推上正式營運伺服器 (Production)。所有針對營運環境的更新、設定修改或重啟，都必須提交一份詳細的文件，經過一個被稱為「CCB / CAB (變更管制委員會)」的最高議會人工審核、批准並敲槌子後，才能在指定的「深夜維護視窗」內進行。請問這個囉嗦但能保命的流程叫什麼？**
A. 軟體組成分析 (SCA)
B. 事件應變計畫 (Incident Response)
C. 變更管理與發布管理 (Change and Release Management)
D. 容量規劃 (Capacity Planning)

*   **答案：C**
*   **解析：** **變更管理 (Change Management)** 是一道鐵閘門，它的唯一使命就是「絕對不允許任何未經授權、未被留存紀錄、或是沒有經過嚴格沙盒測試的黑箱變更」，去碰觸那脆弱且神聖的正式生產環境；從而避免掉因為一個天兵工程師打錯一個字，就導致公司全球服務斷線 24 小時的毀滅性人為災難。

#### **9. 一套古老的會計應用程式在歷經了 15 年的折磨後，終於光榮退役，它的所有工作已經被全新的雲端 ERP 系統取代。針對這台老舊伺服器肚子裡那些「依法必須保存10年，以備國稅局隨時上門查稅」的歷史帳本資料。下列哪一項是生命週期結束 (EOL) 與除役流程中，最正確且合法的資料處置方針？**
A. 用強力磁鐵 (Degaussing) 把硬碟直接銷毀成渣。
B. 將伺服器連上網際網路並保持開機，方便稽核員隨時遠端登入查帳。
C. 執行資料保留與冷封存 / 歸檔 (Data Retention and Archiving) 到安全的離線儲存庫中。
D. 給所有員工發放隨身碟，讓他們各自拷貝一份帶回家備份。

*   **答案：C**
*   **解析：** 系統退場並不代表你可以把資料一燒了之！因為監管法規 (如稅法、病歷法) 往往強硬規定這類敏感資料必須被保留 5 到 10 年不等。正確的做法是將這些依法具有剩餘價值的資料，安全地遷移並 **封存/歸檔 (Archiving)** 到一個離線或受到極度嚴密保護的低成本儲存室中 (Data Retention)，然後再把那台千瘡百孔的舊伺服器砸爛。

#### **10. 一個跨國線上遊戲公司發現：他們位於東京的主資料庫，在無人為介入、沒有改過任何系統設定的情況下，其硬碟與記憶體的消耗量突然從平均的 40% 緩步爬升到 98%，幾乎快要撐爆當機。為了能在系統崩潰的「最後一秒鐘之前」精準發現這個異常並叫醒工程師，這個組織必須高度依賴哪一種維運實務？**
A. 災難復原演練 (Disaster Recovery Simulation)
B. 持續監控與日誌分析 (Continuous Monitoring and Log Analysis / Telemetry)
C. 基礎設施即程式碼 (IaC)
D. 漏洞掃描 (Vulnerability Scanning)

*   **答案：B**
*   **解析：** **持續監控與遙測 (Continuous Monitoring and Telemetry)** 就如同為系統裝上心跳監視器。透過全天候 24/7 不間斷地收集伺服器的效能指標 (資源極限) 以及安控日誌 (Logs)，維運團隊才能在系統資源耗盡、或大批駭客剛拔出刀準備破門而入的極早期，搶先一步偵測到異常活動 (Anomalies) 並發出警報。

#### **11. 工程師發現目前正在線上服役的應用程式 3.2 版存在一個致命錯誤：只要使用者輸入特定的生日，資料庫就會無條件抹除該用戶的所有積蓄。因為災情太過慘烈且根本找不出蟲子在哪裡，高層下令：「馬上按下一顆核彈按鈕，讓系統在一秒內強制時光倒流，回到昨天晚上還沒有這個 Bug 的 3.1 版狀態！」請問這個救命的神級操作稱為什麼？**
A. 藍綠部署 (Blue-Green Deployment)
B. 回復 / 降級部署 (Rollback / Downgrade)
C. 零信任網路 (Zero Trust Network)
D. 虛擬修補 (Virtual Patching)

*   **答案：B**
*   **解析：** 在變更管理與發布流程中，每一次的部署上線，都絕對必須強制綁定一部寫得清清楚楚的 **回復計畫 (Rollback Plan)**。萬一新版本上了線卻發揮出毀滅地球的威力，營運團隊必須有能力在幾分鐘內，將系統無痛切換並 **降級 (Downgrade)** 回上一個已知安全的穩定版本。

#### **12. 當準備汰換並丟棄一批曾經用來處理過極度敏感的「國防機密專案」資料的老舊硬碟時，資安部門堅決反對只是在作業系統裡按下「右鍵 -> 格式化 (Format)」。他們堅持要花大錢買一台巨大的工業碎紙機，把這些硬碟連同裡面的馬達一起絞碎成比米粒還小的金屬粉末。請問為什麼要採取如此極端且暴力的物理破壞手段？**
A. 為了確保這些硬碟能被賣給回收商換錢。
B. 因為軟體格式化只能刪除檔案指標，資料仍存在；唯有經過核准的物理銷毀 (Physical Destruction)，才能保證這場資料殺戮是「絕對不可逆轉 (Irreversible)」且無法復原的。
C. 為了滿足可用性 (Availability) 原則。
D. 為了釋放空間給虛擬記憶體。

*   **答案：B**
*   **解析：** 在生命週期結束的 **資訊處置 / 媒體淨化 (Media Sanitization / Disposal)** 階段，對於最高機密等級的資料，不管是普通的格式化還是寫入亂碼覆寫 (Wiping) 都被認為有遭到鑑識專家還原的風險。只有 **物理銷毀 (Physical Destruction / 碎化 Shredding 或 熔毀 Melting)**，才是能徹徹底底在物理宇宙中保證這些機密資料永遠灰飛煙滅、無人能救的方法。

#### **13. 一位開發者在一套古老且被遺忘的應用程式裡，因為懶得記密碼，於是在程式碼裡面偷偷寫了一段：「只要用戶名稱輸入 'super_admin'，不管密碼打什麼，就無條件讓他登入並給他最高管理員權限。」結果三年後，這段程式碼在正式營運環境中被駭客發現並用來血洗了公司。請問這個由內鬼/天兵留下來的恐怖通道叫什麼？**
A. 第三方相依性漏洞 (Supply Chain Vulnerability)
B. 零時差漏洞 (Zero-Day Vulnerability)
C. 未具備文件的功能 / 惡意後門 (Undocumented functionality / Backdoor)
D. 跨網站指令碼 (XSS)

*   **答案：C**
*   **解析：** 這就是惡名昭彰的 **未具備文件的功能 (Undocumented functionality)** 或是 **後門 (Backdoors)** 甚至是邏輯炸彈！這是極度危險的內部威脅產物。開發者通常是為了方便自己隨時用上帝視角進去除錯，私自鑿開了這個繞過所有正統防護 (Bypassing controls) 的秘密隧道；而當這個系統上線時，這個暗門就成了駭客最愛的免死金牌。

#### **14. 災難降臨了！公司的主要機房被一場無情的颶風徹底夷為平地。慶幸的是，公司高層在半年前有乖乖聽資安長的話，在另一個連著不同電網、且絕對不會有颶風的遙遠州界，花重金包下了一個「隨時插著電、伺服器開著機、連作業系統跟公司應用程式都已經全部灌好準備就緒」的備用基地。當災難發生的短短兩小時內，公司所有的流量就已經全無縫轉移到這個基地並恢復營業。請問這個超豪華的備用基地在術語中被稱為什麼？**
A. 冷站 (Cold Site)
B. 溫站 (Warm Site)
C. 熱站 (Hot Site)
D. 沙箱 (Sandbox)

*   **答案：C**
*   **解析：** 在 **災難復原 (Disaster Recovery)** 的兵法中，**熱站 (Hot Site)** 是最昂貴卻也最能保命的終極防線。它是一個與主要機房一模一樣的完美複製品，設備隨時通著電、資料保持著近乎即時的即時同步 (Real-time replication)。當毀滅到來時，熱站能在幾個小時、甚至幾分鐘內無縫吞下所有的業務流量，保證公司的心跳(可用性) 絕對不會停止。
