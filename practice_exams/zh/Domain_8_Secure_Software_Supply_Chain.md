# 領域 8：安全的軟體供應鏈 (Secure Software Supply Chain) (12題)

#### **1. 一個軟體開發專案從網路上廣泛下載並使用了 35 個開源的第三方 JavaScript 函式庫，以加速開發進度。資安主管要求團隊必須提交一份「極度詳盡、寫滿了這個 Web 應用程式體內所包含的每一個第三方組件、組件版本號碼、以及其依賴層級關係」的清查清單。這份被稱為軟體界「食品成分營養標示」的最關鍵文件是什麼？**
A. 威脅建模報告 (Threat Model Report)
B. 軟體物料清單 (Software Bill of Materials, SBOM)
C. 開源軟體授權合約 (Open Source License Agreement)
D. 供應商層級 SLA (Service Level Agreement)

*   **答案：B**
*   **解析：** **軟體物料清單 (SBOM)** 是管理現代軟體供應鏈風險的絕對基石。它就像是一份食品包裝上的成分表，鉅細靡遺地羅列出一套龐大軟體內部，到底縫合轉接了哪些開源套件、來自哪個供應商、以及它們精確的版本號。沒有這張地圖，當某個開源套件 (如 Log4j) 爆發毀滅性漏洞時，組織根本不知道自己的心臟有沒有被埋下這顆炸彈。

#### **2. 當開發團隊在 GitHub 上透過套件管理器 (如 NPM 或 Maven) 下載一個極度依賴的開源套件時，發現該套件採用的是具有極強「傳染性 (Viral / Copyleft)」的 GNU 通用公共授權條款 (GPL) 授權。如果這個團隊將這個 GPL 套件綁定並編譯進他們打算拿去賣大錢的專有商業軟體中，他們將必須承擔什麼恐怖的法律夢魘風險？**
A. 他們的伺服器會立刻被這個套件植入木馬。
B. 他們的軟體將失去使用對稱式加密演算法的能力。
C. 根據 GPL 的傳染性條款，他們可能被迫必須將自己公司花了幾千萬寫的「專有商業軟體原始碼」全部免費對全世界開源公開。
D. 他們將永遠無法再使用任何雲端服務。

*   **答案：C**
*   **解析：** 這就是身為開發者最大的法律地雷。**軟體授權 (Software Licensing)** 規定了你能拿別人的開源程式碼來做什麼。像 MIT 或 Apache 是極度寬鬆自由的；但是像 **GPL** 這種被稱為 "Copyleft (著佐權/傳染性)" 的授權，它死硬規定：只要你的專有產品敢去連結、或修改它的程式碼並發布，你自己的整個軟體產品也必須同樣「被迫開源」！這對商業軟體公司來說是滅頂之災的 **智財權 (IP) 風險**。

#### **3. 在為公司內部開發的軟體原始碼產線 (CI/CD) 規劃安全藍圖時，資安架構師下達了最高指令：「全線導入 SLSA (Supply-chain Levels for Software Artifacts) 框架！」請問架構師大費周章導入 SLSA 這個 Google 研發的框架，其最瘋狂但也是最核心的資安目的是為了保護什麼？**
A. 確保軟體的字控 UI 介面具備響應式設計。
B. 保護「軟體產物本身的血統純淨度」，斬斷任何駭客試圖在編譯過程中偷偷加料、篡改二進位檔，或是在建置機器上動手腳的供應鏈死路。
C. 為了符合信用卡組織 PCI DSS 的無聊稽核。
D. 取代所有的防火牆與防毒軟體。

*   **答案：B**
*   **解析：** **SLSA (Supply-chain Levels for Software Artifacts)** 框架的存在，就是為了對抗像是 SolarWinds 那種毀滅性、國家級黑客會去摸進去你家編譯伺服器裡「偷換建置腳本」的軟體供應鏈大屠殺。SLSA 透過強制要求血緣出處證明 (Provenance)、強制隔離建置環境、與雙重認證等嚴酷手段，確保從原始碼到最終執行檔的這條生產線上，沒有任何一支看不見的黑手能去竄改 (Tampering) 你的軟體產物。

#### **4. 一家組織正與一家外部的軟體公司簽約，委託他們開發一套關鍵的人力資源應用程式。組織裡的法務部門與資安主管聯手，在合約附錄中白紙黑字寫下一條死規定：「如果在這套軟體上線後，被發現存在任何 CVSS 評分高於 7.0 分的漏洞，貴公司(供應商)必須在被通報的 48 小時之內，無條件提供並部署修補程式 (Patch)；若逾期未修補，貴公司須面臨每日一百萬元的巨額罰款。」請問這條冷酷的條款被寫在哪份神聖的文件中？**
A. 軟體物料清單 (SBOM)
B. 服務層級協定 (Service-Level Agreement, SLA) / 供應商合約
C. 品質保證測試計畫 (QA Test Plan)
D. 威脅模型分析報告 (Threat Modeling Report)

*   **答案：B**
*   **解析：** 在管理供應鏈風險時，你唯一能掐住外部供應商喉嚨的武器，就是那份具有法律拘束力的合約：**服務層級協定 (SLA) 與供應商合約條款**。在合約中明文綁定「漏洞修補的響應時間」、甚至是「資安事件的違約罰則」，才能強制第三方供應商重視你家系統的安全，而不是拿了錢就把滿是漏洞的爛軟體丟著不管。

#### **5. 駭客組織改變了戰術。他們不去攻擊那家已經武裝到牙齒的大防毒軟體公司，而是駭進了那家公司依賴的，一個極其邊緣、用來幫防毒軟體打包安裝檔的「第三方小包商自動化建置伺服器」。駭客在這台沒有人看管的伺服器上，把準備釋放出去的官方防毒軟體更新包裡面塞進了勒索病毒。隔天，全球幾百萬個客戶開開心心地點了「官方更新」，然後全數陣亡。這場教科書級別的大災難被稱為什麼攻擊？**
A. 跨網站腳本攻擊 (XSS)
B. SQL 內部注入攻擊
C. 軟體供應鏈攻擊 (Software Supply Chain Attack)
D. 社交工程釣魚攻擊 (Phishing)

*   **答案：C**
*   **解析：** 這就是最經典、也最難以防禦的 **供應鏈攻擊 (Supply Chain Attack)** (正如現實中震驚全球的 SolarWinds 攻擊事件)。攻擊者避開防守嚴密的主體，轉而從防護最弱的「第三方供應商、開源依賴套件、或是建置基礎設施 (Build Infrastructure)」下手。藉由污染了軟體的「上游源頭」，讓這個含有毒藥的更新包，透過官方且合法的通道，暢行無阻地被派發到成千上萬個無辜客戶的系統深處。

#### **6. 你發現你們公司的軟體竟然偷偷引入了一個由「神秘東方匿名網友」在 GitHub 上維護的密碼學加解密圖書館套件。為了確保這個套件沒有被這位老兄植入後門，並試圖驗證這個套件的最初開發來源，你們團隊應該瘋狂去蒐集並檢驗什麼證據？**
A. 去看這個套件在 GitHub 上有多少顆星星 (Stars)。
B. 去檢查該套件的 血統/出處證明 (Provenance)、原作者的程式碼數位簽章與提交歷史。
C. 去用 DAST 動態掃描工具掃描你們家自己的資料庫。
D. 把那個套件用反組譯工具看一眼就當沒事。

*   **答案：B**
*   **解析：** 當你引入任何第三方軟體時，核實它的 **出處 / 血統 (Provenance)** 是唯一能保命的方法。你必須去驗證這包原始碼到底是不是從那個「宣稱的作者」手中寫出來的？這個套件有沒有打上那個作者專屬的 **數位簽章 (Digital Signature)**？這一切都是為了確保軟體的「真實驗證身分 (Authenticity)」，防堵有人在半路李代桃僵，偷偷抽換掉原本乾淨的套件。

#### **7. 一家金融機構決定每年花費五十萬，聘請一家享譽國際的「獨立第三方資安查核機構 (Third-Party Auditor)」，針對他們那些傲慢的軟體供應商進行了一場極度痛苦的突擊檢查、程式碼白箱盲測與資安評鑑大會。請問花這筆鉅款搞這種第三方評鑑，其最大的好處是什麼？**
A. 可以用這個評鑑結果去威脅供應商幫忙寫免費的新功能。
B. 獲得一份不帶偏見、客觀且具備公信力的診斷報告，驗證供應商「是不是真的有達到他們合約裡吹牛的安全水準」，進而消除組織盲目信任的風險。
C. 為了可以把這個報告賣給其他黑帽駭客賺錢。
D. 自動清除供應商軟體裡面所有的 Bug。

*   **答案：B**
*   **解析：** 你永遠不能只聽供應商在那邊發誓說「我們的軟體超安全」。執行 **第三方評鑑與稽核 (Third-party Audits and Assessments)**，能引入絕對客觀且殘酷的外部眼光，去無情地拆穿供應商的謊言，實打實地驗證他們交出來的軟體，究竟有沒有符合你們之間在簽訂 SLA 時所規定的安全紅線。

#### **8. 公司突然接獲資安大廠緊急通報：某個全球有幾十億台機器都在用的開源日誌套件 (例如：Apache Log4j)，被發現存在一個「打字就能毀滅你整台伺服器伺服器」的最高等級 10.0 分漏洞！此時此刻，你們公司的事件應變團隊 (IR Team)，第一步必須立刻瘋狂去翻閱哪個神聖的檔案，才能在最短時間內判定「我們家自己寫的那幾百支應用程式裡面，到底有沒有用到這個被詛咒的日誌套件」？**
A. 變更管理的審核日誌 (Change Management Logs)
B. 軟體物料清單 (SBOM) 與 依賴套件追蹤系統。
C. 使用者的登入與登出日誌 (Access Logs)
D. 品質保證 (QA) 團隊的效能壓力測試報告

*   **答案：B**
*   **解析：** 當一場突如其來的供應鏈核彈級零時差漏洞爆發時，時間就是生命。此時只有一張極度精準、涵蓋了所有隱藏依賴關係的 **軟體物料清單 (SBOM)**，能讓防禦團隊在幾分鐘內，用關鍵字搜尋出整個企業內「有哪五十支應用程式不小心把這個劇毒套件給裝進了肚子裡」，從而立刻精準下藥，隔離並展開修補行動。

#### **9. 在選擇雲端服務提供商 (CSP) 或採用 SaaS 代管服務時，組織的資安架構師必須非常清醒地認識到一個冷酷的現實：雖然我們把「大樓保全、冷氣、甚至是底層虛擬機器的防毒軟體」的管理責任，通通花錢外包推卸給了 AWS 或 Azure。但是！！這份雲端合約裡有一條鐵律，那就是：「有三樣東西的最終安全與保護責任，無論如何，永遠、永遠都是落在身為客戶的『你們自己』頭上」。這三樣推不掉的核心負擔，在共同責任模型 (Shared Responsibility Model) 裡指的是什麼？**
A. 資料中心的大門鑰匙、網路纜線、主機板
B. 你自己的資料 (Data)、終端實體裝置、與使用者的帳號和存取權限 (Endpoints, Account, and Access management)
C. 客戶端瀏覽器、鍵盤、滑鼠
D. 網路路由器、交換器、負載平衡器

*   **答案：B**
*   **解析：** 請將這句箴言刻在腦袋裡：在雲端的 **共同責任模型 (Shared Responsibility Model)** 中，不管你買的是 IaaS, PaaS, 還是最高級爽躺的 SaaS。身為客戶你永遠不能把**「你自己的資料 (Data)、使用者的身份存取管理 (IAM/帳號密碼)、以及你連接上去的終端設備安全」** 的屎盆子扣到雲端原廠頭上。如果你的員工用最白癡的密碼 "admin/admin" 登入 AWS，導致公司資料被偷光，AWS 是一毛錢法律賠償責任都沒有的。

#### **10. 某企業高度依賴一款由國外小規模工作室所開發的冷門商業軟體。公司的營運長夜不成眠，他擔心：如果這家國外小公司哪天突然宣布破產倒閉，或是被競爭敵手惡意併購而停止維護更新這套軟體，那該怎麼辦？為了防禦這種「商業供應鏈斷鏈死亡」的災難，組織在當初簽約這套軟體時，應該動用法律手段實著哪一項終極的避險救濟安排？**
A. 軟體代碼託管合約 (Software Escrow Agreement)
B. 定期舉辦駭客松 (Hackathons)
C. 與該國政府簽署引渡條款
D. 立刻自己重新寫一套完全一樣的軟體

*   **答案：A**
*   **解析：** **軟體代碼託管合約 (Software/Source Code Escrow)** 是一種終極的保命法律手段。在這種設計下，供應商必須把他們這套軟體最核心的「原始碼 (Source Code)」交給一個絕對中立、信用的第三方機構鎖進金庫保管。合約生效條件是：一旦這家供應商不幸倒閉、停止營運或違約不提供支援，這份原始碼的保險箱就會全自動為「客戶」打開。讓客戶能拿到原始碼，自己僱人繼續維護這套系統，確保業務不用被迫切腹自殺。

#### **11. 一名黑帽駭客發現自己打不進某銀行防守嚴密的內部網路。於是他心生一計，跑去開發者們常逛的開源套件論壇，發布了一款名為「超快速字串處理器」的超好用免費外掛工具。這款工具真的超好用，但它會在處理字串的同時，默默地把所有內容抄送給駭客。幾名傻白甜的銀行內部工程師，為了貪圖方便按下了下載鍵，並把它匯入到了銀行的開發專案中。這種專挑工程師貪小便宜與懶惰心態下手的手段，叫做污染了什麼東西？**
A. 污染了防火牆的 ACL (Access Control List) 設定
B. 污染了 開源軟體庫 / 上游儲存庫相依性 (Open-Source Repositories / Upstream Dependencies)
C. 污染了員工的薪資帳戶
D. 污染了資料庫綱要

*   **答案：B**
*   **解析：** 駭客現在最愛玩的戰術，就是把包含了惡意程式碼的**開源套件、圖書館或函式庫 (Open-source plugins / Dependencies)** 上傳到像是 NPM, PyPI 或是 GitHub 等公開的集散地；或是更狠的「錯字仿冒 (Typosquatting)」，取一個跟知名套件只差一個英文字母的假名。只要有警覺性不高的工程師把它下載 (Pull) 進自己的專案裡，駭客就等於在公司內網裡合法種下了一株致命的食人花。這就是針對 **依賴項生態系 (Dependency ecosystem/Repositories)** 下毒。

#### **12. 面對無孔不入的供應鏈攻擊，當代資安架構師在設計系統連線模型時，徹底拋棄了傳統那種「只要你人在公司大樓裡面，或者你已經連上了公司 VPN，你就是自己人，想看什麼系統都可以」的天真想法。取而代之的是，架構師宣布了一項全新的恐怖統治政策：「即使你是公司的執行長，即便你坐在公司裡用網路線插著內部伺服器，我依然預設『不信任你』！你每按一次滑鼠要去存取一顆資料庫，我都必須在那個當下，從零開始重新驗證你的身分與裝置安全性。」請問這種極致偏執狂的網路安全架構，叫做什麼名字？**
A. 虛擬私人網路 (VPN) 隧道架構
B. 藍綠色部署 (Blue-Green Deployment)
C. 零信任網路架構 (Zero Trust Architecture, ZTA)
D. 傳統周邊防禦堡壘模型 (Perimeter Defense Model)

*   **答案：C**
*   **解析：** **零信任架構 (Zero Trust Architecture, ZTA)** 的核心信仰就是那句響亮的口號：「永遠不要信任，始終都要驗證 (Never Trust, Always Verify)」。它徹底砸爛了傳統「內網就是安全的」這層天真的濾鏡。在零信任的世界觀裡，無論你人站在網際網路上還是坐在機房裡，所有的身分、裝置狀態、與存取意圖，都必須在「每一次」交會的瞬間接受審判與驗證。這是對抗供應鏈滲透與內部威脅擴散時，最無情但也最無敵的最終堡壘。
