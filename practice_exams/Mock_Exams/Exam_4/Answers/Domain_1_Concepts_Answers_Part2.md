# 第四套全真模擬試題 (Exam 4) - 答案與詳解本
## 領域 1：安全的軟體概念 (Secure Software Concepts) (16題) - Part 2 (Q9-Q16)

#### **9. A system architect is designing a high-availability e-commerce platform. To ensure the platform can survive an entire AWS Availability Zone going offline or a massive Distributed Denial of Service (DDoS) attack without inconveniencing customers, they deploy the application across three geographic regions and utilize a global load balancer mapped to an auto-scaling group. Which leg of the CIA triad is the architect primarily focusing on?**
**(一名頂級建城大師，正在督造一座能抵禦天災人禍的『日不落超級皇城大錢莊 (high-availability e-commerce platform)』。為了確保即使是半個地球的板塊沉沒 (an entire AWS AZ going offline)、或者是被幾百萬隻無腦殭屍老鼠發動『萬屍攻城毀滅撞門大屠城 (massive DDoS attack)』，這座皇城裡的買賣客人都依然能談笑風生喝酒吃肉睡好覺 (without inconveniencing customers)！大師決定砸下重金，把這座皇城【像切生日蛋糕一樣，硬生生劈成三份，分別安置在三塊互不相連的大陸神柱上 (across three geographic regions)！】。然後在神柱的正上方，掛上一顆會自動幫客人找空位坐、且發現城牆被老鼠咬破還會自動生出無數座備用城牆的『無限分流自動繁殖大水晶球 (global load balancer + auto-scaling group)』！請問這位建城大師如此發瘋般的地圖佈局，他最在乎、死死捧在手心保護的，是偉大 CIA 三神像裡的哪一尊保命活神仙？)**
A. Confidentiality (機密性是確保只有皇室能看帳本，這堆能殺死幾百萬老鼠的城牆可不懂怎麼幫帳本上密碼鎖)
B. Integrity (完整性是確保帳單數字不被瞎蓋亂改，這堆自動繁殖的城牆也不懂數字校對的加減乘除)
C. Availability (這就是能保證皇城大開、客流不息、日進斗金、絕對不會被人拔掉插頭斷氣關門的：【『無盡壽命與絕對開門做生意之終極可用性大女神 (Availability)』】！這三神柱分佈與無限繁殖的設計，唯一的目標就是：【讓這座皇城永遠活蹦亂跳供人使喚 (High-Availability)】，這是抵擋拔電線與癱瘓攻擊唯一解藥！)
D. Authenticity (真實性是辨別站在城門口的這隻到底是老鼠還是貴客，這堆牆壁不會查身分證)

*   **正確答案 / Correct Answer：C**
*   **[解析邏輯]**
    *   **為何 C 是最佳選擇：** **「可用性 (Availability)」** 的終極體現。
    *   **定義：** 系統能在被授權者需要時，即時且連續不中斷地提供服務與存取權限。
    *   **威脅：** 單點失效 (Single Point of Failure, SPOF) 如機房停電失火，或者是蓄意的 DDoS (分散式阻斷服務) 攻擊塞爆頻寬與 CPU。
    *   **對策 (High Availability Architecture)：** 題幹中使用的 `跨地區冗餘 (Multi-Region Redundancy)`、`全域負載平衡器 (Global Load Balancer)`、`自動擴展 (Auto-Scaling)` 都是現代雲端架構中，確保伺服器就算被打掛幾十台，系統依然能在 1 毫秒內切換到健康的機器上，維持對客戶 99.999% 的服務不中斷承諾 (SLA)。這是純粹為了 Availability 服務的架構設計。

#### **10. A user logs into a corporate VPN using their username and password (Something you know). Immediately after, they are prompted to enter a 6-digit code generated by an authenticator application on their smartphone (Something you have). Once authenticated, they attempt to access a confidential HR folder but are denied because they are in the Engineering department. What concepts have just been demonstrated in chronological sequence?**
**(一名穿著夜行衣的探子，趁著月黑風高，摸到了帝國【『暗夜跨境加密大隧道 (VPN)』】的城門口。他先從嘴巴裡，正確地吐出了那句只有他跟大內總管才知道的通關密語『天王蓋地虎』 (username and password / Something you know)。下一秒！城牆上的雷射光掃描器逼他掏出那支只有他獨佔擁有的『掌上隨身發報機』，並把上面每秒鐘都在亂跳的『六顆血紅大數字』秀給守衛看 (6-digit code on smartphone / Something you have)！這探子連闖兩關成功後，開心地走進大內。但他居然想偷偷摸去皇城深處、那間裝滿了宮女隱私資料的『大內閣梳妝房 (confidential HR folder)』！此時大內侍衛一腳把他踹飛並大吼：「滾！你這狗牌上寫著你是個打鐵的工匠 (Engineering department)！憑你也想進去摸宮女？」探子被死死擋在門外 (denied)。請問！這名探子從報上名號到最後吃閉門羹，他活生生、照順序依序觸發了哪三道審判大天雷 (chronological sequence)？)**
A. Identification -> Authentication -> Authorization (這就是那堪稱資安第一課、絕不能背錯的黃金通關三部曲：【『我是誰 ( Identification，我是打鐵匠王阿火) → 驗明正身 (Authentication，用密語跟手機證明你真的是王阿火而不是西藏來的騙子) → 賦權大審判 (Authorization，證明了你是王阿火後，再看名單說你王阿火只能去打鐵房，沒資格進宮女房被踢出去)』】！)
B. Authentication -> Identification -> Non-Repudiation (不會先查你身上帶什麼證件 (Authentication)，還不知道你是誰 (Identification) 的，更扯不上最後的不容抵賴性)
C. Authorization -> Authentication -> Identification (你連門都還沒進，連自己是誰都沒報上名，誰有空先拿名冊查你能進哪個房間 (Authorization)？大順序全錯)
D. Identification -> Authorization -> Accounting (記帳是進門以後看你吃了幾碗飯的最後一步(Accounting)，中間還少了一個人別驗證與賦權)

*   **正確答案 / Correct Answer：A**
*   **[解析邏輯]**
    *   **為何 A 是最佳選擇：** 這是存取控制 (Access Control) 最經典的 IAA (或 IAAA) 順序：
    *   **第一步 - Identification (身分識別)：** "聲稱我是誰 (Claim of Identity)"。例如你打了王大明的帳號 (Username)。
    *   **第二步 - Authentication (身分驗證/認證)：** "證明我真的是王大明 (Proof of Identity)"。題幹中使用了兩步驗證 (MFA)，密碼 (Type 1) + 手機驗證器 (Type 2)，成功證明了他是合法持有者。
    *   **第三步 - Authorization (授權)：** "驗證通過後，判定這傢伙有沒有資格做某事 (Determining Access Rights)"。系統查詢 Access Control List (ACL)，發現他是工程部的 Role，就把他試圖讀取 HR (人資部) 資料夾的動作封殺 (Access Denied)。

#### **11. A legacy operating system requires users to type complex, 20-character passwords containing symbols, numbers, and mixed-case letters, and forces them to change this password every 14 days. As a result, the helpdesk is overwhelmed with password reset requests, and security auditors find sticky notes with passwords written on them physically attached to over 40% of the office monitors. Which security design principle did the creators of this password policy fail to consider?**
**(在一座迂腐死板、用古老石板打造的長老神廟裡 (legacy OS)。大祭司頒布了一條猶如惡夢般變態恐怖的開門大律法：【『所有來上班的神職人員！每天進神廟，必須要給我背誦一段長達 20 個字、裡面夾帶著火星文符號、甲骨文亂碼、還要大小寫混雜的詭異地獄繞口令 (complex 20-character passwords)！而且這道咒語，大祭司規定每 14 天就要徹底燒毀重寫一遍 (change every 14 days)』】。結果這套自以為傲的防線不但防不了外星人，反而引發了自作孽的大災難：那群負責開門的工頭，每天都被幾千個忘記咒語卡在門外痛哭流涕的牧師們給逼到崩潰 (helpdesk overwhelmed)！更可悲的是，當御史大夫來視察時，差點沒吐血昏倒。他看見這神廟裡【居然有高達四成的祭司，直接把這串猶如天書的密碼，寫在隨風飄揚的黃色便利貼上，然後大喇喇地死死黏在他們那台對著大街敞開的螢幕邊框上示眾 (sticky notes attached to monitors)】！請問，設計這套自作聰明智障律法的大祭司，他在當年寫下這道命令時，傲慢地且徹底地把哪一條兵家防禦心理學大真理，給當成垃圾丟在地上踩踏？)**
A. Complete Mediation (這是每次都要檢查的原理，但大祭司這套規矩的死穴不是沒檢查，而是檢查得太反人類了)
B. Psychological Acceptability (Ease of Use) (這就是逼死人不償命、把人逼到造反走後門的千古絕響：【『極度漠視與強姦了渺小人類大腦極限之心理與人性可接受度 (Psychological Acceptability / Ease of Use)』】！這條真理咆哮指出：【安全機制如果設計得太麻煩、太反骨、太不順手、反人類到逼死這個使用者！這群為了偷懶為了活下去的人類，絕對會發明出世界上最愚蠢但最簡單的方法 (比如把天書密碼寫在螢幕上昭告天下) 去繞過你大費周章蓋起的這面銅牆鐵壁！】這就是安全與好用的終極拉扯法則！)
C. Fail-Safe Defaults (大祭司有預設沒背咒語就不准進，他有守護好這條失效安全底線，只是手段太激進)
D. Defense in Depth (如果大祭司有套用縱深防禦，他就不會只死守一條 20 碼的密碼，他會去買指紋機或雙重認證卡片，但他現在探討的是這單一政策的「人機介面仇恨值」，不是在探討城牆有幾道的問題)

*   **正確答案 / Correct Answer：B**
*   **[解析邏輯]**
    *   **為何 B 是最佳選擇：** **「心理可接受度 (Psychological Acceptability)」** (由 Saltzer 和 Schroeder 在 1975 年提出的經典資安設計原則)。
    *   **核心理念：** 安全機制不應該過度干擾或阻礙使用者的日常工作流程。"Security mechanisms should not make the resource more difficult to access than if the security mechanism were not present."
    *   **現實世界的反噬 (The Human Element)：** 任何極度嚴苛、違反人類記憶與行為模式的資安政策 (如題幹中：20位數亂碼 + 14天強迫更換)。人類大腦根本記不住。於是員工為了解決這個 "麻煩"，會創造出**極度危險但方便的「影子 IT 或繞道行為 (Workarounds)」**，例如把密碼寫在螢幕便利貼上，或是把密碼設成 `Password!@#1`, `Password!@#2`。這反而徹底摧毀了原本設立安全機制的防禦初衷。

#### **12. A development team is creating a new file-sharing application. To protect the files, they implement a web application firewall (WAF) at the network perimeter, require multi-factor authentication (MFA) to access the application, strictly enforce Role-Based Access Control (RBAC) at the application layer, and finally, encrypt the files at rest using AES-256 on the hard drive. What overarching security architectural strategy is this team demonstrating?**
**(一支精良的皇家打造大隊，正在修築一座裝著國家千萬機密的『天下文書交換大寶庫 (file-sharing application)』。為了守護這個寶庫，他們佈下了天羅地網！不僅在城池最遙遠的護城河邊際，架起了一座能把飛彈都攔下來的【神聖網頁防禦大火網 (WAF at network perimeter)】；你要進這個寶庫大廳，還得先交出你的護照跟隨身大印章【進行終極雙重搜身驗證 (require MFA)】；進了大廳，如果你的狗牌上面不見金光刺字，你就絕對別想跨過【大權限禁衛軍把守的角色生死線 (strictly enforce RBAC)】！最後，就算今天有駭客像地鼠一樣從地底下挖地道繞過了所有守衛，直接站在了存放機密的保險箱前。他絕望地打開箱子也會發現！裡面的每一份卷宗，居然都被塗上了【『最高級別、連神都解不開的 AES-256 終極加密塗層 (encrypt files at rest using AES-256)』】！請問，這支大隊一層又一層毫不手軟的心機大布陣。在天下兵法裡被尊稱為何等浩瀚防禦之神跡？)**
A. Security by Obscurity (他們是真刀真槍的蓋鎖頭上加密，絕對不是玩躲貓貓的隱匿式偷雞摸狗)
B. Zero Trust (零信任是這十年才紅的不信任何人的極端查哨流派，這裡提到了城牆邊界 (perimeter)，這更偏嚮傳統宏大的分層設防大思維)
C. Defense in Depth (Layered Defense) (這就是資安建城大業之所以能千秋萬載立於不敗之地的千古絕對正道王法：【『如俄羅斯娃娃般層層扒皮、剝絲抽繭的終極城牆相疊大堡壘術：縱深防禦 (Defense in Depth / Layered Defense)』】！)
D. Separation of Duties (這是防制一個人兼裁判又兼球員的權力制衡，這群大隊現在是在疊城牆，不是在分配兵權)

*   **正確答案 / Correct Answer：C**
*   **[解析邏輯]**
    *   **為何 C 是最佳選擇：** **「縱深防禦 (Defense in Depth)」** 或 **「多層次防禦 (Layered Defense)」** 是資安架構的最高指導原則。
    *   **核心思維：** "單一安全控制措施遲早會失效" (例如 WAF 破功，被 Zero-Day 打穿；或是員工被釣魚信件騙走密碼)。
    *   **多層次架構 (Onion Model 洋蔥模型)：** 必須在每一個層面都加上鎖。
        1.  **Network Layer (網路層)：** WAF 阻擋惡意流量。
        2.  **Identity Layer (身分層)：** MFA 防止密碼被盜用。
        3.  **Application Layer (應用層)：** RBAC 防止越權。
        4.  **Data Layer (資料層)：** AES-256 加密防止就算硬碟或資料庫整顆被駭客偷走，他也看不懂內容。
    *   這樣駭客必須在一次攻擊中，連續破解這「四道」完全不同的鎖，才能拿到寶物，大幅度提高了攻擊的難度與成本 (Cost of Attack)。

#### **13. What is the fundamental difference between "Authentication" and "Authorization" in secure software design?**
**(在所有初入資安殿堂的小菜鳥，第一天都會遇到的一對名為『雙生子大魔神 (Authentication/身分認證 & Authorization/授權賦權)』的終極靈魂大哉問前！請你用最直白、最銳利、連三歲小孩都能聽懂的白話文，一劍劈開並解答這對雙生魔神的『決定性本質與命脈職權大差異 (fundamental difference)』，究竟該去哪一間衙門上奏？)**
A. Authentication is checking *what* the user is allowed to do; Authorization is proving *who* the user is. (大錯特錯，完全把兩間衙門的職位與執掌聖旨給弄反顛覆了)
B. Authentication uses encryption; Authorization uses hashing. (這是在講那群打鐵匠用的密碼學法器，根本就不是這兩間門禁衙門在審判時所管的心法真諦)
C. Authentication verifies the identity of the user (Who are you?); Authorization determines the permissions and actions the authenticated user is allowed to perform (What can you do?). (這就是劈開迷霧的最標準真理解答：【那名叫 Authentication 的大衙門，他們只負責用放大鏡去盤問：「站住！你他媽的到底是誰？拿出證明來啊！(Who are you?)」！而那名叫 Authorization 的內務府大院長，他是等你過了第一關後，拿著聖旨清單來宣告：「查大明律例！既然你已經被驗明正身是西宮總管王阿火了，那你現在只能進去掃地，【這就是你能做的事情與極限、絕對不准你去碰皇帝的龍椅！(What can you do?)】」】！)
D. There is no technical difference; the terms are used interchangeably in modern IAM systems. (這是完全不學無術的懶惰胡扯，現代的最高 IAM 權限中樞正是把它們切得清清楚楚的兩刀)

*   **正確答案 / Correct Answer：C**
*   **[解析邏輯]**
    *   **為何 C 是最佳選擇：** 這是資安領域最基礎但最常被混淆的兩個概念。
    *   **Authentication (身分驗證，常縮寫為 AuthN)：**
        *   **目的：** 證明實質身分。
        *   **口訣：** "Who are you?" (你是誰？)
        *   **機制：** 密碼、生物指紋辨識、OTP、Smart Card 等。
    *   **Authorization (授權，常縮寫為 AuthZ)：**
        *   **目的：** 決定這個合法身分能夠執行的存取權限。
        *   **口訣：** "What are you allowed to do?" (你可以做什麼？/ 可以摸什麼資料？)
        *   **機制：** ACL (存取控制串列)、RBAC (以角色為基礎的存取控制)、ABAC 等。

#### **14. A piece of malware infects a financial system. It does not steal any data (no data exfiltration) and does not crash the system. Instead, it subtly alters the exchange rate variable in the memory by 0.001% every day for six months, leading to massive financial discrepancies in the quarterly reporting. Which aspect of the CIA triad has this specific attack targeted?**
**(一隻極度陰險狡詐的『幽靈微米木馬大惡蠱 (malware)』，悄無聲息地潛入了一家執掌國庫大計的光明金控系統。這隻惡毒的寄生小蟲與眾不同！牠極其自律，既不偷偷把金庫的黃金搬去變賣 (no data exfiltration)，也沒有發瘋似地把整棟金控大樓給炸垮放火燒山 (does not crash the system)。相反地，牠就像一根毒針，悄悄地爬上那台計算國庫黃金與他國白銀兌換率的最精算天秤機心臟上。這半年來，牠每天像鬼魅般，輕輕地、無法察覺地【把那個神聖的心臟法碼，只撥動了微乎其微的 0.001% 小數點 (subtly alters the exchange rate variable)】！直到半年後結算大會，國庫長官看著帳單上那驚濤駭浪般的『幾千億落差假帳金坑巨洞 (massive financial discrepancies)』！才絕望癱軟在地！請問這隻殺人不見血的幽靈惡蠱，牠精準且惡毒地鑿穿了 CIA 三神像中，哪一尊保衛真實不可欺神像的心臟？)**
A. Confidentiality (這隻蠱蟲只負責修改破壞真理數字，這不是要把國家機密偷出去給路邊大媽看的竊密洩漏勾當，隱私依然健在)
B. Integrity (這就是能殺人於無形、將信任體系連根拔起絞碎的：【『絕對真實與不容竄改之完整性大女神 (Integrity)』】！完整性的天條是「確保資料沒有遭受未經授權的修改或毀損」。這種俗稱『Data Diddling (資料微調偷改)』或是『Salami Attack (義大利牙籤臘腸大削切剝皮法)』。駭客不去把你的牆壁砸爛，他只去修改你銀行的存款尾數小數點。當你再也無法相信資料庫報表上的這個數字也是真的時。你這台機器的完整真理信仰，就徹底淪為廢鐵了！)
C. Availability (機台還能結算大報表、還能打單計算，只是算出來的數字全錯了。所以機台一直活著沒關門，保住了可用性)
D. Non-Repudiation (抵賴與否在此刻都已經來不及了，這國庫的帳單巨洞與竄改事實並不歸不可否認性所統領)

*   **正確答案 / Correct Answer：B**
*   **[解析邏輯]**
    *   **為何 B 是最佳選擇：** **「完整性 (Integrity)」**。
    *   **定義：** 防護資料或系統不被 "未經授權地修改或破壞 (unauthorized modification or destruction)"，確保資料的精確度 (Accuracy)、真實性 (Authenticity) 與可靠度 (Reliability)。
    *   **攻擊情境分析：** 此題描述的是經典的 **Salami Attack (薩拉米香腸/切片攻擊)** 或 Data Diddling。駭客每次只偷走金融交易尾數 0.001 元，單獨看一筆你根本不會發現。但累積了六個月巨大的交易量，這就是幾億元的損失。這類攻擊這正是 Integrity 神聖防線之所以極端重要的完美案例：如果資料的數值已經被暗中竄改，那這個系統即使永遠不當機 (Availability)，且沒人去偷裡面的資料 (Confidentiality)，也會是一場造成公司破產的金融災難。

#### **15. When defining Access Control Models for a new military application, the requirement dictates that subjects (users) and objects (files) must be assigned formal security clearance labels (e.g., Top Secret, Secret, Confidential). A user can only access a file if their clearance level is mathematically greater than or equal to the file's classification level, and the system administrator cannot override these mathematical rules. Which historic access control model is being implemented?**
**(在為兵部打造一個可以管理幾百萬刺客名單與將軍密報的『終極大內國防密報系統 (military application)』時。兵部最高統帥用劍指著大工程師，下達了死命令：【『聽好！在這個系統裡，不管是會走路喘氣的那些兵將 (subjects)，還是被鎖在櫃子裡的公文紙 (objects)！在這系統出生的那一瞬間，就必須被強迫在臉上，死死烙印上一個象徵著國家地位高低賤尊卑的『九品大芝麻官階極光大烙印標籤 (formal security clearance labels e.g., Top Secret)』！』】。而且這套階級制度，採用的是猶如鐵壁般殘酷無情的鐵血數學天條：【只要你這個小兵腦袋上烙印的階層級別分數，【沒有數學實質大於或是剛好等於】你想借出來看的那份神祕公文捲軸上烙印的致命危險分數層級 (clearance level must be mathematically greater than or equal to file's classification level)！那哪怕你是拿著玉皇大帝的聖旨或是城裡那個最大權限總管大太監來幫你開門說情，大門也絕對一秒鎖死封殺 (system administrator cannot override rules)！】請問！這等猶如軍規般絕對等級森嚴、鐵血無情連總管也無法逾矩的古老國防存取大陣法，尊號為何？)**
A. Discretionary Access Control (DAC) (這是最隨便的菜市場小老百姓法制，檔案誰寫出來的誰就能當山大王，自己隨便請客決定這隻檔案給誰看，毫無兵部那種皇帝決斷層層管制階級可言)
B. Role-Based Access Control (RBAC) (這是大公司排班表最愛用的群組角色權限控管法，他是看你的名片是寫「工匠」還是「太醫」，然後決定你進不進得了藥房大門，但他不是用這種國防等級的上下階層極端比對標籤法)
C. Mandatory Access Control (MAC) (這就是以鐵血無情、由國家統治中央機器強制打下絕對階層與類別標籤、且無人能推翻的：【『軍規級之強迫性最高指導存取大控制防線陣法 (Mandatory Access Control, MAC)』】！)
D. Attribute-Based Access Control (ABAC) (這是更花俏的條件組合法，除了看階級，他還會看你今天穿什麼衣服、外面有沒有下雨、早上八點還是半夜一點來決定開門放行，這跟此題的純粹階層階級烙印法意旨不搭)

*   **正確答案 / Correct Answer：C**
*   **[解析邏輯]**
    *   **為何 C 是最佳選擇：** **「強制存取控制 (Mandatory Access Control, MAC)」**。
    *   **核心特徵 (Labels & Clearances)：** MAC 是軍方或情報單位的標準配備 (如 SE-Linux)。它依賴兩個絕對的標籤：使用者的「安全許可證 (Clearance)」(如：這人身家清白可以看最高機密)，以及檔案的「安全分類標籤 (Classification/Sensitivity label)」(如：這份文件的等級是極機密 Top Secret)。
    *   **無法被僭越的鐵律 (System-Enforced)：** 在普通系統 (DAC) 中，我是檔案擁有者，我就能自己決定要不要分享給同事看。但在 MAC 世界裡。**即使我是檔案的創造者，或者我是全系統的 System Admin，也【沒有權力】自己更改檔案的讀取權限或分享給不具備該職等身分的同事**。系統的「安全核心 (Security Kernel)」會 100% 根據作業系統最底層內建的數學比對演算法，強制規定只有等級及格的人才能翻閱，這徹底杜絕了人為疏失或內鬼私相授受。

#### **16. A security consultant reviewing an application's architecture discovers that every time a user requests to view an image, the application simply checks if the user's session cookie is valid and then directly serves the image file based on the URL parameter (e.g., `image=receipt_105.jpg`). The application never verifies if the currently logged-in user actually *owns* or has the right to view `receipt_105.jpg`. Which security design principle is being blatantly violated here?**
**(一名安檢總教頭，在巡視這間名為『大帝國相簿管家 (application)』的城防配置時，發現了一條令人髮指的弱智懶惰罪狀！這間相簿管家，在門口迎賓借照片時，居然只叫守衛喵一眼來的人身上有沒有這家店發出的『只要買票進場就會給的合法通關手環憑證 (valid session cookie)』。只要看到有套手環，這管家就瞬間腦死。他連名單都不看一眼，【只要那個買票進場的客人，隨手指著深宮大院的圖片庫，大喊：「給我拿出你們家深鎖在抽屜裡的『第一零五號絕密收據大寶圖』出來給老子看！ (directly serves the image based on URL parameter)`」】，這管家就真的像個耳包瞎子一樣，直接把寶圖掏出來奉上！他這輩子從來沒有想到要再問一句最致命的盤問：【「等一下！老子雖然承認你有買票進我們商城。但你他媽的在我們國庫裡，有那張寫著你是這張絕密收據主人、或你有這個皇室資格能翻閱老王買春發票的授權證書嗎？ (never verifies if the logged-in user actually owns or has the right to view it)」】。請問，這名只查一次門票就放人大放送爽拿的弱智管家，他是把那神聖的哪一道檢查天條，當成大便踩在腳底下踐踏了？)**
A. Complete Mediation (這就是那道絕對殘酷、每次進門前就算你拿著皇帝金牌也得給我停下來搜身驗權限的：【『每次行為之絕對檢驗與終極完全仲裁大真理 (Complete Mediation)』】！這條真理咆哮著：你不能因為他剛進大門時查過一次身分證！當這王八蛋往金庫小房間走想拿照片時，你居然就不驗了！每一次、每一個系統存取的動作，都必須要被大會重新盤查此人的特權夠不夠資格！這家店的管家沒做到這點，完全犯了不可饒恕的懈怠死罪！)
B. Economy of Mechanism (經濟機制是說防禦機關要簡單，不要搞出一堆奇門遁甲自己把自己搞垮，這裡的管家倒是很簡單，但他太簡單所以出包了，這不是違背這個天條的原意)
C. Open Design (這是把自己的架構圖貼到天下武林大會公開讓大家挑毛病改進，與這管家的查證疏失無關)
D. Separation of Duties (這是規定一件大事要兩個人互相牽制避免作弊貪污，不是這個懶惰不管事放空城計的管家現在闖下的死穴)

*   **正確答案 / Correct Answer：A**
*   **[解析邏輯]**
    *   **為何 A 是最佳選擇：** **「完全仲裁 (Complete Mediation)」** 是這題描述的核心漏洞本質。
    *   **定義：** 每一次實體 (Subject/User) 對客體 (Object/File) 的存取請求，都【必須】通過系統安全核心的驗證及授權檢查。不可因為 "他 10 分鐘前登入成功過"，就快取或是跳過後續重要動作的權限檢查。
    *   **OWASP 經典漏洞體現：** 題幹中描述的漏洞，在 OWASP Top 10 中被稱為「不安全直接物件參考 (IDOR / Insecure Direct Object Reference)」或「授權損毀 (Broken Access Control)」。
    *   系統雖然有做 Authentication (確認有 Session Cookie、確實是個合法登入的員工)。但它卻沒有針對「這個特定資源 (receipt_105.jpg)」做 Authorization (這篇收據的所有權人是 User-B，User-A 憑什麼用修改 URL 就能看到？)。這個 "省去檢查授權步驟" 的懶惰實作，直接破壞了 Complete Mediation 的鐵率。
