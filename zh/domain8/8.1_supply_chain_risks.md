# 8.1 軟體供應鏈風險 (Software Supply Chain Risks)

## 學習目標

- 定義何謂軟體供應鏈
- 識別出幾種最致命且常見的供應鏈攻擊手法 (Attack Vectors)
- 了解過度仰賴第三方開源程式碼會帶來什麼樣的毀滅性後果
- 解釋「委外海外開發環境」所暗藏的資安黑洞

---

## 到底什麼是軟體供應鏈？ (What is the Software Supply Chain?)

幾十年前，一間軟體公司會驕傲地宣稱他們的產品 100% 都是由自家工程師在冷氣房裡一行一行敲出來的。但今天時代變了，如今市面上一個典型的企業級應用程式，其骨子裡高達 80% 到 90% 全都是由**第三方開源軟體組件、商業版 SDK 套件，以及串接外部別人的 API** 所拼湊而成的科學怪人。你們公司自己工程師寫的扣，充其量只是負責把這些外面撿回來的積木上點三秒膠黏起來而已。

所謂的**「軟體供應鏈」**，就是指：在你的軟體打包好交到最終客戶手上之前，曾經碰過這套軟體、或成為這套軟體一部分的「所有東西」：
- 它肚子裡的開源函式庫 (例如：jQuery、React、Log4j)。
- 寫出這套軟體的人 (自家薪水小偷工程師、或是遠在印度的便宜外包團隊)。
- 用來蓋房子的各種工具 (開發環境 IDE、編譯器、CI/CD 自動打包產線)。
- 支撐這套軟體活著的基礎設施 (如 AWS、Azure 雲端、或是公開的 Docker Hub 映像檔庫)。

## 駭客為什麼愛打供應鏈？ (The Attack Vector)

駭客為什麼現在都不直接攻擊你了，反而跑去攻擊供應鏈？答案就是：**「規模經濟與超高投資報酬率 (Scale and Efficiency)」。**

與其累得半死、一間一間去硬駭 10,000 家防護森嚴的大企業；聰明的駭客選擇去駭入**「這 10,000 家企業共同使用的一家上游供應商」** (例如：摸走某個超熱門開源套件作者的帳號，或是直接駭進像 SolarWinds 這種專門提供網路監控軟體給大財團的廠商)。駭客只要神不知鬼不覺地在那「一家」供應商的更新包裡塞進一個小小的木馬後門；隔天，當這 10,000 家企業乖乖按下「自動更新」時，駭客就等於在一天之內，同時拿到了這 10,000 家大企業的機房金鑰。這就是供應鏈攻擊的恐怖之處。

## 最常見的供應鏈攻擊手法 (Common Supply Chain Risks)

### 1. 遭駭客攻陷的第三方函式庫 (Compromised Third-Party Libraries)
駭客透過網路釣魚或密碼外洩，成功盜取了一名開源軟體維護者 (例如在 npm、PyPI 或 RubyGems 等公開套件庫上的神人) 的帳號權限。接著，駭客用這個神人的身分，發布了一個夾帶惡意木馬的「全新版本」。全世界的工程師只要一敲指令自動下載最新版，就等於歡天喜地把木馬迎進了自家公司的保險箱裡。

### 2. 相依性困惑攻擊 / 命名佔用 (Dependency Confusion / Namesquatting)
這是一招極度陰險的高級騙術。很多大公司在寫扣時，會同時混用「公司內部私有的套件」與「網路上公開免費的套件」。如果駭客在網路上瞎貓碰到死耗子，猜中了你們公司某個內部私有套件的名稱 (例如叫做 `company-auth-lib`)；駭客就會去公開的 npm 套件庫上，註冊一個名稱「一模一樣」、裡面包著病毒的假套件，並且**故意給它標上一個高到嚇死人的版本號** (例如 `v99.0`)。這時，你們工程師電腦裡的依賴套件管理程式 (package manager) 就會被搞到「精神錯亂 (confused)」：它看到外面有一個 `v99.0` 最新版，以為好棒棒，就會捨棄公司內部真正安全的那包舊程式碼，轉而把網路上那個帶毒的假套件給下載回來。

### 3. 錯字誤導攻擊 (Typo-Squatting)
如果你的套件很紅，駭客就會去註冊一個名稱「拼字只差一兩個英文字母」、看起來幾乎一模一樣的假套件；然後守株待兔，祈禱你們家剛睡醒的工程師在終端機敲安裝指令時打錯字。
- **經典範例**：Python 有個紅透半邊天的網路請求套件叫 `requests`。駭客就故意發布一個帶有病毒的套件，取名叫 `requsts` (少了一個 e)，然後就等著收割那些手殘打錯字的粗心工程師。

### 4. 淪陷的編譯打包產線 (Compromised Build Environments / CI/CD)
CI/CD 管線本身就是一個巨大的資安破口。如果駭客駭進了你們的 Jenkins 伺服器或 GitHub Actions；他們就可以趁著工程師已經做完所有的原始碼安全審查、正準備把程式碼打包成執行檔的**那個瞬間**，神不知鬼不覺地把惡意後門「注射 (inject)」進最終的二進制檔案裡。這招最可怕的地方在於：這份被下了毒的最終安裝檔，上面居然還完美地擁著你們公司「合法且有效」的微軟數位簽章！這是一份經過官方認證的合法毒藥。

### 5. 海外外包開發的安全黑洞 (Outsourced Offshore Development)
為了省錢找外包廠商，等於是你主動敞開大門迎接資安風險。你根本不知道、也管不到遠在印度的這個外包工程師，他每天用什麼爛電腦在寫扣？他們家的網路有沒有中毒？如果這位外包工程師的筆電早就被裝了木馬木馬，當他興高采烈地把寫好的程式碼 (Commit) 推送上你們公司的 GitHub 程式碼倉庫時，那隻木馬也就跟著偷渡進你們公司的內網了。

## 防禦之道 (Mitigating the Risk / High Level)

- **零信任心法 (Zero Trust)**：請用對待「從網頁表單上駭客輸入的有毒字串」一樣的防備心，來質疑每一行第三方開源程式碼。絕對不要只因為這個更新包上面貼著「微軟」或「Google」的標籤，就閉著眼睛無腦相信它。
- **釘死版本號 (Pinning Dependencies)**：這也是一條鐵律。絕對、永遠、打死都**不准**讓你們的套件管理工具 (package managers) 自動去抓網路上所謂的 `latest (最新版)` 或是 `* (萬用字元)` 版本。你必須在設定檔裡，把套件的版本號精準地「釘死 (Pin)」在你們已經測試過、確認安全的特定版本上 (例如只能抓 `1.4.2` 版)。
- **建置內部守門員套件庫 (Internal Repositories)**：嚴格禁止工程師的電腦直接連上外網去 npm 或是 Maven 這種公開市集下載套件。你必須在公司內網架設一道「防火牆代理倉庫 (proxy repository)」（例如：買一套 JFrog Artifactory 或是 Sonatype Nexus）。所有外面的開源套件都必須先下載到這裡，經過資安團隊用防毒掃描儀徹底洗過一次、確認無毒後，工程師才准把它拉進自己的專案裡用。

## 考試重點

1. **攻擊的高效槓桿 (Efficiency of Attack)**：為什麼供應鏈攻擊會讓全世界資安專家聞風喪膽？因為駭客只要駭爛「一家」上游供應商 (例如 SolarWinds)，就能如同核彈爆炸輻射般，瞬間牽連並攻陷下游成千上萬家原本防禦森嚴的超級大企業。
2. **相依性困惑攻擊 (Dependency Confusion)**：利用了安裝工具「盲目崇拜高版本號碼」的弱點，騙系統去下載外網高版本的病毒套件，而不是乖乖下載內網安全的低版本套件。
3. **釘死版本 (Pinning)**：資安實務上最重要的防禦動作之一。把套件版本號寫死鎖定，以防止套件管理工具在半夜擅自下載駭客發布的「惡意自動更新包」。
4. **第三方連帶風險 (Third-Party Risk)**：記住一個殘酷的事實：就算你們公司工程師寫的扣堪稱藝術品、完美無瑕 0 漏洞；只要你們匯入了「一個」充滿破洞的第三方開源函式庫，你們整個應用程式就會瞬間變成資安界的一個大笑話。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Software Supply Chain (軟體供應鏈)** | 由無數個供應商、開發工具箱、以及第三方開源程式碼所交織而成，用來支撐並組合出一個現代應用程式的龐大生態蜘蛛網 |
| **Dependency Confusion (相依性困惑攻擊)** | 一種駭客發明的騙術，專門欺騙無腦的套件安裝程式，讓它捨棄安全的內部私有庫，轉而投向外面帶木馬的公開版本套件懷抱 |
| **Typo-Squatting (錯字誤導攻擊)** | 惡意註冊一個名字跟那些超級爆款熱門套件「長得幾乎一模一樣，只差一個字母」的假套件，專門獵捕那些手速太快打錯字的粗心工程師 |
| **Version Pinning (釘死版本號)** | 在設定檔中，把依賴函式庫的版號給鎖得死死的 (例如嚴格指定 1.4.2)，嚴禁系統自作主張去亂載未經安檢的最新更新檔 |
