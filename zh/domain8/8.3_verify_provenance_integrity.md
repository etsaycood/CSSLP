# 8.3 驗證來源血統與完整性 (Verify Provenance and Integrity)

## 學習目標

- 在供應鏈安全的情境下，精準定義「來源血統 (Provenance)」與「完整性 (Integrity)」的差異
- 解釋「程式碼簽章 (Code Signing)」與「數位憑證 (Digital Certificates)」在防禦上扮演的角色
- 描述如何利用「密碼學雜湊函數 (Cryptographic Hashes)」來驗證檔案是否被動過手腳
- 理解 SLSA (軟體製品的供應鏈層級) 安全框架的防禦理念

---

## 來源血統 vs. 完整性 (Provenance vs. Integrity)

當你們家的工程師，在網路上看到一個超好用的全新開源函式庫，興奮地準備按下「下載 (Download)」鍵，把這包程式碼帶回公司內網之前；資安團隊必須確保系統已經完美回答了以下兩個攸關生死的關鍵問題。

| 概念 | 靈魂的拷問 (The Question) | 驗證神器 (The Tool) | 防禦的資安價值 (Security Value) |
|---------|--------------|----------|----------------|
| **來源血統 (Provenance)** | 「這包程式碼**到底是哪個爹娘生出來的？** 真的是微軟官方寫的嗎？」 | 程式碼簽章 (Code Signing) / 數位憑證 / 數位簽章 | **防止偽造冒牌貨 (Impersonation)** (例如：阻止某個俄羅斯駭客，在網路上戴著面具宣稱自己就是微軟官方，騙你下載他的惡意更新包)。 |
| **完整性 (Integrity)** | 「自從原作者親手把它發布到網路上之後，這包程式碼在漫長的運送途中，**有沒有被任何人偷偷打開來動過手腳？**」 | 密碼學雜湊函數 (Cryptographic Hashing) | **防止惡意竄改 (Modification)** (例如：阻止駭客在半路攔截了你下載到一半的驅動程式，偷偷在裡面塞進一隻木馬後門，然後再假裝沒事寄給你)。 |

這兩大神器加在一起，共同鑄造了資安界的鐵律：**「不可否認性 (Non-Repudiation)」**——如果未來這包程式碼真的被抓到裡面藏著病毒，原作者絕對無法在法庭上賴帳說「這不是我寫的，是別人塞進去的」。

## 驗證完整性：雜湊函數的魔法 (Verifying Integrity: Hashing)

雜湊函數 (像是目前業界最強的 SHA-256) 就像是一台神奇的果汁機；不管你丟進去的是一個 1KB 的純文字檔，還是一個 10GB 的電影檔，它都會經過複雜的數學運算，吐出一串**「長度永遠固定、且全世界獨一無二的亂碼字元」** (這就稱為雜湊值 hash value，或是摘要 digest)。

- **完整性驗證流程 (The Integrity Check)**：
    1. 真實的微軟工程師寫好程式，把它打包成 `library.zip`，然後用 SHA-256 算出它的專屬雜湊值 (例如 `A1B2C3...`)。
    2. 微軟把 `library.zip` 放上公開下載區，並在微軟**最權威的官方網站上**，同步公布這串 `A1B2C3...` 雜湊值。
    3. 你下載了 `library.zip`。接著，你在自己的電腦上，用自己的軟體親手幫這包檔案重算一次 SHA-256 雜湊值。然後把算出來的結果，拿去跟微軟官網上的那一串比對。
    4. **如果兩串亂碼「一模一樣、一字不差」**：恭喜，檔案的完整性過關。這證明檔案在網路傳輸中絕對沒有被黑帽駭客竄改。因為只要駭客在裡面偷偷多加了哪怕只是「一個字元 (a single bit)」的病毒碼，算出來的雜湊值就會瞬間變得面目全非、完全不一樣 (這是密碼學中的「雪崩效應 Avalanche Effect」)。

> **致命弱點 (Weak Hashes)**：現在去考資安證照，**絕對、打死都不准**再依賴古董級的 MD5 或是 SHA-1 來做檔案驗證了。因為它們早就被證實存在「碰撞攻擊 (collision attacks)」的弱點 (意思是：駭客已經有能力，寫出一個充滿病毒的假檔案，但它算出來的 MD5 雜湊值，竟然跟官方原本的一模一樣！這會直接摧毀你的防禦)。

## 驗證來源血統：程式碼簽章的鐵證 (Verifying Provenance: Code Signing)

雜湊函數很棒，它能證明檔案沒被竄改；但問題來了，它沒辦法證明**這檔案「一開始到底是誰寫的」**。如果有個駭客直接把微軟伺服器給駭了，把下載區的檔案換成他的病毒包，然後順便把旁邊公布的那串雜湊值，也換成他算出來的「病毒包雜湊值」呢？這時你下載比對，發現兩者相符，就傻傻地把病毒安裝下去了。

為了解決這個血統危機，我們必須召喚**「程式碼簽章 (Code Signing)」**，這是將雜湊函數與非對稱式密碼學 (公開金鑰基礎建設 PKI) 完美結合的終極防禦。

- **運作流程 (The Process)**：
    1. 原廠工程師一樣先算出 `library.zip` 的雜湊值。
    2. 接著，工程師拿出只有他們公司自己嚴密保管的**「私鑰 (Private Key)」**，把這串雜湊值給「加密鎖起來」。這串被私鑰鎖起來的雜湊值，就是傳說中的**「數位簽章 (Digital Signature)」**。
    3. 廠商把這個數位簽章，連同跟私鑰成對的「公鑰 (Public Key，這個公鑰有經過全球權威且可信的數位憑證授權中心 CA 蓋章認證)」，一起綁在 `library.zip` 身上開放下載。
    4. 你下載完檔案後。你只能用廠商附隨的「公鑰」，去把那個數位簽章給**解密打開**。打開後，你會看到廠商原本算好的那串雜湊值。然後你再自己對檔案算一次雜湊值，兩者一比對。
- **最終防禦結果 (The Result)**：因為全世界**只有那把真正的「公鑰」，才能解開由那把唯一「私鑰」所上的鎖**。所以當解密成功的那一刻，你在數學上得到了絕對的保證：這個檔案 100% 就是握有那把私鑰的微軟原廠所發布的 (**這證明了來源血統 Provenance**)。同時，因為解開防護罩後的雜湊值比對也吻合，你又得到了第二重保證：檔案沒被動過手腳 (**這證明了完整性 Integrity**)。

## 防堵供應鏈攻擊的終極框架：SLSA

**軟體製品的供應鏈層級 (SLSA, Supply-chain Levels for Software Artifacts，發音為 "salsa")** 是一個由 Google 與 Linux 基金會大力推動的跨業界安全防禦框架。它的誕生，就是為了防堵軟體建置產線被竄改，確保我們下載套件的完整性。

SLSA 的核心靈魂拷問是：*「我們剛下載回來的這包開源軟體，我們到底有多大的信心，確信牠在被編譯打包 (build process) 的過程中，沒有被外星人或駭客在中途偷塞奇怪的東西進去？」*

SLSA 定義了循序漸進的四個成熟度安全防護層級：
- **Level 1 (初階)**：整個軟體打包建置流程是全自動腳本化的，而且我們找得到「來源血統證明 (provenance)」(至少知道是哪家工廠打包的)。(這只是最基礎的防護)。
- **Level 2 (進階)**：打包建置的伺服器，會自動產出一份「經過認證的血統證明書 (authenticated provenance)」。(這層級開始能有效防止駭客在軟體打包中途動手腳)。
- **Level 3 (高階)**：開發原始碼與建置產線的平台，必須符合極高標準的資安規定 (例如：使用「用完即丟的臨時建置環境 Ephemeral environments」，以及與世隔絕的建置孤島 isolated builds)。
- **Level 4 (終極防護)**：最高境界。要求全公司所有的程式碼更動，都必須有「不相關的兩位工程師共同審查背書 (two-person review)」，並且強制要求實作**「真空建置 / 密封建置 (Hermetic reproducible builds)」**(整個打包編譯過程是在一台完全拔掉實體網路線、與外界 100% 絕對物理隔離的真空無菌室電腦裡進行，以保證今天跟明天的編譯結果會一模一樣，絕無外力介入)。

> **考試秘訣 (Exam Tip)**：SLSA 是一個正如火如荼發展中的新興資安標準，它的火力徹頭徹尾**聚焦於保護「自動化 CI/CD 建置管線 (build pipeline)」**，使其免遭毀滅性的軟體供應鏈攻擊。

---

## 考試重點

1. **來源 (Provenance) = 「誰寫的 (Origin)」**：這是為了防偽造冒充。主要是透過「數位簽章 (PKI 架構下的私鑰與公鑰)」來證明。
2. **完整性 (Integrity) = 「有沒有被改過 (Unchanged)」**：這是為了防竄改。主要是透過「密碼學雜湊函數 (如 SHA-2 / SHA-256)」來證明。
3. **不可否認性 (Non-Repudiation)**：只要你同時擁有上述兩者 (來源血統 + 檔案完整性)，那個發布勒索病毒的原凶就絕對無法在法院上狡辯說「這軟體不是我發出去的」。
4. **SLSA 框架**：專門為保護「軟體建置產線 (CI/CD)」而生，用以擔保最終產出軟體完整性與清白的新世代資安標準防禦框架。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Provenance (來源血統/出處)** | 一個軟體製品從無到有，每一階段的所有權歷史紀錄與身分產地證明 |
| **Integrity (完整性)** | 一種絕對的保證，確保資料或程式碼在傳輸或儲存中，絕對沒有遭受「未經授權」的惡意竄改 |
| **Code Signing (程式碼簽章)** | 用數位方式在執行檔上簽名背書的流程；用來向世人證明「這確實是我寫的軟體」，同時保證內部檔案「沒有被人偷換過」 |
| **Digital Signature (數位簽章)** | 利用非對稱式加密所打造出的密碼學防護罩；能同時實現身分來源認證、確保資料完整性，並達成無可抵賴的不可否認性 |
| **SLSA** | Supply-chain Levels for Software Artifacts；一套專為防堵駭客入侵 CI/CD 產線而量身訂做的全球資安框架模型 |
| **Hermetic Build (真空建置/密封建置)** | 將軟體打包的伺服器，與外部網際網路進行 100% 絕對拔線物理隔離的極端作法，確保建置過程的純潔無瑕與絕對可重製性 |
