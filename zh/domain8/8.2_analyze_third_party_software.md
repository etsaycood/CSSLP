# 8.2 分析第三方軟體 (Analyze Third-Party Software)

## 學習目標

- 解釋「軟體組成分析 (SCA, Software Composition Analysis)」在現代開發中無可取代的地位
- 清楚分辨「寬鬆型 (permissive)」與「傳染/限制型 (restrictive)」開源軟體授權合約的致命差異
- 識別出用來揪出第三方開源程式碼中潛藏漏洞的自動化利器與方法

---

## 軟體組成分析 (SCA, Software Composition Analysis)

既然現代的應用程式，根本就是一塊由無數個外面撿回來的函式庫所拼湊而成的「百衲被」；那你想要靠自家工程師用肉眼一行一行去進行「人工程式碼審查 (manual code review)」，完全是癡人說夢。用來對付並分析這堆第三方開源軟體的唯一重型武器，叫做**「軟體組成分析 (SCA)」工具**。

**SCA 是怎麼運作的：**
1. 工程師按下編譯建置 (Build) 按鈕，或是把程式碼推上 Git 伺服器。
2. SCA 工具會立刻跳出來，強勢掃描你的套件清單檔 (例如 Node.js 的 `package.json`、Java 的 `pom.xml`、Python 的 `requirements.txt`)，有時候甚至會直接拿放大鏡去掃描編譯好的二進制檔案。
3. 它會精準羅列出你的應用程式到底吃進了哪些函式庫、採用了哪個「精確的版本號碼」；更厲害的是，它還能順藤摸瓜，把你沒看見的**「傳遞性依賴 (transitive dependencies)」**(也就是：你引用的 A 函式庫，它自己私下又偷偷引用了 B 函式庫) 全盤托出。
4. 接著，SCA 會拿著這份極度詳細的清單，連線去比對全世界的「已知漏洞資料庫 (例如美國政府的 NVD 國家弱點資料庫)」。只要發現清單中任何一個組件身上掛著還沒修補的 CVE 漏洞，系統就會立刻亮紅燈。
5. 最後，它還會順便檢查這些開源庫身上貼著什麼樣的**「版權授權合約 (licenses)」**，確保你們公司不會因此吃上官司。

> **資安防禦價值的體現 (Security Value)**：SCA 是用來完美斬殺 OWASP Top 10 中「A06：不安全及過時的組件 (Vulnerable and Outdated Components)」這個致命風險的唯一解藥。

## 拆解開源軟體授權的法律地雷 (Analyzing Open Source Licensing Risks)

搞資安，不只要防駭客偷資料，還要防公司被律師告到破產。要知道，開源軟體 (OSS) 雖然標榜「免費使用」，但免費的背後往往附帶著極度嚴苛的「法律使用條件」。

如果你的天兵工程師，不小心把一個帶有「錯誤版權授權」的開源函式庫，給偷偷塞進了你們公司準備拿出去賣錢的「商業閉源軟體」裡面；最慘的結局是：**你們公司可能會被法院強制勒令，必須把你們家幾百萬行的商業機密程式碼，全部免費公開到網路上供全人類下載。**

### 駭人聽聞的授權條款分類 (License Types)

| 授權類別 | 知名代表作 | 殘酷真相的解釋 | 對企業的商業風險 (Business Risk) |
|------------------|----------|-------------|---------------|
| **寬鬆型授權 (Permissive)** | MIT, Apache 2.0, BSD | 佛心來的。給予開發者極大的自由。你可以隨便用、隨便改，甚至把它包進你們公司那個「打死不公開原始碼的商業軟體」裡面拿去賣錢，原作者也不會逼你公開你們的商業機密。 | **極低 (Very Low)** |
| **著佐權 / 傳染型/ 限制型 (Copyleft / Restrictive / Viral)** | GPL (GNU 通用公眾授權條款) v2 / v3 | 具有恐怖的**「病毒傳染性 (Viral)」**。只要你的商業軟體裡「沾到」了一行 GPL 授權的程式碼，你整個軟體就會被「合法感染」，**強制變成 GPL 授權**。一旦你把這套軟體賣給別人 (散佈 distribute)，在法律上，你**必須、絕對、無條件地把你整個商業軟體的原始碼，連同 GPL 程式碼一起免費對外公開。** | **極致毀滅性 (Extremely High)** (特別是對那些靠賣閉源軟體賺錢的商業公司而言) |

### 開發者的盲區法則 (The Developer Blind Spot)
請認清現實：**工程師根本不懂、也不想懂版權法律。** 他們只會在遇到寫不出來的功能時，上網 Google 搜尋一個能用的函式庫，然後毫不猶豫地按下載匯入。
- **保命的護欄控制 (The Control)**：這就是為什麼你必須在 CI/CD 產線裡裝設 SCA 工具。SCA 會自動幫忙檢查掃描這些套件的授權條款；只要發現有人偷夾帶 GPL 或其他高風險的傳染型版權進來，**SCA 就會直接「折斷建置產線 (break the build)」，不准這段程式碼被合併上去。**

## 揪出第三方世界裡的漏洞 (Discovering Vulnerabilities in Third-Party Code)

當資安界又爆出一個毀天滅地的核彈級漏洞時 (例如 Log4Shell 或 Heartbleed)，你們家的資安長 (CISO) 衝進辦公室問的第一句話絕對是：***「我們公司有沒有中鏢？」***

在古代，要回答這個問題，IT 部門得花上幾個禮拜拋家棄子瘋狂加班，登入幾百台伺服器裡去人工撈看看有沒有裝這個該死的函式庫。在講求速度的現代，這種做法絕對是等死。

### 救命神器：軟體物料清單 (The SBOM, Software Bill of Materials)
為了解決這個搜查噩夢，現代資安的終極解法就是：強制要求公司開發或採購的「每一套軟體」，都必須隨身附帶一份 **SBOM**。

- **嚴謹定義**：SBOM 是一份極度正式、且被設計成「機器可讀 (machine-readable)」的盤點清冊。裡面詳細記錄了這套軟體在打造過程中，吃了哪些第三方組件、它跟它們之間錯綜複雜的血緣供應鏈關係。簡單來說，它就是一份**「程式碼的食品成分標籤 (ingredients label)」**。
- **業界標準格式 (Formats)**：現在最紅的兩大門派是 CycloneDX (由 OWASP 推動) 以及 SPDX (由 Linux 基金會推動)。
- **在災難應變時的無價之寶 (Incident Response Value)**：當資安長怒吼問：「我們有沒有受到 Log4Shell 影響？」時，資安團隊只要轉身在公司中央的 SBOM 戰情資料庫裡下個搜尋指令。**短短幾秒鐘內**，系統就會吐出一張清單，精準告訴你這幾千個專案裡，只有哪 5 套內部系統的肚子裡剛好吞了那個帶毒的 Log4j 版本。你馬上就能派搶修小組過去精準打藥修補，而不是全公司像無頭蒼蠅一樣亂竄。

## 程式碼託管機制 (Code Escrow)

想像一個情境：你的大公司花了一千萬，跟一家看起來很有潛力的 5人新創小公司，買了一套極度關鍵的專利商業系統。結果兩年後，那家新創公司倒閉破產、老闆跑路了。你們手上的這套系統瞬間變成了無人維護、也沒人知道怎麼修 bug 的孤兒。

為了防止這種慘劇，你需要：
- **原始碼託管 (Source Code Escrow)**：這是一種極度高明的法律防衛機制。在當初簽約買軟體時，你就逼著那家新創廠商必須把他們那套「視如命脈、絕對不給你看的商業機密原始碼」，乖乖交給一個受到雙方信任、絕對中立的第三方公證機構（稱為**託管代理人 / Escrow Agent**）鎖進保險箱裡保管。
- 一旦那家新創公司真的不幸倒閉，或是他們違反合約死不肯修 bug 擺爛時（這叫觸發「釋放條件 / release condition」），這個第三方公證機構就會打開保險箱，**把那套價值連城的原始碼直接交到你們手裡**。讓你們公司可以自己接手養活這套系統。

## 考試重點

1. **白箱掃描大對決：SCA vs. SAST**：這兩個都是白箱測試工具。但請這樣記：SAST 是用來抓**「你家工程師自己手寫的笨蛋漏洞」**；而 SCA 則是用來抓**「別人寫的第三方開源庫裡夾帶的 CVE 漏洞與版權糾紛」**。
2. **著佐權 (GPL) 的傳染風險 (Copyleft Risk)**：把帶有 GPL 授權的開源工具包進你們家賣錢的商業軟體裡，在法律上會強迫你必須「開源公開你們家的商業機密原始碼」。而 MIT/Apache 這類的寬鬆授權，才是商業公司的好朋友。
3. **SBOM 軟體物料清單的神聖地位 (SBOM Importance)**：SBOM 就是一套軟體的「成分表」。當外面爆發新的第三方核彈級漏洞時，SBOM 是讓你能在幾秒內確認自己有沒有中鏢、啟動快速災難應變的救命羅盤。
4. **程式碼託管 (Code Escrow)**：這是當你跟小公司買核心軟體時，用來保護自己免於「供應商倒閉爛尾」這種商業風險的保命大絕招。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **SCA** | Software Composition Analysis (軟體組成分析)；專門拿來掃描、盤點並揪出你肚子裡吃了哪些過期、帶毒第三方開源函式庫的 X 光機 |
| **Copyleft / Viral License (著佐權 / 傳染型授權)** | 一種極度霸道的開源版權 (例如 GPL)。它規定只要你的軟體沾到它的一滴血，你整個軟體就必須連帶繼承它的開源規範，強制公開原始碼 |
| **Permissive License (寬鬆型授權)** | 佛心開源版權 (例如 MIT, Apache)。只有極少數的限制，你可以放心地把它拿去改包在你的商業軟體裡賣錢 |
| **SBOM (軟體物料清單)** | Software Bill of Materials；這是一份標準機器可讀的「軟體成分標示表」，清清楚楚列出你的軟體是由哪些套件拼成的 |
| **Escrow (程式碼託管)** | 一種將原廠原始碼交給第三方中立單位保管的法律合約機制；確保當供應商破產倒閉時，買家還能拿到原始碼自力救濟 |
