# 7.4 問題管理 (Problem Management)

## 學習目標

- 清楚且銳利地區分「事件管理 (Incident Management)」與「問題管理 (Problem Management)」之間本質上的不同
- 描述「問題管理」被設立的終極目標與使命
- 能夠實際應用「根本原因分析 (Root Cause Analysis, RCA)」技巧
- 解釋「連續問五次為什麼 (5 Whys)」以及「石川魚骨圖 (Ishikawa/Fishbone diagrams)」的運作邏輯

---

## 救火隊長 vs. 建築結構大師 (Incident Management vs. Problem Management)

這兩者是 ITIL (資訊科技基礎架構資料庫) 框架裡緊密相連、卻常常被搞混的兩個死對頭概念；他們的目標完全南轅北轍。

| 特性比較 | 事件管理 (Incident Management / 救火隊) | 問題管理 (Problem Management / 結構技師) |
|---------|---------------------|--------------------|
| **終極目標 (Goal)** | 不擇手段，**在最短、最快的時間內** 讓服務恢復正常運作。 | 挖出埋在最深處的**根本原因 (Root Cause)**，確保這場災難這輩子永遠不會有機會發生第二次。 |
| **處理步調 (Speed)** | 急如星火 / 被動反應式 (Reactive)。他們只在乎先把眼前的火撲滅。 | 慢條斯理 / 主動預防式 (Proactive)。他們負責事後在廢墟裡拿放大鏡做深度調查與分析。 |
| **生動比喻 (Analogy)** | 頭痛欲裂時，二話不說先吞兩顆普拿疼止痛藥。 | 跑去大醫院掛號照核磁共振，找出你到底*為什麼*會長期慢性頭痛，然後開刀把腦瘤切掉徹底治癒。 |
| **最終產出 (Output)** | 應急的權宜之計 (Workarounds)、拿膠帶貼一貼的臨時修補。 | 永久性的結構大改造、軟體底層架構的翻修改寫。 |

### 兩者相愛相殺的關係 (The Relationship)
當同一個「事件 (Incidents)」連續發生好幾次時，這就是在強烈暗示底下藏著一個巨大的「問題 (Problem)」。 
- **情境**：1 號網頁伺服器突然當機崩潰了 (這叫 Incident 1)。救火隊二話不說直接把它重開機 (恢復服務)。 
- 接著，2 號網頁伺服器也跟著崩潰了 (這叫 Incident 2)。救火隊又把它重開機了。 
- 此時，**問題管理 (Problem Management) 團隊** 正式介入調查。他們不關心怎麼重開機，他們花了一週看懂了程式碼，最後抓到兇手：原來是程式碼裡長著一個致命的「記憶體外洩 (memory leak)」蟲子，這才是導致伺服器每天都會被慢慢塞爆並當機的元凶。

## 根本原因分析 (Root Cause Analysis, RCA)

RCA 是一套系統化、有組織的抽絲剝繭分析流程，目的是要找出引發系統故障的那道「最源頭的罪魁禍首」。 

解決表面症狀 (事件管理) 只能換來短暫的安寧；唯有徹底解決根本原因 (問題管理)，才能為公司帶來一勞永逸的永世太平。

### 抓漏神技 1：連續問五次為什麼 (The "5 Whys")
這是一種極度簡單粗暴、卻異常有效的反覆盤問技巧，用來逼迫大家探究因果關係。你只需要像個執拗的小孩一樣，對著同一個問題連續問大約五次「為什麼？」，直到你挖出最深層的核心爛瘡為止。

**災難情境示範**：公司最賺錢的電商資料庫，被駭客用 SQL Injection (資料隱碼攻擊) 整個端走了。
1. **為什麼資料庫會被駭客端走？** 
   - 答：因為駭客在登入畫面上，成功發動了 SQL Injection 攻擊。
2. **為什麼駭客的 SQL Injection 攻擊會成功？**
   - 答：因為登入畫面的輸入欄位，根本沒有做字元過濾驗證，也沒有套用參數化查詢 (parameterized)。
3. **為什麼工程師寫扣時，沒有套用參數化查詢？**
   - 答：因為這名工程師偷懶，直接複製貼上十年前舊系統裡的老舊資料庫連線語法。
4. **為什麼工程師會去用那種不安全的上古時代語法？**
   - 答：因為這名工程師打從進公司以來，從來沒有受過半小時的「安全程式碼撰寫」教育訓練。
5. **為什麼公司沒有辦安全程式碼的教育訓練？**
   - 答：因為高層根本沒有編列這筆預算，也沒有硬性規定工程師必須上這門課。 *(抓到了！這就是你的 Root Cause 根本原因！)*

> **資安防禦價值的體現 (Security Value)**：與其讓工程師只針對那個登入畫面打上修補程式 (這叫事件管理)；公司高層看了 RCA 報告後痛定思痛，決定實施全公司強制性的「安全寫扣訓練」，這項決策將會在未來幾年內，成功預防數百個潛在漏洞的誕生 (這就是偉大的問題管理)。

### 抓漏神技 2：石川魚骨圖 (Ishikawa / Fishbone Diagram)
這是一套極具視覺衝擊力的工具，也被稱為「因果圖 (Cause-and-Effect diagram)」。它能幫助團隊把可能導致災難的所有零碎原因，有系統地分門別類，進而鎖定真兇。

魚的「頭」代表著爆發的災難問題。而魚背上的「大骨刺」，則代表著幾大類別的潛在肇因，通常包含以下五種：
- **人員 (People)**：(例如：員工沒受過訓練、連續加班太累眼花、或是內鬼惡意破壞)
- **流程 (Processes)**：(例如：QA 品保環節形同虛設、變更管理流程太過隨便)
- **技術裝備 (Technology/Equipment)**：(例如：伺服器硬體太老舊、根本沒買 WAF 防火牆)
- **材料 (Materials)**：(例如：引用了充滿破洞的第三方開源軟體庫)
- **環境 (Environment)**：(例如：機房網路不穩、停電)

透過逼迫團隊針對每一根魚骨頭進行腦力激盪，大家就能用「宏觀且全面的上帝視角」去揪出所有可能的共犯結構，而不是盲目地把錯全推給某單一的硬體故障。

## 懸案與權宜之計 (Known Errors and Workarounds)

在問題管理的世界裡，有時候你雖然已經靠著 RCA 成功抓出了導致系統當機的「根本原因」，但礙於公司當下沒錢、沒時間、或是技術實在太過錯綜複雜，導致團隊**「無法立刻進行永久性修補」**；此時，這個懸案就會被官方正式歸類建檔為**「已知錯誤 (Known Error)」**。

- **已知錯誤資料庫 (KEDB, Known Error Database)**：公司 IT 部門必備的一本藏經閣 (資料庫)。裡面登記了滿滿的已知系統地雷，並且附上了官方認證核准的**「繞路避雷指南 (Workarounds)」**。
- **權宜之計/繞道方案 (Workaround)**：這是一種用來幫系統爭取時間的「臨時續命解法」；它能在永久修復方案出爐前，暫時恢復服務運作或是降低風險的殺傷力。(舉例來說：「這套軟體底層有個改不動的嚴重記憶體外洩問題。*權宜之計：在原廠明年第一季釋出大改版之前，請 IT 人員寫一支腳本，每天凌晨 3 點強迫這台伺服器自動重新開機一次以釋放記憶體。*」)

## 考試重點

1. **事件 vs. 問題 (Incident vs. Problem)**：事件 (Incident) = 目標是「恢復呼吸心跳越快越好」；問題 (Problem) = 目標是「找到病因永絕後患」。
2. **5 Whys (連問五次為什麼)**：一種如剝洋蔥般的簡單反問技巧，用來鑿穿表面症狀，直達深層的流程與管理層面缺陷。
3. **魚骨圖 (Fishbone Diagram)**：將錯綜複雜的潛在肇事原因，強制歸類進 (人員、流程、技術設備) 等類別中的視覺化分類法。
4. **已知錯誤 (Known Error)**：這是一個「已經找到兇手 (根本原因)」，也已經想好怎麼「臨時繞路苟活 (workaround)」，但就是「還沒有辦法徹底根治填平」的大坑洞。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Problem Management (問題管理)** | 掌管發掘並根治系統疾病的流程；唯一目標就是揪出導致災情不斷爆發的「根本原因」 |
| **Root Cause Analysis (RCA, 根本原因分析)** | 一套有系統、有邏輯的辦案手法，用以追溯系統故障的最原始源頭 |
| **5 Whys (五個為什麼)** | 一種極其死纏爛打的疊代式追問分析技巧，用以鑿出問題的核心病灶 |
| **Ishikawa / Fishbone Diagram (石川圖/魚骨圖)** | 一種將可能導致災難的各種病因，以魚骨形狀進行視覺化分門別類的因果分析圖 |
| **Known Error (已知錯誤)** | 一個被官方登記在案、已經查出是因為什麼根本原因所引發，且備有臨時應急方案的系統地雷 |
| **Workaround (權宜之計 / 繞道方案)** | 在偉大的永久修復程式被開發出來之前，用來臨時恢復系統運作、或是稍微減輕資安暴擊傷害的狗皮膏藥解答 |
