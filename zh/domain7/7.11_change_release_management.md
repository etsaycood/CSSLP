# 7.11 變更管理與發布管理 (Change and Release Management)

## 學習目標

- 描述「變更管理 (Change Management)」的核心流程與運作邏輯
- 解釋「變更諮詢委員會 (CAB, Change Advisory Board)」所掌握的生殺大權角色
- 定義何謂安全的發布戰略，以及版本控制的神聖性
- 清楚辨識在軟體發布上線流程中，絕對不可妥協的「職責分離 (separation of duties)」原則

## 變更管理流程 (The Change Management Process)

在 Production 真實營運環境中，那種「未經授權的隨意異動」或是「計畫不周延的草率更新」，絕對是導致系統大斷線與遭駭客攻陷的頭號戰犯。變更管理的終極使命，就是為了拿一副手銬，把那些老愛自作主張「偷連上正式主機順手改個錯字」的暴走工程師與系統管理員給銬起來。

**刻在石板上的黃金鐵律 (The Golden Rule)**：任何人、任何情況下，若沒有經歷過「正式提出申請、層層審查、委員會批准、沙箱測試，以及寫妥退路計畫 (rollback plan)」，**絕對不准**對 Production 營運環境的系統做哪怕只有一行字元的修改。

### 變更管理的核心生死六步 (Core Steps of Change Management)
1. **提出變更申請單 (RFC, Request for Change)**：寫一份文情並茂的公文，鉅細靡遺地交代你到底想對系統動什麼手腳 (不管是打補丁、推新功能上線，還是改個設定檔的參數)。
2. **審查與風險評估 (Review & Risk Assessment)**：相關單位的長官開始拿放大鏡審核：你這包改動，會不會偷偷把系統門戶大開(資安風險)？會不會把主機卡死(效能衝擊)？客戶今天晚上在使用時會不會斷線爆氣？
3. **核准放行 (Approval - The CAB 審判庭)**：「變更諮詢委員會 (CAB)」會召開大會，公審這張 RFC，最後決定要給你蓋章通過還是退件。
4. **沙盒試錯 (Testing)**：你那包程式碼，必須先送進一個跟外面真實世界 (Production) 長得一模一樣的實驗室 (Non-Production / Staging / QA 區) 裡跑跑看，確認它真的不會把系統搞炸。
5. **發布實作 (Implementation / Release)**：等到月黑風高的「核准例行維護時段 (maintenance window)」，才敢小心翼翼地把新版程式碼推上線。
6. **事後驗屍檢討 (Post-Implementation Review)**：上線後，去看看這包改版到底活得好不好？有沒有引發客訴大崩潰 (incidents)？如果有的話，二話不說，立刻啟動備用的「退版復原計畫 (rollback plan)」退回舊版。

### 非常時期的緊急變更 (Emergency Changes)
如果今天公司剛中勒索軟體、系統大斷線，或是面臨一個在外面把人劈成兩半的零日漏洞 (zero-day vulnerability)，你要是還叫工程師去填 RFC 表單，然後傻傻等下下禮拜二的 CAB 委員會開會來審批，那公司早就先破產了。
- **緊急變更諮詢委員會 (ECAB, Emergency Change Advisory Board)**：這是一個由極少數最高決策長官組成、手握無限權限的「特種救火特戰隊」。他們有權力為了公司存亡，直接在一通半夜打來的電話裡或通訊軟體上，拍板「立刻打上這包熱修復補丁 (out-of-band patches)」。
- **特權背後的代價 (Rule)**：在這種緊急狀況下，繁文縟節可以先省略；**但是！所有的文書審查手續與公文，事後必須「全部溯及既往 (retroactively)」補齊銷案。**

## 發布管理 (Release Management)

如果說「變更管理」是那掌握生殺大權、負責蓋「准奏」印章的**大腦**；那「發布管理」就是負責在戰場第一線、把真刀真槍的軟體安全裝到伺服器上的**手腳**。

### 發布流程中的保命控制措施 (Key Controls in Release Management)
- **職責分離 (SoD, Separation of Duties / 最容易考)**：那個負責在鍵盤上敲出新版程式碼的開發工程師，**打死也不准**握有可以直接把扣推上 Production 正式站台的帳號密碼。必須由一位專心致志的「發布管理工程師 (Release Engineer)」(或是無情的自動化 CI/CD 管線) 來代為執行發布上傳的動作。這道防火牆，是為了防止某個暴走的內鬼工程師，偷偷繞過同儕代碼審查，趁半夜把寫好後門木馬的程式庫悄悄塞進伺服器裡。
- **組態建構基準檔 (Configuration baselines)**：發布單位必須有一本清冊，清楚記載著「今晚要發布的這個偉大的『2.0 版本』，裡面到底是由哪 150 支特定檔案，與哪 38 個指定版本的第三方函式庫所拼圖拼成的」。這樣未來系統如果爛掉，他們才有辦法照圖完美重組出來。
- **退版 / 撤退逃生計畫 (Rollback / Back-out Plans)**：發布管理的鐵血軍令：**如果你不知道怎麼撤退，你就連進去的資格都沒有 (You cannot deploy if you cannot undo it)**。如果新版推上去後像核彈一樣引發連環錯誤，發布管理手冊裡必須寫得清清楚楚：要怎麼下指令，才能在一秒內讓時光倒流，安全退回那個大家都愛的 Version 1.9 老版本。

## 版本控制的強迫症 (Versioning and Semantic Versioning)

要做好資安弱點管理，你必須精準掌控「到底是哪一版解掉了哪個漏洞」。在如今的軟體工程界，大家最愛用的一套標籤法寶叫做**「語義化版本控制 (Semantic Versioning)」**(格式長這樣：`主版號 MAJOR . 次版號 MINOR . 修訂號 PATCH`)。

舉例：你的程式現在是 `1.4.2` 版
- **1 (主版號 MAJOR)**：這代表公司剛經歷一場驚天動地的底層程式碼大清洗；如果你現在升級，舊的 API 保證全部死給你看，一切都不相容。
- **4 (次版號 MINOR)**：工程師寫了幾個新奇的功能放進去。不用怕，這些新功能有乖乖向後相容 (backwards-compatible)，舊系統升級上去也不會炸掉。
- **2 (修訂號 PATCH)**：這只是在幫舊功能抓蟲子 (bug fixes)。**所有千鈞一髮的安全修補程式 (security patches)，通常都是躲在這個數字裡。**

> **資安救火的關鍵意涵 (Security Implication)**：如果今天資安處衝進來，勒令大家必須馬上把某個老套件從 `1.4.2` 急升到 `1.4.3` 來防堵那個會死人的 CVE 漏洞；這時，長官們看到版號後就可以安心地在同意書上簽字並下令自動升級——因為只有最後一碼 `PATCH` 修訂號變動了，這意味著這只不過是包無害的漏洞補丁，不會去動到底層的大骨架 (MINOR/MAJOR 沒動)，所以理論上系統絕對不會被弄壞。

## 考試重點

1. **職責分離 (Separation of Duties)**：這是發布管理世界裡的磐石。記住一個口訣：「寫扣的人不准去部署；負責部署機器的手絕對不准去碰原始碼。」兩邊絕對不能是同一個人。
2. **退版計畫是發布的門票 (Rollback Plan Requirement)**：只要遞交給 CAB 審查的 RFC 申請單裡，敢忘記附上一份「實兵演練過的防爆退路計畫」，CAB 委員連看都不會看一眼，直接丟進碎紙機。
3. **生死存亡的緊急變更 (Emergency Changes)**：資安世界不能死守不知變通的僵化教條。當面臨零日漏洞時，我們必須繞過那冗長的常規 CAB 流程直接打藥急救；但請記住，開完刀後，所有相關的手術同意書 (紀錄) 事後必須乖乖補上。
4. **變更 = 伴隨極大風險 (Change equals Risk)**：這是一道殘酷的統計學真相。不管是把系統搞崩潰，還是因為設錯參數引狼入室開啟後門，絕大多數悲劇發生的起點，就落在工程師實施完那場「自信滿滿的變更作業」後的下一秒。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **RFC (變更申請單)** | Request for Change；一份呈遞給高層長官、懇求准許你對現有穩定營運系統進行更動的正式投名狀公文 |
| **CAB (變更諮詢委員會)** | Change Advisory Board；一群坐在八角桌前，手握生殺大權決定系統更新能否上線的高階審判官 |
| **ECAB (緊急變更委員會)** | Emergency CAB；當系統快要爆炸時被緊急叩醒，擁有無限授權能一秒核准救命補丁的特種委員會 |
| **Separation of Duties (SoD / 職責分離)** | 資安鐵律；確保這世界上絕對沒有一個人可以身兼球員與裁判——你寫的程式碼，必須由另一個人負責安檢並按鈕投放到正式機器上 |
| **Semantic Versioning (語義化版本控制)** | 業界通用的 `主版號.次版號.修訂號` 三節式軟體命名學；能讓人一眼看出這次更新到底是毀天滅地的大改版，還是無傷大雅的抓蟲小補丁 |
| **Rollback Plan (退版/撤退復原計畫)** | 對於一場悲劇且翻車的軟體更新署行動，唯一能讓時光一秒倒流、返回原本歲月靜好舊版本的救命藥丸 |
