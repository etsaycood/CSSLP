# 7.3 資安事件管理 (Incident Management)

## 學習目標

- 清楚分辨「一般事件 (Event)」與「資安事件 (Incident)」之間的天壤之別
- 倒背如流標準的資安事件應變生命週期 (PICERL 六步法)
- 辨識出災情控制 (containment)、威脅根除 (eradication) 與系統復原 (recovery) 各自的核心目標
- 解釋為何「事後檢討會 (post-incident review / lessons learned)」是整個流程中最具價值的一環

---

## 只是個事件 (Event) 還是 危機爆發 (Incident)？

在 IT 維運的世界裡，這兩個詞代表著完全不同的嚴重層級：

- **一般事件 (Event)**：在系統或網路上任何「可被觀察到的正常動靜」。(例如：有員工登入系統、防火牆按規矩擋下了一個夾帶不明 IP 的封包、某台伺服器正常重新開機)。所有的 Event 都只是日常風景。
- **不良事件 (Adverse Event)**：帶來了負面後果的事件 (例如：某台伺服器突然當機崩潰了)。這可能只是硬體故障，不一定跟資安有關。
- **資安事件 (Security Incident)**：這是拉響防空警報的時刻。它代表著公司的資安政策、可接受使用規範 (AUP)，或是標準的安全防線，已經**遭到實質破壞，或是正處於被破壞的極度危險邊緣**。(例如：駭客成功竊走客戶資料庫、勒索軟體大規模感染了內網、網站遭到猛烈的 DoS 阻斷服務攻擊而癱瘓)。

> **總結**：所有的 Incident (資安事件) 都是 Event (一般事件) 的一種；但幾萬個 Event 裡面，可能才藏著一個會要命的 Incident。

## 資安事件應變的黃金生命週期 (The Incident Response Lifecycle - PICERL)

業界處理資安事件的最高標準聖經，是遵循 NIST SP 800-61 所制定的 **PICERL 六階段方法論**。考卷上一定會考這個！

### 1. 準備期 (Preparation)
這絕對是整個生命週期中**最生死攸關、但往往被忽視的一環**。它發生在災難真正降臨「之前」。
- **建立 CSIRT** (電腦資安事件應變小組 / Computer Security Incident Response Team)：這群人就是你們公司的特種救火隊。
- **撰寫作戰教戰手冊 (Runbooks / Playbooks)**：針對特定災難準備的標準作業程序 (SOP)。(例如：「遭遇勒索軟體劇本」、「遭遇大流量 DDoS 癱瘓劇本」)。
- **整備火力與通訊備案 (Tools and Access)**：確保團隊手邊有裝滿急救工具的「緊急應變包 (jump bags)」、最新的網路架構圖；最重要的是，必須準備**「帶外通訊工具 (out-of-band communication)」** (因為如果公司內網被攻陷了，你們原本在用的 Slack 或 Teams 聊天室很可能早就被駭客偷看監聽了！)。
- **教育訓練與桌上沙盤推演 (Tabletop Exercises)**：長官與應變小組定期坐在會議室裡，模擬各種極端攻擊情境，看著劇本實際推演戰術。

### 2. 識別/偵測與分析 (Identification / Detection & Analysis)
負責判定「我們是不是真的被打了？」、災情有多慘，以及影響範圍有多大。
- 分析來自 SIEM、WAF 或是熱心員工通報的警報。
- **初步分流研判 (Triage)**：這是虛驚一場的誤報 (false positive)？還是真的被破門而入了 (true positive)？
- **清查爆炸半徑 (Scope)**：駭客只有摸到最外層的網頁伺服器？還是他們已經像土撥鼠一樣鑽進內網，連進了命脈資料庫？
- **正式宣告 (Declaration)**：一旦由長官「正式宣告」這是一起 Incident，整個公司就會立刻切換成戰時狀態，啟動最高層級的應變程序。

### 3. 災情控制 (Containment)
**止血**。這階段的唯一目標，就是在最短的時間內把駭客關在籠子裡，阻止災情擴大蔓延。
- **短期控制 (Short-term containment)**：立刻採取斷然處置 (例如：直接拔掉那台受感染伺服器的網路線、強制凍結被盜用的高階主管帳號)，但同時必須小心保護好「犯罪現場 (數位證據)」。
- **長期控制 (Long-term containment)**：在一邊準備最終修補方案的同時，先架設臨時圍牆。例如用乾淨的映像檔重建系統、打上臨時修補程式，甚至是故意把駭客的流量導進一個名為「沉洞 (sinkhole / 黑洞網域)」的死胡同裡，藉此偷偷觀察駭客的後續動作。

> **致命警告 (Warning)**：發現伺服器被駭客入侵時，**絕對不要立刻直接把它「重新開機 (Reboot)」！** 重新開機這個愚蠢的動作，會直接清空揮發性記憶體 (RAM) 裡最值錢的法醫證據 (例如駭客正在跑的木馬程式、當下活躍的網路連線)。請拔網路線 (隔離)，不要拔電源線。

### 4. 根除威脅 (Eradication)
這是「大掃除」階段。把系統裡的毒瘤 (根本原因與駭客本人) 給徹底挖除乾淨。
- 刪除每一隻惡意軟體、砍掉駭客偷偷建立的後門帳號、補上當初被他們用來穿牆的那個防火牆漏洞。
- 這個階段通常會跟「問題管理 (Problem Management)」團隊緊密合作；開發工程師會熬夜寫出一個熱修復程式 (hotfix) 來永遠堵住系統底層的破洞 (例如把 SQL Injection 的弱點程式碼給改掉)。

### 5. 系統復原 (Recovery)
確認系統乾淨無毒後，小心翼翼地把受災戶重新引導回正常的商業營運軌道。
- 從「確認乾淨無毒」的備份檔中還原資料。
- 從無瑕的 IaC (基礎設施即程式碼) 模板中，重新召喚出全新的 OS 作業環境。
- **嚴密監控驗證 (Validation)**：系統上線後不能鬆懈，必須加派重兵站崗監視這台機器好幾天；以確保駭客沒有趁你們不注意時，利用某個大家都沒發現的暗門再次殺個回馬槍。

### 6. 經驗教訓 / 事後檢討會 (Lessons Learned / Post-Incident Review)
這是整個生命週期中最常被草草帶過，**但卻是價值連城的最重要階段**。必須在事件落幕的幾週內趁著記憶猶新盡快召開。
- **終極目標**：為了讓整個應變團隊「下次能做得更好」。
- **靈魂拷問 (Questions asked)**：到底發生了什麼事？我們為什麼沒有更早發現他？這瞎編的防禦劇本管用嗎？我們還缺什麼裝備或工具要買？
- 這個階段的產出，會**直接回饋並升級第 1 階段的「準備期 (Preparation)」**，形成一個生生不息的防禦進化迴圈。

## 數位鑑識與證據保全 (Forensics & Evidence Handling)

如果這起事件未來可能會鬧上法庭、或是需要把內鬼移送法辦，那你們在蒐集數位證據時，就必須像鑑識科警察一樣吹毛求疵。

- **揮發性順序 (Order of Volatility)**：在犯罪現場蒐證時，必須**「從最容易消失的東西開始抓」**，一路抓到最不容易消失的東西。
    1. CPU 暫存器 (Registers) / 快取 (Cache)（一閃即逝）
    2. RAM (主記憶體) (關機就沒了)
    3. 網路連線狀態 / 路由表
    4. 實體硬碟 (Hard Drives)
    5. 放冷宮的備份磁帶 (最穩定)
- **監管鏈 (Chain of Custody)**：這是一份具法律效力的「流水帳鐵證記錄」。上面清楚記載了：這顆硬碟到底是誰拔下來的？誰運送的？中途誰碰過？什麼時候碰的？**只要監管鏈中間斷了一次 (例如交接沒簽名)，這份證據在法庭上就會直接被法官當成垃圾退回 (inadmissible)。**
- **防寫裝置 (Write Blockers)**：辦案鐵律：**永遠不要在「原始」硬碟上直接跑分析軟體進行鑑識。** 你必須先用一個名為「硬體防寫裝置 (hardware write blocker)」的物理神器把硬碟保護起來，再對它進行 bit-for-bit (位元對位元) 的完美複製拷貝。然後把原始硬碟鎖進保險箱，所有的逆向分析跟調查，都只能對那份「拷貝檔」進行。

## 考試重點

1. **死背 PICERL 的順序**：準備 (Preparation) ➔ 識別 (Identification) ➔ 控制 (Containment) ➔ 根除 (Eradication) ➔ 復原 (Recovery) ➔ 教訓檢討 (Lessons Learned)。
2. **「準備 (Preparation)」是重中之重**：平時不燒香寫劇本演練，災難來臨時大家只會像無頭蒼蠅一樣亂竄。
3. **千萬不要重新開機 (Do not reboot)**：重開機會直接謀殺掉 RAM 裡最珍貴的鑑識證據。要控制災情，請「拔網路線 (環境隔離)」，不是去拔插頭。
4. **揮發性順序 (Order of Volatility)**：蒐證第一步永遠是先用特殊工具把 RAM 裡的資料倒出來 (dump)，之後才去碰硬碟。
5. **監管鏈 (Chain of Custody)**：這是一份用來保證證據沒有被掉包或污染的法律追蹤證明文件。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Incident (資安事件)** | 嚴重違反公司安全政策、對系統防禦造成實質性衝擊與破壞的危急狀況 |
| **CSIRT** | 電腦資安事件應變小組 (Computer Security Incident Response Team)；負責在著火時衝進機房救火的特種菁英部隊 |
| **Runbook / Playbook (作戰劇本/教戰手冊)** | 預先寫好在地窖裡的標準作業程序，詳細列出當遇到特定攻擊 (如勒索軟體) 時，每個人第一秒該做什麼的 SOP |
| **Containment (災情控制)** | 俗稱止血；應急處置的最高指導原則，目的是把威脅死死隔離在一個小圈圈裡，防止它蔓延燒毀全公司 |
| **Order of Volatility (揮發性順序)** | 數位鑑識蒐證的黃金法則；規定必須按照資料消失的快慢速度，從最脆弱的 RAM 開始依序抓取證據 |
| **Chain of Custody (證據監管鏈)** | 嚴格記錄任何一份數位證據「從案發現場被提取開始，一路到送進法庭途中」，經手過所有人員與時間點的歷史鐵證流水帳 |
