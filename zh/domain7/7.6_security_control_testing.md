# 7.6 安全控制措施測試 (部署上線後 / Post-Deployment)

## 學習目標

- 解釋為什麼在軟體正式部署上線後，還必須持續對安全控制措施進行測試
- 區分「部署前」的 AST 掃描測試 (Pre-deployment AST) 與「部署後」的控制測試 (Post-deployment control testing) 有何本質上的不同
- 描述內部與外部「持續性稽核 (continuous auditing)」機制所扮演的角色
- 徹底弄懂 SLA 監控，以及針對雲端服務供應商 (CSP) 的稽核報告 (如 SOC 2)

---

## 為什麼上線後還要繼續測？ (Why Test Controls After Deployment?)

在領域 6 (測試階段) 中，我們已經在 Staging 預備環境中對應用程式拿放大鏡做過各種地毯式弱點掃描了。但問題是，一旦軟體正式被推上 Production 真實戰場，整個外在運作環境就變了：真實使用者的詭異操作行為會引入全新的變數，而且外面的駭客攻擊手法也是每天在進化的。

**部署上線後的安全控制測試 (Security control testing post-deployment)，其核心目的是為了「驗證」以下三件事：**
1. 那些號稱有買的防禦機制，在 Production 環境中「**真的有被打開 (actually turned on)**」。
2. 這些防禦機制，在真實流量的衝擊下「**設定是正確無誤的 (configured correctly)**」。
3. 這些機制沒有因為時間久了而「**嚴重退化 (degraded)**」，或是被某個嫌它太擋路妨礙效能的工程師給「**偷偷關掉 (secretly turned off)**」。

| 部署前 (領域 6 / 溫室裡) | 部署後 (領域 7 / 真實戰場) |
|-------------|------------|
| 核心叩問：「這包程式碼裡面有沒有藏著漏洞？」 | 核心叩問：「上線後外圍那一圈動態防禦機制，現在有沒有在認真擋子彈？」 |
| 武器：SAST 靜態掃描、同儕程式碼審查、DAST 動態測試 | 武器：WAF 阻擋規則微調、SIEM 警報演習驗證、組態設定檔稽核 |

## 持續性稽核與弱點掃描 (Continuous Auditing and Vulnerability Scanning)

成熟的維運團隊，早就不會在那邊苦苦等待「一年一次的年度滲透測試大拜拜」了；他們會在 Production 環境裡直接常駐「持續性掃描 (continuous scanning)」的雷達。

### 1. 產品環境的弱點掃描 (Production Vulnerability Scanning)
每天或每週，定期啟動自動化的「已驗證 (authenticated)」與「未驗證 (unauthenticated)」掃描程式，對著活生生的正式基礎設施 (如網頁伺服器、資料庫、負載平衡器) 進行掃雷。
- **終極目標**：抓出那些忘記按時打修補程式的主機、比對新公布的 CVE 看看現有軟體有沒有中鏢，或是揪出致命的「組態漂移 (configuration drift / 例如某個天兵工程師昨晚為了圖方便，不小心把遠端桌面的 Port 3389 曝露到外網)」。
- **保命天條 (Rule)**：這些掃描動作**絕對必須**被排程在「半夜離峰時段 (maintenance windows)」進行；否則掃描器的強大流量，極有可能會把正式機台給活活打死，造成一起難堪的無差別 DoS (阻斷服務攻擊)。

### 2. WAF 與 IDS/IPS 的微調校準 (WAF and IDS/IPS Tuning)
網頁應用程式防火牆 (WAF) 這種精密武器，絕對不是那種「插了電就可以下班 (set-and-forget)」的傻瓜電器。
- 在系統剛上線部署的初期，為了怕 WAF 亂砍人把正常的付費客戶也給擋了 (這叫誤報 False Positives)，工程師通常只敢把它切成「學習模式 (Learning)」或是「只記錄不阻擋模式 (Log Only)」。
- **安全控制措施測試 (Security Control Testing)**：維運團隊必須持續不懈地拿「已知的惡意攻擊字串」(例如：無害的 EICAR 防毒測試檔，或是經典但無殺傷力的 SQL 注入字串) 去戳自家的 WAF。唯有確認 WAF 真的有成功把這個惡意封包給「擋下來」，並且後台的 SIEM 戰情室裡有確實「響起警報」，這個測試才算過關。

## 第三方與雲端供應商稽核 (Third-Party and Cloud Auditing)

如果你的應用程式是架在 AWS, Azure 上，或是你買了 SaaS 服務，記住一個鐵則：你可以把機房管理外包，但一旦發生資料外洩，**這筆賠償帳還是會算在「你」的頭上 (you still own the risk)**。問題來了，你總不能帶隊衝進 AWS 的資料中心去查機房吧？

### 服務級別協議 (SLAs, Service Level Agreements)
你必須派人死盯著這家雲端廠商，看他們有沒有遵守當初合約裡白紙黑字的 SLA 承諾 (例如：承諾 99.99% 的服務連線可用性 (uptime)，或是保證在發現嚴重漏洞後 4 小時內一定打上修補程式)。

### SOC 2 稽核報告 (System and Organization Controls)
既然你不能親自去踹 AWS 的防火牆，你就必須依賴公正第三方的查核報告。在雲端運算界，**SOC 2 Type II** 報告就是驗證廠商資安實力的「黃金白金級國際通用護照」。這是一份由獨立外部稽核會計師事務所出具的報告，用來證明這家雲端廠商的資安防護做得有多滴水不漏。

- **Type I 報告 (第一型)**：只是個「某一天」的靜態快照。會計師只看你的規定寫得好不好 (Controls are designed correctly)。
- **Type II 報告 (第二型)**：這才是真劍決鬥。會計師會派人進駐觀察長達半年到一年的一段時間 (a period of time)；他們要驗證你不僅規定寫得好，而且每天都有真真切切地照著做 (Controls actually work day-to-day)。**在價值與可信度上，Type II 徹底碾壓 Type I。**

## 合成交易演練 (Synthetic Transactions)

為了 24 小時不間斷地驗證應用程式裡的安全控制機制 (例如登入身分驗證、結帳流程) 是不是真的活著，維運團隊會施放名為「合成交易 (synthetic transactions)」的機器人。

- 寫一支測試腳本 (script)，每 5 分鐘就自動假裝成真實客戶：登入網站、把商品丟進購物車、然後按下結帳。
- 如果這支腳本在中途突然報錯跑不下去了，或是它在結帳時「竟然沒有跳出預期的 MFA 多重因素驗證畫面就直接結帳成功了」；維運團隊就會立刻收到血紅色的尖叫警報——因為這代表 Production 環境裡某個致命的安全關卡剛剛當機報銷了。

## 考試重點

1. **持續性稽核 (Continuous Auditing)**：它的存在是為了取代過去那種一年只拜拜做一次的老舊資安評估法。但記住，掃描必須排在半夜離峰時段，否則你會把自家系統打到 DoS 斷線。
2. **WAF 微調 (WAF Tuning)**：部署上線後最關鍵的苦力活。新裝的 WAF 不是瘋狂亂擋正常流量 (False Positives)，就是像瞎子一樣放駭客進來 (False Negatives)，這一切都需要靠人工日夜餵放惡意字串來進行調教。
3. **SOC 2 Type II 報告**：要驗證一家雲端廠商 (CSP) 在「過去這半年內」有沒有認真做好資安，唯一指定的認證標準護照就是它。
4. **組態漂移 (Configuration Drift)**：這是為什麼系統上線後還必須瘋狂進行測試的最主要原因——總有管理員會為了貪圖方便，偷偷關掉防火牆然後永遠忘記把它開回來。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Control Testing (控制測試)** | 在系統正式上線後，拿真實世界的子彈去驗證防禦機制 (如 WAF 或權限控管) 是否真的有乖乖運轉的常態性演習 |
| **SOC 2 Type II** | 一份極具公信力的外部稽核報告，證明雲端服務商在長達半年以上的時間內，其資安防禦措施每天都有確實落地執行 |
| **Synthetic Transaction (合成交易)** | 派出自動化機器人腳本不斷在網站上假裝購物或登入，用以24小時即時測量系統健康度與安全機制是否失效的探針 |
| **Configuration Drift (組態漂移)** | 隨著時間推移，工程師因為無數次未經記錄的微小設定修改，導致伺服器長相跟當初藍圖完全不一樣、變得千瘡百孔的現象 |
| **WAF Tuning (WAF 微調/調教)** | 拿著各種攻擊字串反覆測試網頁防火牆，藉此不斷調整阻擋規則，使其既不會錯殺好人也能完美攔下駭客的過程 |
