# 5.3 實作安全控制措施 (Implement Security Controls)

## 學習目標

- 選擇並實作正確合適的密碼學演算法與通訊協定
- 實作安全的身分認證 (authentication) 與身分管理機制
- 妥善地管理加密金鑰 (cryptographic keys) 與機密憑證 (secrets)
- 定義安全的錯誤處理 (error handling)、日誌記錄 (logging) 與監控實務作法

---

## 實作密碼學防護 (Implementing Cryptography)

密碼學是出了名的「難以正確實作」。在密碼學領域裡，最核心的鐵律/天條就是：**永遠不要實作你自己的密碼學演算法 (Never roll your own crypto / 禁止自創演算法)。**

### 演算法選擇指南

| 使用情境 | 推薦使用之演算法/國際標準 | 應當淘汰/避免使用的舊東西 |
|----------|----------------------------------|--------------|
| **對稱式加密 (Symmetric)** (保護靜止資料 Data at Rest) | **AES-GCM** (如 AES-256), ChaCha20 | DES, 3DES, RC4 |
| **非對稱式加密 (Asymmetric)** (用於金鑰交換、數位簽章) | **RSA** (金鑰長度 2048+ bit), **ECC** (橢圓曲線加密), Diffie-Hellman | RSA < 2048 bit |
| **雜湊運算 (Hashing)** (用於完整性確認) | **SHA-2** (包含 SHA-256, SHA-512), SHA-3 | MD5, SHA-1 |
| **密碼雜湊 (Password Hashing)** (用於密碼安全儲存) | **Argon2**, **Bcrypt**, PBKDF2 | 原味沒加料的 MD5, 原味沒加料的 SHA-2, 絕對禁止明文 (cleartext) |
| **傳輸中資料加密 (Data in Transit)** | **TLS 1.3**, TLS 1.2 | SSL (所有版本都死透了), TLS 1.0, TLS 1.1 |

### 密碼學實作的關鍵概念

- **加鹽 (Salting, 專用於密碼)**：這是一串具有唯一性、隨機生成的字串，在進行雜湊運算*之前*，會先偷偷加到使用者的密碼中。它的主要目的是防止**彩虹表攻擊 (Rainbow Table attacks)**，並能確保就算兩個使用者設定了一模一樣的弱密碼，算出來的雜湊值也會截然不同。
- **金鑰延伸函數 / 工作乘數 (Key Stretching / Work Factor)**：像 Bcrypt 與 PBKDF2 這類演算法，在設計上是**刻意刁難、讓運算變得非常緩慢且極度消耗資源**的。這能大幅提高計算成本，讓大規模的暴力破解攻擊在運算時間上變得毫無效益 (computationally impractical)。
- **亂數熵值 (Entropy / Randomness)**：凡是需要用來生成金鑰、隨機數 (nonces) 以及 Session IDs 的場合，都需要極高品質的隨機性。永遠只准使用**密碼學安全的偽亂數生成器 (CSPRNG**；例如 Linux 上的 `/dev/urandom`)，絕對不能使用程式語言內建的普通隨機函數 (譬如標準庫的 `Math.random()`)。
- **初始化向量 (IVs, Initialization Vectors) / 隨機數 (Nonces)**：為了確保即使是相同的明文，每次加密出來的密文也一定會長得不一樣，每次重啟加密操作時所使用的 IV 都必須是全新、唯一且無法預測的。

## 機密憑證與金鑰的管理 (Secrets and Key Management)

將機密資訊 (如 API keys、資料庫連線密碼、私鑰) 寫死 (hardcoded) 在程式碼裡，是引發重大資料洩露災難的頭號戰犯。

### 金鑰管理生命週期 (Key Management Lifecycle)

1. **生成 (Generation)**：使用高強度的 CSPRNGs 來產生。
2. **派發/散佈 (Distribution)**：強制要求透過帶外通訊 (out of band, OOB) 的安全管道來傳遞金鑰。
3. **儲存 (Storage)**：使用專用且嚴密保護的**機密管理金鑰庫 (Secrets Management Vaults)** (例如：HashiCorp Vault、AWS Secrets Manager、Azure Key Vault，或是實體的硬體安全模組 HSMs)。**絕對嚴禁將機密跟著原始碼一起 push 進版本控制系統裡。**
4. **輪替 (Rotation)**：定期更換新金鑰，特別是在有人員離職異動，或是懷疑金鑰可能已經遭妥協外洩時。
5. **撤銷/銷毀 (Revocation/Destruction)**：當金鑰退役不再使用時，必須透過密碼學手段將其徹底抹除 (erase keys)。

## 實作身分與存取管理 (Implementing Identity and Access Management)

### 身分驗證的實作

- **多因素認證 (MFA)**：強制要求提供兩種以上的認證因素 (包含你所知道的 Knowledge、你所擁有的 Possession、你天生具備的特徵 Inherence)。對於所有來自外部的存取，以及任何高權限的系統管理功能，都應該無條件強制導入 MFA。
- **聯合身分 (Federation) / 單一登入 (SSO)**：請擁抱並採用成熟的國際標準協定，不要自己土炮發明一套新的身分認證框架：
    - **SAML**：基於 XML 格式，大量應用於大型企業內部，或 B2B (企業對企業) 之間的 SSO 整合。
    - **OAuth 2.0**：這是一套專門用來處理**存取權限委派 (delegating access)** 的授權框架 (Token-based 權杖式技術)。
    - **OpenID Connect (OIDC)**：這是直接架構在 OAuth 2.0 肩膀上，進一步延伸出來專門處理**身分認證 (Authentication)** 用的協議層。

### 權杖代幣的管理 (Token Management, JWT)

JSON Web Tokens (JWTs) 是現代無狀態 (stateless) API 身分驗證中的當紅炸子雞。
- **必須做 (Do)**：一定要使用高強度的演算法 (如 RS256) 為 JWT 蓋上簽章 (變成 JWS)，以確保其內容未遭竄改的完整性。
- **不准做 (Don't)**：絕對不可以盲目接受/允許演算法被設定為 "none" (無簽章模式) 的 JWT。
- **不准做 (Don't)**：絕對不可以把敏感機密給塞在 JWT 的酬載 (payload) 中 (除非你有額外使用 JWE 進行加密)。
- **必須做 (Do)**：請把通行權杖 (access tokens) 的有效期限 (TTL) 設得越短越好，並搭配使用更新權杖 (refresh tokens) 的機制來換取新的通行證。

## 安全的例外狀況處理 (Secure Exception Handling)

在實作錯誤處理時，必須在「方便開發者除錯」與「避免不慎洩漏系統情資 (information disclosure)」之間取得完美的平衡。

| 最佳實務作法 | 背後原理解釋 |
|----------|-------------|
| **精準地捕捉特定的例外 (Catch Specific Exceptions)** | 不要偷懶直接在最頂層丟一個會包山包海的 `Exception` 大類別去接炸彈。你應該精準捕捉特定的錯誤病灶 (例如：`SQLException`)，才能對症下藥給予妥善處置。 |
| **絕對不要對外暴露系統內部運作細節** | 絕對不要把記憶體堆疊資訊 (stack traces)、資料庫下的原始查詢明碼，或是伺服器上的實體路徑給毫無保留地傾印回給終端使用者看。請一律給他們罐頭、但有禮貌的通用錯誤訊息 (例如：「發生未預期的錯誤，請稍後再試。」) |
| **把真正的細節安全地記錄到日誌中** | 讓使用者看罐頭訊息的同時，系統後台則必須盡責地將完整的技術發病細節 (包含 stack trace) 給記錄到受嚴密保護、集中化管理的日誌伺服器上，供開發人員事後緝凶除錯用。 |

## 資安日誌與稽核軌跡 (Security Logging and Audit Trails)

完備的日誌記錄，是實現不可否認性 (non-repudiation)、驅動事件緊急應變 (incident response)，以及滿足法規遵從的命脈樞紐。

### 什麼東西「必須」記錄下來

- 所有身分驗證行為，包含成功的登入與失敗的撞門嘗試。
- 授權失敗 / 企圖越權存取慘遭滑鐵盧被拒絕 (access control denied) 的事件。
- 帳號遭到鎖定 (Account lockouts) 與密碼變更事件。
- 牽涉到龐大利益或高價值操作的交易行為 (例如：鉅額轉帳、存取病患的 PHI 就醫紀錄)。
- 針對應用程式組態設定所進行的任何變更。

### 什麼東西「絕對不可以」出現在日誌中 (敏感資料)

- 明文未加密的密碼，或是目前有效活著的 Session 權杖 (session tokens)。
- 毫無遮掩的完整信用卡卡號 (PAN) 或是背後的 CVV 驗證碼。
- 社會安全碼 (SSN)、身分證字號，或是任何未經遮蔽 (unmasked) 的 PII/PHI 個資。
- **防範日誌偽造攻擊 (Log Forging Prevention)**：在把使用者的任何輸入給寫進日誌檔*之前*，務必先進行內容清洗 (sanitize) 與驗證過濾。這是為了防堵攻擊者偷偷塞入換行字元 (CRLF injection) 或試圖發動日誌下毒 (log poisoning) 攻擊。

### 日誌資料自身的安全防護

- 這些日誌本身必須具備防竄改能力 (定期執行完整性校驗、存放在強調 WORM — Write-Once-Read-Many 一次寫入多次讀取技術的專屬儲存媒體中)。
- 日誌應該被**集中化管理** (將日誌從本地端伺服器立刻抽走，實時發送到遠端防護嚴密的 SIEM 系統去存放)，以此防堵攻擊者在攻陷台機器後，隨手直接把日誌檔刪除好達到毀屍滅跡 (covering their tracks) 的目的。

## 考試重點

1. **絕對禁止自創演算法 (Never roll your own crypto)**：永遠只准使用那些經過千錘百鍊、已被業界廣泛認可的標準函式庫與演算法。
2. **密碼的儲存法則**：密碼永遠必須經歷「先加鹽 (Salt) + 後雜湊 (Hash)」，且這個雜湊函數必須是個慢郎中/金鑰延伸演算法 (如 Argon2, Bcrypt, PBKDF2)。拿原味直接的 MD5/SHA 來存密碼，在現代標準下是完全不及格的。
3. **CSPRNG (密碼學安全之偽亂數生成器)**：語言內建的普通 `random()` 函數其規律是可以被數學預測出來的；只要是牽涉到安全功能的隨機產出，無一例外都必須採用 CSPRNG。
4. **錯誤處理的兩面手法**：對站在外面的使用者，只丟給他們籠統空泛的罐頭錯誤；對躲在安全堡壘裡的日誌檔，則要鉅細靡遺地寫下所有的技術報錯細節。
5. **機密金鑰庫 (Secrets Vaults)**：如果考題問你該把 API 金鑰或資料庫密碼存哪裡，正確解答永遠是「Vault / 金鑰庫 / HSM 硬體安全模組」；若是出現選「原始碼版本控管系統」或「本地端設定檔」的選項，連考慮都不用考慮直接劃掉。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **CSPRNG** | Cryptographically Secure Pseudo-Random Number Generator (密碼學安全的偽亂數生成器) |
| **Salting (加鹽)** | 在幫密碼執行不可逆的雜湊運算之前，先故意在密碼字串中偷偷摻入一長串極度隨機的雜湊前綴資料機制 |
| **Key Stretching (金鑰延伸 / 工作乘數)** | 一種故意將雜湊函數的運算過程弄得很慢、很耗資源的演算法設計策略 (例如：Bcrypt)，旨在用大量的拖延戰術來遏止暴力破解的野心 |
| **JWT** | JSON Web Token (現代無狀態 Stateless APIs 中最被廣泛使用的認證權杖技術標準) |
| **SAML/OIDC** | 業界用來實現身分聯盟 (federation) 驗證與跨系統單一登入 (SSO) 的主流標準通訊協定 |
| **WORM** | Write-Once-Read-Many (只能寫入一次但可被讀取無限次，這是用來確保存出不可竄改之安全稽核日誌檔的金氏世界標準硬體規格特性) |
