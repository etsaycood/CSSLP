# 5.1 應用安全程式設計實務 (Apply Secure Coding Practices)

## 學習目標

- 識別並緩解 OWASP Top 10 漏洞
- 應用宣告式安全 (declarative security) 與命令式安全 (imperative security) 的概念
- 實作安全狀態管理 (secure state management)
- 描述安全記憶體管理的各項原則

## The OWASP Top 10 (十大網站應用程式安全風險)

OWASP Top 10 是全球公認，針對網站應用程式最嚴重安全風險的標準指南。

### 1. 權限控制失效 (Broken Access Control)
未能有效強制執行已認證使用者究竟被允許做什麼事。
- **範例實例**：不安全的直接物件參考 (IDOR)、權限提升、繞過存取控制檢查、中繼資料竄改 (metadata manipulation，例如竄改 JWT)。
- **緩解措施**：預設拒絕 (Default deny)、在中央端點 (如中介軟體/過濾器) 統一實施存取控制、強制檢查資料紀錄擁有權、記錄所有存取控制失敗的事件。

### 2. 加密機制失效 (Cryptographic Failures) (前稱為「敏感資料外洩」)
與密碼學實作相關的缺失，進而導致敏感資料外洩。
- **範例實例**：以明文傳輸資料、使用防護力薄弱或已被棄用的演算法 (如 MD5, SHA1)、將加密金鑰寫死 (hardcoding) 在程式碼中、亂數熵值 (entropy) 不足。
- **緩解措施**：對所有靜止與傳輸中的敏感資料進行加密、使用高強度的標準演算法 (AES, RSA, TLS 1.3)、安全地管理金鑰 (存放於 vaults 金鑰庫中)。

### 3. 注入攻擊 (Injection)
未被信任的資料被當成指令或查詢的一部分，強行送給直譯器 (interpreter) 執行。
- **範例實例**：SQL 資料隱碼攻擊 (SQL injection)、OS 指令注入、LDAP 注入、跨站腳本攻擊 (XSS 廣義上也算一種注入)。
- **緩解措施**：使用**參數化查詢 (parameterized queries / prepared statements)**、將受信任指令與非受信任資料徹底抽離分開、在伺服器端實施嚴格的**白名單輸入驗證 (allowlist input validation)**、使用適當的輸出編碼 (output encoding)。

### 4. 不安全設計 (Insecure Design)
這類缺陷歸咎於設計階段就遺漏或是設計了無效的安全控制項。(此點強調問題出在「編寫程式碼之前」的設計階段)。
- **範例實例**：缺乏速率限制 (rate limiting) 導致系統慘遭憑證填充攻擊 (credential stuffing) 灌爆、缺乏針對商業邏輯的驗證。
- **緩解措施**：在設計階段導入威脅建模 (threat modeling)、採用安全設計模式、建立安全的參考架構套用。

### 5. 安全設定缺陷 (Security Misconfiguration)
不安全的預設設定、門戶大開的雲端儲存桶、喋喋不休洩漏機密的錯誤訊息。
- **範例實例**：保留預設密碼未更改、未修補的已知系統缺陷、啟用了根本沒用到的功能、錯誤訊息直接把堆疊追蹤 (stack traces) 傾印在畫面上。
- **緩解措施**：組態設定自動化 (基礎設施即程式碼 IaC)、移除不用到的功能/服務、進行網路架構分段隔離、導入可重複被執行的系統強固 (hardening) 流程。

### 6. 使用含有已知漏洞的元件 (Vulnerable and Outdated Components)
使用了帶有已知安全漏洞的第三方軟體元件。
- **範例實例**：使用舊版的 Log4j (導致 Log4Shell)、作業系統沒打補丁、依賴那些早就被原作者拋棄不再維護的開源函式庫。
- **緩解措施**：維護一份 SBOM (軟體物料清單)、使用 SCA (軟體組成分析) 工具、建立持續修補更新 (continuous patching) 的流程、移除沒用到的相依套件。

### 7. 認證與身分驗證失效 (Identification and Authentication Failures)
身分驗證與工作階段 (session) 管理機制未被正確實作。
- **範例實例**：擋不住憑證填充攻擊、允許暴力破解攻擊、密碼強度政策薄弱、把 Session ID 暴露在 URL 網址中。
- **緩解措施**：導入多因素認證 (MFA)、強制實施強密碼政策 (參考 NIST SP 800-63b)、實作帳號鎖定 (account lockout)/速率限制、使用框架內建且安全的工作階段管理器 (session managers)。

### 8. 軟體及資料完整性失效 (Software and Data Integrity Failures)
程式碼與周邊基礎設施未能有效防範完整性遭破壞的問題。
- **範例實例**：對不受信任的資料進行反序列化 (Deserialization)、依賴來自不可靠來源的擴充套件/函式庫、採用未實驗證把關的 CI/CD 管線。
- **緩解措施**：使用數位簽章、驗證套件產出物 (artifact) 的完整性、**絕對不要**反序列化不受信任的資料 (或是改用 JSON 等安全的序列化格式)、保護 CI/CD 管線的安全。

### 9. 資安日誌與監控失效 (Security Logging and Monitoring Failures)
無法及時察覺、通報並回應正在進行中的入侵事件。
- **範例實例**：沒有針對關鍵事件 (如登入、存取失敗、高價值交易) 留存日誌、日誌只存放在本機端結果在被駭時一併慘遭滅口刪除、完全沒有設定警報。
- **緩解措施**：確保所有登入/存取失敗紀錄以及高價值交易都有被記錄下來、使用中控式集中化日誌管理系統 (SIEM)、為主動可疑活動建立觸發警報。

### 10. 伺服器端請求偽造 (SSRF, Server-Side Request Forgery)
網站應用程式在未經驗證使用者所提供 URL 的情況下，就乖乖地去抓取遠端資源。
- **範例實例**：攻擊者強迫伺服器去連線攻擊其他本來只對內部開放的服務，或是去讀取內部檔案 (例如去打 AWS 的內部中繼資料端點以此撈取金鑰)。
- **緩解措施**：針對允許連線的目的地網域/URL 使用白名單機制 (allowlists)、停用 HTTP 重新導向 (redirections)、隔離網路存取權限、不要把撈回來的原始回覆 (raw responses) 長相直接吐回給客戶端看。

## 常見弱點列舉 (CWE, Common Weakness Enumeration)

OWASP Top 10 追蹤的是整體風險類別，而 **CWE 清單 (CWE list)** 則是由社群共同發展，針對各種軟體常見「底層技術性安全弱點 (underlying technical flaws)」的詳細列舉名冊。
- 範例：CWE-89 (在 SQL 指令中對特殊元素的不當過濾)、CWE-79 (跨站腳本攻擊 XSS)。
- **CWE/SANS Top 25**：這是一份彙整了目前散佈最廣、也是最致命危險，極有可能導致嚴重資安事件的 25 大常見軟體錯誤清單。

## 安全程式設計概念 (Secure Coding Concepts)

### 宣告式安全 (Declarative) vs. 命令式安全 (Imperative Security)

| 類型 | 說明 | 程式碼範例 |
|------|-------------|---------|
| **宣告式安全 (Declarative Security)** | 將安全規則定義在程式碼之外 (例如：寫在組態檔 config 或是使用標註 annotations 中)。它是由底層框架/容器負責去強制執行的。 | XML 中的 `<security-constraint>`，或是加在函數上的 `@RolesAllowed("ADMIN")` 標註。 |
| **命令式安全 (Imperative Security)** | 開發人員把安全邏輯親自/手動寫死在應用程式的核心程式碼之中。 | `if (user.getRole().equals("ADMIN")) { doSomething(); }` |

**最佳實務做法**：盡可能全面採用**宣告式安全 (declarative security)**。它更容易被稽核、易於維護，而且能大幅降低開發人員寫出血汗手殘蟲 (developer error) 的機率。命令式安全 (Imperative security) 應該只保留用在那些非常細緻、且綁定特定商業邏輯的複雜權限判斷上 (例如：「使用者只能編輯*自己*建立的文件」)。

### 狀態管理 (State Management)

HTTP 協定本質上是**無狀態的 (stateless)**。應用程式必須靠自己來實作出安全的狀態管理機制。

- **Cookies (餅乾/小型文字檔案)**：由伺服器發送並儲存在客戶端瀏覽器上的一小段資料。
    - **Secure 屬性標記 (Secure Flag)**：確保該 Cookie 絕對只會透過加密的 HTTPS 連線進行傳輸。
    - **HttpOnly 屬性標記 (HttpOnly Flag)**：嚴格禁止用戶端腳本 (JavaScript) 存取該 Cookie 的內容，這是緩解 XSS 攻擊竊取 Session 的神兵利器。
    - **SameSite 屬性標記 (SameSite Flag)**：控制 Cookie 是否允許伴隨跨站請求 (cross-site requests) 被一起發送出去，是緩解 CSRF 攻擊的關鍵。
- **權杖 Tokens (如 JWT)**：必須被**密碼學簽章過 (cryptographically signed)** 以防止被竄改。敏感資料絕對不能被塞在 JWT 的 payload(酬載) 中 (除非有先經過加密處理)，因為 payload 預設只經過 base64 編碼處理，那**不是加密 (not encrypted)**，任何人都能輕易解開來看。
- **工作階段識別碼 (Session IDs)**：必須使用高亂度 (high entropy) 的隨機生成器產生、每次收到都要經過嚴格驗證。最重要的是，在使用者發生權限階級變動 (privilege changes，也就是登入或是登出) 時，**必須強制重新生成一個全新的 Session ID (或是讓舊的失效)**，藉以防範**工作階段固定攻擊 (Session Fixation)**。

### 記憶體管理 (Memory Management)

不當的記憶體管理會導致諸如緩衝區溢位 (buffer overflows, CWE-119) 等嚴重漏洞。

| 問題性 | 說明 | 預防策略 |
|-------|-------------|------------|
| **緩衝區溢位 (Buffer Overflow)** | 寫入的資料長度超越了原本配置緩衝區的邊界死線，導致強行覆蓋掉了相鄰在旁邊的無辜記憶體內容。 | 多使用安全的字串處理函數 (例如：用 `strncpy` 汰換掉危險的 `strcpy`)，或是乾脆改用具備記憶體安全特性的現代程式語言 (如 Java, C#, Rust, Go)。 |
| **記憶體外洩 (Memory Leak)** | 當配置借用來的記憶體區塊不再需要使用時，卻忘記釋放歸還系統，最終導致系統資源被吃乾抹淨而引發阻斷服務 (DoS)。 | 多使用自帶垃圾回收 (garbage-collected) 機制的語言；若是非得用 C/C++ 不可，必須嚴格落實記憶體追蹤與生命週期管理。 |
| **釋放後使用 (Use After Free)** | 程式試圖去存取一塊其實早就已經被釋放掉 (freed) 的記憶體位址，這極有可能會讓程式不小心去執行到攻擊者精心佈置在那邊的惡意資料。 | 養成好習慣：在釋放 (free) 指標之後，立刻將該指標設為 NULL。 |

### 併發與多執行緒 (Concurrency and Threading)

當多個執行緒在未經適當同步管控的情況下，同時爭搶存取某項共享資源時，就會引發併發問題。

- **競爭危害 (Race Conditions, TOC/TOU)**：Time of Check / Time of Use (檢查時間與使用時間差)。資源的狀態，在「系統對其進行檢查確認的當下」與「實際動手使用它」的這段空檔空窗期之間，被人偷偷發生了改變。
    - **預防策略**：使用具備執行緒安全性 (thread-safe) 的函式庫、實施互斥鎖機制 (mutual exclusion, 如 mutexes, locks)，或是使用原子操作 (atomic operations)。
- **死結 (Deadlocks)**：兩個以上的執行緒永遠被卡死凍結在那邊，因為大家都在互相等待對方先釋放手上的鎖。
    - **預防策略**：明訂出獲取鎖定的一致性順序 (lock acquisition ordering)、設定逾時機制 (timeouts)。

## 考試重點

1. **OWASP Top 10**：務必熟記最新版 Top 10 排行榜上的常客，以及對付它們的主力緩解策略 (特別常考：對付注入攻擊 -> 唯一正解是使用參數化查詢 / Parameterized Queries)。
2. **Cookie 的安全標籤**：記住口訣 — Secure = 限定 HTTPS 傳輸；HttpOnly = 沒收 JS 讀取權限 (防 XSS)；SameSite = 防治跨站請求偽造 (防 CSRF)。
3. **宣告式 (Declarative) vs. 命令式 (Imperative)**：永遠是「宣告式 (寫在設定檔/標註裡)」優先於「命令式 (自己硬幹寫在程式碼裡)」。
4. **記憶體管理**：緩衝區溢位 (Buffer overflows) 問題幾乎是專屬於非記憶體安全語言 (如 C/C++) 的特產。
5. **SSRF (伺服器端請求偽造)**：SSRF 攻擊的目標通常都鎖定在那些本該被妥善保護在防火牆後方的「內部服務/資源」。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **OWASP** | Open Web Application Security Project (開放全球網際網路應用程式安全計畫) |
| **CWE** | Common Weakness Enumeration (常見弱點列舉清單) |
| **Declarative Security (宣告式安全)** | 透過組態設定檔或標註/修飾子 (annotations) 來定義安全規則，而不是藉由撰寫具體的程式碼邏輯來實現 |
| **SSRF** | Server-Side Request Forgery (伺服器端請求偽造) — 誘騙伺服器去代為向內部網路發起對非預期目標請求的攻擊手段 |
| **IDOR** | Insecure Direct Object Reference (不安全的直接物件參考) — 歸類在 OWASP 第1名 Broken Access Control 下的一種常見漏洞 |
| **HttpOnly** | 一種 Cookie 屬性標記，設定後可阻擋 JavaScript 存取該 Cookie，是防範 XSS 竊取 Session 的良藥 |
| **TOC/TOU** | Time of Check to Time of Use (檢查到使用的時間差，又稱為競爭危害) — 利用系統在檢查條件狀態與實際執行操作之間短暫的時間差發動攻擊 |
