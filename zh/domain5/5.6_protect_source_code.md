# 5.6 保護原始碼 (Protect Source Code)

## 學習目標

- 定義版本控制系統 (VCS) 的安全性
- 描述安全的分支 (branching) 與合併 (merging) 策略
- 解釋程式碼簽章 (code signing) 與校驗和 (checksums) 的重要性
- 概述針對原始碼存放庫 (repositories) 的實體與邏輯保護措施

---

## 版本控制系統的安全 (Version Control System / VCS Security)

版本控制系統 (例如：Git, Subversion, Perforce) 是一套應用程式原始碼的「唯一真相來源 (authoritative source of truth)」。一旦 VCS 遭到攻陷，可能會導致公司智慧財產 (IP) 遭竊，或是讓駭客有機會在產品中偷偷安插惡意後門。

### 保衛資源庫

| 控制措施 | 說明 |
|---------|-------------|
| **身分認證 (Authentication)** | 強制所有存取資源庫的開發人員啟用多因素認證 (MFA)。要求使用 SSH 金鑰或個人存取權杖 (PATs) 來取代傳統的密碼登入。 |
| **存取控制 (RBAC)** | 基於角色 (如：讀者、貢獻者、維護者、管理員) 嚴格落實最小權限原則。限制到底「誰」有權限可以直接 push (推播) 程式碼到預設的主分支，或是誰能刪除整個專案資源庫。 |
| **稽核日誌 (Audit Logging)** | 完整記錄所有的 clones (複製下載)、pulls (拉取) 以及 pushes (推播) 行為。監控任何異常舉動 (例如：某個工程師在週末半夜突然把公司整個 codebase 全部 clone 下載打包帶走)。 |
| **機密掃描 (Secret Scanning)** | 使用自動化的 pre-receive hooks (推播接收前攔截腳本) 或是原生工具 (如 GitHub Advanced Security)，來掃描即將 commit 上來的程式碼，一旦發現夾帶 API keys、密碼或權杖就立刻阻擋其 push。 |
| **網路隔離 (Network Isolation)** | 對於視為公司最高機密的智慧財產 (IP)，應該將其 VCS 架設在地端 (on-premises) 或是被 VPN 嚴格限制存取的私有雲 VPC 中，而不是直接大方放到公開的網際網路上。 |

## 安全的分支與合併策略 (Secure Branching and Merging Strategies)

分支模型 (Branching models) 規定了開發人員之間該如何協作，以及如何將他們修改的程式碼整合在一起。安全機制必須被深深嵌合進這些工作流程 (workflows) 之中。

### 分支保護規則 (Branch Protection Rules)

最主要/發布用的分支 (main/master) 必須受到嚴密保護，以確保在程式碼被正式整合進去之前，其品質與安全性都有達標。

1. **強制要求發起拉取請求/合併請求 (Require Pull Requests / PRs)**：嚴格禁止開發人員直接 `push` 程式碼到 `main` 分支。所有的變更都必須開一個新的分支，然後透過發起 Pull Request 來提案。
2. **強制要求同儕審查 (Require Peer Reviews)**：在 PR 能夠被准許合併之前，必須至少要有另一位 (最好是兩位) 具備核准權限的開發人員，看過並點擊 Approve 批准該 PR (這是為了強制落實「職責分離 Separation of Duties」)。
3. **強制要求狀態檢查 (Require Status Checks)**：在允許合併之前，必須先讓自動化的 CI 管線跑完且全數通過。這包含：
    - 建置成功 (沒有冒出編譯錯誤)。
    - 單元測試 (Unit tests) 全數綠燈通過。
    - SAST / SCA 資安掃描全數通過，且沒有抓到任何 Critical 等級的漏洞。
    - 程式碼的測試涵蓋率 (Code coverage) 有達到最低要求門檻。
4. **強制要求線性歷史 (Require Linear History)**：禁止使用那種會把你搞得頭暈轉向、甚至會讓惡意程式碼矇混過關的複雜合併 (merge commits)；強制採用 `rebase` 或是 `squash merging` (將所有 commit 壓縮成一個) 的方式來保持歷史紀錄乾淨一條線。

### 提交簽章 (Commit Signatures)

你該如何證明眼前這個署名為「Alice」的 commit，真的是坐在你對面的 Alice 本人寫的，而不是某個駭客假冒她的 Email 地址偽造送上來的？
- **有簽章的提交 (Signed Commits)**：開發人員使用他們個人的 GPG/PGP 金鑰或是 S/MIME 憑證，以密碼學的方式為自己送出的每一個 Git commit 蓋上私章。
- **驗證 (Verification)**：VCS 平台 (如 GitHub, GitLab) 會拿使用者事前登錄的公鑰，去驗證這個簽章是否為真；如果驗證通過，就會在該 commit 旁邊打上一個閃亮的「已驗證 (Verified)」徽章。

## 程式碼簽章與驗證 (Code Signing and Validation)

如果說「提交簽章 (Commit signatures)」是用來證明這段程式碼是*哪個開發人員*寫的；那**「程式碼簽章 (Code signing)」**就是用來向最終的客戶保證，他們手上拿到的最終編譯成品 (如 `.exe`, `.apk` 或 Docker 映像檔) 的**完整性 (integrity)** 與**來源產地 (origin)**。

### 程式碼簽章的目的
1. **身分鑑別 / 不可否認性 (Authentication / Non-Repudiation)**：向使用者保證，這套軟體確實是由它所宣稱的發行商 (如 Microsoft, Apple) 所合法產出的。
2. **完整性 (Integrity)**：向使用者保證，這套軟體自從發行商為其蓋上簽章出廠後，就沒有在半路上被任何人惡意竄改或是被偷塞入惡意軟體 (malware)。

### 程式碼簽章是如何運作的？
1. 發行商先針對這個最終編譯好的執行檔，計算出一組密碼學雜湊值 (hash)。
2. 發行商拿自己的**私鑰 (Private Key)**（這把鑰匙是跟受信任的憑證授權中心 CA 申請買來的），對這個雜湊值進行加密。
3. 這一小段被加密出來的雜湊值 (這就是所謂的「數位簽章」)，會連同發行商的「公鑰/憑證」一起被附加打包進執行檔裡面。
4. 當今天有位使用者下載了這套軟體，使用者的作業系統就會拿出附在裡面的公鑰，把雜湊值給解密還原出來；接著系統自己也對手邊剛剛下載好的檔案重算一次雜湊值，然後把兩個雜湊值拿來比對。如果一模一樣，就代表這個軟體值得信任。

### 保護簽章流程的安全
「用來簽署程式碼的那把私鑰」價值連城；如果不幸被偷走，駭客就可以肆無忌憚地拿它來幫他們寫的木馬病毒蓋上簽證，完美偽裝成你們家公司出產的合法軟體。
- 程式碼簽章的動作應該統一交由 CI/CD 管線全自動執行，絕對禁止讓開發人員手動去簽。
- 程式碼簽章的私鑰 (Private Key) 必須被牢牢鎖在 **HSM (硬體安全模組 Hardware Security Module)** 或是防護森嚴的雲端金鑰庫中，絕對禁止匯出。
- 想要觸發啟動簽章的流程，必須受到嚴格的職責分離管制與主管的批准授權。

## 反竄改與混淆技術 (Anti-Tampering and Obfuscation)

對於那些必須被部署到「充滿敵意之環境」中的應用程式 (例如：會被下載到使用者個人裝置上的遊戲、手機行動 App、IoT 設備韌體)，它們的原始碼與二進位執行檔就必須擁有能夠抵禦被惡意逆向工程 (reverse engineering) 的自我防護能力。

| 技術名稱 | 說明 |
|-----------|-------------|
| **混淆 (Obfuscation)** | 將原始碼或二進位代碼故意轉換得奇形怪狀，讓人類(駭客) 極難閱讀與理解，但在轉換過程中完全不會改變它原本的執行功能 (常見手法如：把變數名稱全部重新命名成毫無意義的鬼畫符亂碼、把字串加密起來)。 |
| **反偵錯 (Anti-Debugging)** | 在程式碼中埋入偵測機制，一旦程式發現自己正被放在除錯器 (debugger) 或模擬器中執行時，就會改變原本的行為，或是乾脆直接當機死給你看了事，以此來阻撓駭客的分析進度。 |
| **反竄改 (Anti-Tampering)** | 在程式碼的執行時期 (runtime) 掛上校驗和 (checksum) 驗證機制，來檢查程式「自己的」完整性。如果程式發現自己的二進位檔被人動過手腳，就會立刻拒絕執行。 |

> **考試提示**：記住！「混淆技術 (Obfuscation)」*並不是*一種正規的首要安全控制措施 (因為它說穿了就是一種「靠自欺欺人來達成安全的隱匿式安全 / security through obscurity」)。它充其量只能用來「拖延」那些死纏爛打的攻擊者；它**絕對無法取代強悍的密碼學防禦或是優良的安全架構設計**。

## 考試重點

1. **分支保護 (Branch Protection)**：你必須要在程式碼被整併主分支 (main) 之前，強制設立層層關卡：要求發 PR、要求同儕人工審查，以及必須全數通過自動化的 SAST/SCA 資安掃描。
2. **提交簽章 vs. 程式碼簽章 (Commit Signing vs. Code Signing)**：簽 Commit (利用 GPG) 是在程式碼倉庫裡用來證明*開發人員自己的身分*的。而簽程式碼 (利用 PKI 憑證體系) 則是在軟體出貨時，向終端消費者保證*該軟體產出物的完整性與來源產地*的。
3. **機密掃描 (Secret Scanning)**：必須在程式庫 (repository) 的大門口裝設感測器攔截，以防止粗心大意的開發人員把明碼 API 金鑰給 commit 推進去。
4. **混淆技術的極限 (Obfuscation Limits)**：混淆的作用只在於保護公司的智慧財產 (IP) 並拖延駭客發動逆向工程的時間；它並不會讓資料變得更安全，也無法自動把漏洞修好。
5. **職責分離 (Separation of Duties)**：同儕審查 (Peer reviews) 就是落實這項精神的最佳體現 — 寫出這段程式碼的人，絕對不可以同時兼任那個能把這段程式碼批准放行上線的人。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **VCS** | Version Control System (版本控制系統，例如大家最熟悉的 Git) |
| **Branch Protection (分支保護)** | 透過設定檔的種種規則限制，來約束到底誰才有權限推送程式碼，以及程式碼在合併前必須全數通過哪些狀態與安檢檢查 |
| **Pull Request (PR / 拉取請求)** | 一種請求將自己修改的程式碼分支合併回主線的正式提案，此舉大大促進了團隊同儕審查的實施 |
| **Code Signing (程式碼簽章)** | 為即將出貨的執行檔蓋上密碼學數位簽章，藉以向使用者擔保該檔案是本店正牌出品且途中未遭竄改的保證書 |
| **Signed Commit (已簽章的提交)** | 開發人員使用私人的 GPG/PGP 金鑰對自己送出的 Git commit 進行密碼學簽章，用以確保這段程式碼真的是原作者本人親自發布的 |
| **Obfuscation (混淆)** | 將原本條理分明的程式碼故意弄得奇形怪狀、難以閱讀，企圖藉此挫敗對手逆向工程意圖的障眼法技術 |
| **HSM** | Hardware Security Module (硬體安全模組 — 專門用來把具有核彈級身分地位的「程式碼簽章金鑰」鎖在裡面保護的獨立硬體金庫) |
