# 5.2 分析程式碼的安全漏洞 (Analyze Code for Security Vulnerabilities)

## 學習目標

- 區分人工手動 (manual) 與自動化 (automated) 程式碼審查技術的差異
- 比較 SAST, DAST, IAST 以及 SCA 等不同的分析方法
- 為各種不同的分析類型，找出其最適合介入的軟體開發生命週期 (SDLC) 階段
- 描述模糊測試 (fuzz testing) 的目的與執行流程

---

## 人工程式碼分析 (Manual Code Analysis)

人工分析時至今日，依然是挑出那些「自動化工具根本抓不到的業務邏輯缺陷 (logic flaws)」絕不可或缺的關鍵手段。

### 同儕程式碼審查 (Peer Code Review)

程式碼審查 (Code review) 是一種由一位或多位開發人員，來檢查由另一位同事所撰寫之原始碼的實務作法。

| 審查類型 | 說明 |
|------|-------------|
| **背後靈式審查 (Over-the-shoulder)** | 原作者盯著同一個螢幕，親自向身旁的審查者解釋程式碼。隨興且快速。 |
| **結對程式設計 (Pair programming)** | 兩位開發人員共用一台工作站一起寫扣。這是一種持續不間斷的審查過程。 |
| **工具輔助審查 (Pull Requests / PR)** | 審查者利用工具 (如 GitHub, GitLab) 在合併程式碼 (merging) 的差異檔 (diffs) 上留下評論。這是現代業界最標準的作法。 |
| **正式檢驗 (Formal/Fagan Inspection)** | 一種高度結構化、極度嚴謹的審查會，每個參與者都有預先定義好的嚴格角色分工 (主持人、作者、朗讀者、測試員)。這在崇尚敏捷開發的現代很少見，通常只用於高保證 (high-assurance) 級別的續命系統或航太系統。 |

**帶來的好處**：這是找出**業務邏輯缺陷 (business logic flaws)**、整體架構設計問題，以及評估程式碼可維護性的「最強」手段。
**缺點/侷限**：極度耗時、極端仰賴審查者個人的資安專業水準與經驗，而且很難大規模推廣 (不具擴展性)。

## 自動化應用程式安全測試 (AST, Automated Application Security Testing)

自動化工具能透過分析程式碼或是實際與正在運行中的應用程式互動，來抓出常見的已知漏洞特徵。

### SAST (靜態應用程式安全測試, Static Application Security Testing)

SAST 負責分析**原始碼、位元組碼 (bytecode) 或二進位執行檔，且「不需要」把程式跑起來** (屬於白箱測試 White-box testing)。

| 特性 | 詳細說明 |
|----------------|--------|
| **何時該用它** | 越早越好，佈署在 SDLC 的極早期階段 (開發人員的 IDE 內、commit hooks 攔截器、CI/CD 管線中)。 |
| **它能抓到什麼** | 緩衝區溢位、常見的語法級注入缺陷、原始碼中寫死的機密 (hardcoded credentials)、不安全的加密函式呼叫。 |
| **優點 Pros** | 能精準指出有問題的「檔案與行號位置」；能在程式碼被編譯/部署之前就揪出問題；具備極高的擴展性。 |
| **缺點 Cons** | **誤報率極高 (High false positive rate)**；無法抓出因執行環境設置不當所引發的問題，也抓不到複雜的邏輯缺陷；深度綁定並受限於特定的程式語言。 |

### DAST (動態應用程式安全測試, Dynamic Application Security Testing)

DAST 從**外部與一個「正在執行中」的應用程式進行互動**，行為模式就像個真實的攻擊者一樣 (屬於黑箱測試 Black-box testing)。

| 特性 | 詳細說明 |
|----------------|--------|
| **何時該用它** | 佈署在 SDLC 的較晚期階段 (測試/QA、Staging 測試環境、Pre-production 預備上線環境)。前提條件是應用程式「必須先跑起來」。 |
| **它能抓到什麼** | 身分驗證繞過、伺服器組態設定不當、XSS、(透過觀察系統吐錯的回應來判斷的) 注入攻擊。 |
| **優點 Pros** | **誤報率極低 (Low false positive rate)** (只要它抓到的，通常都是真的能打穿的漏洞)；與程式語言無關；能抓出環境/執行時期的組態問題。 |
| **缺點 Cons** | 無法告訴你到底是哪一行程式碼寫錯了；需要一個穩定且執行中的環境；測試的涵蓋率 (test coverage) 完全取決於它背後那隻網路爬蟲/腳本夠不夠聰明。 |

### IAST (互動式應用程式安全測試, Interactive Application Security Testing)

IAST 結合了 SAST 與 DAST 的優點。它透過在應用程式運行時，**從內部植入探針 (instrumenting) 來進行監控** (屬於玻璃箱/灰箱測試 Glass-box testing)。

| 特性 | 詳細說明 |
|----------------|--------|
| **何時該用它** | 通常佈署在測試/QA 階段 (這時通常會搭配自動化功能測試一起跑)。 |
| **它是如何運作的** | 會有一個代理程式 (agent) 潛伏在應用程式內部，當外部的測試腳本 (或是 DAST 工具) 在狂打這支 App 時，代理程式負責在內部同步監控記憶體、流量與程式碼執行流向。 |
| **優點 Pros** | 準確率極高 (誤報極低)；能精準指出有問題的程式行號；能抓出執行時期的缺陷。 |
| **缺點 Cons** | 需要特定的語言/框架支援 (不是什麼語言都能掛 agent)；可能會拖慢應用程式的執行效能；極度依賴要有夠好的功能測試涵蓋率 (有跑到的路徑才測得到)。 |

### RASP (執行時期應用程式自我保護, Runtime Application Self-Protection)

和 IAST 一樣，RASP 也是潛伏執行在應用程式內部。但 RASP 的重點在於**防護阻擋 (protection) 而不是測試**，它負責在正式生產環境 (production) 中，即時攔截並阻斷攻擊。(這部分會在領域 7 深入探討，但概念上與 AST 系出同源)。

### SCA (軟體組成分析, Software Composition Analysis)

SCA 工具負責分析專案原始碼與建置產出物，藉以盤點出**所有使用到的第三方與開源元件**，並拿它們去跟已知的漏洞資料庫 (如 NVD) 進行比對。

| 特性 | 詳細說明 |
|----------------|--------|
| **何時該用它** | 佈署在每一次的建置 (build) 或程式碼提交 (commit) 階段。 |
| **它是如何運作的** | 自動產出一份 SBOM (軟體物料清單 SBOM)；精準標記出帶有已知漏洞的元件；揪出開源授權條款 (license) 違反合規的問題。 |
| **優點 Pros** | 完美解決 OWASP Top 10 中「包含已知漏洞元件」的痛點；掃描速度極快。 |
| **缺點 Cons** | 只能抓出*第三方*程式碼中那些*已經被公開紀錄*的漏洞 (CVEs)；它完全不管/也不掃你自己寫的客製化程式碼。 |

## AST 分析工具大車拚 (AST Comparison Summary)

| 工具類型 | 分析類型 | 應用程式當下狀態 | 能否精準指出程式碼行號？ | 誤報率 (False Positives) |
|------|---------------|---------------|---------------------|-----------------|
| **SAST** | 靜態 (白箱) | 不須執行 (Not running) | 可以 (Yes) | 很高 (High) |
| **DAST** | 動態 (黑箱) | 執行中 (Running) | 不行 (No) | 很低 (Low) |
| **IAST** | 互動式 (玻璃箱) | 執行中 (Running) | 可以 (Yes) | 極低 (Very Low) |
| **SCA** | 組成成分 | 不須執行 (Not running) | 可以 (列出相依套件清單) | 很低 (Low) |

## 模糊測試 (Fuzzing / Fuzz Testing)

模糊測試指的是將**無效的、非預期的、或是隨機生成的垃圾資料**當作輸入，強行塞給一支電腦程式，藉由觀察系統是否會拋出例外狀況、發生崩潰當機或記憶體外洩，來找出潛在的脆弱點。

### 模糊測試的兩大流派

| 類型 | 說明 |
|------|-------------|
| **基於生成的模糊測試 (Generation-based / 聰明 Fuzzing)** | 預先了解目標預期的資料格式，然後「有智慧地」生成出格式半對半錯的誘餌 (例如：基於 RFC 標準故意產生格式畸形的 HTTP 請求)。 |
| **基於突變的模糊測試 (Mutation-based / 笨蛋 Fuzzing)** | 拿一份原本合法正常的輸入檔案，然後在完全不理解其通訊協定或格式內容的情況下，隨機且盲目地對其進行突變惡搞 (例如：隨機翻轉 bit、亂塞資料截斷檔尾)。 |

**進行模糊測試的主要目標**：
- 揪出緩衝區溢位 (buffer overflows) 的死角。
- 發掘未被妥善處理的例外狀況 (unhandled exceptions) 以及會導致系統當機的特殊情境。
- 發現記憶體外洩漏洞 (memory leaks)。

模糊測試在尋找負責「解析邏輯 (parsing logic)」與「網路通訊協定」模組中的**零日漏洞 (zero-day vulnerabilities)** 時表現極其優異，特別是針對 C/C++ 寫出的老舊應用程式。

## 考試重點

1. **SAST vs. DAST 的強烈對比**：SAST = 早期介入、掃原始碼、誤報率偏高；DAST = 晚期介入、掃運行中的 App、誤報率很低。
2. **IAST**：從 App 內部插針監控，完美融合了 SAST (懂原始碼) 與 DAST (懂運行狀態) 兩者的優勢。
3. **SCA**：用途非常單純且專一，就是用來抓「開源軟體/第三方相依套件」身上的已知舊傷 (CVE) 用的。
4. **模糊測試 (Fuzzing) 的核心目標**：最愛用來找尋那些能搞崩應用程式 (crashes)、逼出未能妥善處理的例外狀況，以及榨出記憶體外洩漏洞 (特別是緩衝區溢位)。
5. **邏輯缺陷 (Logic flaws) 剋星**：自動化工具 (SAST/DAST) 通常對揪出複雜的商業邏輯缺陷表現極差；這部分還是得仰賴**人工的同儕程式碼審查 (manual peer review)** 或威脅建模來把關。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **SAST** | Static Application Security Testing (靜態應用程式安全測試) |
| **DAST** | Dynamic Application Security Testing (動態應用程式安全測試) |
| **IAST** | Interactive Application Security Testing (互動式應用程式安全測試) |
| **SCA** | Software Composition Analysis (軟體組成分析) |
| **Fagan Inspection** | 一種極度正式、採用高度結構化且分工細膩的人工程式碼審查流程 |
| **Fuzzing (模糊測試)** | 直接將無效的、非預期的、或是隨機的亂數資料灌給應用程式，藉以誘發並觀察其當機/崩潰等異常行為的測試手法 |
