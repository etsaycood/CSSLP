# 6.1 發展安全測試策略 (Develop Security Testing Strategy)

## 學習目標

- 定義安全測試策略的目的與組成要素
- 解釋風險 (risk) 是如何驅動/決定測試的範圍與深度
- 區分測試策略 (test strategy) 與測試計畫 (test plan) 的不同
- 識別在制定測試策略時所參考的各項國際標準與指南

---

## 什麼是安全測試策略 (Security Testing Strategy)？

安全測試策略是一份**高階的指導文件 (high-level document)**，用於定義對軟體進行測試的整體方針、最終目標以及邊界參數。它是用來銜接「組織高大上的資安政策方針」與「第一線實際動手測試執行」之間落差的重要橋樑。

| 特性 | 說明 |
|----------------|-------------|
| **關注焦點 (Focus)** | 高階宏觀視角（探討「要測什麼 What」與「為什麼要測 Why」） |
| **目標讀者 (Audience)** | 管理階層、利害關係人 (Stakeholders)、專案經理 |
| **生命週期 (Lifespan)** | 屬於長期規劃；通常適用於跨越多個專案，甚至涵蓋整個組織層級 |
| **源頭/依據 (Origin)** | 衍生自組織的安全政策要求、風險胃納量 (risk appetite) 以及法規合規性要求 |

### 策略 (Strategy) vs. 計畫 (Plan)

> **考試提示 (Exam Tip)**：策略 (STRATEGY) = 著眼於巨觀的高階方針（例如：「本公司將全面採用自動化 SAST 掃描搭配人工滲透測試技術」）。計畫 (PLAN) = 鉅細靡遺的戰術執行細節（例如：「工程師 Jane 將於本週二上午 10 點整啟動 Checkmarx 工具掃描原始碼」）。

---

## 測試策略的組成要素 (Components of a Testing Strategy)

一份全面的測試策略，通常會涵蓋以下幾個關鍵元素：

### 1. 目標 (Objectives)
這場測試到底想達成什麼目的？（例如：「確保這次的系統能通過 PCI DSS 發卡組織稽核大考」，或是「保證在軟體正式發佈前，絕對不能殘留任何 OWASP Top 10 的漏洞。」）

### 2. 基於風險的測試方針 (Risk-Based Testing Approach)
要在哪裡花多少力氣測試、測試要鋪奪廣挖多深，完全取決於該應用程式的**風險輪廓 (risk profile)**。
- **高風險應用程式 (High Risk App)**（例如：牽涉金流的金融交易系統）：必須享受全套頂級待遇，包含 SAST, DAST, IAST, 人工程式碼細部審查，再外加上花錢請外部第三方團隊來打滲透測試 (penetration testing)。
- **低風險應用程式 (Low Risk App)**（例如：純內網觀看、不牽涉個資的員工餐廳每週菜單系統）：可能只需要設定跑個自動化的 SAST 跟 DAST 掃描交代一下就夠了。

### 3. 可依循的標準與指南 (Standards and Guidelines)
策略文件中必須明確點出，這次的測試將對齊並遵從哪些業界公認標準：
- **OWASP ASVS** (應用程式安全驗證標準 Application Security Verification Standard)
- **NIST SP 800-115** (資訊安全測試與評估技術指南 Technical Guide to Information Security Testing and Assessment)
- **PTES** (滲透測試執行標準 Penetration Testing Execution Standard)
- **OSSTMM** (開源安全測試方法論手冊 Open Source Security Testing Methodology Manual)

### 4. 允許的測試環境 (Environments)
明確規範「哪些環境可以被拿來測試」。（例如：「一切具破壞性的打擊測試，只准在 Staging 預備環境中進行。嚴禁在未經核准且未排定維護公告時間的情況下，直接拿自動化掃描軟體去轟炸 Production 正式環境。」）

### 5. 自動化策略 (Automation Strategy)
定義在整場測試中「人工作業」與「自動化掃描」兩者間的心力分配比例；並且規劃這些測試工具該如何與現有的 CI/CD 佈署管線完美融合 (也就是實踐 DevSecOps 精神)。

## 安全測試的類型 (Types of Security Testing)

測試策略就是由上面這群高層，來拍板決定這次專案該端出*哪些測試手法的組合套餐*：

| 測試類型 | 說明 |
|-----------|-------------|
| **白箱測試 (White-Box Testing)** | 在對系統內部架構與原始碼瞭若指掌 (甚至擁有原始碼) 的情況下進行的測試 (例如：SAST 工具掃描、人工作業的原始碼逐行審查)。 |
| **黑箱測試 (Black-Box Testing)** | 對系統內部運作邏輯一無所知，完全瞎子摸象般站在外部駭客視角發動攻擊的測試 (例如：DAST 動態掃描、外部滲透測試)。 |
| **灰箱測試 (Gray-Box Testing)** | 具備一定程度的內部情報 (例如：雖然沒有原始碼，但測試者擁有登入帳號密碼，能以一般使用者身分從內部觀察系統反應) 所進行的測試 (例如：IAST 互動式掃描、已身分驗證的 DAST 掃描)。 |
| **模糊測試 (Fuzz Testing)** | 故意亂塞一堆無效的、格式錯亂的、或是隨機亂數的垃圾資料餵給系統，企圖把它搞崩潰或誘發例外狀況 (exception)。 |
| **迴歸測試 (Regression Testing)** | 針對以前曾經發現過、且宣稱已經被修好的舊漏洞「重新再測一次」，為了是確保新修改的程式碼沒有不小心又把這些老病根給喚醒 (reintroduced)。 |

---

## 考試重點

1. **策略 (Strategy) vs. 計畫 (Plan) 的差別**：策略看的是長遠廣闊的高階宏觀；計畫則是負責落實且訂出具體時程表的戰術執行。
2. **基於風險的測試 (Risk-based testing)**：好鋼要用在刀刃上。投入的測試資源與心力，必須與該應用程式所承受的風險等級成正比。
3. **黑箱、白箱與灰箱的差異**：必須清楚理解這三種測試法中，測試人員各自掌握了多少系統內部情報與原始碼。
4. **必考國際標準**：只要談到資安測試標準，請立刻聯想到 NIST 800-115 測評指南以及 OWASP ASVS。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Test Strategy (測試策略)** | 一份定義測試整體方針、目標與框架的高階層級指導文件 |
| **Risk-Based Testing (基於風險的測試)** | 一種依據應用程式曝險輪廓與衝擊大小，來決定該分配多少測試火力資源的實踐哲學 |
| **White-Box Testing (白箱測試)** | 測試者手握系統完整原始碼與架構藍圖，在毫無秘密的情況下進行的深度測試 |
| **Black-Box Testing (黑箱測試)** | 測試者對系統內部運作邏輯一無所知，純粹憑藉外部探測摸索進行的盲眼測試 |
| **Regression Testing (迴歸測試)** | 在程式碼發生變動後重新執行測試，以確保新的修改沒有意外造成舊 BUG 復發重現 |
| **ASVS** | OWASP Application Security Verification Standard (OWASP 發布的應用程式安全驗證標準規範) |
