# 6.3 發展安全測試案例 (Develop Security Test Cases)

## 學習目標

- 區分正向測試 (positive testing) 與負向測試 (negative testing) 之間的差異
- 設計出有效的正向與負向測試案例
- 應用邊界值分析 (BVA, Boundary Value Analysis) 與等價類劃分 (Equivalence Partitioning) 技巧
- 構思並制定出防範誤用與惡意濫用 (misuse and abuse) 的測試案例

---

## 測試案例的解剖學 (The Anatomy of a Test Case)

測試案例代表著一組經過精心設計的**特定條件、測試資料，以及預期的系統反應結果**，用來檢驗某項安全需求到底有沒有被確實做到。

一個標準的測試案例會包含下列元件：
- **測試 ID / 名稱 (Test ID / Name)**：供識別用的專屬編號，通常用來反向對應追溯回 SRTM (安全需求追溯矩陣) 上的需求項目。
- **前置條件 (Preconditions)**：在正式發動測試之前，系統必須先被設定好的狀態（例如：「使用者目前已處於成功登入狀態」）。
- **測試步驟 (Test Steps)**：測試人員（或自動化腳本）照表操課必須滑鼠點擊/敲鍵盤的精確操作步驟。
- **測試資料 (Test Data)**：實際用來塞進系統裡的那些輸入值（例如：假造的帳號、密碼，或是夾帶特製的惡意攻擊字串 payload）。
- **預期結果 (Expected Result)**：如果系統真的有乖乖做好防禦安全機制，它*應該*要作出的反應。
- **實際結果 (Actual Result)**：測試打下去後，系統*實際上*真正幹了什麼好事 (判斷為 通過 Pass / 失敗 Fail)。

---

## 正向測試 vs. 負向測試 (Positive vs. Negative Testing)

資安測試之所以與眾不同，很大一部分取決於你必須深刻理解系統在面對「守規矩的輸入」與「來搞破壞的輸入」時，會有什麼截然不同的反應。

### 正向測試 / 快樂路徑 (Positive Testing / Happy Path)
旨在判定當我們餵給系統合法、乖巧正常的輸入，並執行被授權允許的操作時，應用程式到底有沒有**如預期般正常運作 (works as expected)**。
- **關注焦點**：系統的功能性與易用性。核心靈魂拷問：「它有沒有乖乖做好它份內該做的事？」
- **情境範例**：在登入畫面輸入完全正確的帳號與密碼，預期系統會放行並引導登入者進入管理儀表板畫面。
- **資安上的意義**：確保合法合規的善良使用者不會被自家保全無端攔阻拒絕存取（這是為了保障 CIA 三要素中的「可用性 Availability」）。

### 負向測試 / 悲慘路徑 (Negative Testing / Sad Path)
旨在驗證當應用程式遭遇到非法、格式錯亂的輸入時，能否**優雅且安全地應對處理 (handles invalid input gracefully)**，並果斷拒絕任何未經授權的僭越操作。
- **關注焦點**：系統的韌性與安全性。核心靈魂拷問：「它能不能強勢阻擋那些它*不該做/不被允許做*的事？」
- **情境範例**：在登入畫面的帳號欄位輸入惡意代碼 `' OR 1=1 --`，預期系統會識破輸入異常並果斷拒絕登入，然後在畫面上只拋出一個模糊籠統的錯誤訊息；而不是傻傻地放行駭客登入，或是直接崩潰導致資料庫內部錯誤訊息噴滿整個畫面。
- **資安上的意義**：這是用來驗證系統的錯誤處理機制是否嚴謹、系統在出錯時是否能退回安全狀態保護自己 (失效安全 fail-secure)，以及它抵禦各種惡意注入攻擊 (injection) 的防禦力。

> **考試提示 (Exam Tip)**：在傳統的功能性軟體品保測試 (functional testing) 中，絕大部分的心力都花在測「正向測試 (Positive testing)」求神明保佑它會動。但在**資安測試 (security testing)** 的領域裡，絕對多數的心力與資源，都瘋狂砸在了想盡各種辦法把系統搞殘的**負向測試 (Negative testing)** 上。

## 輸入測試設計技巧 (Input Testing Techniques)

在構思負向測試用的「惡意輸入資料」時，你不可能窮舉測試這個世界上無限可能的所有排列組合。因此，測試專家會運用特定的科學技巧，用「數量最少」的測試案例，榨出「覆蓋率最廣」的防禦死角。

### 1. 邊界值分析 (BVA, Boundary Value Analysis)
軟體的 Bug (蟲子) 最喜歡躲在輸入條件規範的邊緣 (邊界/縫隙) 處。BVA 抓蟲戰術就是精準打擊那些「剛好在邊界線上」、「差一點點就摸到線上 (低於)」，以及「剛好跨過那條線一步 (高於)」的臨界數值。
- **情境設定**：規定密碼長度必須介於 8 到 12 個字元之間。
- **使用 BVA 設計出的測試案例**：
    - 輸入 7 個字元 (無效 Invalid - 剛好低於最低門檻一步)
    - 輸入 8 個字元 (有效 Valid - 剛好踩在最低門檻線上)
    - 輸入 12 個字元 (有效 Valid - 剛好踩在最高門檻線上)
    - 輸入 13 個字元 (無效 Invalid - 剛好超速越過最高門檻一步)

### 2. 等價類劃分法 (Equivalence Class Partitioning)
把所有理論上可能的無數數值，像切蛋糕一樣劃分成幾個群組 (partitions/等價類)。其核心邏輯是：只要系統面對某個群組內的一筆資料做出了特定的反應，那麼我們就大膽假設系統面對該群組內的「任何一筆」資料，都會做出「一模一樣」的反應。因此，你每一種狀況都只需要挑選*一個*代表性數值來測試就收工了。
- **情境設定**：系統只允許 18 到 65 歲的人註冊。
- **切分出來的等價類 Partition群組**：
    - 群組 1：`年齡 < 18` (無效的年齡) -> 派代表 `10` 歲去測試是否被拒絕
    - 群組 2：`18 <= 年齡 <= 65` (合法的年齡) -> 派代表 `30` 歲去測試是否放行
    - 群組 3：`年齡 > 65` (無效的年齡) -> 派代表 `80` 歲去測試是否被拒絕

### 3. 防範誤用與濫用的測試案例 (Misuse and Abuse Case Testing)
直接拿我們在領域 3 與領域 4 所辛苦建立的「威脅模型」以及「誤用/濫用案例」，把它們當作劇本，一比一還原拿來當作測試手段。
- **正常使用情境 (Use Case)**：使用者上傳大頭照來更新個人檔案。
- **無心之過/誤用情境 (Misuse Case)**：使用者手滑，不小心把一份高達 500MB 的 PDF 檔當成大頭照傳了上去。 (測試此情境：驗證系統是否有做檔案大小限制與附檔名驗證攔截)。
- **心存惡意/濫用情境 (Abuse Case)**：駭客故意上傳了一隻極度危險的 `.php` 一句話木馬網頁後門，但狡猾地把副檔名竄改偽裝成小畫家的 `.jpg` 檔，企圖藉此獲取遠端伺服器控制權 (RCE)。 (測試此情境：驗證系統是否有深扒並檢查檔案的實際內容型別 (content-type)，以及伺服器儲存該照片的目錄是否被拔除了執行權限)。

## 狀態演進與工作流程測試 (State and Workflow Testing)

高竿的資安測試絕對不僅止於傻傻地在單一輸入框裡塞字串，你必須跳出來以宏觀視角，去挑戰應用程式骨子裡的狀態機 (state machine) 邏輯。

- **強行瀏覽 / URL 竄改 (Forced Browsing / URL Manipulation)**：考驗系統的邏輯順序防線。例如買東西時，惡意不把結帳手續的第一步跟第二步走完，就直接在網址列強行跳到「步驟三：訂單完成畫面」，看系統會不會被糊弄過去。
- **參數竄改 (Parameter Tampering)**：當網頁把填好的表單拋回給伺服器時，用中途攔截代理伺服器 (如 Burp Suite) 把它截下來，偷偷把裡面隱藏的欄位、購物車商品的結帳金額，或是代表身分的 User ID 竄改掉後，再放行送去給伺服器，看伺服器會不會笨笨地照單全收。
- **競爭危害/時間差攻擊 (Race Conditions, TOC/TOU)**：利用系統處理多工的時間差縫隙。例如幾乎在同一毫秒內，一口氣連發了好幾個「使用折價券」的請求給伺服器，測試系統會不會在餘額或是狀態還來不及更新的空檔，笨笨地讓你一張折價券抵用兩次。

---

## 考試重點

1. **正向測試 (Positive Testing)**：餵給系統合情合理的資料，求個「系統能如預期般順利運作」的平安結果。
2. **負向測試 (Negative Testing)**：故意拿破銅爛鐵或惡毒字串去戳系統，驗證系統能否在遭到侵犯時「退回安全狀態防禦自己並優雅死機 (fails securely)」。這在資安測試領域是重中之重。
3. **等價類劃分 (Equivalence Partitioning)**：把資料分門別類，然後每個類別只派一個「代表人物」出來受測就好，藉此大幅減少測試工時。
4. **邊界值分析 (BVA)**：專打輸入限制的最邊緣地帶 (也就是 n-1，剛好命中 n，以及越軌跑出界線的 n+1) 來抓出防禦縫隙。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Positive Testing (正向測試)** | 餵給系統完全合法、乖順的測試資料，為了確保系統本身預期的功能可以正常無誤地被執行 |
| **Negative Testing (負向測試)** | 刻意拿無效的、格式畸形的資料去挑釁系統，為了驗證系統在出事時能妥善且安全地應對錯誤，而不會崩潰或被駭客趁虛而入 |
| **BVA** | Boundary Value Analysis (邊界值分析)；一種專挑輸入容許值極限邊緣的線上、剛剛好超過一點點與低於一點點的位置來發動測試的技巧 |
| **Equivalence Partitioning (等價類劃分)** | 將輸入域大餅像切蛋糕一樣劃分出多個區塊(等價類)，並大膽假設只要測其中一個數值，就能代表整個區塊測試結果的技巧 |
| **Test Case (測試案例)** | 為了驗證某一條資安防護需求是否真的有效運作，所詳細撰寫包含前置條件、執行步驟、注入資料及預期結果的劇本腳本 |
