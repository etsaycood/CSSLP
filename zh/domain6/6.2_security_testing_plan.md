# 6.2 發展安全測試計畫 (Develop Security Testing Plan)

## 學習目標

- 區分測試計畫 (test plan) 與測試策略 (test strategy) 之間的不同
- 識別出一份安全測試計畫的核心組成要素
- 明確定出測試範圍 (scope)、排程 (schedule) 以及資源分配 (resource allocation)
- 解釋安全需求追溯矩陣 (SRTM) 在測試規劃階段所扮演的關鍵角色

---

## 什麼是安全測試計畫 (Security Testing Plan)？

安全測試計畫是一份**戰術層面、與特定專案綁定 (tactical, project-specific) 的執行文件**，它的源頭與大方向是繼承自組織的整體測試策略 (Testing Strategy)。計畫書中鉅細靡遺地羅列了為特定版本發布或特定應用程式執行測試時所需的**邏輯流程、實作方法、時程表安排，以及所需動用的各項資源**。

| 特性 | 說明 |
|----------------|-------------|
| **關注焦點 (Focus)** | 低階微觀、戰術層面（探討「要怎麼做 How」、「誰來做 Who」、「何時做 When」） |
| **目標讀者 (Audience)** | QA 品保團隊、第一線資安測試人員、開發工程師 |
| **生命週期 (Lifespan)** | 屬於短期規劃；它的壽命僅侷限於單一特定專案、一次 sprint 衝刺週期，或是一次軟體版本發布 |
| **源頭/依據 (Origin)** | 衍生自組織高階的測試策略 (Test Strategy) 以及該特定專案自身獨有的安全需求 |

---

## 安全測試計畫的組成要素 (Components of a Security Testing Plan)

一份堅若磐石的測試計畫必須能完美解答所有後勤調度的疑問。其關鍵章節包含：

### 1. 測試範圍 (Scope: In-Scope and Out-of-Scope)
白紙黑字明確畫出紅線，規定到底「什麼東西能測」，以及更重要的「什麼東西絕對不准碰」。
- **範圍內 (In-Scope)**：「主要 Web 應用程式的對外 API、使用者登入入口網頁，以及後端的資料庫。」
- **範圍外 (Out-of-Scope)**：「串接第三方的金流支付閘道伺服器、架設 Staging 測試環境的底層網路硬體設備。」
> **資安特別提醒**：在執行滲透測試時，清楚劃定「範圍外 (out-of-scope)」的紅線攸關生死，這能確保駭客(測試員) 的砲火不會不小心打歪，意外擊沉（癱瘓）與系統相連的其他無辜第三方服務或共用系統。

### 2. 時程與後勤安排 (Schedule and Logistics)
- **時間軸 (Timelines)**：這場測試大戲何時開鏡 (開始)、何時殺青 (結束)？
- **攻擊視窗 (Test Windows)**：到底允許在每天的哪幾個時段發動具破壞性癱瘓風險的測試或壓力測試？（例如：「為避免影響白天正常營運，只准許在美東時間凌晨 2:00 到 4:00 之間動手」）。
- **里程碑 (Milestones)**：訂出完成初步弱點掃描、完成人工作業打擊測試，以及提交最終驗收報告的死線 (Deadlines)。

### 3. 資源分配調度 (Resource Allocation)
- **人力 (Personnel)**：究竟派大軍是誰？（用內部 QA、自家的紅隊/資安人員，還是花錢請外面的第三方滲透測試公司）。
- **武裝工具 (Tools)**：明訂出精確的武器清單（例如：「規定使用 Burp Suite Pro 作為攔截代理伺服器、使用 Nessus 負責底層基礎架構弱掃」）。
- **靶場環境 (Environments)**：明確指出到底要打哪個特定環境的 URL 網址或 IP 位址。

### 4. 測試資料需求 (Test Data Requirements)
測試員需要準備什麼武器彈藥(資料) 才能發動測試？這些測試用的機敏資料該如何保護？
- 例如：「QA 團隊請先開好 5 個處於啟用狀態、且具備不同權限位階（如 Admin 管理員、User 一般用戶、Read-Only 唯讀訪客）的測試用帳號供測試員使用。」

### 5. 准入與退場條件 (Entrance and Exit Criteria)
- **准入條件 (Entrance Criteria)**：在測試團隊鳴槍開測前，有哪些前置條件必須先被滿足？（例如：「必須確認開發團隊已經把新功能全數寫完 (feature complete)、程式碼已部署到 Staging 環境，而且最基本的冒煙測試 (smoke tests) 必須過關系統沒死機。」）
- **退場條件 (Exit Criteria)**：怎樣才算大功告成測試圓滿結束？（例如：「計畫書內所有預定的測試案例皆已打完收工，而且系統目前身上**已經沒有殘留任何**尚未修復的 Critical (極危險) 或 High (高危險) 等級漏洞。」）

### 6. 缺陷與事件管理 (Defect Management / Incident Response)
當測試員在打擊過程中挖到寶(漏洞)時，該如何通報？萬一不小心把伺服器打死該找誰求救？
- 抓到的蟲 (bugs) 該上報登錄到哪個系統去？（例如：統一開 Jira 工單）
- 萬一測試員下手太重，不小心把 Staging 伺服器打到崩潰當機時的**緊急聯絡人應變通報程序 (emergency contact procedure)** 是什麼？

## 安全需求追溯矩陣的角色 (The Role of the SRTM)

那張我們在領域 3 (Domain 3) 辛苦刻出來的**安全需求追溯矩陣 (SRTM)**，到了這裡將會化身為產出測試計畫最重要的一本起訴書/輸入來源。
- 測試計畫就是靠著 SRTM 來確保這套系統的防護被**全面覆蓋 (Full Coverage)** 檢驗，絕無漏網之魚。
- 對於 SRTM 上頭洋洋灑灑列出的「每一條」安全需求，你在這份測試計畫書裡，都必須變出**至少一個**對應的測試案例 (Test Case) 去驗證它到底有沒有做到。
- **雙向追溯性 (Bi-directional Traceability)**：確保每一項測試背後都有其為了滿足某條需求的正當理由；同時也確保每一條寫在紙上的安全需求，都有實際被拿去測試考驗過。

---

## 簽署與核准放行 (Approvals and Sign-off)

由於資安測試（特別是滿懷惡意的 DAST 動態掃描與滲透測試 Pentesting）具有極強的破壞性與干擾性，因此這份測試計畫書絕對不能測試團隊自己說了算，必須取得各方利害關係人的**正式簽字背書 (formal sign-off)**：
- **專案經理 / 資產擁有人 (Project Manager / Asset Owner)**：簽字確認他們清楚了解並同意這場測試的打擊範圍 (scope) 與時程。
- **IT / 維運團隊 (IT / Operations)**：簽字確認他們知道未來的哪幾個晚上系統可能會因為被打而效能變差或異常，並做好心理準備。
- **資安高層 (Security Leadership)**：簽字擔保這份計畫書的打擊力道與深度，足夠揪出該應用程式所面臨的各種潛在資安風險。

## 考試重點

1. **計畫 (Plan) vs. 策略 (Strategy)**：計畫是為了「單一特定專案」量身訂做、充滿執行細節 (人事時地物) 的文件；策略則是全公司一體適用、宏觀的指導方針。
2. **範圍邊界 (Scope boundary)**：在計畫書中把「範圍外 (out-of-scope)」也就是絕對不准碰的紅線給畫清楚，重要性絕對不亞於告訴測試員該去打哪裡，這是為了避免惹禍上身引發越權測試。
3. **准入與退場條件 (Entrance/Exit Criteria)**：這就是兩道鐵門，嚴格規定了想要讓測試起跑，以及想要測試宣布結案時，必須強制滿足的各項嚴苛條件。
4. **SRTM (安全需求追溯矩陣)**：測試計畫的精神導師。計畫書必須確保 SRTM 上的每一條安全需求，都有派上用場的對應測試案例去負責「照妖鏡」驗收 (確保測試涵蓋率 Coverage)。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Test Plan (測試計畫)** | 充滿戰術執行細節的文件，明確記載針對單一特定測試任務的時程表安排、打擊範圍與調度的各項後勤資源 |
| **In-Scope (範圍內)** | 被正式授權列入靶場名單，允許對其發動測試與攻擊的系統、功能或網路區段 |
| **Out-of-Scope (範圍外)** | 那些被明令列入黑名單，絕對禁止對其發射火力測試的無辜/非標的系統 |
| **Entrance Criteria (准入條件)** | 一道前置的檢驗關卡，規定在正式啟動打擊測試之前，專案本身必須先滿足的各項準備就緒狀態條件 |
| **Exit Criteria (退場條件)** | 一道結案的檢驗關卡，規定當滿足什麼條件 (例如：沒有存活的高危漏洞) 時，這場測試活動才算圓滿落幕宣告完工 |
| **SRTM** | Security Requirements Traceability Matrix (安全需求追溯矩陣) |
