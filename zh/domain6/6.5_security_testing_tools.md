# 6.5 執行安全測試：工具與實作 (Perform Security Testing: Tools and Execution)

## 學習目標

- 能在 SDLC 的各個階段中，精準挑選出最合適的「應用程式安全測試 (AST)」工具
- 區別結構化/白箱測試 (structural / white-box) 與動態/黑箱測試 (dynamic / black-box) 在原理上的核心差異
- 解釋滲透測試 (Penetration Testing) 在整個資安體系中所扮演的終極角色，以及它無奈的侷限性
- 徹底理解模糊測試 (Fuzzing) 的概念，以及程式碼測試涵蓋率 (Code Coverage) 的各項評分指標

---

## 應用程式安全測試 (AST) 工具總覽大亂鬥

要想做好軟體資安測試，你必須學會打出一套華麗的「工具組合拳」；因為這個世界上**絕對沒有任何一套單一神武工具，能幫你抓出所有種類的漏洞。**

| 工具類型 | 分析手法 / 視角 | 最適合介入的戰場 (Use Case) | 專長優勢 (Strengths) | 致命弱點 (Weaknesses) |
|-----------|-----------------|------------------|-----------|------------|
| **SAST** (靜態掃描) | 白箱 (透視原始碼) | 佈置於 SDLC 最早期最前線；裝在開發者的 IDE 裡、或是 CI/CD 的攔截器中。 | 能精準點出那隻該死的蟲子躲在「第幾行」；擴展性極佳，掃掃萬行代碼不費力。 | **無可救藥的高誤報率 (High false positives)**；而且它是個瞎子，完全抓不到那種程式跑起來後才發生的「伺服器設定錯誤」。 |
| **SCA** (套件組成分析) | 白箱 (透視相依套件) | 專門用來盤點/起底你偷偷引用的那些第三方開源函式庫有沒有黑歷史。 | 掃描速度極快；能幫你生出一份精美的 SBOM (軟體物料清單 SBOM，這是防範現代供應鏈攻擊的必備護身符)。 | 只會抓「別人寫的」開源套件裡「已經被公開登記在案 (CVE)」的舊傷；它完全不鳥你自己手寫的那些狗屁不通的客製化邏輯錯誤。 |
| **DAST** (動態掃描) | 黑箱 (對著活的應用程式猛打) | 佈署於 SDLC 的後半後段班；在 QA 測試站或 Staging 預備環境中發威。 | **誤報率極低 (Low false positives)** (只要它打得穿的，通常都是真的能殺人的洞)；專門獵殺執行時期的破綻 (如身分繞過、XSS 跨站腳本)。 | 無法告訴你到底是原始碼的哪一行寫爛了；而且它非常嬌貴，需要一個穩定活著且跑得起來的測試環境才能動工。 |
| **IAST** (互動式掃描) | 玻璃箱 (深入敵營App體內) | 伴隨在 QA 功能測試的過程中一起跑；它會在應用程式活著運作時，像寄生蟲一樣在內部同步監控程式碼流向。 | 準確度極高；能一次給你兩個願望：「指出精確行號」加上「抓出真實執行時期的動態缺陷」。 | 會拖慢應用程式運作的效能；而且它極度依賴 QA 人員的測試涵蓋率 (QA 沒測到的功能路徑，它身上這隻寄生蟲也連帶看不見任何破綻)。 |

> **溫馨提醒**：這些大名鼎鼎的 AST 兄弟們，我們其實在領域 5 (Domain 5) 的時候就已經給他們點過名了；但在領域 6 這裡，我們的焦點將放在：**如何像個指揮官一樣，把他們整合成一套拳拳到肉的整體測試策略 (testing strategy) 去驗證軟體安全。**

## 滲透測試 (Penetration Testing / Pentesting)

滲透測試，說穿了就是一場經過官方正式授權、真槍實彈針對自家電腦系統發動的「模擬網路攻擊」，目的是為了檢驗系統的真實抗打擊能力。

- **它在 SDLC 中的定位**：滲透測試是一項被排在 SDLC 大結局 (Pre-production 或 Production 階段) 的**「最終確認 (Validation)」**活動。它**絕對不應該**被當成你在開發過程中拿來抓臭蟲 (bugs) 的主力農具。它的存在，是為了驗證你前面花了幾百萬建置的「整套資安防禦體系」到底有沒有效。
- **終極目標**：化身成真實世界中陰險狡詐的攻擊者，去發掘那些冰冷的自動化掃描工具 (SAST/DAST) 永遠看不懂的貓膩：例如「商業邏輯缺陷 (business logic flaws)」、令人拍案叫絕的「組合技漏洞 (chained vulnerabilities)」（把兩個看起來無害的低風險小破綻，巧妙組合成一發致命的核彈級高風險攻擊），以及各種荒唐的伺服器組態設定失誤。

### 迷信滲透測試的悲慘代價 (The Problem with Pentesting)
- **太晚發現，病入膏肓 (Late in the Cycle)**：如果你直到新系統準備上線剪綵前夕，才靠著滲透測試抓出一個深植於底層架構的致命缺陷，那要回去打掉重練的成本將會是天文數字 (這嚴重違背了資安左移 Shift-Left 的精神)。
- **只是一瞬間的假象 (Point-in-Time)**：那份要價不菲的滲透測試報告，永遠只代表著「受測當天、那個特定版本程式碼」的虛幻安全承諾。在如今標榜敏捷開發、一天推版五六次的時代，你需要的是無間斷的自動化持續測試，而不是一年才求神拜佛辦一次的滲透測試法會。

## 模糊測試 (Fuzzing)

模糊測試的操作手法就是如此粗暴：準備海量無效的、預期外的，或者是完全亂碼的垃圾資料 (也就是所謂的 "fuzz/雜訊")，瘋狂地傾瀉灌入應用程式的嘴裡，直到把它灌到當機崩潰、觸發未處理的例外狀況，或是逼它把記憶體給吐出來 (記憶體外洩 leaking)。

- **核心目標**：揪出 C/C++ 老古董最愛的緩衝區溢位漏洞 (buffer overflows)、那些程式設計師忘記接住的例外狀況 (unhandled exceptions)，以及潛藏在負責「解析檔案 (parsers)」或「解讀網路通訊協定」模組裡的底層邏輯錯誤。
- **基於生成 (Generation-based / 聰明的 Fuzzer)**：這隻 Fuzzer 是個讀書人，它熟讀過目標預期的檔案格式或通訊標準的 RFC 文件 (例如 HTTP 協定)，然後它會「用心險惡」地照著這些規則，量身打造出骨架合法但內藏劇毒的格式畸形惡意輸入。
- **基於突變 (Mutation-based / 笨蛋卻暴力的 Fuzzer)**：這隻 Fuzzer 就是個文盲，它隨手抓起一個原本一切合法正常的輸入檔案，然後在完全不理解其資料結構的情況下，開始胡亂地翻轉裡面的 bits、亂砍字串或瞎塞亂碼，看看能瞎貓碰上什麼死耗子。

## 程式碼測試涵蓋率指標 (Code Coverage Metrics)

你宣稱你測試完了？那請問這套龐大的應用程式裡，到底有「多少比例」的扣是真的有被打到的？測試涵蓋率的各項指標，就是用來冷酷量化你這套測試腳本到底有「多徹底」。

| 評估指標 | 高深莫測的定義 | 資安戰場上的大白話解釋 |
|--------|------------|------------------|
| **敘述句涵蓋率 (Statement Coverage)** | 在測試大軍掃過後，程式碼裡到底有百分之幾的「執行語句」有被真的踩到過。 | 想拿及格分數最簡單的方法，但也是防禦力最弱的安慰劑指標。(例如：隨便跑個超隨便的正向測試 Happy Path，就能輕鬆踩過大部分的基礎語句)。 |
| **分支/決策涵蓋率 (Branch/Decision Coverage)** | 針對程式碼裡那些掌握生殺大權的布林判斷式 (if/else 岔路口)，是否每一條路口的 `True` (左轉) 與 `False` (右轉) 兩條命運的選擇，都有被測試腳本完整給走過一遍？ | 水準之作。這確保了系統中那些至關重要的邏輯分叉路徑，都已經被嚴格驗證過不會不小心把人推下懸崖。 |
| **條件涵蓋率 (Condition Coverage)** | 針對那些被綁在一起的複雜布林判斷式裡的「每一個微小的子條件 (sub-conditions)」，它是不是都有被獨立拿出來，強迫翻轉測試過 `True` 與 `False` 的極端情況？ | 大師級的最強防禦測試。這對於檢驗那些錯綜複雜、牽一髮動全身的權限控制邏輯至關重要 (`if (身分 == Admin AND 檔案權限 == Public)`：這句裡面的每一個條件都必須被嚴刑拷打過)。 |
| **函式/方法涵蓋率 (Function/Method Coverage)** | 在這場測試大秀中，到底有百分之幾的函式/模組有被實際呼叫/點名上台過？ | 這只是一個最基礎的出發點，讓你知道還有哪些冷宮裡的模組是完全處於「測試真空」狀態、一次都沒被寵幸過的。 |

> **考試提示 (Exam Tip)**：在架構稍微複雜一點的現代軟體中，想要追求 100% 的絕對路徑涵蓋率 (path coverage) 根本是痴人說夢，因為那代表著指數型爆炸、永無止境的排列組合。測試工程師必須學會妥協，把珍貴的涵蓋率火力集中在「風險最高 (risk-based)」的核心組件上。

## 考試重點

1. **SAST 與 DAST 的致命傷宿命對決**：請死背：SAST 的痛點是「誤報率極高」；DAST 的痛點是「它絕對沒辦法告訴你到底是哪一行寫錯了」。
2. **滲透測試永遠只配當「確認 (Validation)」的壓軸**：滲透測試的價值在於找出隱密複雜的「商業邏輯缺陷」以及巧奪天工的「組合技攻擊 exploiting」；如果你花大錢找人來滲透測試，結果對方的報告裡指出有滿坑滿谷最低級的 SQL 注入漏洞，那代表你早該在前面階段先推出去擋子彈的掃描工具 (SAST/DAST) 根本是個廢物。
3. **模糊測試 (Fuzzing) 的狂熱信仰**：唯一真理：把它搞崩潰 (Inducing crashes)。它就是那個用來挖出 C 語言最痛恨的「緩衝區溢位 (buffer overflows)」與「未處理例外狀況」的神級電鑽。
4. **測試涵蓋率的鄙視鏈 (Code Coverage)**：**分支/決策涵蓋率 (Branch/Decision coverage)** 所能帶來的資安信心水準，絕對遠遠把最基礎可悲的敘述句涵蓋率 (Statement coverage) 甩在幾條街之外。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **SAST** | 靜態應用程式安全測試 (具備白箱視野，適合佈署於 SDLC 早期階段糾錯) |
| **DAST** | 動態應用程式安全測試 (具備黑箱視野，適合佈署於 SDLC 後期階段猛攻活體App) |
| **Penetration Testing (滲透測試)** | 模擬真實世界中陰險的駭客攻擊手法，用來期末考驗證整體防護體系是否真能扛揍的終極測試 |
| **Fuzzing (模糊測試)** | 狂暴地灌入海量無效、畸形或純亂碼的資料，企圖把系統逼向崩潰死機邊緣的極端測試法 |
| **Statement Coverage (敘述句涵蓋率)** | 一個用來衡量測試徹底程度的指標，計算有多少比例的程式碼語句在測試期間有被路過踩到 |
| **Branch Coverage (分支/決策涵蓋率)** | 一個進階的指標，計算程式中所有宛如岔路口般的邏輯判斷路徑，是否都有被徹底走訪過 |
