# 2.2 識別與採用安全標準 (Identify and Adopt Security Standards)

## 學習目標

- 識別與安全軟體開發相關的各大主要安全框架與標準
- 區分規範性框架 (prescriptive frameworks) 與成熟度模型 (maturity models) 的差異
- 解釋如何跨組織推廣資安意識 (security awareness)
- 描述標準在建立可衡量、可重複的安全實務過程中所扮演的角色

---

## 安全標準的角色

安全標準提供了**明確定義的活動水準**，可供衡量並監控其合規性。它們在組織內部以及整個業界建立了一套共通語言和期望基準。採用公認的安全標準能帶來若干好處：

- **一致性 (Consistency)**：確保跨團隊與各專案間的流程是可重複的
- **可衡量性 (Measurability)**：定義用以評估合規性的明確標準
- **可信度 (Credibility)**：獲得第三方的認可以及客戶的信賴
- **法律保護 (Legal protection)**：能展示有盡到應有之注意義務 (due diligence) 與應有之作為 (due care)
- **效率 (Efficiency)**：利用經過驗證的實務經驗，而非一切從頭閉門造車

## 安全框架與標準

### ISO 標準

| 標準 | 全名 | 重點 |
|----------|-----------|-------|
| **ISO 27001** | 資訊安全管理系統 (Information Security Management Systems) | 建立、實施、維護及持續改善 ISMS 的各項要求 |
| **ISO 27002** | 資訊安全控制實務準則 (Code of Practice for Information Security Controls) | 選擇與實施安全控制項的指引 |
| **ISO/IEC 15408** | 通用準則 (Common Criteria, CC) | 評估 IT 產品安全性的框架 |
| **ISO/IEC 9126** | 軟體產品品質 (Software Product Quality) | 六項品質特徵：功能性、可靠性、易用性、效率、可維護性、可移植性 |
| **ISO/IEC 12207** | 軟體生命週期流程 (Software Life Cycle Processes) | 定義涵蓋整個軟體生命週期的各項流程 |
| **ISO/IEC 33001:2015** | 流程評估 (Process Assessment) | 流程能力判定 (層級 0–5) |

#### 通用準則 (Common Criteria, ISO/IEC 15408)

通用準則框架允許廠商對其產品提出安全聲明 (security claims)，並讓這些聲明接受第三方的獨立評估：

| 概念 | 說明 |
|---------|-------------|
| **TOE** (Target of Evaluation，評估標的) | 正在被評估的產品或系統 |
| **ST** (Security Target，安全目標) | 與特定 TOE 相關聯的安全屬性 |
| **PP** (Protection Profile，保護特徵描述) | 針對一類特定產品所制定的安全需求 |
| **EAL 1–7** (Evaluation Assurance Levels，評估保證等級) | 1 = 最低，7 = 最高 |

> **考試提示**：EAL 等級表示**評估的深度 (depth of evaluation)**，而不是安全的強度。EAL 4 是在經濟效益上，能夠回溯改造並融入現有產品的最高等級。

### NIST 框架

| 框架 | 重點 |
|-----------|-------|
| **NIST SP 800 系列** | 綜合性安全指引（密碼學協定、風險管理、系統安全） |
| **NIST RMF** (Risk Management Framework) | 六個步驟流程：分類 (Categorize) → 選擇 (Select) → 實作 (Implement) → 評估 (Assess) → 授權 (Authorize) → 監控 (Monitor) |
| **NIST SSDF** (SP 800-218) | 安全軟體開發框架 (包含 PO, PS, PW, RV 四大群組) |
| **FIPS** (Federal Info Processing Standards) | 針對美國聯邦機構與特定承包商的強制性要求 |
| **FedRAMP** | 針對聯邦機構使用之雲端產品所制定的標準化安全評估 |
| **SCAP** | Security Content Automation Protocol (安全內容自動化協定) — 標準化的安全性衡量方法 |

### OWASP (Open Web Application Security Project，開放式網站應用程式安全計畫)

OWASP 是一個開放社群，提供可免費取得的各種安全資源：

| 資源 | 說明 |
|----------|-------------|
| **OWASP Top 10** | 網站應用程式最致命的十大安全風險 |
| **OWASP ASVS** | 應用程式安全驗證標準 (Application Security Verification Standard) — 劃分為三個驗證等級 |
| **OWASP SAMM** | 軟體保證成熟度模型 (Software Assurance Maturity Model) — 用於衡量與改善安全開發實務 |
| **OWASP Testing Guide** | 綜合性的網站應用程式安全測試方法論 |
| **OWASP Secure Coding Practices** | 安全編碼快速參考指南 |

### 軟體安全成熟度模型 (Software Security Maturity Models)

成熟度模型幫助組織**評估與改善**其軟體安全態勢並隨著時間進步。不同於告訴您「要做什麼」的規範性框架，成熟度模型幫助您了解「您現在在哪裡」以及「您應該往哪裡走」。

#### BSIMM (Building Security In Maturity Model，內建安全性成熟度模型)

BSIMM 是一個**描述性模型 (descriptive model)**，其建立基礎是觀察許多組織實際執行的軟體安全倡議。它衡量的是您「正在做什麼」，而不是規定您「應該做什麼」。

| 特徵 | 說明 |
|---------------|-------------|
| **方法 (Approach)** | 數據導向、觀察性的 — 基於真實世界的實務做法 |
| **領域 (Domains)** | 治理 (Governance)、情報 (Intelligence)、SDLC 觸及點 (SSDL Touchpoints)、部署 (Deployment) |
| **用途 (Use)** | 將您的軟體安全倡議 (SSI) 與業界同行進行標竿比較基準 (Benchmark) |
| **活動 (Activities)** | 約 120 項活動，歸類劃分為 12 項實務 |

#### SAMM (Software Assurance Maturity Model，軟體保證成熟度模型)

SAMM 是由 OWASP 所維護的**規範性模型 (prescriptive model)**，它提供了一種結構化的方法來改善軟體安全性。

| 特徵 | 說明 |
|---------------|-------------|
| **方法 (Approach)** | 規範性的 — 告訴您應當採用哪些實務做法 |
| **業務功能 (Business Functions)** | 治理、設計、實作、驗證、維運 |
| **成熟度層級 (Maturity Levels)** | 每個業務功能內的各項實務皆劃分四個層級 (0–3) |
| **用途 (Use)** | 建立用以改善安全態勢的發展藍圖 (roadmap) |

#### BSIMM 與 SAMM 的比較

| 層面 | BSIMM | SAMM |
|--------|-------|------|
| **類型** | 描述性的（別人都在做什麼） | 規範性的（你應該做什麼） |
| **基礎** | 來自組織的實證經驗數據 | 由專家定義的最佳實務 |
| **主要用途** | 標竿比較 (Benchmarking) | 建立策略藍圖 (Roadmap creation) |
| **維護單位** | Synopsys (商業機構) | OWASP (開源社群) |

> **考試提示**：BSIMM **描述 (describes)** 當前的業界實務做法；SAMM **規定 (prescribes)** 您應該做些什麼。兩者皆具備其價值 — BSIMM 適用於做標竿比較，而 SAMM 則適用於規劃改善計畫。

### SAFECode (Software Assurance Forum for Excellence in Code)

SAFECode 是一個由**業界支援的組織**，致力於加強軟體保證的溝通交流：
- 發布免費的最佳實務指引
- 適用於**各種規模**的組織（不局限於大型企業）
- 專注於務實、可執行的安全開發實務
- 內容包含有關開發與發布使用案例的指導原則

## 推廣資安意識 (Promoting Security Awareness)

安全標準的採用只有在所有參與者都理解並擁抱它們時才會有效。資安意識培訓計畫能確保所有利害關係人 — 從開發人員到高階主管 — 都能明白自己的職責。

### 資安意識的組成要件

| 要件 | 目標受眾 | 內容 |
|-----------|----------------|---------|
| **高階主管簡報** | C-Level 主管、董事會 | 業務風險、法規要求、投資合理性 (investment justification) |
| **開發人員培訓** | 工程團隊 | 安全編碼實務、常見漏洞、工具使用方式 |
| **安全擁護者計畫** | 被選定的團隊成員 | 進階安全培訓、導師指導機制 (mentoring)、內部倡導推廣 |
| **入職培訓** | 新進員工 | 安全政策、可接受使用政策 (acceptable use)、事件通報流程 |
| **持續性意識宣導** | 全體員工 | 網路釣魚防範意識、社交工程防範、密碼衛生 (password hygiene) |
| **依角色區分培訓** | 專業角色 | DBA安全、網管安全、架構師安全審查 |

### 安全文化 (Security Culture)

建立安全文化需要：
- **高層定調 (Tone from the top)**：高階主管的支持與明確的承諾
- **與績效整合**：將安全目標納入職務說明書與績效考核中
- **正向肯定**：認可並獎勵具備資安意識的行為
- **無指責的檢討會議 (Blameless post-mortems)**：鼓勵通報事件，而非處處擔憂受罰
- **持續學習**：定期更新培訓內容以應對不斷演進的威脅

## 考試重點

1. **ISO 15408 通用準則**：了解 TOE、ST、PP 以及 EAL 等級 (1–7)。
2. **BSIMM 與 SAMM 的比較**：描述性（標竿比較基準）與規範性（策略藍圖）的區分。
3. **NIST RMF 六大步驟**：分類 → 選擇 → 實作 → 評估 → 授權 → 監控。
4. **FedRAMP**：專為美國聯邦政府使用所制定的標準化雲端安全評估。
5. **OWASP 資源**：Top 10（風險指標）、ASVS（驗證標準）、SAMM（成熟度模型）。
6. **SAFECode**：由業界支援，適用於各種組織規模。
7. **資安意識**：必須包含高階主管的支持、開發人員的培訓，並融入組織文化中。
8. **ISO 9126**：軟體品質的六項特徵（功能性、可靠性、易用性、效率、可維護性、可移植性）。
