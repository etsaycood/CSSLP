# 3.1 定義軟體安全需求 (Define Software Security Requirements)

## 學習目標

- 區分功能性 (functional) 與非功能性 (non-functional) 安全需求
- 解釋主體-客體-活動矩陣 (Subject-Object-Activity Matrix)
- 開發整合了安全考量的使用案例 (use cases)
- 識別時序 (timing) 與競態條件 (race condition) 相關的需求
- 描述工作階段管理 (session management)、錯誤處理 (error handling) 與組態設定的需求

---

## 功能性安全需求 (Functional Security Requirements)

功能性需求描述的是**軟體必須做什麼 (what the software must do)** — 它們定義了具體的功能、行為與能力。從安全的角度來看，功能性需求定義了系統必須展現的安全行為。

### 從業務需求到功能性需求

業務需求必須被**轉換為功能性安全需求**。每一個業務目標都伴隨著必須被明確定義的安全隱涵要求。

| 業務需求 | 功能性安全需求 |
|---------------------|-------------------------------|
| 「客戶可以查看其帳戶資訊」 | 在顯示帳戶資料前，必須對使用者身分進行認證 (Authenticate) |
| 「員工可以處理訂單」 | 根據角色對使用者進行授權 (Authorize)；記錄所有對訂單的修改行為 |
| 「系統的可用性必須達到 99.9%」 | 實作備援機制、容錯移轉 (failover) 與健康狀態監控 |

### 主體-客體-活動矩陣 (Subject-Object-Activity Matrix)

主體-客體-活動矩陣建立起**使用者與資源之間的關係**：

| 概念 | 說明 |
|---------|-------------|
| **主體 (Subject)** | 系統中參與操作的使用者或處理程序 (processes) |
| **客體 (Object)** | 主體互動存取之目標資源 |
| **活動 (Activity)** | 針對每一組主客體配對，被允許或被拒絕的動作 |

**矩陣範例：**

| 主體 | 客體 | 允許的活動 | 拒絕的活動 |
|---------|--------|---------------------|-------------------|
| 客戶 | 自己的帳戶資料 | 讀取 (Read), 更新 (Update) | 刪除 (Delete) |
| 客戶 | 其他人的帳戶 | — | 讀取, 更新, 刪除 |
| 管理員 | 所有帳戶 | 讀取, 更新, 刪除 | — |
| API 服務 | 交易日誌 (Transaction log) | 寫入 (Write) | 刪除 |

> **考試提示**：主體-客體-活動矩陣提供了一種**簡潔的溝通方式**，用以表達系統允許的互動行為。這是一項用來定義授權需求 (authorization requirements) 的工具。

### 使用案例 (Use Cases)

使用案例描述的是系統**預期行為的具體範例**。

| 特徵 | 說明 |
|---------------|-------------|
| **目的** | 定義系統的預期行為（功能性需求） |
| **範圍** | **不**適用於「所有」的主客體互動 — 用於處理複雜或有模糊地帶的情境 |
| **格式** | 通常以**圖形化格式**呈現（使用案例圖/use-case diagrams） |
| **效益** | 減少業務需求中表達不清與不完整的模糊地帶 |

**使用案例建模 (Use-case modeling)** 展示了參與者/行為者 (actors, 即使用者) 對系統預期的行為 (活動)，並且明確地規範出系統會做什麼以及不做什麼。

### 時序與競態條件 (Timing and Race Conditions)

軟體必須處理與時序相關的安全漏洞：

| 漏洞 | 說明 |
|--------------|-------------|
| **TOC/TOU** (Time of Check/Time of Use - 檢查時與使用時) | 攻擊者在資源「被檢查」與「被使用」的時間差之間對資源進行了竄改 |
| **競態條件 (Race Condition)** | 併發執行的執行緒 (concurrent threads) 在競爭空窗期內，為了修改同一個客體而發生衝突與相互競爭的狀態 |
| **無窮迴圈 (Infinite Loop)** | 永遠不會終止的處理程序，持續消耗運算資源 |
| **互斥鎖 (Mutual Exclusion)** | 一種防止多個主體同時存取共享資源的機制 |

## 非功能性安全需求 (Non-Functional Security Requirements)

非功能性需求定義的是**系統執行的效能與品質 (how well the system performs)**，而不是它做什麼。它們具體規範了系統的品質屬性，包含了安全性、效能、易用性與可靠性。

### 核心安全需求（非功能性）

| 需求 | 說明 |
|-------------|-------------|
| **機密性 (Confidentiality)** | 資料加密、存取控制、隱寫術 (steganography)、遮罩 (masking) |
| **完整性 (Integrity)** | 輸入驗證、雜湊 (hashing)、校驗和 (checksums)、CRC、同位元檢查、數位簽章 |
| **可用性 (Availability)** | BIA (營運衝擊分析)、MTD (最大可容忍停機時間)、RTO (復原時間目標)、備援、容錯移轉、DRP (災難復原計畫) |
| **認證 (Authentication)** | MFA (多因素認證)、摘要認證 (digest authentication)、憑證驗證 |
| **授權 (Authorization)** | DAC (自主式)、MAC (強制式)、RBAC (角色基礎)、ABAC (屬性基礎)、委派模型 |
| **課責性/當責 (Accountability)** | 稽核軌跡 (Audit trails)、日誌記錄、SIEM (安全性資訊與事件管理) 整合 |

#### 完整性驗證方法

| 方法 | 強度 | 適用情境 |
|--------|----------|----------|
| **同位元檢查 (Parity bit check)** | 弱 | 資料傳輸的基本錯誤偵測 |
| **CRC** (循環冗餘校驗) | 中等 | 網路通訊協定中的資料完整性 |
| **校驗和 (Checksum)** | 中等 | 檔案完整性驗證 |
| **密碼學雜湊 (Cryptographic hash)** | 強 | 最強形式的資料完整性保證；若結合加密技術，還能兼具機密性 |

### 一般需求

#### 工作階段管理 (Session Management)

- 在無狀態的通訊協定 (stateless protocols) 中維持**狀態 (state)**
- 實踐**完全中介 (complete mediation)** 與**心理可接受度 (psychological acceptability)** 原則
- 使用**工作階段識別碼 (Session IDs)**，且這些 ID 必須是不可預測且受保護的
- 必須確保建立起來的工作階段不容易遭到劫持 (hijacking)、固定 (fixation) 或重播 (replay) 攻擊

#### 錯誤與例外權管理 (Error and Exception Management)

- 錯誤與例外情況往往是**資訊外洩 (information disclosure) 的潛在來源**
- 錯誤訊息絕對不能揭露系統內部細節、堆疊追蹤 (stack traces) 或組態配置資訊
- 必須捕抓 (catch) 所有的例外情況並優雅地處理它們 — 絕對不要拋出原始的錯誤輸出
- 安全故障 (Secure failure)：當錯誤發生時，系統必須預設退回到**安全狀態 (secure state)**

#### 組態參數 (Configuration Parameters)

- 識別並掌握會影響安全性的組態設定
- 確保對組態資料提供適當等級的保護
- 必須定義並強制執行基準線安全配置 (baseline security configurations)

### 維運需求 (Operational Requirements)

| 需求 | 說明 |
|-------------|-------------|
| **防盜版 (Anti-piracy)** | 程式碼混淆 (Code obfuscation)、程式碼簽章、防竄改、授權機制、IP (智慧財產) 保護 |
| **封存歸檔 (Archiving)** | 組織的保留政策 — 存放位置、保存期限與格式 |
| **部署環境 (Deployment environment)** | 設計即安全 (Secure by design)、預設安全 (Secure by default)、部署也安全 (Secure in deployment) |

> **核心原則**：預設配置必須維持安全性。如果一個使用者接受了系統所有的預設值，那麼該系統就應該處於**安全狀態 (secure state)**。

### 國際化需求 (International Requirements)

| 考量事項 | 說明 |
|--------------|-------------|
| **法律與技術面** | 確保符合當地的法規要求 |
| **字元編碼 (Character encoding)** | 支援不同的字元集與顯示方向 |
| **正規化 (Canonicalization)** | 將資料轉換為標準形式，以防止基於編碼方式繞過機制的攻擊 |

### 採購需求 (Procurement Requirements)

- **合約與 SLA** 為軟體採購提供了法律保護
- **軟體託管 (Software escrow)** 確保了若供應商倒閉或無法提供服務時，企業仍可取得原始碼
- 第三方元件在整合前必須經過安全評估

### 保護需求誘發/蒐集 (PNE, Protection Needs Elicitation)

這是向利害關係人**蒐集與引導出安全需求**的過程：

| 技術 | 說明 |
|-----------|-------------|
| **調查/問卷 (Surveys/Questionnaires)** | 收集功能性與保證 (assurance) 需求 |
| **訪談 (Interviews)** | 與利害關係人就安全需求進行詳細討論 |
| **工作坊 (Workshops)** | 協同合作式的會議，用以識別並排定安全需求的優先順序 |
| **文件分析 (Document analysis)** | 審閱現有的政策、標準以及法規遵循要求 |

PNE 需要與利害關係人進行**深度的溝通與協作**，且能應對解決以下風險：
- 商業風險 (Business risk)
- 專案風險 (Project risk)
- 技術風險 (Technology risk)

## 考試重點

1. **功能性 vs. 非功能性**：功能性 = 系統做什麼 (what)；非功能性 = 系統做得多好/品質 (how well)。
2. **主體-客體-活動矩陣**：定義了使用者與資源之間允許/拒絕的操作行為。
3. **TOC/TOU**：Time of Check / Time of Use 競態條件下的資安漏洞。
4. **工作階段管理**：Session IDs 必須具備不可預測性並受到保護。
5. **錯誤訊息**：絕對不能揭露系統內部的運作細節。
6. **預設安全 (Secure by default)**：系統的預設配置必須是安全的。
7. **正規化 (Canonicalization)**：防止攻擊者透過不同的編碼方式來繞過防護機制。
8. **PNE (保護需求誘發)**：為蒐集安全需求所採取的一種結構化方法。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Functional Requirement (功能性需求)** | 定義軟體必須做什麼（具體的行為與能力） |
| **Non-Functional Requirement (非功能性需求)** | 定義系統的品質屬性（安全性、效能、易用性等） |
| **Subject (主體)** | 提出存取請求的實體（使用者、處理程序、服務） |
| **Object (客體)** | 被存取的資源標的（檔案、資料庫、API 等） |
| **Use Case (使用案例)** | 系統預期行為的具體範例描述 |
| **TOC/TOU** | Time of Check / Time of Use — 檢查時與使用時的競態條件漏洞 |
| **Race Condition (競態條件/競爭狀態)** | 併發執行的執行緒為了修改同一個資源而相互競爭的情境 |
| **Mutual Exclusion (互斥鎖)** | 防止同時存取共享資源的防護機制 |
| **PNE** | Protection Needs Elicitation (保護需求誘發/引導) |
| **Canonicalization (正規化)** | 將資料轉換為標準化、唯一表示形式的過程 |
| **BIA** | Business Impact Analysis (營運衝擊分析) |
| **CRC** | Cyclic Redundancy Check (循環冗餘校驗) |
| **CONOPS** | Concept of Operations (營運概念) |
