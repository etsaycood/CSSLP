# 3.4 識別隱私權需求 (Identify Privacy Requirements)

## 學習目標

- 定義資料收集範圍與資料最小化 (data minimization) 原則
- 區分匿名化 (anonymization)、去識別化/假名化 (pseudo-anonymization) 與權杖化 (tokenization)
- 描述主要隱私權法規賦予使用者的權利
- 解釋資料保留 (data retention) 要求與最佳實務
- 識別跨境資料傳輸 (cross-border data transfer) 的相關規範與要求

---

## 資料收集範圍 (Data Collection Scope)

隱私保護始於**限制收集哪些資料**。**資料最小化 (data minimization)** 原則指出，組織應當僅收集用以達成特定且明確定義之目的時，所絕對必要的資料。

| 原則 | 說明 |
|-----------|-------------|
| **目的限制 (Purpose limitation)** | 資料收集只能用於具體、明確且合法的目的 |
| **資料最小化 (Data minimization)** | 僅收集必要之資訊 — 這是保護隱私最有效、最強大的技術之一 |
| **儲存限制 (Storage limitation)** | 只有在為了達成特定目的所必要之期間內，才將資料保留下來 |
| **準確性 (Accuracy)** | 確保資料的正確無誤且維持在最新狀態 |
| **合法性 (Lawfulness)** | 確保處理個人資料時具有合法依據 |

## 資料匿名化與隱私強化技術 (PET, Privacy-Enhancing Technologies)

### 匿名化 (Anonymization) vs. 去識別化/假名化 (Pseudo-Anonymization)

| 技術 | 可逆嗎？ (Reversible) | 說明 |
|-----------|:-----------:|-------------|
| **匿名化 (Anonymization)** | **否 (No)** | 永久性地移除識別資訊；資料**絕對無法**被重新連結拼湊回特定個人 |
| **去識別化/假名化 (Pseudo-Anonymization)** | **是 (Yes)** | 用假名 (pseudonyms) 來替換識別資訊；只要有額外的資訊配合，資料就**可以**被還原連回特定個人 |

> **考試提示**：在多數的隱私權法規中，已經過匿名化處理的資料就**不再被視為個人資料 (personal data)** 了。而經過假名化/去識別化處理的資料仍然被認定是個人資料，因為其過程是可逆轉 (reversed) 的。

### 匿名化技術 (Anonymization Techniques)

| 技術 | 說明 | 範例 |
|-----------|-------------|---------|
| **替換/代換 (Replacement / Substitution)** | 將可識別身分的資訊替換為無識別意義的值 | 把人名替換成「Person A (某甲)」 |
| **抑制/省略 (Suppression / Omission)** | 將會暴露身分的資訊完全剔除/抹除 | 從資料集中移除「社會安全碼 (SSN)」整個欄位 |
| **泛化 (Generalization)** | 將高度具象的資訊替換為較寬泛/概括性的形式 | 把確切的出生年月日，替換成只顯示「出生年份」 |
| **擾動/隨機化 (Perturbation / Randomization)** | 在資料數值中加入隨機的變更差異 | 在實際年齡數值中加入隨機的干擾雜訊 (加減些微數值) |
| **攪亂 (Scrambling)** | 刪除或是混淆機密資料 | 把某個欄位內的字元順序全部打亂重排 |

### 其他隱私強化技術 (Other Privacy-Enhancing Technologies)

| 技術 | 說明 | 可逆嗎？ |
|------------|-------------|:-----------:|
| **資料遮罩 (Data Masking)** | 置換掉變更過的值以建立一個鏡像版本 (例如：`****1234`) | 是 (這是「去識別化/假名化」的一種形式) |
| **權杖化 (Tokenization)** | 將敏感資料替換為一組隨機生成的權杖 (tokens)，這組權杖未來可被追溯回原始值 | 是 |
| **資料最小化 (Data Minimization)** | 根本打從一開始就不去收集或儲存那些非必要的資料 | 不適用 (N/A) |

> **關鍵區分**：**權杖化 (Tokenization)** 分配的是一個隨機值，這個過程是**可逆轉的 (can be reversed)** 以便隨時取回原始資料。而諸如抑制 (suppression) 與泛化 (generalization) 這類的**匿名化 (Anonymization)** 技術則是**不可逆轉的 (irreversible)**。

## 使用者權利 (User Rights)

主要的隱私權法規賦予了個人對其個人資料擁抱特定權利：

### 核心使用者權利（以 GDPR 框架為例）

| 權利 | 說明 |
|-------|-------------|
| **知情權 (Right to Information)** | 獲知其個人資料被如何收集與使用的權利 |
| **存取權 (Right to Access)** | 取得正在被處理之其個人資料副本的權利 |
| **更正權 (Right to Rectification)** | 要求更正不準確或不完整的個人資料的權利 |
| **被遺忘權/刪除權 (Right to Erasure / "Right to be Forgotten")** | 當個資不再被需要時，要求將其刪除的權利 |
| **限制處理權 (Right to Restrict Processing)** | 要求限制其個人資料被處理之方式的權利 |
| **拒絕權 (Right to Object)** | 基於合法利益或直接行銷目的，拒絕其資料被處理的權利 |
| **資料可攜權 (Right to Data Portability)** | 以結構化、機器可讀的格式，接收其個人資料的權利 |

### 使用者同意 (User Consent)

| 同意類型 | 說明 |
|-------------|-------------|
| **知情/積極同意 (Informed/Affirmative consent)** | 使用者主動明白地同意其資料被處理 (選擇加入，opt-in) |
| **明確同意 (Explicit consent)** | 若要處理特殊類別資料（如：醫療健康、生物特徵等），必須取得此種同意 |
| **家長同意 (Parental consent)** | 收集兒童資料時所必須（COPPA 規定為 13 歲以下；GDPR 則視各成員國法規而異） |
| **撤回同意 (Withdrawal)** | 資料主體可以**隨時撤回其先前的同意** |

### 外洩通報 (Breach Notification)

當發生資料外洩事件時，組織必須通知受影響的個人與主管機關：

| 法規規範 | 通報時限 |
|-----------|----------------------|
| **GDPR** | 在 72 小時內通知監管機關；如有高風險需「毫不延遲地 (without undue delay)」通知資料主體 |
| **HIPAA** | 在 60 天內通知受影響之個人 |
| **美國州法** | 一般通常要求在 30–90 天內通報 |

## 資料保留 (Data Retention)

資料保留定義了資料保存的**時間長度、存放位置，以及保存哪些內容**。

| 考量要素 | 說明 |
|---------|--------------|
| **期間 (Duration)** | 資料必須保留多久？（依據法規最低要求與業務需求而定） |
| **位置 (Location)** | 資料存放於何處？（涉及司法管轄權/Jurisdictional 問題） |
| **內容 (Content)** | 具體要保留哪些特定資料元素？ |
| **存取 (Access)** | 在保留期間內，誰可以存取這些被保留下來的資料？ |
| **銷毀 (Disposal)** | 當保留期限屆滿時，資料將如何被銷毀？ |

### 常見之資料保留要求

| 法規縮寫 | 保留期限規定 |
|-----------|-----------------|
| **SOX (沙賓法案)** | 財務相關紀錄：7 年 |
| **HIPAA** | 醫療紀錄：自建立日期起算 6 年 |
| **PCI DSS** | 稽核日誌：1 年（且最近 3 個月的紀錄必須能被即時存取調閱） |
| **GDPR** | 未明定具體期限 — 原則為「只要達成聲明目的即不得再繼續保留」 |

## 跨境傳輸要求 (Cross-Border Requirements)

當資料傳輸跨越了國界時，會有額外的法規要求適用：

### 關鍵概念 (Key Concepts)

| 概念 | 說明 |
|---------|-------------|
| **資料落地/駐留限制 (Data Residency)** | 要求資料必須實體儲存在特定地理位置/國境內的要求 |
| **司法管轄權 (Jurisdiction)** | 基於資料的存放位置或是資料主體的所在地，對資料行使之法律權力 |
| **多國界限 (Multi-national boundaries)** | 橫跨多個擁有不同隱私法律之國家的資料處理行為 |

### GDPR 跨境傳輸機制

| 機制 | 說明 |
|-----------|-------------|
| **約束性企業規則 (BCRs, Binding Corporate Rules)** | 針對跨國企業集團內部進行海外資料傳輸所制定的內部政策原則 |
| **標準合約條款 (SCCs, Standard Contractual Clauses)** | 作為資料傳輸依據、經由歐盟執委會預先核准的合約範本（又稱為示範合約條款 Model Contract Clauses / MCCs） |
| **適足性認定 (Adequacy decisions)** | 歐盟執委會作出的決議，認可某個非歐盟國家能夠提供具備「適足 (adequate)」水準的資料保護 |

### 其他隱私權法規 (Other Privacy Regulations)

| 法規 | 司法管轄範圍 | 主要特色/功能 |
|-----------|-------------|-------------|
| **GDPR** | 歐盟 (EU) / 歐洲經濟區 (EEA) | 具備「域外效力 (extraterritorial reach)」的全面性資料保護機制 |
| **CCPA** (即 AB 375 法案) | 美國加州 | 給予消費者知悉權、刪除權，以及拒絕資料被販售的權利 (opt-out) |
| **LGPD** | 巴西 | 類似 GDPR 框架的當地個資法 |
| **POPIA** | 南非 | 個人資訊保護法案 |

## 考試重點

1. **資料最小化 (Data minimization)**：最強大的 PET (隱私強化技術) — 如果你不需要用到，一開始就不要收集它。
2. **匿名化 vs. 假名化/去識別化**：匿名化 (Anonymization) = 不可逆轉的；假名化 (pseudo-anonymization) = 可逆轉的。
3. **權杖化 (Tokenization)**：將敏感資料替換成隨機權杖 (tokens)，這是一種可逆轉找回原始資料的做法。
4. **被遺忘權 (Right to be forgotten)**：使用者有權利要求刪除與其有關的個人資料。
5. **GDPR 外洩通報**：必須在 72 小時內通報主管機關 (supervisory authority)。
6. **跨境傳輸機制**：包含 BCRs (約束性企業規則)、SCCs (標準合約條款)、Adequacy decisions (適足性認定)。
7. **撤回同意 (Consent withdrawal)**：資料主體擁有能夠在任何時間點撤回其同意授權的權利。
8. **資料落地/駐留 (Data residency)**：某些司法管轄區會強制要求資料不可離境，必須保留在其國界範圍內。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Data Minimization (資料最小化)** | 為了能達成定義好之目的，僅局限收集最少最低限度所需資料的原則 |
| **Anonymization (匿名化)** | 從資料中進行不可逆轉地移除識別身分資訊的過程 |
| **Pseudo-Anonymization (去識別化/假名化)** | 使用假名將資料進行身分隱蔽，但此過程是可逆轉 (可被還原) 的 |
| **Tokenization (權杖化)** | 將敏感資料替換為隨機的權杖，且此權杖是可以透過系統設計被逆轉還原回原始值的 |
| **Data Masking (資料遮罩)** | 藉由替換掉變更後的值以隱藏原始資料 (例如：顯示成 `****1234`) |
| **Right to Erasure (刪除權)** | 資料主體要求刪除其個人資料的法定權利 |
| **Data Residency (資料駐留/落地限制)** | 要求資料必須存放儲存在一個特定的地理區域/國家境內的法律限制 |
| **BCR** | Binding Corporate Rules (約束性企業規則) — 供跨國集團內部進行國際資料傳輸的合法機制 |
| **SCC** | Standard Contractual Clauses (標準合約條款) — 合法進行國際資料轉移傳輸的合約機制之一 |
| **PET** | Privacy-Enhancing Technology (隱私強化技術) |
| **GDPR** | General Data Protection Regulation (歐盟一般資料保護規則) |
| **CCPA** | California Consumer Privacy Act (加州消費者隱私法) |
