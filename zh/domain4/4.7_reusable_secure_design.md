# 4.7 評估並選擇可重複使用的安全設計 (Evaluate and Select Reusable Secure Design)

## 學習目標

- 識別各項安全設計模式 (secure design patterns) 及其應用場景
- 評估各種框架與平台所能提供的安全防護能力
- 解釋重複使用既有設計元件的效益與潛在風險
- 懂得如何將合適的設計模式應用於解決常見的安全問題

---

## 安全設計模式 (Secure Design Patterns)

設計模式代表著針對常見軟體設計問題所歸納出的**經過實戰驗證、且可重複使用的解決方案 (proven, reusable solutions)**。而安全設計模式，則是專門被用來解決各種與資訊安全相關的具體疑難雜症。

### 核心安全設計模式

| 模式名稱 | 說明 | 應用場景範例 |
|---------|-------------|------------|
| **輸入驗證閘道器 (Input Validation Gateway)** | 在系統的統一入口點進行集中式的輸入資料驗證 | Web 應用程式的前端控制器 (front controllers)、API 閘道器 (API gateways) |
| **身分驗證執行者 (Authentication Enforcer)** | 可攔截所有請求，並進行集中式身分驗證檢查的關卡 | 過濾器鏈 (Filter chains)、中介軟體 (middleware)、反向代理伺服器 (reverse proxies) |
| **授權執行者 (Authorization Enforcer)** | 集中把關所有的資源存取請求看是否具備權限 | XACML 規範中的 PDP/PEP 元件、中介軟體 |
| **安全工作階段管理 (Secure Session Management)** | 集中管理所有的工作階段(Session)之建立、驗證與銷毀 | 具備產生安全權杖(token) 功能的 Session 中介軟體 |
| **例外狀況管理器 (Exception Manager)** | 集中處理錯誤與例外拋出，並在輸出前清洗掉敏感資訊 | 自訂的錯誤處理機制 (在後台記錄詳細 log，但吐給前端只顯示籠統概括的罐頭訊息) |
| **安全日誌記錄器 (Secure Logger)** | 標準化的日誌框架，內建防護機制以抵禦日誌注入攻擊 (log injection) | 具備輸入過濾以及防竄改機制的強固版日誌框架 |
| **攔截與驗證器 (Intercepting Validator)** | 在觸摸到核心業務邏輯之前，就先預先處理並驗證資料 | Servlet 過濾器、中介軟體 |

### 滿足安全性考量的架構模式 (Architectural Patterns for Security)

| 架構模式 | 說明 | 帶來的資安效益 |
|---------|-------------|-----------------|
| **分層式架構 (Layered architecture)** | 將展示層、業務邏輯層與資料存取層各自剝離切開 | 實現縱深防禦 (Defense in depth)；即使其中一層被攻破，也不至於讓整座系統立刻全面失守 |
| **微服務 (Microservices)** | 拆分成各種體積小巧、可獨立部署的服務群 | 有效限縮災情控制爆炸半徑 (Blast radius containment)；允許各服務能制定套用各自獨立的安全政策 |
| **服務網格 (Service mesh)** | 專門負責管理服務對服務間通訊的底層基礎設施層 | 提供 mTLS (雙向 TLS 認證)、高度可觀測性 (observability) 以及流量管控能力 |
| **API 閘道器 (API gateway)** | 將所有外部 API 存取請求統整收攏到唯一的一個單一入口點 | 提供集中式的認證/授權管理、速率限制 (rate limiting) 與統一日誌記錄 |
| **事件驅動架構 (Event-driven)** | 系統元件間透過發布/訂閱事件(訊號)或訊息來進行非同步溝通 | 實現低耦合 (Loose coupling)、自帶完整的事件稽核軌跡 (audit trail) 以及非同步處理優勢 |
| **零信任架構 (Zero-trust)** | 永不信任、始終驗證 (Never trust, always verify) — 完全無視其來自哪個網段或位置 | 對於防堵駭客在內部網路橫向越權移動 (lateral movement) 具備最強韌的防禦姿態 |

## 框架與平台的選擇 (Framework and Platform Selection)

### 評估基準指標

| 評估指標 | 說明 |
|-----------|-------------|
| **內建的安全功能 (Built-in security features)** | 是否已內建好身分認證、授權管理、CSRF 防護以及 XSS 防止機制 |
| **安全過往紀錄 (Security track record)** | 觀察其過去曾爆發過的歷史漏洞數量，以及官方提供修補回應的速度 |
| **社群與技術支援 (Community and support)** | 專案是否還有在活躍維護更新、安全補丁釋出的頻率、開發者社群的規模大小 |
| **法規遵循支援 (Compliance support)** | 該平台是否提供能滿足法規稽核要求的功能（如完備的日誌功能、金鑰加密支援） |
| **組態安全性 (Configuration security)** | 該框架/平台的「出廠預設值 (secure default)」是否是安全的 |
| **可擴充性 (Extensibility)** | 是否允許開發者能輕鬆外掛或自行添加客製化的安全控制項 |
| **文件品質 (Documentation)** | 官方提供的資安相關文件與最佳實務(best practices) 指南之品質優劣 |

### 沿用/重複使用元件的效益與風險

| 比較面向 | 帶來的效益 (Benefits) | 隱含的風險 (Risks) |
|--------|----------|-------|
| **安全性背書** | 歷經廣大社群實戰測試與同儕審查過 (peer-reviewed) 的程式碼 | 如果忘了定時更新，只不過是帶著「舉世皆知」的已知漏洞在裸奔 |
| **開發時程成本** | 省下大把時間，無須重新發明安全性輪子 | 過度依賴第三方廠商或開源社群的維護與更新節奏 |
| **實作的一致性** | 確保跨專案之間的各項安全套件實作模式口徑一致 | 提升了遭遇軟體供應鏈攻擊 (Supply chain compromise) 的風險暴露面 |
| **合乎標準規範** | 這類套件通常都已經為符合各種國際標準 (如 FIPS, OWASP) 做過最佳化 | 可能會踩到開源軟體授權條款 (License) 不相容的法律地雷 |

## 常見的資安反模式 (Common Security Anti-Patterns)

反模式 (Anti-patterns) 指的是那些看似直覺合理，但實際上卻會導致嚴重後果或安全漏洞的錯誤做法：

| 反模式名稱 | 導致的問題 | 正確的處置模式 (Correct Pattern) |
|-------------|---------|----------------|
| **隱匿式安全 (Security through obscurity)** | 過度依賴將系統運作細節藏起來(不讓人知道)來作為防禦手段 | 應依賴佈署多重控制防護手段的縱深防禦 (Defense in depth) 架構 |
| **硬編碼憑證 (Hardcoded credentials)** | 將密碼或金鑰之類的機密資料寫死在原始碼中 | 使用專門的機密管理工具 (secrets management，如金鑰庫 vaults 或環境變數) |
| **扁平化網路 (Flat network)** | 整個內部網路毫無阻攔，缺乏任何內部網段隔離 | 導入網路分段隔離機制 (Network segmentation)、微切分 (micro-segmentation) |
| **全能神明物件 (God object)** | 讓包含過多特權的單一龐大元件一手包辦所有事情 | 落實最小權限原則、關注點分離 (separation of concerns) |
| **攔截過於寬泛的例外 (Catching generic exceptions)** | 使用 `catch(Exception e)` 大雜燴網羅所有錯誤，這會掩蓋掉真正的資安報警訊號 | 應該去攔截具體特定的自訂例外狀況，並個別進行妥善的調適處理 |
| **純用戶端安全檢查 (Client-side security)** | 將安全性檢查與放行權力完全寄奉在脆弱的前端/用戶端上 | 所有檢查與強制執行權力都必須移交保留給後端(伺服器端)執行，前端只負責處理 UX 體驗 |
| **盲目信任所有輸入 (Trust all input)** | 毫無實做任何輸入驗證防護機制 | 在每一次資料跨越信任邊界時，嚴格執行輸入驗證 |

## 考試重點

1. **安全設計模式 (Secure design patterns)**：理解以下幾種模式的作用 — 輸入驗證閘道器、認證/授權執行者、例外狀況管理器等。
2. **零信任架構 (Zero-trust architecture)**：核心精神是「永不信任，始終驗證」；這是針對防堵攻擊者在內網發動橫向擴散 (lateral movement) 最有效的強硬姿態。
3. **API 閘道器模式 (API gateway pattern)**：作為集中管理所有外部連線存取控制與防護政策的統一入口點。
4. **反模式 (Anti-patterns)**：隱匿式安全 (透過把東西藏起來不讓別人知道藉以自保)、寫死帳密 (hardcoded credentials)、扁平化毫無隔離防護的內部網路。
5. **重複使用元件 (Component reuse)**：必須能權衡好處（經過實戰考驗的穩定、省時間）與風險（供應鏈被埋毒、依賴別人來修補維護）之間的取捨。
6. **框架評選指標 (Framework selection)**：看其過往的安全歷史紀錄、是否內建安全防護功能，以及其預設值是否夠堅固安全。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Design Pattern (設計模式)** | 針對某一類常見的軟體設計問題，所歸納出來的經過實戰驗證、且具備高度可重複使用價值的標準解決方案範本 |
| **Anti-Pattern (反模式)** | 指在實務中經常被採用，但最終卻會導致適得其反，甚至帶來更糟糕負面效果的糟糕設計慣例 |
| **API Gateway (API 閘道器)** | 作為一個統一的集中化端點入口大門，專門負責管理、保護並派送後端眾多 API 的存取請求 |
| **Service Mesh (服務網格)** | 專門用於管理與保護微服務(microservices) 對微服務之間互相通訊連線的底層基礎設施網路控制層 |
| **Zero-Trust Architecture (零信任架構)** | 徹底捨棄「內網就是安全的」這套迷思，要求針對任何存取請求(無論來自何方)，都必須經過嚴格驗證的安全模型 |
| **Defense in Depth (縱深防禦)** | 在系統架構中佈署層層疊疊、多重且多樣化的安全防禦控制線之策略 |
| **Input Validation Gateway (輸入驗證閘道器)** | 一種設計模式，規定在系統所有的外部入口點集中設立攔截關卡，以對所有流入資料進行統整驗證 |
| **Exception Manager (例外狀況管理器)** | 一種設計模式，負責集中攔截處理所有的錯誤訊息，並在對外輸出前過濾洗淨掉任何可能洩漏系統內部實作細節的敏感資訊 |
