# 4.3 執行安全介面設計 (Performing Secure Interface Design)

## 學習目標

- 描述安全 API 與介面設計的各項原則
- 解釋輸入驗證 (input validation)、輸出編碼 (output encoding) 以及正規化 (canonicalization)
- 將契約式設計 (contract-based design) 應用於介面上
- 識別常見的介面漏洞以及相應的緩解措施

---

## 安全介面原則 (Secure Interface Principles)

介面就是**攻擊面 (attack surfaces)** — 系統的每一個 API 端點、網頁表單、檔案上傳功能，以及跨微服務間的通訊管道，全都是攻擊者潛在的入侵大門。

### 介面類型

| 類型 | 說明 | 資安關注點 |
|------|-------------|-----------------|
| **使用者介面 (UI)** | 網頁表單 (Web forms)、手機 App 畫面、桌面端 GUI | 輸入驗證、CSRF (跨站請求偽造)、點擊劫持 (clickjacking) |
| **APIs (REST/SOAP/GraphQL)** | 系統間的程式化溝通介面 | 身分驗證 (Authentication)、授權 (authorization)、注入攻擊 (injection)、速率限制 (rate limiting) |
| **檔案上傳 (File uploads)** | 接收來自使用者提供的檔案 | 惡意軟體 (Malware)、路徑穿越攻擊 (path traversal)、檔案類型驗證 |
| **服務間通訊 (Inter-service)** | 微服務 (microservices) 之間的互相呼叫 | mTLS (雙向 TLS 認證)、服務網格 (service mesh) 認證 |
| **資料庫介面** | 應用程式連接至資料庫的連線 | SQL 資料隱碼攻擊 (SQL injection)、連線安全性、憑證(連線字串)管理 |

---

## 輸入驗證 (Input Validation)

輸入驗證是防禦絕大多數注入式攻擊的**第一道防線 (first line of defense)**。所有從外部跨越信任邊界進入系統的資料，都必須經過嚴格的驗證過濾。

### 驗證策略

| 策略 | 說明 | 偏好程度 |
|----------|-------------|:----------:|
| **白名單/允許清單 (Allowlist / Whitelist)** | 僅接受且放行那些符合預先定義好的模式規則之已知安全輸入 | **最推薦/首選 (Preferred)** |
| **黑名單/拒絕清單 (Denylist / Blacklist)** | 攔截並拒絕那些包含已知惡意特徵的輸入 | 較不可靠 — 因為攻擊者總是能發明出新的繞過花樣 |
| **清洗過濾 (Sanitization)** | 將輸入的內容進行清理或轉換，剔除掉危險的部分 | 當白名單策略難以實行時的備用方案 |

### 驗證規則

| 規則 | 說明 |
|------|-------------|
| **資料類型 (Type)** | 驗證資料是否符合預期的基本型態（例如：整數、字串、日期時間） |
| **長度 (Length)** | 強制限制字串的最小與最大長度 |
| **範圍 (Range)** | 確保數值界在合理的可接受範圍內 |
| **格式/模式 (Format/Pattern)** | 比對輸入內容是否符合預期的格式（例如：正規表示式 regex、格式遮罩 format masks） |
| **編碼 (Encoding)** | 驗證其編碼方式與字元集 (character set) |
| **正規化 (Canonicalization)** | 在進行驗證之前，必須先將輸入內容還原轉化回最標準統一的基底格式 |

> **考試提示**：永遠要在**伺服器端 (server side)** 執行驗證。用戶端驗證 (Client-side validation) 只是為了給善良使用者提供即時回饋與便利性，它**完全無法提供任何實質安全性 (no security)** — 因為它隨時都可以被駭客輕鬆繞過。

### 正規化 (Canonicalization)

正規化 (Canonicalization) 的作用是在進行驗證之前，先將所有五花八門的輸入轉換為其**最簡單、最標準的表示形式 (simplest, most standard representation)**：

| 攻擊手法 | 範例 | 防禦方式 |
|--------|---------|-----------|
| **URL 編碼混淆** | 用 `%27` 來代替 `'` (單引號) | 在驗證前必須先解碼還原 |
| **Unicode 編碼混淆** | 用 `\u0027` 來代替 `'` | 將其正規化為標準格式 |
| **雙重編碼 (Double encoding)** | `%2527` (對單引號 `'` 連續編碼兩次) | 必須採取迭代迴圈式解碼，直到內容不再發生變化為止 |
| **路徑穿越 (Path traversal)** | `../../../etc/passwd` | 對路徑字串進行正規化，並確保其沒有跳出允許存取的白名單目錄範圍 |

## 輸出編碼 (Output Encoding)

當要將使用者提供的資料拋到畫面上顯示時，輸出編碼 (Output encoding) 是防止**注入攻擊 (injection attacks)** 的保命符：

| 輸出情境脈絡 | 編碼方式 | 目的/防禦對象 |
|---------|----------|---------|
| **HTML 主體 (body)** | HTML 實體編碼 (HTML entity encoding，例如 `<` 變成 `&lt;`) | 防止 XSS (跨站腳本攻擊) |
| **HTML 屬性 (attributes)** | 屬性編碼 (Attribute encoding) | 防止屬性注入攻擊 (attribute injection) |
| **JavaScript 內** | JavaScript 編碼 (JavaScript encoding) | 防止 JS 注入 |
| **URL 網址內** | URL 編碼 (又稱百分比編碼 percent encoding) | 防止 URL 參數遭惡意竄改 |
| **CSS 樣式內** | CSS 編碼 (CSS encoding) | 防止 CSS 注入 |
| **SQL 查詢內** | 參數化查詢 (Parameterized queries)（這雖非編碼，但同屬防注入手段） | 防止 SQL 資料隱碼攻擊 (SQL injection) |

## 契約式設計 (Contract-Based Design / Design by Contract)

契約式設計 (DbC) 透過以下三個核心組件，為介面定義了**嚴謹的形式化規範 (formal specifications)**：

| 組件 | 說明 | 範例 |
|-----------|-------------|---------|
| **前置條件 (Preconditions)** | 在呼叫該介面之前，**什麼條件必須為真** | 輸入參數必須是有效合法的、使用者必須先取得認證 |
| **後置條件 (Postconditions)** | 介面執行完畢後，**保證能兌現的承諾是什麼** | 資料已被成功加密、稽核日誌已經被確實寫入 |
| **不變條件 (Invariants)** | 在該介面的整個生命週期中，**某些事物會永遠保持為真** | 帳戶的存款餘額永遠不會是負數 |

### 帶來的好處與資安效益

- **責任釐清 (Clear responsibility)**：呼叫者 (Caller) 必須對滿足「前置條件」負責；被呼叫者 (Callee) 則必須為兌現「後置條件」負責。
- **快速失敗 (Fail-fast)**：能盡早偵測出違規狀況 — 在觸碰到關鍵敏感情境或操作之前就先擋下。
- **完備的文件化 (Documentation)**：提供了一份針對預期行為與保證結果的形式化規範。
- **可測試性 (Testability)**：契約中明確定義了各種具體條件，這讓編寫測試案例變得極度直接且容易。

## API 安全最佳實務

| 實務做法 | 說明 |
|----------|-------------|
| **身分認證 (Authentication)** | 所有的 API 呼叫都必須使用強認證機制 (如 OAuth 2.0, API keys, mTLS) |
| **權限授權 (Authorization)** | 每一個 API 端點都必須獨立實施授權檢查 — 絕對不能依賴前端畫面來隱藏不該按的按鈕 |
| **速率限制 (Rate limiting)** | 限制每個客戶端在特定時間內的請求次數限制，藉以防範濫用與 DoS 阻斷服務攻擊 |
| **輸入驗證 (Input validation)** | 必須對所有輸入參數、HTTP 標頭 (headers) 以及請求的本體內容 (body) 進行驗證 |
| **輸出過濾 (Output filtering)** | 永遠只回傳該客戶端有權限看到的那部分資料欄位 |
| **全線 HTTPS (HTTPS only)** | 用 TLS 加密所有的 API 流量 |
| **版本控制 (Versioning)** | 妥善替 API 制定版本號，以應對各種會破壞原本相容性的更新 (breaking changes) |
| **錯誤處理 (Error handling)** | 發生錯誤時只回傳泛用模糊的錯誤訊息 — 絕對不要把系統內部的運作細節暴露出來 |
| **CORS (跨來源資源共用)** | 正確且安全地設定跨來源資源共用 (CORS) 政策 |
| **嚴格執行內容類型 (Content type enforcement)** | 必須驗證並強制要求客戶端傳來預期中的 `Content-Type` 標頭 |

## 考試重點

1. **白名單 (Allowlisting) 遠勝於黑名單 (Denylisting)**：在進行輸入驗證時，永遠白名單優先。
2. **務必執行伺服器端驗證 (Server-side validation)**：客戶端驗證形同虛設隨時可被繞過 — 永遠要在後端 (伺服器端) 進行驗證把關。
3. **正規化 (Canonicalization)**：在開始進行驗證邏輯之前，必須先將輸入還原/正規化，藉以防止基於編碼混淆的攻擊花招。
4. **輸出編碼 (Output encoding)**：必須依據輸出的情境 (如 HTML、JS、URL、CSS) 選擇相對應的編碼方式。
5. **契約式設計 (Design by Contract)**：前置條件 (Preconditions)、後置條件 (Postconditions)、不變條件 (Invariants)。
6. **參數化查詢 (Parameterized queries)**：這是防範 SQL Injection (資料隱碼攻擊) 最具體且正確的防禦做法 (這不歸類在輸出編碼中)。
7. **API 速率限制 (Rate limiting)**：作為防禦 DoS (阻斷服務) 攻擊與惡意濫用的關鍵利器。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Input Validation (輸入驗證)** | 驗證使用者的輸入內容是否符合其預期的格式、資料型態以及範圍大小的過程 |
| **Allowlist (白名單/允許清單)** | 只針對已知且被判定為安全良好之輸入模式予以放行的驗證清單 |
| **Denylist (黑名單/拒絕清單)** | 阻攔防堵已知惡意特徵之輸入模式的驗證清單 |
| **Canonicalization (正規化)** | 在進行驗證檢查之前，將各種經過編碼與變形的輸入，還原轉換成最標準統一的基底表示法的過程 |
| **Output Encoding (輸出編碼)** | 為了確保資料在其目標環境(如瀏覽器渲染)中被安全地視為純文字而非可執行代碼，而進行的資料轉換過程 |
| **Design by Contract (契約式設計)** | 透過定義明確的前置條件、後置條件與不變條件，來作為軟體元件間互動之形式化規範的方法論 |
| **Precondition (前置條件)** | 在執行某個函數或呼叫某個介面之前，必須滿足「為真 (true)」的狀態條件 |
| **Postcondition (後置條件)** | 在函數或介面順利執行完成後，保證一定會兌現/會滿足的結果條件 |
| **Invariant (不變條件)** | 在執行操作的整個生命週期內，無論發生什麼事都會一直停留在「為真 (true)」的恆定條件 |
| **XSS** | Cross-Site Scripting (跨站腳本攻擊) — 透過將惡意腳本注入到其他終端使用者瀏覽的無害網頁中來發動的攻擊 |
| **CSRF** | Cross-Site Request Forgery (跨站請求偽造) — 騙取/利用使用者已獲認證的瀏覽器身分來執行未授權操作的攻擊 |
| **CORS** | Cross-Origin Resource Sharing (跨來源資源共用) — 用於規範瀏覽器如何處理跨來源請求的安全機制 |
