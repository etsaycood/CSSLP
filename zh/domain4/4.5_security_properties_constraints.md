# 4.5 塑造 (非功能性) 安全屬性與限制的模型 (Model Security Properties and Constraints)

## 學習目標

- 定義架構層級中的非功能性安全屬性 (non-functional security properties)
- 將機密性、完整性與可用性 (CIA) 限制條件應用於架構決策上
- 解釋身分認證 (authentication)、授權 (authorization) 以及可歸責性 (accountability) 的架構設計模式
- 描述非功能性安全需求是如何轉變成為架構設計上的限制 (constraints) 的

---

## 非功能性安全屬性 (Non-Functional Security Properties)

非功能性安全屬性定義了系統架構必須滿足的**品質特徵 (quality attributes)**。與功能性需求（系統「做什麼」）不同，這些屬性主要是用來限制或規範系統在執行其安全功能時，應該要**表現得有多好 (how well)**。

### 在架構層級落實 CIA 三要素

| 屬性 | 衍生的架構含義與配置 |
|----------|---------------------------|
| **機密性 (Confidentiality)** | 加密服務、金鑰管理基礎設施 (KMS)、存取控制的強制執行點 (enforcement points)、資料遮罩 (data masking) 服務 |
| **完整性 (Integrity)** | 雜湊運算服務 (Hashing services)、數位簽章基礎設施、驗證中介軟體 (validation middleware)、具有不可竄改性 (immutable) 的稽核日誌 |
| **可用性 (Availability)** | 負載平衡器 (Load balancers)、容錯轉移叢集 (failover clusters)、自動擴展 (auto-scaling) 機制、健康狀態監控、災難復原 (DR) 備援站點 |

### 身分認證架構 (Authentication Architecture)

| 設計模式 | 說明 | 適用情境 (Use Case) |
|---------|-------------|----------|
| **集中式認證 (Centralized)** | 建立一個讓所有應用程式共用/共享的單一認證服務 | 企業內部單一登入 (SSO)、IAM 平台 |
| **基於權杖 (Token-based)** | 使用憑證權杖 (如 JWT, OAuth) 進行**無狀態 (stateless)** 的身分驗證 | 微服務架構 (Microservices)、APIs、手機行動 App |
| **憑證式 (Certificate-based)** | 使用 X.509 憑證進行雙向 TLS 認證 (mTLS) | 服務對服務間 (Service-to-service) 的溝通、高安全性要求的環境 |
| **聯合認證 (Federated)** | 將身分驗證工作委由外部的身分提供者 (IdP) 來執行 | 跨組織合作系統、B2B (企業對企業) 系統整合 |

### 授權控制架構 (Authorization Architecture)

| 設計模式 | 說明 | 適用情境 (Use Case) |
|---------|-------------|----------|
| **集中式政策引擎 (Centralized policy engine)** | 由一個統一的政策決策點 (PDP) 來評估所有的存取請求 | 大型跨部門企業系統 |
| **內嵌式授權 (Embedded authorization)** | 將授權邏輯直接寫死/內嵌在每個個別服務之中 | 簡單的小型應用程式、單體式 (monolithic) 舊架構應用程式 |
| **API 閘道器強制執行 (API gateway enforcement)** | 在 API 閘道器 (Gateway) 層級就直接強制實施授權檢查攔截 | 微服務架構、API 優先開發 (API-first) 架構 |
| **基於屬性的存取控制 (ABAC)** | 綜合考量主體、客體以及環境屬性，來進行動態授權判定 | 需要極高細緻度控制 (Fine-grained)、且會依據當下情境上下文 (context-aware) 變動的存取控制場景 |

#### XACML 架構元件

| 元件名稱 | 扮演角色與職責 |
|-----------|------|
| **PEP** (Policy Enforcement Point, 政策執行點) | 負責在第一線**攔截 (Intercepts)** 存取請求，並負責**強制執行 (enforces)** 後端做出的最終決策 |
| **PDP** (Policy Decision Point, 政策決策點) | 依據政策規則來**評估 (Evaluates)** 該存取請求究竟該放行還是拒絕 |
| **PAP** (Policy Administration Point, 政策管理點) | 負責**建立與管理 (Creates and manages)** 所有這些授權政策規則的集中控制台 |
| **PIP** (Policy Information Point, 政策資訊點) | 負責**提供外部屬性資料 (Provides attribute data)** 給 PDP 讓它作為決策判斷的參考依據 |

### 可歸責性架構 (Accountability Architecture)

| 組成元件 | 建立目的 |
|-----------|---------|
| **集中式日誌管理 (Centralized logging)** | 將散落各地的元件日誌，統一彙整收攏到一個中央儲存庫中 |
| **SIEM 系統整合 (SIEM integration)** | 跨系統對來自四面八方的事件進行關聯分析 (Correlate events)，以揪出潛在威脅 |
| **不可竄改的稽核軌跡 (Immutable audit trails)** | 採用只能寫入一次 (Write-once)、無法被事後修改的日誌保存技術 (具備防竄改特徵 tamper-evident) |
| **日誌保護 (Log protection)** | 對日誌檔案進行加密保護，防止任何未經授權的修改破壞 |
| **保留政策 (Retention policies)** | 依據法規遵循要求，明確定義各類型日誌檔案的法定保留期限 |

## 安全限制條件 (Security Constraints)

安全限制條件代表的是**被強加於架構設計上的各種制約與框架 (restrictions imposed on the architecture)**，這些制約可能源自於最初提出的安全需求、外部法規，或是組織內部的強制性政策：

| 限制條件類型 | 具體範例 |
|----------------|---------|
| **技術層面限制 (Technology constraints)** | 強制規定必須採用經過 FIPS 140-2 標準驗證通過的密碼學模組 |
| **網路層面限制 (Network constraints)** | 規定資料庫儲存層 (Database tier) 絕對不可以直接曝露在網際網路上讓外部存取 |
| **資料層面限制 (Data constraints)** | PII (個人識別資訊) 在靜止儲存時與傳輸過程中，都強制必須維持加密狀態 |
| **效能層面限制 (Performance constraints)** | 規定身分認證過程從發起請求到回應，必須在 2 秒鐘內完成 |
| **法規層面限制 (Regulatory constraints)** | 依據 SOX (沙賓法案) 規定，所有稽核日誌都必須強制保留存放 7 年 |
| **組織內部限制 (Organizational constraints)** | 公司內所有新開發的服務，都必須強制串接並使用公司統一配發的 IAM 平台 |

## 考試重點

1. **架構層級的 CIA 實作**：必須了解 CIA (機密性、完整性、可用性) 每一個要素是如何轉化為具體的架構元件實作的。
2. **XACML 的四大元件**：PEP(執行)、PDP(決策判斷)、PAP(管理維護)、PIP(資訊提供) — 務必弄清楚它們各自扮演的角色。
3. **基於權杖的身分認證 (Token-based auth)**：它是無狀態的 (Stateless)，非常適合用於現代微服務架構與 Web API 環境。
4. **集中式日誌管理 (Centralized logging)**：這是實現可歸責性 (accountability) 與跨系統事件關聯分析 (correlation) 絕對不可或缺的核心基礎。
5. **安全限制條件 (Security constraints)**：理解這六大類別 — 技術、網路、資料、效能、法規、組織層面的限制是如何牽制影響架構設計的。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Non-Functional Requirement (非功能性需求)** | 定義系統執行表現「品質與能力」的各種品質屬性，而非系統具備什麼實質「功能」 |
| **Security Constraint (安全限制)** | 由安全需求發展而來，並強加限制在系統架構設計與選擇上的各種框架或制約條件 |
| **PEP** | Policy Enforcement Point (政策執行點) — 站在第一線負責攔截請求，並負責實際「動手」執行放行或阻擋動作的節點 |
| **PDP** | Policy Decision Point (政策決策點) — 安全架構的「大腦」，負責讀取政策並動起腦筋評估判斷該請求是否符合放行標準 |
| **PAP** | Policy Administration Point (政策管理點) — 讓管理員能夠新增、修改、維護各種存取政策規則的管理介面與後台 |
| **PIP** | Policy Information Point (政策資訊點) — 負責提供如地理位置、時間、職位等外部環境屬性資料給 PDP 作為決策參考的資料庫 |
| **XACML** | eXtensible Access Control Markup Language (可延伸存取控制標記語言) — 這是一套用來實作基於屬性存取控制 (ABAC) 的標準框架與技術 |
| **JWT** | JSON Web Token — 一種結構緊湊且對 URL 友善的權杖格式，用來在雙方之間安全可靠地傳遞聲明 (claims) |
| **Mutual TLS (mTLS, 雙向 TLS)** | 伺服器端與客戶端雙方都必須出示憑證，來證明彼此身分的雙向身分驗證機制 |
| **SIEM** | Security Information and Event Management (安全資訊與事件管理系統) — 專門收集並分析全公司資安日誌的大型關聯分析平台 |
