# 4.4 執行架構層級的風險評估 (Performing Architectural Risk Assessment)

## 學習目標

- 解釋架構層級風險評估 (architectural risk assessment) 的目的與流程
- 描述如何評估並縮小攻擊面 (attack surface)
- 應用風險排序/分級 (risk ranking) 技巧來排定架構風險的優先處理順序
- 針對高風險的架構元件，識別並選擇合適的對策 (countermeasures)

---

## 架構層級的風險評估 (Architectural Risk Assessment)

架構風險評估旨在**評估系統設計的安全態勢 (security posture of the system design)**，以便在真正進入程式碼實作之前，先一步發掘出潛在的弱點。它是建立在威脅模型 (threat model) 的基礎上，進一步在系統架構的脈絡下，評估每一個已識別威脅其發生的可能性 (likelihood) 與衝擊影響 (impact)。

---

## 攻擊面評估 (Attack Surface Evaluation)

攻擊面 (attack surface) 指的是**所有進入點與離開點的總和 (sum of all entry and exit points)** — 攻擊者可以透過這些管道與系統產生互動。縮小攻擊面，是提升系統安全性最有效的方法之一。

### 攻擊面的組成要素

| 要素類別 | 說明 | 範例 |
|-----------|-------------|---------|
| **網路攻擊面 (Network attack surface)** | 暴露在外部網路的開放連接埠 (ports)、通訊協定與服務 | HTTP (80), HTTPS (443), SSH (22), 資料庫連接埠 |
| **軟體攻擊面 (Software attack surface)** | 允許未經驗證或不被信任之使用者存取的程式碼路徑 | 登入頁面、對外公開的 API、檔案上傳端點 |
| **實體攻擊面 (Physical attack surface)** | 能直接接觸到硬體與基礎設施的實體存取點 | 伺服器機房、USB 插槽、主機控制台 (console) 存取介面 |
| **人為攻擊面 (Human attack surface)** | 那些可能會成為社交工程 (social engineering) 攻擊目標的人員 | 內部員工、IT 客服 (helpdesk)、外包商 |

### 衡量攻擊面的指標

| 指標 | 說明 |
|--------|-------------|
| **進入點 (Entry points)** | 資料能夠進入系統的管道數量 |
| **離開點 (Exit points)** | 資料能夠離開系統的管道數量 |
| **暴露的服務 (Exposed services)** | 可從外部網路直接存取呼叫的服務數量 |
| **權限等級 (Privilege levels)** | 系統內劃分了多少種不同的權限層級 |
| **跨越信任邊界的次數 (Trust boundaries crossed)** | 資料流動時發生信任邊界轉換/交接的總次數 |

### 縮小攻擊面的策略 (Attack Surface Reduction Strategies)

| 策略 | 說明 |
|----------|-------------|
| **減少進入點 (Reduce entry points)** | 停用非必要的服務系統，關閉沒有用到的連接埠 (ports) |
| **降低權限 (Reduce privilege)** | 讓服務以「滿足需求/剛好夠用」的最低權限來執行 |
| **減少程式碼 (Reduce code)** | 刪除並移除沒有用到的死碼 (dead code)、廢棄功能，以及拔除 debug 專用端點 |
| **減少資料儲存 (Reduce data)** | 不需要的資料，打從一開始就不要存 |
| **降低信任程度 (Reduce trust levels)** | 嚴格限制「是誰」以及「什麼東西」有資格存取各個元件 |
| **網路分段隔離 (Network segmentation)** | 將系統各元件隔離打散到各自獨立的網路區段中 |
| **API 閘道器 (API gateway)** | 透過單一且受到嚴格管控的端點，來收攏/集中管理所有的外部存取請求 |

## 風險排序與等級劃分 (Risk Ranking)

風險分級旨在根據風險發生的「可能性 (likelihood)」與「衝擊影響 (impact)」，為已識別出的架構風險分配**相對的優先處理順序 (relative priority)**。

### 風險矩陣 (Risk Matrix)

| 發生的可能性 ↓ \ 造成的衝擊 → | **低衝擊 (Low Impact)** | **中等衝擊 (Medium Impact)** | **高衝擊 (High Impact)** |
|--|:-:|:-:|:-:|
| **高度可能 (High Likelihood)** | 中等 (Medium) | 高 (High) | 極危險 (Critical) |
| **中度可能 (Medium Likelihood)** | 低 (Low) | 中等 (Medium) | 高 (High) |
| **低度可能 (Low Likelihood)** | 低 (Low) | 低 (Low) | 中等 (Medium) |

### 影響風險評分等級的因素

| 因素 | 說明 |
|--------|-------------|
| **威脅者的能力 (Threat capability)** | 潛在發動攻擊之威脅來源所具備的技術水準與可用資源 |
| **漏洞的可利用性 (Vulnerability exploitability)** | 該漏洞被實際利用來發動攻擊的難易程度 |
| **業務衝擊 (Business impact)** | 在財務面、聲譽面、法律層面，以及日常營運上所造成的影響 |
| **暴露程度 (Exposure)** | 包含漏洞的元件有多容易被接觸或存取到 |
| **現有的安全控制項 (Existing controls)** | 目前已經實施之防護對策的實際防禦成效 |

## 選擇防護對策 (Countermeasure Selection)

針對每一項被評為「高風險」的架構弱點，都必須挑選適當的防護對策予以反制：

| 存在的架構層風險 | 推薦的防護對策 (Countermeasure) |
|-------------------|----------------|
| 攻擊面過分龐大 | 減少對外暴露的服務、導入 API 閘道器 (API gateway) |
| 違反信任邊界原則 | 在每一次跨越邊界時落實輸入驗證 (input validation) 與身分認證 |
| 存在單點故障 (Single point of failure, SPOF) 性隱患 | 建立備援機制 (Redundancy)、故障轉移 (failover) 機制、叢集部署 (clustering) |
| 隔離防護不足 | 落實容器化 (Containerization)、網路分段隔離、處理程序隔離 (process isolation) |
| 資料傳輸未經加密 | 在資料傳輸過程中 (in transit) 採用 TLS 加密，對於靜止資料 (at rest) 實施硬碟/資料庫加密 |
| 身分驗證機制薄弱 | 導入多因素認證 (MFA) 或是憑證式認證 (certificate-based authentication) |
| 賦予元件的權限過大 | 嚴格落實最小權限原則，配置僅具備最低必要權限的服務專用帳戶 (service accounts) |

---

## 考試重點

1. **攻擊面 (Attack surface)**：所有進入點與離開點的總和 — 必須盡可能減少它以提升整體安全性。
2. **四大攻擊面分類**：網路 (Network)、軟體 (software)、實體 (physical)、人為 (human)。
3. **縮小攻擊面 (Attack surface reduction)**：歸納手段包含：減少進入點、降低權限、刪減不必要的程式碼、減少資料落地儲存、降低信任層級過度依賴。
4. **風險評分排序 (Risk ranking)**：永遠是「可能性 (Likelihood) × 衝擊影響 (Impact)」這兩項變數相乘所得出的矩陣結果。
5. **選擇對策 (Countermeasure selection)**：必須能夠將特定的架構風險，精準地對應到正確的控制防護措施上。

## 關鍵術語表

| 術語 | 定義 |
|------|-----------|
| **Attack Surface (攻擊面)** | 攻擊者能夠藉由各種管道與目標系統產生接觸互動的所有切入點之總和 |
| **Attack Surface Reduction (縮小攻擊面)** | 盡最大努力減少系統對外接觸進入點以及暴露元件數量的安全實務作為 |
| **Risk Ranking (風險排序/分級)** | 根據風險發生的「可能性」與「衝擊」，來排定各項風險優先處置順序的過程 |
| **Risk Matrix (風險評估矩陣)** | 將發生機率與衝擊力道進行交叉比對，以判定最終風險等級矩陣網格圖 |
| **Countermeasure (防護對策/控制措施)** | 針對已被識別出之特定風險，所特別建置部署的安全控制手段 |
| **Network Segmentation (網路分段隔離)** | 將一個大網路切割成諸多小網段，並在區段間的通訊路徑上實施嚴格的存取控制 |
| **API Gateway (API 閘道器)** | 作為單一集中化端點入口，統一負責管理並保護後端諸多 API 存取安全的中介服務 |
